# ドメインモデルと値オブジェクトの実装が設計より不足している

## 内容

`docs/03_内部設計/01_アーキテクチャ設計/ディレクトリ構成.md` と `docs/03_内部設計/02_クラス・モジュール設計/01_バックエンド/クラス一覧.md` では、次のdomain配下にドメインモデル・値オブジェクト・方針クラスを置く設計になっている。

- `src/backend/domain/chat`
  - `Chat`
  - `UserInstruction`
  - `ChatTitlePolicy`
- `src/backend/domain/validation`
  - `ValidationResult`
  - `RetryPolicy`
- `src/backend/domain/references`
  - `PdfReference`
  - `PdfLocator`
- `src/backend/domain/artifacts`
  - `ArtifactReference`
- `src/backend/domain/shared/ids`
  - `ChatId`
  - `RunId`
  - `UserId`
  - `UserInstructionId`
  - `ReferenceId`
  - `ArtifactId`
  - `SessionId`
- `src/backend/domain/shared/time`
  - ドメイン日時値

一方、現行実装でdomain配下に実体があるのは主に `domain/answer` と `domain/execution` であり、上記ディレクトリの多くは空になっている。チャットタイトル生成、ユーザ指示の正規化、ID生成、参照元や成果物の単純値は、Repository、application DTO、補助関数、UUID/文字列で扱われている。

## 影響

設計書ではdomain層が「副作用を持たない業務判断」を保持する前提だが、現行実装では一部の業務ルールがRepositoryやapplication DTOへ分散している。

特に、ユーザ指示の正規化、チャットタイトル生成、参照元locator、成果物参照、再試行可否は、DB実装やCodex実装から独立してテストできる余地がある。反対に、全IDを値オブジェクト化するだけの実装は修正量に比べて効果が薄い可能性がある。

## 設計と実装の評価

第3案がよい。設計どおり全てのデータ保持クラスを一括で増やすより、業務ルールや不変条件を持つものからdomainへ移すのが現実的である。

優先度は、`UserInstruction`、`ChatTitlePolicy`、`RetryPolicy`、`PdfReference` / `PdfLocator`、`ArtifactReference` の順が妥当である。ID値オブジェクトは、Runtime Provider IFの整理と合わせて、port境界やRepository境界で効果が出る範囲から導入する。

単なるDTOは引き続き `application/ports/*/dto.py` に置き、domainへ寄せすぎない方がよい。
