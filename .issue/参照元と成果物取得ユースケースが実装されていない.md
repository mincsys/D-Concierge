# 参照元と成果物取得ユースケースが実装されていない

## 内容

`docs/03_内部設計/02_クラス・モジュール設計/01_バックエンド/src/backend/application/references/GetReferenceDataUseCaseクラス設計.md`、`docs/03_内部設計/04_処理設計/参照元PDF取得処理設計.md`、`docs/03_内部設計/04_処理設計/Codex成果物配信処理設計.md` では、参照元PDF取得とCodex成果物配信をapplication層のUseCaseが調停し、ファイル取得は `FileReferenceStore` / `FileArtifactStore` へ委譲する前提になっている。

一方、現行実装では `GetReferenceDataUseCase`、`GetArtifactUseCase`、`FileReferenceStore` が存在せず、`src/backend/app/factory.py` のRESTエンドポイント内でRepository取得、PathSecurityService呼び出し、FileResponse生成を直接行っている。

## 影響

presentation/app層が、DB取得、パス安全検証、ファイル配信判断を直接持っている。処理設計にあるトレースログ、エラー変換、許可範囲判定の責務境界が曖昧になり、単体テストの境界も設計とずれる。

## 設計と実装の評価

設計の方がよい。参照元と成果物取得は利用者向け配信処理であり、パス安全性・MIME・存在確認・エラー分類をapplicationのUseCaseとして閉じた方が、REST境界を薄く保てる。

対応は、`application/references` と `application/artifacts` に取得UseCaseを追加し、`application/ports/filesystem` に参照元ファイル読込portを足す。RESTエンドポイントはUseCase結果を `FileResponse` へ変換するだけにする。
