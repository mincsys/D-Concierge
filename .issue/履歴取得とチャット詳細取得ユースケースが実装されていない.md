# 履歴取得とチャット詳細取得ユースケースが実装されていない

## 内容

`docs/03_内部設計/01_アーキテクチャ設計/ディレクトリ構成.md` では、`src/backend/application/history/` を「履歴一覧、履歴再表示、タイトル生成」の配置先としている。

また、`docs/03_内部設計/02_クラス・モジュール設計/01_バックエンド/クラス一覧.md` では、次のクラスを設計対象としている。

- `src/backend/application/chat` の `GetChatDetailUseCase`
- `src/backend/application/history` の `ListChatHistoriesUseCase`

一方、現行実装では `src/backend/application/history/` は空であり、`GetChatDetailUseCase` と `ListChatHistoriesUseCase` は存在しない。履歴一覧取得とチャット詳細取得は `src/backend/app/factory.py` のRESTエンドポイントからRepositoryを直接呼び出し、同じ場所でAPI応答スキーマへ変換している。

## 影響

REST境界が参照系ユースケースの責務を直接持っているため、設計書にある「presentationはHTTP入力検証と応答変換、applicationは取得処理の調停」という分担とずれている。

履歴一覧・詳細取得は薄い処理だが、trace_id、ローカル利用者ID、対象なし、DB接続失敗、応答変換失敗などの扱いは処理設計に記載されており、REST関数へ直書きすると他のユースケースと境界が不揃いになる。

## 設計と実装の評価

設計の方がよい。履歴一覧取得とチャット詳細取得は薄いユースケースとして実装し、REST層はUseCase呼び出しとHTTP応答変換に限定した方が、処理設計・クラス一覧・ディレクトリ構成と整合する。

対応は、`ListChatHistoriesUseCase` を `application/history` に、`GetChatDetailUseCase` を `application/chat` に追加し、`app/factory.py` または移動後の `presentation/rest` からRepositoryを直接参照しない形へ寄せる。
