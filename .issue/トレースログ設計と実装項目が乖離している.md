# トレースログ設計と実装項目が乖離している

## 内容

`docs/03_内部設計/03_内部IF設計/トレースログIF.md` と `docs/03_内部設計/06_共通設計/共通設計.md` では、trace_id値の受け渡しを `shared/tracing` に置き、トレースログ項目として `user_id`、`stacktrace`、`codex_exit_status`、`runner_type`、`os_name`、`process_result`、`validation_comment` などを必要に応じて記録する設計になっている。

一方、現行実装では `TraceLogRecord` は `src/backend/application/ports/trace_log/dto.py` にあり、`src/backend/shared/tracing` の実装は存在しない。`TraceLogRecord` の項目も設計書の入力項目より少なく、`stacktrace` や `codex_exit_status` などは保持していない。

## 影響

障害調査時に設計書が想定する情報がログへ出ない可能性がある。また、trace_idの値型・マスク方針・ログDTOの所有場所が設計と実装で異なるため、今後ログ出力を増やす際の基準がぶれる。

## 設計と実装の評価

第3案がよい。`TraceLoggerPort` と `TraceLogRecord` を `application/ports/trace_log` に置く実装方針は、副作用境界として自然である。一方で、trace_id値やマスク方針を `shared/tracing` 相当に分離する設計意図も妥当である。

対応は、`TraceLoggerPort` は `application/ports/trace_log` に置いたまま、trace_idの値型・マスク補助・ログ項目の標準化を `shared/tracing` または共通モジュールに戻す。ログ項目は設計書の全項目を実装するか、MVPで記録しない項目を設計から整理する。
