# 要件定義 質問チェックリスト

## 使い方

- リポジトリ、既存文書、ユーザ指定から一意に分かる内容は質問しない。
- 既存の要件定義書がある場合は、それを主入力として扱う。
- ユーザから追加指示がある場合は、既存要件定義書への追加入力として扱う。
- 既存の要件定義書がない場合は、ユーザ要望、業務依頼、議事メモ、既存資料を入力として聞き取りを始める。
- 既存要件定義書、ユーザ指示、既存資料、README、モックが衝突する場合は、後続の設計や実装に影響する箇所ごとに質問する。
- 未確定事項は本文作成前にユーザへ提示し、合意してから要件定義書へ反映する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 各質問では、確認が必要な理由、影響する章、未確定のまま進めた場合に曖昧になる内容を説明する。
- 選択肢を提示する場合は、採用時の影響が明確に異なるものだけにする。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 選択肢に `など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- 推奨選択肢は、既存要件定義書、ユーザ指定、業務上の自然さ、後続工程の安全性を基準に選ぶ。
- 要件定義書本文を作成する直前に、全質問項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理し、未合意項目がある場合は本文作成へ進まない。
- 質問回答とリポジトリ事実の整理がそろった場合、質問表にある全質問項目と合意結果を一覧化してユーザへ表示する。
- 合意結果一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイル、設定、資料から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。

## 1. 入力資料と衝突確認

- どの既存要件定義書を主入力にするか。
- ユーザ指示、業務依頼、議事メモ、README、モック、既存仕様書のどれを追加入力として扱うか。
- 既存要件定義書と追加指示が衝突する場合、どの内容を正として要件定義へ反映するか。

## 2. 背景と目的

- 何が不便で、何を改善したいのか。
- なぜ今このシステムが必要なのか。
- 成果として何ができれば成功なのか。
- システム化の目的を、利用者価値や業務上の課題としてどう表現するか。

## 3. 利用者と利用シーン

- 誰が使うのか。
- どの端末、どの環境、どの頻度で使うのか。
- 管理者、運用担当者、外部利用者、システム連携先は存在するのか。
- 利用者区分ごとに必要な業務や権限が異なるか。

## 4. 初期リリース範囲と対象外

- 初期リリースで必須の機能は何か。
- 後回しでよい機能は何か。
- 対象外として明示すべきものは何か。
- 将来構想や後続フェーズを要件定義書内でどう分けるか。

## 5. 業務

- 利用者はどの順で操作または業務を行うのか。
- 業務の開始条件、終了条件、途中終了、例外時の扱いは何か。
- 管理業務、運用業務、承認、確認、通知、再実行が必要か。

## 6. 機能

- 保存、参照、検索、集計、登録、更新、削除、承認、通知など、必要な機能は何か。
- 機能ごとの入力、出力、成功条件、失敗時の扱いは何か。
- 初期リリースで実現する機能と、対象外または将来拡張に回す機能をどう分けるか。

## 7. 画面

- 最低限必要な画面、ダイアログ、一覧、詳細、設定画面は何か。
- 各画面で何を表示し、何を入力し、何を操作するのか。
- PC、スマートフォン、特定ブラウザ、特定端末の前提はあるか。
- モックがある場合、画面構成、導線、項目名、表示文言を要件定義へどう反映するか。

## 8. データ

- 何を保存する必要があるのか。
- 誰のデータか、どの単位で持つのか。
- 一覧、詳細、集計、検索、履歴、監査で必要なデータは何か。
- 保存期間、削除、復元、個人情報、機密情報の扱いをどう定義するか。

## 9. 外部連携

- どの外部サービスや外部システムと連携するのか。
- 入力、出力、連携タイミング、同期または非同期のどれを要求するか。
- 連携失敗、再試行、外部サービス停止、応答遅延を要件としてどう扱うか。

## 10. 非機能

- どの程度の速度、同時利用、データ量を想定するか。
- どの程度の信頼性、可用性、復旧性が必要か。
- セキュリティ、権限、認証、操作性、アクセシビリティ、監査、保存期間、通知要件はあるか。

## 11. 運用要件・制約

- 利用環境、運用体制、保守体制、バックアップ、ログ、監視、リリース、移行の制約は何か。
- 利用できる外部サービス、禁止事項、手動運用で許容する範囲は何か。
- 障害時、問い合わせ時、データ修正時の運用要求は何か。

## 12. 実装へ踏み込む判断

- 技術製品名、API パス、DB カラム名、内部モジュール分割を要件として固定する必要があるか。
- 実装方式を固定しない場合、要求レベルの表現へどう戻すか。
- 運用導線や外部公開経路のように、要求そのものへ影響する技術制約をどう記載するか。

## 13. 作成対象ファイル

- どの要件定義書を作成または更新するか。
- 将来構想、補助資料、別紙を分ける必要があるか。
- 同じ要求を複数文書へ重複記載しない前提でよいか。

## 14. 質問順の目安

1. 入力資料と衝突確認
2. 背景、目的、利用者
3. 初期リリース範囲と対象外
4. 主要業務フロー
5. 主要機能、画面、データ
6. 外部連携、非機能、運用要件・制約
7. 実装へ踏み込む判断、作成対象ファイル
