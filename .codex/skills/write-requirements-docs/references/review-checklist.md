# 要件定義書レビュー観点

## 目的と範囲

- 背景、目的、対象利用者が明確か。
- 初期リリース対象と対象外が分かれているか。
- 要件定義として必要な範囲に収まっているか。

## 章間整合

- 業務要件にある流れが機能要件と画面要件へ反映されているか。
- 画面要件に出る画面が業務要件や機能要件とつながっているか。
- データ要件が機能要件と整合しているか。
- 外部連携要件が機能要件と矛盾していないか。

## 記述の明確さ

- 要求が断定形で書かれているか。
- あいまいな表現や主語不明の記述がないか。
- 実装方針と要求事項が混ざっていないか。
- 既存要件定義書、ユーザ指示、既存資料、README、モックの衝突箇所を、ユーザ合意なしで固定していないか。

## 非機能と制約

- 利用環境や前提条件が明示されているか。

## 考慮漏れ

- 主要業務フローだけでなく、準正常系、異常系、取消、再実行、失敗時の扱いが必要な範囲で要件化されているか。
- 利用者区分、権限、操作可能範囲、参照可能範囲が漏れていないか。
- データの作成、更新、削除、参照、保持、破棄、外部連携に必要な要件が漏れていないか。
- 初期対象外、将来対象、制約事項が本文に混ざらず整理されているか。
- 非機能要件が性能、可用性、セキュリティ、運用、監査、バックアップ、障害対応の必要範囲を覆っているか。

## 完成度

- 差分説明、検討メモ、旧仕様の痕跡が残っていないか。
- 既存文書と用語、章番号、文体がそろっているか。
- 次工程の設計担当者が、追加解釈なしで外部設計へ進めるだけの材料があるか。
