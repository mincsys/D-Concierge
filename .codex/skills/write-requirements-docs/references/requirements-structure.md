# 要件定義書の標準構成

## 基本方針

- ユーザ指定のフォーマットがある場合は、その章立てとファイル名を優先する。
- 指定がない場合は、以下の章構成を基本とする。
- 要件定義書では、設計解や技術選定よりも「何が必要か」「どこまでやるか」を優先して明記する。

## 標準章立て

```text
1. 文書の目的
2. システム概要
  2.1 背景
  2.2 システム化の目的
  2.3 対象利用者
3. 開発対象範囲
  3.1 初期リリース対象
  3.2 将来拡張対象（必要時のみ）
4. 業務要件
  4.1 業務概要
  4.2 業務フロー
  4.3 管理業務
5. 機能要件
  5.x 機能カテゴリ別要件
6. 画面要件
  6.x 画面別要件
7. データ要件
8. 外部連携要件
9. 非機能要件
10. 運用要件・制約
```

## 章ごとの役割

| 章 | 役割 |
| --- | --- |
| 文書の目的 | 文書の利用目的と読者を示す |
| システム概要 | 背景、目的、利用者を整理する |
| 開発対象範囲 | 初期リリース対象と対象外を明確にする |
| 業務要件 | 利用者や管理者の業務の流れを示す |
| 機能要件 | システムが提供する機能を分類して定義する |
| 画面要件 | 画面ごとの目的、表示項目、操作を整理する |
| データ要件 | 保持すべき主要データを整理する |
| 外部連携要件 | 外部サービスや外部システムとの連携を定義する |
| 非機能要件 | 性能、信頼性、操作性などを定義する |
| 運用要件・制約 | 利用環境、前提、制限事項を整理する |

## 記述粒度

- 業務要件は、誰が、何を、どの順で行うかが分かる粒度にする。
- 機能要件は、1項目ごとに機能の可否が判定できる粒度にする。
- 画面要件は、主要表示項目と主要操作が抜けない粒度にする。
- データ要件は、識別、関連、主要属性が分かる粒度にする。
- 非機能要件は、可能なら定量、難しければ運用判断可能な定性で書く。

## 実装へ踏み込みすぎない目安

- 技術製品名の固定は、要件として必要な場合に限る。
- APIパス、DBカラム名、内部モジュール分割などは通常は書かない。
- ただし、運用導線や外部公開経路のように要求そのものへ影響するものは要件に含めてよい。
