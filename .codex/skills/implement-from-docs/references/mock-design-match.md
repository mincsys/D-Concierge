# Mock Design Match

`mock/` が存在する場合、mock はデザインの正とする。ただし正式実装対象ではない。

## 最初に確認すること

- `mock/` の有無を確認する。
- mock が存在する場合は、画面仕様書の画面イメージが mock から作られている前提で扱う。
- mock の起動方法、主要画面、スタイル定義、画像、表示部品を読む。
- 画面仕様書の対象画面と mock の画面を対応付ける。

## コピーしてよいもの

- 色
- 余白
- フォント指定
- レイアウト
- 画面部品、操作部品、入力欄、ヘッダ、ナビゲーションの見た目
- アイコンや装飾の見た目
- レスポンシブ挙動
- スタイル定義のうち、純粋に見た目を構成する部分
- 表示構造やテンプレートのうち、純粋に見た目を構成する構造

## コピーしてはいけないもの

- mock 固有のデータ
- デモ ID
- 固定 fixture
- mock 用の状態管理
- mock 用の遷移制御
- mock のディレクトリ構成
- mock のクラス、モジュール、部品分割
- mock 用の業務ロジック
- mock 用の外部 IF 代替実装
- mock のテストコード
- 正式 docs にない画面、導線、メッセージ、操作

## 実装方針

- 初期実装では、コードレベルでデザイン実装部分を可能な限り完全コピーする。
- コピー後、正式実装のディレクトリ、モジュール、責務分割へ再配置する。
- データ取得、状態、外部 IF、権限、入力検証、エラー表示は docs と本実装の設計に従って作り直す。
- mock の都合と docs が矛盾する場合は docs を優先し、必要なら `.issue/` に記録する。
- mock が存在しない場合は、画面設計書と既存デザインシステムに従う。

## 確認方針

- 実装中は、mock と本実装の主要表示サイズを見比べる。
- 総合テスト段階でスクリーンショットを撮り、画面仕様書の画面イメージと目視確認する。
- 見た目の差分を直す場合も、mock 固有の業務実装や構成を取り込まない。
- 最終確認では、mock 由来のデモ文言、デモデータ、未接続状態が残っていないことを確認する。
