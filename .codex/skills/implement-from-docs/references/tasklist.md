# タスクリスト

実装前に、作業順序を管理するタスクリストを作成する。タスクリストは作業中の一時成果物として扱い、完了したタスクを順番にチェックする。

## 保存先

- タスクリストは全体で 1 つだけ作成する。
- 保存先は `.tmp/implement-from-docs/tasklist/implementation-tasklist.md` とする。
- 複数の実装対象を扱う場合も同じファイルにまとめ、作業順が分かるように章と小見出しで整理する。

## 更新ルール

- 各タスクは `- [ ]` 形式で書く。
- タスクが完了したら、上から順番に `- [x]` へ更新する。
- 未完了タスクを飛ばして後続タスクを進める場合は、理由をタスクリストへ記録する。
- 対象外になった作業は削除せず、対象外理由を添えて処理済みにする。
- ユーザが確定した対象範囲に未チェック項目が残る場合、最終報告で完了と表現しない。

## 章構成

タスクリストには、少なくとも次の章を作成する。必要に応じて、各章の下に対象機能、実装単位、公開境界、処理契約ごとの小見出しを追加する。

```markdown
# 実装タスクリスト

## 対象範囲と前提確認

## 設計紐づけ

## 契約マトリクス

## 実装・単体テスト

### <実装単位名>

## 結合テスト

### <機能名または公開境界名>

## 品質ゲート

## 総合テスト

## issue と対象外整理

## 最終報告
```

## 各章に含めるタスク

次の項目を基本形として、実装対象に合わせて具体化してタスクリストへ記載する。

### 対象範囲と前提確認

- [ ] 実装対象、対象外、完了判定基準を確認する。
- [ ] 実装対象が未指定の場合は、ユーザへ質問して対象範囲を確定する。
- [ ] 既存 `.issue/` を今回対象に含めるか確認する。
- [ ] 未解消 `.issue/`、未反映の任意成果物、設計補助資料の扱いを確認する。

### 設計紐づけ

- [ ] 対象実装に対応する要件を確認する。
- [ ] 対象実装に対応する外部 IF、画面、データ、メッセージ、ログを確認する。
- [ ] 対象実装に対応する内部設計、クラス設計、処理設計、物理データ設計、共通設計を確認する。
- [ ] 対象実装に対応する静的テスト、単体テスト、結合テスト、総合テストの方針を確認する。
- [ ] 対象実装に対応するコーディング規約と実行コマンドを確認する。

### 契約マトリクス

- [ ] 単体テストで確認する契約を `.tmp/implement-from-docs/contract-matrix/unit/` に作成する。
- [ ] 結合テストで確認する契約を `.tmp/implement-from-docs/contract-matrix/integration/` に作成する。
- [ ] 各契約行を `未処理` として初期化する。
- [ ] 対象外契約がある場合は、対象外理由を記録する。

### 実装・単体テスト

実装単位ごとに小見出しを作成し、単体テストを伴う実装は Red、Green、Refactor の順で進める。

- [ ] Red: 事前条件、事後条件、不変条件、正常系、異常系、境界値の失敗テストを書く。
- [ ] Red: 失敗理由が未実装または設計上の未充足であることを確認する。
- [ ] Green: 失敗テストを通す最小実装を行う。
- [ ] Green: 対象単体テストを再実行する。
- [ ] Refactor: 責務、命名、型、重複、コメント、例外変換、ログ、fixture を整理する。
- [ ] Refactor: 対象範囲の静的テストを実行する。
- [ ] 単体テスト用の契約マトリクスを更新する。

### 結合テスト

結合テストは、機能、公開境界、処理契約ごとに必要なタイミングで追加する。単体テストの Red、Green、Refactor 固定章に押し込めない。

- [ ] 処理設計書の事前条件、事後条件、不変条件、副作用、例外処理、ログ、トランザクション境界を確認する。
- [ ] 外部 IF、DB、外部サービス、通信・入出力境界、実行基盤を通した契約を確認する。
- [ ] fake または mock で確認する契約と、実環境経由で確認する契約を分ける。
- [ ] 結合テストを追加または更新する。
- [ ] 対象結合テストを実行する。
- [ ] 結合テスト用の契約マトリクスを更新する。

### 品質ゲート

- [ ] 契約マトリクスに `未処理` が残っていないことを確認する。
- [ ] 対象範囲の静的テストをすべて成功させる。
- [ ] 単体テストをカバレッジ付きで実行し、方針書の目標を満たすことを確認する。
- [ ] 結合テストをカバレッジ付きで実行し、方針書の目標を満たすことを確認する。
- [ ] evidence 方針に従って最新の実行ログまたは coverage summary を保存する。

### 総合テスト

- [ ] `docs/04_テスト/04_総合テスト/` の対象範囲、対象外範囲、実行方法、証跡保存先を確認する。
- [ ] 総合テストを今回対象に含めるか、ユーザ確定範囲と照合する。
- [ ] 今回対象の場合、総合テストを実行する。
- [ ] 総合テストの証跡を方針書どおりに保存または更新する。
- [ ] 実行できない総合テストがある場合、理由と代替確認を記録する。
- [ ] 今回対象外の場合、対象外理由を記録する。

### issue と対象外整理

- [ ] 新しい仕様、設計、実装の乖離を `.issue/` に記録する。
- [ ] 解消済み issue を削除または更新する。
- [ ] 今回対象外の未実装、未解消 issue、未反映の任意成果物を整理する。
- [ ] 対象外理由をタスクリストと最終報告に残す。

### 最終報告

- [ ] 実装済み範囲を整理する。
- [ ] 今回対象外を整理する。
- [ ] 未解消 issue を整理する。
- [ ] 実行したテスト、実行できなかったテスト、証跡を整理する。
- [ ] タスクリストに未チェック項目が残っていないことを確認する。
