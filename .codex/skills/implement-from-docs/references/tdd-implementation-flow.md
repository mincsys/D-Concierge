# TDD Implementation Flow

実装は対象単位ごとに Red、Green、Refactor の順で進める。順序を飛ばさない。

## Red

- Red の前に、`.tmp/implement-from-docs/` を一時作業ルート、`.tmp/implement-from-docs/contract-matrix/` を契約マトリクス専用領域として作成する。
- 単体テストで確認する契約は `.tmp/implement-from-docs/contract-matrix/unit/` 配下へ作成し、結合テストで確認する契約は `.tmp/implement-from-docs/contract-matrix/integration/` 配下へ作成または更新する。ファイル名は対象が分かる名前にする。
- 契約マトリクスには、次を設計書から転記してテスト観点へ分解する。
  - クラス設計の責務、事前条件、事後条件、不変条件、公開プロパティ、公開メソッド
  - 処理設計の事前条件、事後条件、不変条件、副作用、例外処理、ログ出力、トランザクション境界
  - 外部 IF の入力、出力、状態コード、結果コード、または同等の成否表現、エラー応答
- 契約マトリクスの各行には、実装前は `未処理` を記録し、実装とテストの進行に合わせて `単体テスト済み`、`結合テスト済み`、`対象外理由あり`、`未実装として.issue化` のいずれかへ更新する。
- 対応する `docs/03_内部設計/02_クラス設計/` を読み、対象クラスやユースケースの `事前条件`、`事後条件`、`不変条件` を失敗テストにする。
- クラス設計がない対象は、関連する処理設計、外部 IF、共通設計から同等の契約条件を整理してテストにする。
- 既存機能を変更する場合は、先に既存テストを実行し、不足する契約テストを追加してから実装する。
- テストには方針書指定の観点・確認コメントまたは docstring を付ける。
- Red の失敗理由が、未実装または設計上の未充足であることを確認する。

## Green

- Red を通すための最小実装だけを行う。
- docs の責務分割、依存方向、IF、エラーメッセージ、ログ方針を崩さない。
- 副作用は定義された境界へ閉じ込める。
- 中核ロジックへ未検証 payload、広い辞書型、安易な `Any` や `cast` を持ち込まない。
- Green 後、対象テストを再実行して成功を確認する。

## Refactor

- 責務分割、命名、型、重複、コメント、docstring、例外変換、ログ出力、テスト fixture を整理する。
- コメントや docstring に、修正経緯、廃止、暫定、後回しの痕跡を残さない。
- 不要 import、不要 fixture、不要ログ、不要分岐を削除する。
- Refactor ごとに、変更対象範囲の静的テストを実行する。
- 変更対象のテスト方針、静的テスト方針、開発標準に定義された lint、format check、type check、その他の静的解析を実行する。
- 静的テストが不合格の場合は、同じ TDD サイクル内で修正して再実行する。
- カバレッジ達成は契約網羅の代替にしない。契約マトリクスに `未処理` 行が残る場合は、カバレッジ目標を満たしていても完了扱いしない。

## 実装単位の推奨順

docs の内部設計、ディレクトリ構成、処理設計に従って順序を調整する。特定の技術、層名、実行形態に固定せず、依存される側から依存する側へ進める。

1. 仕様上の最小概念、値、純粋判定、変換ロジック
2. 主要処理の手順制御、契約、境界インターフェース
3. 副作用を持つ入出力、永続化、外部連携、実行環境アダプタ
4. 利用者または外部システムから見える公開境界
5. 複数単位を接続する結合部分
6. 運用、検証、総合確認に必要な仕上げ

## 禁止事項

- Red なしで本体コードを先に実装しない。
- Green で設計外の先回り機能を入れない。
- Refactor で挙動を変える場合は、新しい Red から始める。
- 静的テスト失敗を既知不具合として残したまま次の実装単位へ進まない。
