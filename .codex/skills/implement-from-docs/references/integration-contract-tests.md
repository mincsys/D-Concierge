# Integration Contract Tests

結合テストでは、処理設計書に書かれた契約を自動テストで確認する。

## 契約の拾い方

- `docs/03_内部設計/04_処理設計/*処理設計.md` を読み、各処理の次をテスト観点へ入れる。
  - `事前条件`
  - `事後条件`
  - `不変条件`
  - 副作用
  - 例外処理
  - ログ出力
  - トランザクション境界
- 対応する外部 IF、エラーメッセージ、共通設計、物理データ設計も同時に確認する。
- 処理設計書の契約は全項目を `.tmp/implement-from-docs/contract-matrix/integration/` の契約マトリクスへ記録する。未確認のまま完了しない。
- 処理設計書が存在しない場合は、外部 IF と共通設計を最低限の契約として扱い、必要なら `.issue/` に記録する。

## 処理境界と副作用の結合で確認すること

- 公開経路、呼び出し方式、入力形式、結果コード、返却内容が外部 IF と一致する。
- 認証済み、未認証、権限不正、対象なし、入力不備が設計どおりに分岐する。
- DB 保存、参照、表示順、集計値、制約、関連テーブルが物理データ設計どおりである。
- 更新系では、成功時に必要な関連データがそろい、失敗時に部分保存が残らない。
- 外部連携は実接続先へ接続せず、モック応答で正常、欠落、失敗、タイムアウト相当を確認する。
- ログ出力は必要なイベントだけを確認し、パスワード、認証状態を示す秘密値、画像バイナリ、外部応答本文などを出力しない。
- fake または mock で確認した契約と、実行基盤、DB、外部サービス、通信・入出力境界など実環境経由で確認すべき契約を結合テスト用の契約マトリクス上で分ける。
- 実行基盤または通信・入出力境界を実装または変更する場合は、最低 1 件は実行基盤経由の結合テストで接続契約を確認する。
- ログ出力契約や機微情報非出力を実装できない場合は `.issue/` に記録し、契約マトリクスでは `未実装として.issue化` とする。対象範囲内の未実装契約を完了扱いしない。

## 画面・呼び出し側の結合で確認すること

- 画面、遷移、共通接続、外部 IF 呼び出し部品、状態更新が接続されている。
- 外部 IF 成功応答、未認証、入力エラー、権限不正、対象なし、データなしが画面仕様どおり表示される。
- 単体テストで確認済みの表示部品や純粋関数の詳細ロジックを重複確認しない。
- 認証前提画面はセッション確認結果に従って表示可否を制御する。

## テスト記述ルール

- 1 テストで 1 つの観点と確認内容を明確にする。
- テスト名は、対象の外部 IF または画面、条件、期待結果が読める名前にする。
- テストコードには方針書指定の観点・確認コメントまたは docstring を付ける。
- テスト間の実行順序へ依存しない。
- 本番データ、実外部サービス認証情報、実パスワード、実利用者画像を使わない。
