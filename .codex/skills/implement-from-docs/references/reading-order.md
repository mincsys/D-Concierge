# Reading Order

docs 固定構成を正として、実装前に事実を集める。ユーザへ質問する前に、リポジトリから解決できる不明点を解消する。

## 必読順序

1. `AGENTS.md`
   - 応答言語、実行コマンド、TDD、コメント、sudo、言語・依存管理・実行ツール、`.issue/` 運用を確認する。
2. `docs/01_要件定義/`
   - 初期リリース範囲、利用者、機能要件、画面要件、データ要件、非機能要件を確認する。
3. `docs/02_外部設計/`
   - システム構成、業務フロー、機能一覧、画面設計、論理データ、外部 IF、エラーメッセージ、ログを確認する。
4. `docs/03_内部設計/`
   - アーキテクチャ、ディレクトリ構成、クラス設計、処理設計、物理データ、共通設計を確認する。
5. `docs/04_テスト/`
   - 静的、単体、結合、総合テストの対象範囲、対象外範囲、コマンド、証跡、カバレッジ目標を確認する。
6. `docs/05_開発標準/`
   - 環境構築、起動方法、コーディング規約、標準実装手順を確認する。
7. README、依存定義、設定ファイル、依存サービス定義、実行基盤設定、既存テスト、既存実装
   - docs の実行手順や構成が現行コードと一致するか確認する。
8. `.issue/`
   - 既存の仕様・設計・実装乖離を確認し、今回対象へ影響するものを先に把握する。

## 判断ルール

- docs から一意に分かる事項は質問しない。
- docs と実装、docs 同士、または環境構築書と実ファイルに矛盾がある場合は、独自判断で握りつぶさず `.issue/` に記録する。
- 既存 `.issue/` が今回作業で解消される場合は、解消後に削除または内容を更新する。
- docs に定義されていない機能は、ユーザ承認または設計更新なしに実装しない。
- 実装対象を決めるときは、要件、外部 IF、画面、内部処理、クラス、テスト方針の対応を明示してから着手する。

## 読み取り結果として整理すること

- 対象機能と対象外機能
- 外部 IF、画面、データ構造、メッセージ、ログ
- 層分割、依存方向、副作用境界
- 単体テストで確認するクラス契約
- 結合テストで確認する処理契約
- 静的テスト、単体テスト、結合テスト、総合テストの実行コマンド
- mock の有無と、デザイン反映に使うファイル
