# TDD Flow

生成役は、テストコード検証が完了してから本実装へ進む。

## Red

- テストコード先行作成フェーズで、単体テストまたは結合テストの Red を確認する。
- Red の失敗理由が、未実装または設計上の未充足であることを確認する。
- 既存実装により Red にならない場合は、不足している設計観点だけを Red とし、理由を生成役の完了報告に含める。管理役は `state.md` の `## Red確認結果` に要約する。
- Red 確認結果や Red が作れない理由を、テストコード、docstring、コメント、総合テスト仕様へ指定外に書かない。

## Green

- Red を通すための最小実装だけを行う。
- docs の責務分割、依存方向、IF、エラーメッセージ、ログ方針を崩さない。
- 副作用は定義された境界へ閉じ込める。
- 中核ロジックへ未検証 payload、広い辞書型、安易な `Any` や `cast` を持ち込まない。
- Green 後、対象テストを再実行して成功を確認する。

## Refactor

- 責務分割、命名、型、重複、コメント、docstring、例外変換、ログ出力、テスト fixture を整理する。
- コメントや docstring に、修正経緯、廃止、暫定、後回しの痕跡を残さない。
- 不要 import、不要 fixture、不要ログ、不要分岐を削除する。
- Refactor ごとに、変更対象範囲の静的テストを実行する。
- 静的テストが不合格の場合は、同じ TDD サイクル内で修正して再実行する。

## 完了条件

- 検証役がテスト方針書・設計書に対するテストコードの網羅性を承認している。
- 単体テスト、結合テスト、静的テストが対象範囲で成功している。
- カバレッジ達成をテスト方針書・設計書の観点網羅の代替にしていない。
- 未実装または対象外は issue または対象外理由として記録されている。
