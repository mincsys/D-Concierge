# Test First Flow

生成役は、本実装より前にテスト方針書と設計書から直接、単体・結合テストコードを作成する。総合テストはこのフェーズで扱わず、既存の総合テスト方針と `テスト仕様・結果` に従って総合テストフェーズで実行、記録、証跡保存する。

## 順序

1. 生成役が単体・結合テスト方針書、クラス・モジュール設計、処理設計、外部 IF、画面設計を読む。
2. 生成役が単体・結合テストコードを先に作る。
3. 検証役がテストコードを検証する。
4. 管理役がレビュー完了ごとのステージングを行う。
5. 指摘がある場合は生成役が指摘を修正し、検証役の再レビュー完了ごとに管理役がステージングを行う。
6. 生成役が本実装へ進む。

## 入力資料

- `docs/04_テスト/02_単体テスト/単体テスト方針.md`
- `docs/04_テスト/03_結合テスト/結合テスト方針.md`
- `docs/04_テスト/04_総合テスト/` は、総合テストフェーズの入力として読む。
- クラス設計、モジュール設計、処理設計
- 外部 IF、画面設計、データ設計、共通設計
- 静的テスト方針、コーディング規約、証跡保存ルール

## 作成対象

- 単体テストコード
- 結合テストコード
- 単体・結合テストに必要な fixture、fake、stub、mock、seed、helper
- coverage に必要な補助設定または補助ファイル

このフェーズでは次を作らない。

- 総合テストコード
- 総合テスト仕様
- 総合テスト結果
- Playwright CLI 実行用の固定テストコード

## Red の扱い

- 単体テストまたは結合テストで Red を作れる対象は、本実装前に失敗を確認する。
- 既存実装がすでにある場合は、失敗しない理由を生成役の完了報告に含め、不足している設計観点だけ Red にする。
- Red が作れない理由は生成役の完了報告に含め、管理役が `state.md` の `## Red確認結果` に要約する。
- Red 確認結果や Red が作れない理由を、テストコード、docstring、コメント、総合テスト仕様へ指定外に書かない。

## docstring とコメント

テストコードの docstring または確認コメントは、単体テスト方針書と結合テスト方針書で要求された形式、要否、記載内容だけに従う。

方針書で確認コメントや docstring の形式が定義されていない場合は、既存テストのスタイルと開発標準に従い、不要なコメントを追加しない。

## テストコードレビュー

検証役は `review-artifacts` の `test` checklist を作業用ディレクトリへコピーして確認する。

- テスト名、コメント、docstring がテスト方針と一致している。
- 単体、結合テストの役割が混ざっていない。
- fixture、fake、stub、mock が適切に使われている。
- 実外部サービス、本番データ、秘密情報へ依存していない。
- テストが内部実装詳細へ過度に密結合していない。
- Red 確認結果は生成役の完了報告に含まれ、管理役が `state.md` に要約できる。
- Red 確認結果や Red が作れない理由が、テストコード、docstring、コメント、総合テスト仕様へ指定外に書かれていない。
