# Feature Decomposition

管理役は、実装対象を結合テストと総合テストを通しやすい検証可能な業務機能単位へ分解する。管理役の責務は分解と進行管理までであり、編集候補ファイル一覧や個別契約行は作らない。

## 機能単位の基準

1 機能は、次を一通り回せる単位にする。

- テストコード先行作成
- テストコード検証
- 実装、単体テスト、結合テスト
- 結合テスト完了検証と実装コード品質レビュー
- 機能別総合テストで、対応する総合テストケースを仕様書どおりに試せる範囲の確認

機能単位は、実装内部の都合だけで決めない。利用者操作、外部 IF、業務状態、画面または API から観測できる完了状態を持つ単位を優先する。結果的に内部設計上の機能単位と一致する場合でも、分解理由は結合テストと総合テストで確認できる業務上のまとまりとして説明できる状態にする。

## 分解方針

- 要件、外部 IF、画面、内部処理、テスト仕様が追跡できる粒度にする。
- 総合テスト方針と `docs/04_テスト/04_総合テスト/テスト仕様・結果/*.md` を読み、各機能に対応する総合テスト仕様ファイル、主対象ケース ID、後続依存ケース ID を整理する。
- 1 機能が大きすぎる場合は、利用者が確認できる振る舞い、外部 IF、画面操作、または処理契約単位へ分ける。
- 依存される機能から依存する機能へ進める。ただし、依存関係だけでなく、総合テストケースを仕様書どおりに実施できる最短経路を優先して順序を決める。
- データ構造、設定、共通処理など複数機能にまたがる基盤は、最小限の検証可能単位として先に切り出す。
- 全機能後の総合テストで利用者視点のシナリオが通せるよう、機能間依存と結合確認可能性を重視する。
- 単体または結合テストで検証できない断片だけを 1 機能にしない。やむを得ない場合は、結合テストで確認する範囲と保留理由を明確にする。
- 主業務パスに該当する総合テストケースがある場合は、そのケースを機能別総合テストで仕様書どおりに実施できる状態を優先する。どうしても後続機能、外部認証、自動操作不可、環境制約が必要な場合は、未確認範囲と再実施条件を明確にする。

## 管理役が書く内容

- 機能 ID
- 機能名
- 機能概要
- 関連 docs の代表パス
- 依存する前提機能
- 対応する総合テスト仕様ファイル
- 主対象ケース ID
- 後続依存ケース ID
- 結合完了時に検証役へ確認させる範囲

管理役は詳細な編集範囲を決めない。ただし、機能別総合テストで生成役へ渡すため、対応する総合テスト仕様ファイルとケース ID は整理する。生成役と検証役は、docs、指定ケース、差分から必要な確認範囲を判断する。
