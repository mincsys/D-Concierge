# Manager Tasklist Format

管理役は、作業順序を管理するタスクリストを作成する。タスクリストを編集できるのは管理役だけである。

## 保存先

```text
.tmp/implement-from-docs-v2/tasklist/implementation-tasklist.md
```

## 章構成

```markdown
# 実装タスクリスト v2

## 対象範囲と前提確認

## 初期 docs 読込

## サブエージェント起動

- [ ] 生成役へ `SKILL.md` の軽読と初期 docs 読込を依頼する
- [ ] 検証役へ `SKILL.md` の軽読と初期 docs 読込を依頼する

## サブエージェント中断・再開（発生時のみ追加）

- [ ] 空完了メッセージをリミット到達による一時中断として state に記録する
- [ ] ユーザのリミット解除指示後、前回の同一サブエージェントへ再利用再開を依頼する
- [ ] 再利用できない場合だけ同じ役割のサブエージェントを新規起動し、引き継ぎを渡す
- [ ] 再開依頼で `SKILL.md` の軽読を指示する
- [ ] 再開後の正式な完了報告を受けてから対応フェーズを進める

## 機能分解

## 機能別開発

### F001 <機能名>

- [ ] 単体・結合テストコード先行作成を生成役へ依頼する
- [ ] テストコード検証を検証役へ依頼する
- [ ] テストコード初回レビュー・各再レビュー後のステージングを行う
- [ ] 実装、単体テスト、結合テストを生成役へ依頼する
- [ ] 結合テスト完了検証と実装コード品質レビューを検証役へ依頼する
- [ ] 結合テスト・実装品質の初回レビュー・各再レビュー後のステージングを行う
- [ ] 機能別総合テストを生成役へ依頼する
- [ ] 機能別総合テストレビューを検証役へ依頼する
- [ ] 機能別総合テストレビュー後のステージングを行う
- [ ] 検証役の結合完了判定に基づいて機能結合完了にする

## 横断レビュー

- [ ] 全機能の機能結合完了を確認する
- [ ] 各機能 state の `正式総合テストへの持ち越し` を集約する
- [ ] `.tmp/implement-from-docs-v2/system-test/state.md` を作成または更新する
- [ ] 正式総合テスト前の横断レビューを検証役へ依頼する
- [ ] 横断レビュー issue がある場合は生成役へ修正を依頼する
- [ ] 横断レビュー再レビューとステージングを最大 3 回まで実施する
- [ ] 横断レビュー結果を `.tmp/implement-from-docs-v2/final-review.md` と全体 state に記録する

## 正式総合テスト

- [ ] 正式総合テストと証跡保存を生成役へ依頼する
- [ ] 正式総合テスト結果分析レビューと最終実装品質チェックを検証役へ依頼する
- [ ] 不合格・保留の修正可能 issue を生成役へ修正依頼する
- [ ] 修正後の関連テスト・該当総合テスト再実行を生成役へ依頼する
- [ ] 正式総合テスト再レビューとステージングを最大 3 回まで実施する
- [ ] テスト不能項目と TBC を state に記録する

## issue と TBC 整理

## 最終報告
```

## 更新ルール

- 管理役だけが `- [ ]` を `- [x]` に変更する。
- 生成役と検証役にタスクリストを編集させない。
- 検証役の判定を受けてから、対応するチェックを付ける。
- 検証役レビューでは、依頼した `review-artifacts` checklist が作業用ディレクトリに配置され、未処理項目と根拠なし `- [x]` がなく、指摘あり、対象外、判断不能の件数が完了可否へ反映されていることを確認する。
- サブエージェントの完了メッセージが空、または実質的な完了報告を含まない場合は、対応フェーズの完了チェックを付けない。
- サブエージェント中断・再開の章とチェックは発生時だけ追加する。発生していない場合は章ごと作らなくてよい。
- 検証打ち切りで TBC が残る場合も、独立作業を継続するためのチェックだけを付け、備考に `TBC 残存、未解決` と記録する。
- 正式総合テスト前の横断レビューで issue が残る場合、正式総合テストへ進むチェックを付けない。
- 正式総合テスト後の不合格・保留が検証役により原因分析されていない場合、正式総合テスト結果分析レビューのチェックを付けない。
- テスト不能項目以外に不合格・保留が残る場合、総合テスト issue 修正ループ完了のチェックを付けない。
- 未完了タスクを飛ばす場合は理由を記録する。
- ユーザが確定した対象範囲に未処理機能が残る場合、最終報告で完了と表現しない。
- 機能別総合テストで `不合格` または未分類の未実施が残る場合、機能結合完了扱いにしない。
- `部分確認`、`後続機能待ち`、`環境・承認待ち`、`対象外` は、検証役が妥当と判定し state に正式総合テストへの持ち越しを記録した場合だけ機能結合完了へ進める。
- 結合完了済みでも、正式総合テストで問題が残った機能は最終的な完了扱いにしない。
- `.tmp` 削除や削除ステージングのチェックは作らない。`.tmp` 削除はユーザが行う。
