# State File Format

`.tmp/implement-from-docs-v2/features/<機能ID>/state.md` は、管理役が機能ごとの進行状態を記録するためのファイルである。機能別 state はテストコード、実装、結合レビュー、機能別総合テストレビューまでを扱う。正式総合テストは `.tmp/implement-from-docs-v2/system-test/state.md` に記録する。

## 保存先

```text
.tmp/implement-from-docs-v2/features/F001_<機能名>/state.md
```

## 章構成

```markdown
# F001 <機能名>

## 機能概要

## 関連 docs

## 前提機能

## 現在フェーズ

未着手 / テスト作成 / テスト検証 / 実装中 / 結合検証 / 機能別総合テスト実行中 / 機能別総合テスト検証 / サブエージェント中断中 / 機能結合完了 / 打ち切りTBCあり

## ループ回数

## サブエージェント状態

## Red確認結果

## テストコード作成結果

## テストコード検証結果

## テストコードレビュー指摘

## テストコードレビュー checklist

## 結合テスト検証結果

## 実装品質レビュー結果

## 結合レビュー checklist

## 機能別総合テスト実行結果

## 機能別総合テスト証跡

## 機能別総合テスト保留事項

## 機能別総合テストレビュー結果

## 機能別総合テストレビュー checklist

## 正式総合テストへの持ち越し

## TBC issue

## 備考
```

各 `checklist` 章には、保存先だけでなく checklist 処理サマリを記録する。

## checklist 処理サマリの記録形式

```markdown
- 保存先:
- 総項目数:
- 処理済み項目数:
- 未処理項目数:
- 指摘あり件数:
- 対象外件数:
- 判断不能件数:
- 根拠なし `- [x]`: あり / なし
- 完了可否への反映:
```

## 機能別総合テスト保存先

生成役は、機能別総合テストの開始時に `docs/04_テスト/04_総合テスト/` 全体を次の場所へコピーする。

```text
.tmp/implement-from-docs-v2/features/<機能ID>/system-test/
```

コピー後の標準構成は次のとおり。

```text
.tmp/implement-from-docs-v2/features/<機能ID>/system-test/総合テスト方針.md
.tmp/implement-from-docs-v2/features/<機能ID>/system-test/テスト仕様・結果/
.tmp/implement-from-docs-v2/features/<機能ID>/system-test/evidence/
```

機能別総合テストでは、この `.tmp` 側へ正式総合テストと同じ形式で結果と証跡を保存する。`docs/04_テスト/04_総合テスト/` は変更しない。

## 記録ルール

- 管理役が更新する。
- 生成役と検証役は読み取り専用とし、更新が必要な内容は管理役へ報告する。
- 個別テスト項目はここへ転記しない。
- 編集候補ファイル一覧はここへ書かない。
- 生成役の Red 確認結果は要約して記録する。
- 検証役の判定は要約して記録する。
- 検証役レビューで使った `review-artifacts` checklist 保存先と checklist 処理サマリを記録する。
- ループ回数、対象 issue、検証結果、TBC issue はここへ記録する。
- 未処理項目、根拠なし `- [x]`、未反映の判断不能がある場合は、当該レビューを完了扱いしない。
- 機能別総合テスト結果は個別テスト項目を全量転記せず、分類別件数、主な確認結果、証跡保存先、保留理由、正式総合テストへの持ち越しを要約して記録する。
- `正式総合テストへの持ち越し` には、`部分確認`、`後続機能待ち`、`環境・承認待ち`、`対象外` のうち、最終正式総合テストで再確認すべき項目を記録する。
- 打ち切り時に issue が残る場合は、TBC 移動後のパスを記録する。
- 検証打ち切りで High issue が残る場合でもフェーズ自体は閉じ、備考に `High issue 残存、TBC 移動済み` と記録する。
- サブエージェントの完了メッセージが空、または実質的な完了報告を含まない場合は、`サブエージェント状態` にリミット到達による一時中断として記録し、issue や TBC へ自動分類しない。
- リミット解除後に再開した場合は、再利用再開か新規再起動か、前回サブエージェントを再利用できなかった場合の理由、渡した引き継ぎ要約、再開後の完了報告有無を記録する。
- 修正ループの個別ラウンド記録ディレクトリは作らない。

## サブエージェント状態の記録形式

```markdown
## サブエージェント状態

- 対象役割:
- 起動状態: 初回 / 通常再利用 / 再利用再開 / 新規再起動
- 直前フェーズ:
- 最終依頼:
- 最終応答: 通常完了 / 空完了メッセージ / 利用不能 / 文脈喪失 / なし
- 中断理由:
- 再開方針:
- 新規再起動理由:
- 引き継ぎ要約:
- `SKILL.md` 軽読指示: 済 / 未
- 再開後の完了報告:
```

## 正式総合テスト state

全機能の機能結合完了後、管理役は `.tmp/implement-from-docs-v2/system-test/state.md` を作成または更新する。

```markdown
# 正式総合テスト

## 対象機能一覧

## 結合完了済み機能

## 参照する総合テスト仕様

## 機能別総合テスト集約

## 正式総合テスト重点確認項目

## 現在フェーズ

未着手 / 持ち越し集約 / 正式総合テスト実行中 / 正式総合テスト検証 / 横断レビュー / サブエージェント中断中 / 完了 / 打ち切りTBCあり

## ループ回数

## サブエージェント状態

## 正式総合テスト実行結果

## 証跡

## 保留事項

## 正式総合テストレビュー結果

## 正式総合テストレビュー checklist

## 最終実装品質レビュー結果

## 横断レビュー結果

## 横断レビュー checklist

## TBC issue

## 備考
```

全体 state には、個別機能の詳細なテスト項目を転記しない。機能別総合テスト集約、正式総合テスト結果、証跡、保留事項、全体レビュー結果、TBC issue の要約だけを記録する。

全体 state の `正式総合テストレビュー checklist` と `横断レビュー checklist` にも、保存先、総項目数、処理済み項目数、未処理項目数、指摘あり件数、対象外件数、判断不能件数、根拠なし `- [x]` の有無、完了可否への反映を記録する。

`.tmp/implement-from-docs-v2/` は最終成果物ではないが、削除はユーザが行う。state に `.tmp` 削除確認章を作らない。
