# Reading Order

docs 固定構成を正として、実装前に事実を集める。ユーザへ質問する前に、リポジトリから解決できる不明点を解消する。

## 必読順序

1. `AGENTS.md`
   - 応答言語、実行コマンド、TDD、コメント、sudo、言語・依存管理・実行ツール、issue 運用、ステージング上の制約を確認する。
2. `docs/01_要件定義/`
   - 対象範囲、対象外範囲、利用者、機能要件、画面要件、データ要件、非機能要件を確認する。
3. `docs/02_外部設計/`
   - システム構成、業務フロー、機能一覧、画面設計、論理データ、外部 IF、エラーメッセージ、ログを確認する。
4. `docs/03_内部設計/`
   - アーキテクチャ、ディレクトリ構成、クラス設計、処理設計、物理データ、共通設計を確認する。
5. `docs/04_テスト/`
   - 静的、単体、結合、総合テストの対象範囲、対象外範囲、コマンド、証跡、カバレッジ目標を確認する。
6. `docs/05_開発標準/`
   - 環境構築、起動方法、コーディング規約、標準実装手順を確認する。
7. README、依存定義、設定ファイル、依存サービス定義、実行基盤設定、既存テスト、既存実装
   - docs の実行手順や構成が現行コードと一致するか確認する。
8. `.issue/implement-from-docs/`
   - v2 実行中または過去実行で残った issue が今回対象へ影響するか確認する。
9. 既存証跡
   - 単体、結合、総合テストの evidence や coverage summary がある場合、今回更新対象か確認する。

## 判断ルール

- docs から一意に分かる事項は質問しない。
- docs と実装、docs 同士、または環境構築書と実ファイルに矛盾がある場合は、独自判断で握りつぶさず `.issue/implement-from-docs/` に記録する。
- docs に定義されていない機能は、ユーザ承認または設計更新なしに実装しない。
- 実装対象を決めるときは、要件、外部 IF、画面、内部処理、テスト方針の対応を確認してから着手する。
- 作業開始時に未コミット変更がある場合は、v2 作業差分と混ざらないようユーザ確認を挟む。
