# Issue Lifecycle

v2 実行中の指摘は `.issue/implement-from-docs/` 配下で管理する。

## 保存先

```text
.issue/implement-from-docs/
```

打ち切り時の残 issue と仕様書側修正 issue は次へ移動する。

```text
.issue/implement-from-docs/TBC/
```

## ファイル名

```text
YYYY-MM-DD_HH-MM-SS_<指摘タイトル>.md
```

## 章構成

```markdown
# 指摘タイトル

## 重要度

High / Medium / Low

## レビュー対象

## 指摘内容

## 根拠

## 影響

## 修正方針
```

## 運用

- 指摘は 1 指摘 1 ファイルで保存する。
- issue ファイルは、管理役が削除または TBC 移動するまで前回指摘の正本として残す。
- 検証役が対象 issue を `解消済み / 未解消 / 判断不能 / TBC候補 / 仕様書側修正` のいずれかに分類する。
- 検証役が `解消済み` と判定し、`削除可 issue` に列挙した issue だけを管理役が削除する。
- `未解消`、`判断不能`、`TBC候補`、`仕様書側修正` に分類された issue は削除しない。
- 管理役は issue 本文を読んで独自に解消判断しない。検証役の明示判定だけを根拠に削除する。
- 同一趣旨の issue が既にある場合は重複作成しない。
- 3 回のループで解消できない issue は重大度に関係なく TBC へ移動する。
- 仕様書側修正が必要な issue は、仕様書を修正した後も削除せず TBC へ移動する。
- TBC へ移動した issue は state とタスクリスト備考へ記録する。

## 削除前確認

管理役は issue 削除前に、検証役の報告に次がそろっていることを確認する。

- 対象 issue ごとの分類。
- `削除可 issue` のパス一覧。
- `削除禁止 issue` のパス一覧。
- `解消済み` 判定の根拠となる差分、証跡、生成役報告の要約。

分類がない issue、削除可否が曖昧な issue、または根拠が不足する issue は削除しない。

## 仕様書側修正 issue

レビュー指摘の修正方針が仕様書、テスト方針、証跡方針、参照元文書の修正を求める場合、ユーザ確認なしに修正してよい。修正後も指摘ファイルは削除せず、TBC へ移動する。

## 打ち切り

検証を続けると指摘が無限に増える場合があるため、同一フェーズの生成・検証ループは最大 3 回で打ち切る。打ち切り後は TBC 移動を行い、次フェーズまたは次機能へ進む。
