# Staging Boundary

検証役がレビューまたは再レビューを完了するたび、合否に関係なく v2 作業差分をステージングする。これにより、次の修正依頼、次フェーズ、次機能で発生した変更を未ステージ差分として判別しやすくする。

## 開始前確認

- `git status --short` で既存の未コミット変更を確認する。
- 既存のユーザ変更がある場合は、v2 作業差分と混ざらないようユーザ確認を挟む。
- ユーザ変更を勝手にステージングしない。

## ステージングタイミング

- テストコード初回レビュー完了直後
- テストコード修正ループの各再レビュー完了直後
- 結合テスト、カバレッジ、実装品質の初回レビュー完了直後
- 結合テスト、カバレッジ、実装品質の修正ループの各再レビュー完了直後
- 機能別総合テストの初回レビュー完了直後
- 機能別総合テストの修正ループの各再レビュー完了直後
- 全機能後の正式総合テスト、未ステージ内容、最終実装品質の初回レビュー完了直後
- 全機能後の正式総合テスト、未ステージ内容、最終実装品質の修正ループの各再レビュー完了直後

review-artifacts checklist の作業用コピーだけではステージングしない。ステージングは、検証役がレビューまたは再レビューを終え、checklist 更新、issue 作成、issue 解消分類、完了可否を依頼元へ返した後に行う。

管理役は、検証結果に基づく最低限の `state.md` 更新、issue 削除、TBC 移動が必要な場合はそれを反映してからステージングする。issue 削除は、検証役が `削除可 issue` に列挙した issue だけを対象にする。次の生成依頼、修正依頼、次フェーズへの移行は、このステージング後に行う。

機能別レビュー後のステージングは、次フェーズ、次機能、または正式総合テストへ進む前の境界である。正式総合テストレビュー後のステージングは、横断レビューまたは最終報告へ進む前の境界である。

## ステージング対象

- v2 実行で作成または変更した成果物
- `.tmp/implement-from-docs-v2/` の管理ファイル
- `.tmp/implement-from-docs-v2/` の review-artifacts 作業用 checklist
- `.tmp/implement-from-docs-v2/features/<機能>/system-test/` の機能別総合テスト仕様コピー、結果、evidence
- `.issue/implement-from-docs/` の issue 作成、削除、TBC 移動
- 実装、テスト、docs、証跡の v2 作業差分

`.tmp/implement-from-docs-v2/` は最終成果物ではないが、スキル側では削除しない。削除はユーザが行うため、`.tmp` 削除や削除ステージングはこの skill の作業に含めない。

## 検証役へ渡す差分情報

検証役はコマンドを実行しない。管理役が必要な差分情報を取得し、検証役への依頼に含める。

- `git status --short`
- `git diff`
- 必要に応じて staged 差分
- 未追跡ファイル
- `.issue/implement-from-docs/` の新規、削除、移動
- 機能別総合テストでは `git diff -- docs/04_テスト/04_総合テスト` により `docs` 正本が変更されていないことを示す情報

## 禁止

- ユーザの既存変更を勝手にステージングしない。
- ステージング済みのユーザ変更を戻さない。
- 検証役のレビューまたは再レビュー完了後のステージングを行わずに、修正依頼、次フェーズ、次機能へ進めない。
