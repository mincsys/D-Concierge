# Write Environment Setup Docs

## Overview

要件定義書、外部設計書、内部設計書、テスト設計書、README、設定ファイルを読み、日本のウォーターフォール開発向けの環境構築書を作成する。既定の成果物は `docs/05_開発標準/01_環境構築/開発環境構築.md` と `docs/05_開発標準/01_環境構築/本番環境構築.md` の 2 ファイルだけとし、配置先はユーザへ質問しない。

## Workflow

### 1. 入力資料を読む

- 先に `AGENTS.md`、`README.md`、`docs/` 配下、環境変数サンプル、Docker/Compose、CI/CD、package/lock、infra 関連設定、既存の環境構築関連資料を読む。
- 既存文書がある場合は、文体、用語、章番号、リンク表記、ID体系を合わせる。
- ユーザが文書名や章構成を指定している場合は、その指示を最優先にする。
- 成果物配置は `docs/05_開発標準/01_環境構築/` 固定とし、配置先を質問しない。
- 要件定義、外部設計、内部設計、テスト設計から、必要な実行環境、依存サービス、外部サービス、環境変数、監視、ログ、バックアップ、動作確認の候補を整理する。
- 仕様書と実装・設定ファイルに乖離を見つけた場合は、既存規約どおり `.issue/` へメモする。

### 2. 確定事項と質問事項を分ける

- 既存 docs、README、設定ファイルから一意に読み取れる内容は質問しない。
- 未確定事項は本文作成前にユーザへ質問し、合意してから本文へ反映する。
- 環境構築書では質問を必須化せず、既存資料から一意に確定できない場合のみ質問する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 各質問では、確認が必要な理由、影響する章、未確定のまま進めた場合に曖昧になる内容を丁寧に説明する。
- 選択肢を提示する場合は、`など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- 選択肢は、採用時の影響が明確に異なるものだけにする。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 推奨選択肢は、リポジトリ事実、既存方針、ウォーターフォール文書としての安全性を基準に選ぶ。
- `references/question-checklist.md` を使い、特に次の項目は、既存資料から一意に確定できない場合のみ確認する。
  - 対象環境
  - 対象 OS
  - 本番環境の実行基盤
  - 機密情報の扱い
  - 外部サービス接続
  - 初期データ・マイグレーション
  - 監視・ログ・バックアップ
  - 動作確認の範囲
  - 障害時対応の範囲

### 3. 本文作成直前に合意状況を再確認する

- 設計書本文を作成する直前に、全質問項目についてユーザ合意が取れているかを再確認する。
- 再確認では、各項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理する。
- 質問回答がすべて得られた場合、`references/question-checklist.md` にある全質問項目を一覧化してユーザへ表示する。
- 一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `根拠` には、その合意結果を本文へ反映してよいと判断した理由を書く。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイルや設定から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 未合意項目がある場合は、環境構築書の本文作成へ進まない。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。
- 明確な作成指示がない場合は、合意結果一覧の確認後に待機し、本文作成を開始しない。

### 4. テンプレートを選ぶ

- 開発環境構築の既定テンプレートは `assets/05_環境構築/開発環境構築.md` を使う。
- 本番環境構築の既定テンプレートは `assets/05_環境構築/本番環境構築.md` を使う。
- 章や本文の採否を迷う場合は、ユーザ指定と `references/review-checklist.md` の完成条件を優先する。
- テンプレートは特定クラウド、OS、言語、DB、CI/CD に固定しすぎず、既存資料またはユーザ合意に基づいて具体化する。

### 5. 環境構築書へ落とす

- 文書は日本語で、ウォーターフォール開発向けの正式文書として断定形で書く。
- 差分説明、検討メモ、会話ログ、旧仕様の痕跡を書かない。
- `【template-guidance】` ブロック、`【質問】`、未置換プレースホルダを最終成果物に残さない。
- 開発環境構築書は、開発者がローカルまたは合意済みの開発環境で作業を開始できる状態にする手順書として書く。
- 本番環境構築書は、実運用できる本番環境を構築し、起動、確認、監視、バックアップ、障害時対応の入口までを定義する手順書として書く。
- 秘密値そのもの、個人アカウント情報、実運用の認証情報は本文へ書かない。
- `開発環境構築.md` に `トラブルシューティング` 章を作らない。
- 環境構築書以外の設計書、テスト方針、運用手順書は作らない。

### 6. 整合確認する

- `references/review-checklist.md` で、文書範囲、開発環境構築、本番環境構築、機密情報、既存資料との整合、正式文書表現を確認する。
- `README.md`、Docker/Compose、CI/CD、package/lock、infra 関連設定と、記載したコマンド、サービス名、環境変数、ポート、起動条件が矛盾しないことを確認する。
- 最終成果物は `docs/05_開発標準/01_環境構築/開発環境構築.md` と `docs/05_開発標準/01_環境構築/本番環境構築.md` だけにする。
