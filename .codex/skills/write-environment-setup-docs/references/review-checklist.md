# 環境構築書レビュー観点

## 文書範囲

- 成果物が `docs/05_開発標準/01_環境構築/開発環境構築.md` と `docs/05_開発標準/01_環境構築/本番環境構築.md` の 2 ファイルに限定されているか。
- 配置先をユーザへ質問していないか。
- 環境構築書以外の設計書、テスト方針、運用手順書が混在していないか。
- 日本のウォーターフォール開発向けの正式文書として読めるか。

## 質問合意

- 既存 docs、README、設定ファイルから一意に分かる内容を質問していないか。
- 不明点がある場合だけ質問しているか。
- `【質問】` に相当する未確定項目が本文に残っていないか。
- 質問が 1 つずつ行われ、合意内容が整理されているか。
- 選択肢に `など`、`等`、`その他いろいろ` のような曖昧な表現が使われていないか。
- 選択肢が `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のような番号付きリストになっているか。
- ユーザが番号だけで回答できることを質問文で明記しているか。
- 推奨選択肢が `1. 選択肢名（推奨）` のように 1 つだけ明示されているか。
- 本文作成直前に、質問表の全質問項目と合意結果がユーザへ表示されているか。
- 合意結果一覧に `質問項目`、`合意結果`、`状態`、`根拠` が含まれているか。
- 合意結果一覧の表示後、ユーザから明確な作成指示を受けてから本文作成へ進んでいるか。
- `確認しました`、`OKです`、`問題ありません` だけで本文作成へ進んでいないか。

## 開発環境構築

- `開発環境構築.md` が指定章構成になっているか。
- `トラブルシューティング` 章が作られていないか。
- 必要なツール、環境変数、依存サービス、セットアップ、起動・停止、動作確認が書かれているか。
- README、Docker/Compose、package/lock、設定ファイルとコマンド、サービス名、ポート、環境変数が矛盾していないか。

## 本番環境構築

- `本番環境構築.md` が指定章構成になっているか。
- 本番環境構成、必要なツール・アカウント、環境変数・機密情報、インフラ準備、アプリケーション配置、初期データ・マイグレーション、起動・停止、動作確認、監視・ログ・バックアップ、障害時対応が書かれているか。
- 特定クラウド、OS、言語、DB、CI/CD に根拠なく固定されていないか。
- 本番で実施してよい確認だけが動作確認に書かれているか。

## 機密情報・安全性

- 秘密値そのもの、個人アカウント情報、実運用の認証情報が本文へ書かれていないか。
- 機密情報の管理方法がユーザ合意または既存資料に基づいているか。
- 本番データや秘密値を例示値として使っていないか。

## 既存資料との整合

- 要件定義、外部設計、内部設計、テスト設計、README、設定ファイルと矛盾していないか。
- 仕様書と実装・設定ファイルの乖離を見つけた場合に `.issue/` へメモしているか。
- 実在しないコマンド、サービス名、ファイル名、環境変数を推測で書いていないか。

## 表現

- 文書が断定形で書かれているか。
- 差分説明、検討メモ、会話ログ、旧仕様の痕跡が残っていないか。
- `【template-guidance】` ブロック、`【質問】`、未置換プレースホルダが最終成果物に残っていないか。
- 既存 docs と用語、文体、リンク表記がそろっているか。
