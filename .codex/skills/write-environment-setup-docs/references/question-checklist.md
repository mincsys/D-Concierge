# 環境構築書 質問チェックリスト

## 使い方

- リポジトリや既存文書から一意に分かる内容は質問しない。
- 配置先は `docs/05_開発標準/01_環境構築/` 固定とし、ユーザへ質問しない。
- 環境構築書では質問を必須化せず、既存 docs、README、設定ファイルから一意に確定できない場合のみ質問する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 各質問では、確認が必要な理由、影響する章、未確定のまま進めた場合に曖昧になる内容を説明する。
- 選択肢を提示する場合は、`など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- 選択肢は、採用時の影響が明確に異なるものだけにする。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 推奨選択肢は、リポジトリ事実、既存方針、ウォーターフォール文書としての安全性を基準に選ぶ。
- 設計書本文を作成する直前に、全質問項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理し、未合意項目がある場合は本文作成へ進まない。
- 質問回答がすべて得られた場合、質問表にある全質問項目と合意結果を一覧化してユーザへ表示する。
- 合意結果一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `根拠` には、その合意結果を本文へ反映してよいと判断した理由を書く。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイルや設定から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。

## 1. 対象環境

- 開発環境はローカルのみ、Docker 環境、共通開発環境のどれを含めるか。
- 既存 README や Compose から一意に分かる場合は質問しない。

## 2. 対象 OS

- 開発環境の手順を Linux、macOS、Windows のどれに対応させるか。
- リポジトリや既存 docs で対応 OS が定義されている場合は質問しない。

## 3. 本番環境の実行基盤

- VPS、Docker Compose、Kubernetes、クラウド PaaS、サーバレスのどれを前提にするか。
- 外部設計、内部設計、infra 設定から一意に分かる場合は質問しない。

## 4. 機密情報の扱い

- `.env`、クラウドシークレット、CI/CD シークレット、手動設定のどれを正式な管理方法にするか。
- 秘密値そのものは本文へ書かない。

## 5. 外部サービス接続

- 認証、メール、ストレージ、決済、外部 API を本番で実接続するか、未確定として残すか。
- 外部IF設計や設定ファイルから一意に分かる場合は質問しない。

## 6. 初期データ・マイグレーション

- DB マイグレーション、初期データ投入、管理者ユーザ作成を本番構築手順に含めるか。
- migration や seed の実在が確認できる場合は、その事実を優先する。

## 7. 監視・ログ・バックアップ

- 本番構築書に監視、ログ収集、バックアップ、リストア確認をどこまで含めるか。
- 非機能要件や運用設計から一意に分かる場合は質問しない。

## 8. 動作確認の範囲

- 起動確認、ヘルスチェック、主要画面確認、主要 API 確認、総合テストへの接続確認をどこまで含めるか。
- テスト設計書や README に確認方法がある場合は、その内容を優先する。

## 9. 障害時対応の範囲

- 再起動、ロールバック、バックアップ復元、ログ確認、連絡先を含めるか。
- 運用要件や非機能要件から一意に分かる場合は質問しない。
