# 総合テスト設計 質問チェックリスト

## 使い方

- リポジトリや既存文書から一意に分かる内容は質問しない。
- 本文作成前に、以下の未確定項目をユーザへ提示し、合意してから総合テスト設計書へ反映する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 各質問では、確認が必要な理由、影響する章、未確定のまま進めた場合に曖昧になる内容を説明する。
- 選択肢を提示する場合は、`など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- 選択肢は、採用時の影響が明確に異なるものだけにする。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 推奨選択肢は、リポジトリ事実、既存方針、ウォーターフォール文書としての安全性を基準に選ぶ。
- 設計書本文を作成する直前に、全質問項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理し、未合意項目がある場合は本文作成へ進まない。
- 質問回答がすべて得られた場合、質問表にある全質問項目と合意結果を一覧化してユーザへ表示する。
- 合意結果一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `根拠` には、その合意結果を本文へ反映してよいと判断した理由を書く。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイルや設定から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。
- 明確な作成指示がない場合は、合意結果一覧の確認後に待機し、本文作成を開始しない。

## 1. 総合テスト対象範囲

- どのシステム、サブシステム、機能、画面、API、バッチ、外部IFを総合テスト対象にするか。
- 静的テスト、単体テスト、結合テストとの境界をどう分けるか。
- 初期リリース対象外や後続工程へ委ねる範囲があるか。

## 2. 総合テスト分割単位

- `テスト仕様・結果/〇〇テスト.md` を何単位で作成するか。
- 画面単位、業務シナリオ単位、機能単位、ユーザロール単位、外部IF単位のどれを採用するか。
- 汎用的でシンプルな構成を優先する場合は、画面単位を推奨候補にする。

## 3. 総合テスト観点

- 業務シナリオ、画面遷移、入力検証、権限、登録・更新・削除、外部IF、異常系、メッセージ表示、ブラウザ表示のうち必須観点をどう扱うか。
- 対象システム固有の観点を追加する必要があるか。
- Playwright で検査する観点と、人が目視確認する観点をどう分けるか。

## 4. Playwright 利用方針

- Playwright で検査できる範囲をすべて Playwright で実施する方針でよいか。
- Playwright で検査する範囲は何か。
- Playwright で検査しない範囲は何か。
- Playwright で検査できない範囲は何か。
- Playwright のスクリーンショット、動画、レポートを証跡として扱うか。

## 5. 手動テスト実施方針

- Playwright で検査できない確認を、人がどのように操作、目視確認、証跡取得、結果記入するか。
- 手動確認時の証跡として、スクリーンショット、動画、操作ログ、不具合票のどれを扱うか。
- 不合格、保留、未実施をどのように記録するか。

## 6. テスト環境方針

- ローカル、検証環境、共通開発環境、Playwright 実行環境をどう扱うか。
- 利用ブラウザは何か。
- 外部サービス接続先は何か。
- 環境差分がある場合に結果へどう記録するか。

## 7. 成果物・証跡方針

- 文書成果物は `総合テスト方針.md` と `テスト仕様・結果/〇〇テスト.md` とする。
- スクリーンショットを保存するか。
- 動画を保存するか。
- Playwright レポートを保存するか。
- 不具合票を証跡とするか。
- 再テスト結果をどう保存するか。
- 証跡の保存場所はどこか。
- 証跡の命名規則は何か。
- 証跡の保存期間や世代管理を定めるか。
- 不具合修正時および再テスト時に証跡を更新するか。

## 8. 完了条件

- 全ケース実施を必須にするか。
- 全ケース合格を必須にするか。
- 不具合残件がある場合の扱いをどうするか。
- 再テスト未完了項目がある場合の扱いをどうするか。
- 未実施ケースがある状態で完了扱いにする場合、承認者と記録方法をどう定めるか。

## 9. 作成対象ファイル

- 総合テスト分割単位に基づいて、どの `テスト仕様・結果/〇〇テスト.md` を作成するか。
- 具体ファイル名に `〇〇` を残さない前提でよいか。
- 同じ対象を複数ファイルへ重複記載しない前提でよいか。
