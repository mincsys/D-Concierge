# 〇〇テスト

## 1. 文書の目的
【template-guidance】<br>
- ここには、本書が総合テスト方針で定義した分割単位に対応するテストケースと実施結果を書く文書であることを書く。
- 他の分割単位のテストケースや、総合テスト全体の方針には広げない。

【/template-guidance】<br>

本書は、〇〇テストのテストケース、実施結果および証跡を記録することを目的とする。

## 2. 前提
【template-guidance】<br>
- ここには、`総合テスト方針.md` に基づくテスト仕様・結果文書であることを書く。
- 対象環境、実施手段、参照資料、結果記入ルールを必要に応じて書く。
- 同じテスト対象を複数のテスト仕様・結果文書へ重複記載しないこともここで明記する。

【/template-guidance】<br>

- 本書は、`総合テスト方針.md` に基づいて作成する。
- 本書の対象は、総合テスト分割単位で定義した〇〇テストのみとする。
- 同じテスト対象を他のテスト仕様・結果文書へ重複記載しない。

## 3. テスト対象
【template-guidance】<br>
- ここには、このテスト仕様・結果文書で扱う画面、業務シナリオ、機能、ユーザロール、外部IFを書く。
- 必要に応じて、各対象が関連する機能名または機能IDを `対象` または `確認目的` に含める。ただし、総合テスト分割単位を機能ごと固定にしない。
- 対象外範囲を長く書かず、必要最小限の境界説明にとどめる。

## 4. テストケース一覧
【template-guidance】<br>
- ここには、総合テストとして実施するテストケースを表で書く。
- `観点` には、総合テスト方針で定義した観点または対象固有の観点を書く。
- 必要に応じて、`観点` または `備考` に対応機能、関連機能、関連外部IFを記録し、実装時にケースID単位で追跡できるようにする。
- `前提条件` には、ログイン状態、権限、初期表示、既存データ、外部サービス状態など、実施前に満たすべき条件を書く。
- `操作手順` には、人または Playwright が実施する操作を順番に書く。1 セル内で複数手順を書く場合は `<br>` を使う。
- `期待結果` には、画面表示、データ状態、メッセージ、遷移先、外部IF結果、証跡として確認する内容を書く。
- `実施結果` は、未実施、合格、不合格、保留のいずれかで記録する。
- `実施日` には、テストケースを実施した日付を書く。未実施の場合は空欄とする。
- `実施者` には、テストケースを実施した担当者名を書く。未実施の場合は空欄とする。
- `証跡` には、スクリーンショット、動画、Playwright レポート、不具合票などへの相対リンクまたは識別子を書く。
- 未実施、不合格、保留の理由や扱いは、`備考` に簡潔に書く。
