# Write System Test Design Docs

## Overview

要件定義書、外部設計書、内部設計書、静的テスト方針、単体テスト方針、結合テスト方針、既存資料を読み、日本のウォーターフォール開発向けの総合テスト設計書を作成する。既定の成果物は `docs/04_テスト/04_総合テスト/総合テスト方針.md` と、総合テスト分割単位に従って作成する `docs/04_テスト/04_総合テスト/テスト仕様・結果/〇〇テスト.md` のみとする。

## Workflow

### 1. 入力資料を読む

- 先に `AGENTS.md`、`docs/` 配下、README、既存のテスト関連資料、ユーザ指定メモを読む。
- 既存文書がある場合は、文体、用語、章番号、リンク表記、ID体系を合わせる。
- ユーザが文書名、保存先、章構成を指定している場合は、その指示を最優先にする。
- 先行するテスト方針がある場合は、静的テスト、単体テスト、結合テストと総合テストの境界、証跡方針、完了条件を読み取る。
- 画面設計、業務フロー、機能一覧、外部IF設計、処理設計がある場合は、総合テスト対象範囲、分割単位、観点の候補として整理する。

### 2. 確定事項と質問事項を分ける

- 設計書、ディレクトリ構成、既存テスト設定、README から一意に読み取れる内容は質問しない。
- 未確定事項は本文作成前にユーザへ質問し、合意してから本文へ反映する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 各質問では、確認が必要な理由、影響する章、未確定のまま進めた場合に曖昧になる内容を丁寧に説明する。
- 選択肢を提示する場合は、`など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- 選択肢は、採用時の影響が明確に異なるものだけにする。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 推奨選択肢は、リポジトリ事実、既存方針、ウォーターフォール文書としての安全性を基準に選ぶ。
- `references/question-checklist.md` を使い、特に次の項目は合意必須として扱う。
  - 総合テスト対象範囲
  - 総合テスト分割単位
  - 総合テスト観点
  - Playwright 利用方針
  - 手動テスト実施方針
  - テスト環境方針
  - 成果物・証跡方針
  - 完了条件
  - 作成対象ファイル

### 3. 本文作成直前に合意状況を再確認する

- 設計書本文を作成する直前に、全質問項目についてユーザ合意が取れているかを再確認する。
- 再確認では、各項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理する。
- 質問回答がすべて得られた場合、`references/question-checklist.md` にある全質問項目を一覧化してユーザへ表示する。
- 一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `根拠` には、その合意結果を本文へ反映してよいと判断した理由を書く。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイルや設定から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 未合意項目がある場合は、総合テスト設計書の本文作成へ進まない。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。
- 明確な作成指示がない場合は、合意結果一覧の確認後に待機し、本文作成を開始しない。

### 4. テンプレートを選ぶ

- 総合テスト方針の既定テンプレートは `assets/04_テスト/04_総合テスト/総合テスト方針.md` を使う。
- テスト仕様・結果の既定テンプレートは `assets/04_テスト/04_総合テスト/テスト仕様・結果/〇〇テスト.md` を使う。
- `〇〇テスト.md` は汎用ひな型であり、総合テスト分割単位に従って具体名へ複製して使う。成果物に `〇〇` を残さない。
- テスト観点の詳細化が必要な場合は `references/test-viewpoints.md` を読む。
- 章や本文の採否を迷う場合は、ユーザ指定と `references/review-checklist.md` の完成条件を優先する。

### 5. 総合テスト設計書へ落とす

- 文書は日本語で、ウォーターフォール開発向けの正式文書として断定形で書く。
- 差分説明、検討メモ、会話ログ、旧仕様の痕跡を書かない。
- `【template-guidance】` ブロック、`【質問】`、未置換プレースホルダを最終成果物に残さない。
- 総合テストは手動テストを基本とする。
- Playwright が利用できる環境では、Playwright で検査できる範囲をすべて Playwright で実施する方針とする。
- Playwright で検査しない範囲、検査できない範囲は、`総合テスト方針.md` の Playwright 利用方針に明記する。
- `総合テスト方針.md` には、`総合テスト対象外範囲`、`テストデータ方針`、`テスト仕様・結果ファイルの記述方針` を独立章として作らない。
- `テスト仕様・結果/〇〇テスト.md` には、テストケースとケースごとの実施結果を記載する。
- 静的テスト方針、単体テスト方針、結合テスト方針は作らない。

### 6. 整合確認する

- `references/review-checklist.md` で、文書範囲、質問合意、方針書章構成、テスト仕様・結果テンプレート、観点網羅、Playwright、手動実施、環境、成果物・証跡、完了条件、正式文書表現を確認する。
- `総合テスト方針.md` の分割単位と、作成する `テスト仕様・結果/〇〇テスト.md` の具体ファイル名が対応していることを確認する。
- 最終成果物は `docs/04_テスト/04_総合テスト/総合テスト方針.md` と、合意済みの `docs/04_テスト/04_総合テスト/テスト仕様・結果/〇〇テスト.md` だけにする。
