# 結合テスト観点整理

## 共通観点

- 層間連携: 画面、API、アプリケーション、ドメイン、永続化、外部IFの連携が設計どおりであること。
- IF契約: 呼出元、呼出先、入力、出力、エラーが設計された契約どおりであること。
- DB永続化: 登録、更新、削除、参照、制約違反時の扱いが設計どおりであること。
- トランザクション: 成功時の確定、失敗時のロールバック、部分失敗時の状態が設計どおりであること。
- 認証認可: 未認証、認証済み、権限不足、所有者違いの扱いが設計どおりであること。
- エラー伝播: 下位層、DB、外部IFの失敗が上位層へ設計どおりに変換されること。
- ログ: 成功、失敗、例外、監査対象操作で必要なログ出力が依頼されること。
- リトライ: リトライ対象、非対象、最大試行回数、最終失敗時の扱いが設計どおりであること。
- タイムアウト: タイムアウト時の中断、エラー変換、後続処理抑止が設計どおりであること。
- 処理設計の事前条件: 処理設計書で定義された開始条件を満たす場合に処理を開始できること。
- 処理設計の事後条件: 処理設計書で定義された成功保証または失敗時保証が満たされること。
- 処理設計の不変条件: 処理設計書で定義された処理中に破ってはならない条件が維持されること。

## API 結合

- URL、HTTPメソッド、リクエスト、レスポンス、ステータスコード、エラー形式を確認する。
- 認証状態、権限、所有者制約、入力検証、業務例外の変換を確認する。
- DB更新、DB参照、外部IF呼出、ログ出力の連携を確認する。

## 画面と API の結合

- 画面入力が API payload へ正しく変換されることを確認する。
- APIレスポンスが画面表示、状態、エラー表示へ正しく反映されることを確認する。
- 画面遷移、再読込、二重送信防止、通信失敗時の扱いを確認する。

## バッチと DB の結合

- 対象データ抽出、処理済み判定、登録、更新、削除、失敗時の状態を確認する。
- 再実行、部分失敗、ロールバック、処理順序、排他制御を確認する。

## 外部IF結合

- 外部IFへの送信内容、受信内容、応答検証、失敗時変換を確認する。
- 外部サービスを代替する場合は、fake、mock、stub の応答が契約に沿っていることを確認する。
- タイムアウト、リトライ、レート制限、認証失敗の扱いを確認する。

## データ結合

- ORM、SQL、migration、制約、初期データ、参照クエリの整合を確認する。
- 日付境界、権限境界、件数境界、重複、欠落、関連データ削除時の扱いを確認する。

## 対象外にする観点

- 本番相当負荷、長時間耐久、災害復旧、運用手順確認、実ブラウザ差分、課金を伴う実外部サービス確認は、結合テスト観点から除外する。
- 外部IFは fake、mock、stub、テスト用サンドボックスのいずれかで自動実行できる場合だけ結合テスト観点に含める。
- 自動テストコードで再現できない確認は、結合テスト方針本文の確認観点として書かない。
