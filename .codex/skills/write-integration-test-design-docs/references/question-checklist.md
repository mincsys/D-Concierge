# 結合テスト方針 質問チェックリスト

## 使い方

- リポジトリや既存文書から一意に分かる内容は質問しない。
- 本文作成前に、以下の未確定項目をユーザへ提示し、合意してから `結合テスト方針.md` へ反映する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 各質問では、確認が必要な理由、影響する章、未確定のまま進めた場合に曖昧になる内容を説明する。
- 選択肢を提示する場合は、`など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- 選択肢は、採用時の影響が明確に異なるものだけにする。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 推奨選択肢は、リポジトリ事実、既存方針、ウォーターフォール文書としての安全性を基準に選ぶ。
- 設計書本文を作成する直前に、全質問項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理し、未合意項目がある場合は本文作成へ進まない。
- 質問回答がすべて得られた場合、質問表にある全質問項目と合意結果を一覧化してユーザへ表示する。
- 合意結果一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `根拠` には、その合意結果を本文へ反映してよいと判断した理由を書く。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイルや設定から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで `結合テスト方針.md` の本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。
- 明確な作成指示がない場合は、合意結果一覧の確認後に待機し、本文作成を開始しない。

## 1. 結合テスト対象範囲

- どのシステム、サブシステム、機能、画面、API、バッチ、外部IFを結合テスト対象にするか。
- backend、frontend、DB、外部API、認証基盤、管理画面を含めるか。
- 単体テスト、E2Eテスト、総合テストとの境界をどう分けるか。

## 2. 結合テスト対象単位

- 何を結合テストの 1 単位とみなすか。
- API単位、画面操作単位、ユースケース単位、バッチ処理単位、外部IF単位のどれを採用するか。
- DB、外部サービス、認証、ストレージ、キューを実接続するか、fake、mock、stub で代替するか。

## 3. 結合テスト対象外範囲

- 自動テストコードで扱わない範囲は何か。
- 実外部サービス、本番相当データ、性能負荷、ブラウザ実機差分、運用手順確認を結合テスト方針の確認対象から除外するか。
- 対象外範囲を結合テスト方針本文の確認観点へ混ぜない方針でよいか。

## 4. 結合テスト観点

- 層間連携、IF契約、DB永続化、トランザクション、認証認可、エラー伝播、ログ、リトライ、タイムアウトをどう扱うか。
- 内部設計に処理設計書がある場合は、ユーザへ質問せず、各処理設計書の `事前条件`、`事後条件`、`不変条件` を結合テスト観点へ反映する。
- 処理設計書がない場合は、機能一覧、API一覧、画面一覧、業務フロー、内部IF設計のどれを結合観点の基準にするかを確認する。

## 5. 網羅基準

- 結合テストの網羅基準を何で定義するか。
- 機能一覧、API一覧、画面一覧、処理一覧、外部IF一覧、CRUD種別、権限種別のどれを主基準にするか。
- コードカバレッジを取得するか。
- コードカバレッジを取得する場合、主基準ではなく補助指標として扱うか。

## 6. テストデータ方針

- テストデータは seed、fixture、factory のどれで用意するか。
- テストごとに DB を初期化するか、シナリオ単位で共有するか。
- 外部API応答、画像、ファイル、時刻、認証状態をどう固定するか。
- テストデータに機微情報または本番データを含めない方針にするか。

## 7. テスト環境方針

- ローカル環境、共通開発環境、CI のどこで実行するか。
- DB、外部サービス、認証、ストレージ、メール、キューを実物にするか代替にするか。
- コンテナまたは一時DBを使うか。
- 自動実行に必要なサービス起動条件をどう定義するか。

## 8. テスト実施手順

- 結合テストをいつ実行するか。
- 機能追加時、API変更時、DB変更時、外部IF変更時に必須実行するか。
- 標準実行コマンドは何か。
- 失敗時のログ確認、DB状態確認、外部応答確認、再実行、データ初期化手順をどうするか。

## 9. 成果物・証跡方針

- 文書成果物は `結合テスト方針.md` のみとする。
- テストコードを正式成果物に含めるか。
- 実行ログ、JUnit XML、カバレッジ、DB初期化ログ、CI結果を保存するか。
- 証跡の保存場所はどこか。
- 証跡の命名規則は何か。
- 証跡の保存期間や世代管理を定めるか。
- 不具合修正時および再テスト時に証跡を更新するか。

## 10. 完了条件

- どの条件を満たしたら結合テスト完了とするか。
- 未確認項目、対象外項目、不具合残件がある場合の扱いをどうするか。
- 未確認項目または不具合残件がある状態で完了扱いにする場合、承認者と記録方法をどう定めるか。
