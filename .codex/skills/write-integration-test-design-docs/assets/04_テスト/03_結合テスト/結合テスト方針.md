# 結合テスト方針

## 1. 文書の目的
【template-guidance】<br>
- 本書が結合テストの方針、基準、実施ルールを定義する文書であることを書く。
- 個別テストケース一覧やテスト結果一覧ではないことが分かるようにする。

【/template-guidance】<br>

本書は、対象システムの結合テストにおける対象範囲、結合テスト対象単位、テスト観点、網羅基準、テストデータ、実施環境、実施手順および成果物・証跡の扱いを定義することを目的とする。

## 2. 前提
【template-guidance】<br>
- 参照する要件定義書、外部設計書、内部設計書、単体テスト方針、開発標準、ユーザ合意事項を書く。
- どのシステム、サブシステム、機能、画面、API、バッチ、外部IFを対象にするかはユーザ合意に基づいて書く。
- 実装手順や詳細な環境構築手順は書かない。

【/template-guidance】<br>

- 本書は、要件定義書、外部設計書、内部設計書、単体テスト方針および開発標準を前提とする。
- 本書の対象範囲は、ユーザと合意した結合テスト対象に限定する。
- 結合テストは、テストコードを作成して自動実行する。
- 本書に記載する確認対象は、自動テストコードで実行できるものに限定する。
- 文書成果物は `結合テスト方針.md` のみとする。

## 3. 結合テスト対象範囲
【template-guidance】<br>
- 結合テストで確認するシステム、サブシステム、機能、画面、API、バッチ、外部IFを書く。
- 単体テスト、E2Eテスト、総合テストとの境界が分かるようにする。

【/template-guidance】<br>

| 対象区分 | 対象範囲 | 確認目的 |
| --- | --- | --- |
| ユーザと合意した対象区分 | ユーザと合意した対象範囲 | ユーザと合意した確認目的 |

## 4. 結合テスト対象単位
【template-guidance】<br>
- 何を結合テストの 1 単位とみなすかを書く。
- API単位、画面操作単位、ユースケース単位、バッチ処理単位、外部IF単位のうち、ユーザ合意に基づく単位を書く。
- DB、外部サービス、認証、ストレージ、キューを実接続するか代替するかを書く。

【/template-guidance】<br>

| 区分 | 結合テスト対象単位 | 結合する要素 | 補足 |
| --- | --- | --- | --- |
| ユーザと合意した区分 | ユーザと合意した単位 | ユーザと合意した結合要素 | ユーザと合意した補足 |

## 5. 結合テスト対象外範囲
【template-guidance】<br>
- 結合テスト対象範囲で決めた範囲のうち、自動テストコードで扱わない内容のみを書く。
- 手動確認を前提にした確認内容、手動操作でしか再現できない確認内容、手動データ投入が必要な確認内容は、結合テストの確認観点として書かない。
- 対象外とする理由を書く。別工程での確認手順や手動確認手順は書かない。

【/template-guidance】<br>

| 対象外範囲 | 対象外理由 |
| --- | --- |
| ユーザと合意した対象外範囲 | ユーザと合意した理由 |

## 6. 結合テスト観点
【template-guidance】<br>
- 結合テストで必ず確認する共通観点を書く。
- 内部設計に処理設計書がある場合は、各処理設計書の `事前条件`、`事後条件`、`不変条件` を結合テスト観点へ必ず反映する。
- 処理設計書がない場合は、ユーザ合意済みの基準資料に基づいて観点を書く。
- 詳細化が必要な場合は `references/test-viewpoints.md` を参照する。

【/template-guidance】<br>

結合テストでは、少なくとも以下の観点を確認する。

| 観点 | 確認内容 |
| --- | --- |
| 層間連携 | 画面、API、アプリケーション、ドメイン、永続化、外部IFの連携が設計どおりであること |
| IF契約 | 呼出元、呼出先、入力、出力、エラーが設計された契約どおりであること |
| DB永続化 | 登録、更新、削除、参照、制約違反時の扱いが設計どおりであること |
| トランザクション | 成功時の確定、失敗時のロールバック、部分失敗時の状態が設計どおりであること |
| 認証認可 | 未認証、認証済み、権限不足、所有者違いの扱いが設計どおりであること |
| エラー伝播 | 下位層、DB、外部IFの失敗が上位層へ設計どおりに変換されること |
| ログ | 成功、失敗、例外、監査対象操作で必要なログ出力が依頼されること |
| リトライ | リトライ対象、非対象、最大試行回数、最終失敗時の扱いが設計どおりであること |
| タイムアウト | タイムアウト時の中断、エラー変換、後続処理抑止が設計どおりであること |
| 処理設計の事前条件 | 処理設計書で定義された開始条件を満たす場合に処理を開始できること |
| 処理設計の事後条件 | 処理設計書で定義された成功保証または失敗時保証が満たされること |
| 処理設計の不変条件 | 処理設計書で定義された処理中に破ってはならない条件が維持されること |

## 7. 網羅基準
【template-guidance】<br>
- 機能一覧、API一覧、画面一覧、処理一覧、外部IF一覧、CRUD種別、権限種別のうち、主基準として採用するものを書く。
- コードカバレッジは主基準ではなく補助指標として扱う。
- 網羅基準を満たせない対象がある場合の例外扱いを書く。

【/template-guidance】<br>

- 結合テストの網羅基準は、ユーザと合意した設計書または一覧資料を主基準とする。
- コードカバレッジを取得する場合は、結合テストの補助指標として扱う。
- 網羅基準未達の場合は、未達理由を確認し、必要なテストコードを追加して再実行する。

- 手動データ投入が必要な確認は結合テスト方針の対象に含めない。
- テストの再現性、独立性、データ削除、機微情報の扱いを書く。

## 10. テストコード記述方針
【template-guidance】<br>
- テストコードの配置、命名、docstring、fixture、setup/teardown、データ初期化の扱いを書く。
- `観点` と `確認` の docstring 記載はユーザへ質問せず必須とする。

【/template-guidance】<br>

- 結合テストは、テストコードを作成して実行する。
- テストコードは、結合テスト対象単位と結合する要素に対応する場所へ配置する。
- 1 つのテストでは、原則として 1 つの観点と確認内容を明確にする。
- すべての結合テストコードの docstring に以下を記載する。

```text
観点：確認対象の責務や振る舞いが分かる文章を記載する
確認：検証する結果を具体的に記載する
```

- テストごとに必要なデータを初期化し、別テストの実行順序に依存しないようにする。

## 11. テスト実施手順
【template-guidance】<br>
- 実施タイミング、標準実行コマンド、失敗時調査、再実行、データ初期化手順を書く。
- テスト実行コマンドはユーザ合意またはリポジトリで確認したコマンドに基づいて書く。

【/template-guidance】<br>

- 結合対象の機能、API、画面、処理、外部IFを作成または変更した場合は、関連する結合テストを実施する。
- 結合テスト失敗時は、ログ、DB状態、外部応答、入力データ、実行環境を確認する。
- 再実行時は、テストデータを初期化してから実行する。
- テスト実行コマンドは、ローカルまたはCIで自動実行できるユーザ合意済みのコマンドを使用する。

- テストコード、実行ログ、JUnit XML、カバレッジ、DB初期化ログ、CI結果、保存場所、命名、保存期間、再テスト時更新方針を書く。
- 文書成果物以外の証跡の扱いは必ずユーザに質問し、合意した内容だけを書く。

## 13. 完了条件
【template-guidance】<br>
- 対象範囲、網羅基準、証跡、未確認事項、不具合残件の扱いを書く。
- 何を満たしたら結合テスト完了とするかを断定形で書く。

【/template-guidance】<br>

- ユーザと合意した結合テスト対象範囲が確認されている。
- ユーザと合意した網羅基準を満たしている。
- 結合テストコードが作成され、実行結果が成功している。
- ユーザと合意した証跡が保存されている。
- 未確認事項、不具合残件、対象外事項がある場合は、扱いが明確になっている。

## 14. 留意事項
【template-guidance】<br>
- 実環境差分、外部サービス制約、総合テストへ委ねる事項、個別テスト設計で注意すべき事項を書く。
- 検討メモや会話ログは書かない。

【/template-guidance】<br>

- 結合テスト方針に記載しない個別テストケースは、後続のテスト設計またはテストコードで管理する。
