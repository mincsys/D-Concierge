# 内部設計 質問チェックリスト

## 使い方

- リポジトリ、既存文書、ユーザ指定から一意に分かる内容は質問しない。
- 内部設計では `docs/02_外部設計` を主資料として扱い、`docs/` 全体は整合確認と補助資料として読む。
- 既存の `src/` がある場合は補助資料として現状把握に使い、外部設計やユーザ合意と衝突する場合は衝突箇所ごとに質問する。
- 未確定事項は本文作成前にユーザへ提示し、合意してから内部設計書へ反映する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 各質問では、確認が必要な理由、影響する文書、未確定のまま進めた場合に曖昧になる内容を説明する。
- 選択肢を提示する場合は、採用時の影響が明確に異なるものだけにする。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 選択肢に `など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- 推奨選択肢は、外部設計書、既存内部設計、ユーザ指定、実装容易性、契約安全性を基準に選ぶ。
- 設計書本文を作成する直前に、全質問項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理し、未合意項目がある場合は本文作成へ進まない。
- 質問回答とリポジトリ事実の整理がそろった場合、質問表にある全質問項目と合意結果を一覧化してユーザへ表示する。
- 合意結果一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイル、設定、補助資料から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。

## 1. アーキテクチャ

- 実装単位を何で分けるか。レイヤ、モジュール、app、package、feature のどれを主軸にするか。
- どの責務をどの単位が所有するか。画面制御、業務ルール、永続化、外部通信、集計、認可をどう分けるか。
- 依存方向をどう固定するか。逆依存を許す境界、許さない境界をどこに置くか。
- 公開口を何にするか。同期 API、イベント、ジョブ、CLI、バッチ、共有ライブラリのどれでつなぐか。
- 状態をどこが所有するか。永続データ、セッション、キャッシュ、一時ファイル、メモリ保持をどこへ置くか。
- 純粋ロジックをどこへ寄せ、DB 書込、外部 API 呼出、ファイル操作、時刻取得、ID 発番などの副作用をどこへ隔離するか。
- 単体テストの対象単位を何にするか。ユーザ指定がなければ各クラスの公開メソッド単位を既定にする。
- 結合テストの対象単位を何にするか。ユーザ指定がなければ処理単位を既定にする。
- テスト容易性を成立させるために、どの依存を注入し、どの境界を fake / stub へ差し替え可能にするか。
- 結合テスト対象単位を成立させるために、どの境界を結合対象として残すか。
- 生成時刻、相関 ID、乱数、外部応答をどう抽象化するか。
- テスト方針自体は後工程で設計する前提とし、ここでは対象単位とその対象単位を成立させる設計上の境界だけを合意する。

- 中間ディレクトリと末端ディレクトリの責務をどう分けるか。
- app、module、layer、feature のどれをフォルダ構成の主軸にするか。
- 生成物ディレクトリ、キャッシュディレクトリ、ローカル専用ディレクトリを何として除外するか。
- テスト資材、fixture、共通契約、共通部品をどこへ置くか。
- 既存 `src/` と合意した構成が衝突する場合、どちらを正として直すか。

- 採用しない任意文書について、設計上の欠落や後工程の判断不能が残らないか。
- 採用する任意文書の一覧、入口文書、詳細文書の作成単位をどう定義するか。
