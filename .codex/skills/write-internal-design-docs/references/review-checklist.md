# 内部設計書レビュー観点

## 文書構成

- `アーキテクチャ設計` と `ディレクトリ構成` が必ずあるか。
- 任意文書はユーザ合意に基づいて採用されているか。
- `IF一覧.md` `処理一覧.md` `クラス一覧.md` を採用した場合、対応する詳細文書と整合しているか。
- assets 上の `汎用ひな型` が具体名へ置き換えられ、成果物へそのまま残っていないか。

## 外部設計との関係

- 外部設計のソフトウェア構成、画面仕様、IF 仕様、論理データの再掲になっていないか。
- 外部設計で確定している用語、ID、境界条件と矛盾していないか。
- 内部設計として追加で決めた実装判断だけが書かれているか。

## アーキテクチャ品質

- `カプセル化` `関心の分離` `契約による設計` `副作用の隔離` `テスト容易性` の 5 観点が反映されているか。
- レイヤまたはモジュールごとの責務分割が明確か。
- 構成要素一覧が `構成要素` `責務` の 2 列で整理され、最小構成要素を含めていないか。
- 構成要素一覧に、関連図へ出てくるすべての構成要素が載っているか。
- 構成要素一覧と各階層関連図で名称不一致がないか。
- 構成要素関連図に `第1階層関連図` があるか。
- 構成要素関連図で階層構造が `subgraph` で表現され、矢印には関係ラベルが付いているか。
- `第2階層関連図` 以降が階層深さごとに整理されているか。
- 各階層図でその深さの関係性が省略されていないか。
- `第N階層関連図` で、対象階層に至る親階層がルートから `subgraph` で残っているか。
- 依存方向、公開口、契約境界、副作用境界が関係性図から読めるか。
- 副作用のある処理が純粋ロジックと混在していないか。
- 単体テスト対象単位を成立させるための DI / 副作用隔離ができているか。
- 結合テスト対象単位を成立させるための境界設計ができているか。

## 契約の書き分け

- `クラス設計` の `事前条件 / 事後条件 / 不変条件` がクラス粒度になっているか。
- `処理設計` の `事前条件 / 事後条件 / 不変条件` が処理全体の粒度になっているか。
- `内部IF設計` で呼出元、呼出先、入出力、失敗条件が明確か。
- 例外処理と契約の説明が混ざっていないか。
- `クラス設計` の `事前条件` `事後条件` `不変条件` が型名やシグネチャ断片の羅列ではなく、日本語中心で書かれているか。
- `クラス設計` の `事前条件` `事後条件` `不変条件` が、実装やテストに落とし込める具体度で書かれているか。

## 詳細文書の整合

- 詳細設計ありのクラスで `詳細ファイル除外理由` が `-` になっているか。
- `クラス一覧.md` の `区分` がディレクトリパスになっているか。
- 詳細設計なしのクラスで `詳細ファイル除外理由` に除外理由が書かれているか。
- `クラス一覧.md` の `クラス名` リンクの有無と実際の詳細ファイル配置が一致しているか。
- クラス詳細文書がディレクトリ構成.md のソースコード階層と一致しているか。
- 同じクラスが複数の詳細ファイルへ二重記載されていないか。
- `〇〇クラス設計.md` が成果物では具体名と具体パスへ展開されているか。
- `IF一覧.md` と `処理一覧.md` の一覧粒度が詳細文書と一致しているか。
- `責務` `不変条件` `公開プロパティ` `公開メソッド` `公開イベント` の章が揃っているか。
- `公開メソッド` の `入力` `出力` が日本語で書かれているか。
- `公開メソッド` の `メソッド` 表記が冗長なシグネチャ例になっていないか。

## 表現

- 日本語の正式文書として自然な文体になっているか。
- 差分説明、旧仕様の痕跡、検討メモが残っていないか。
- `【template-guidance】` ブロックが最終成果物に残っていないか。
- `〇〇` などの未置換プレースホルダが残っていないか。
- 内部設計書本文にテスト方針、テスト対象単位、テストケース一覧が書かれていないか。
- ただし、テスト容易性を意識した責務分割、DI 境界、副作用隔離の記述は残っていてよいか。
