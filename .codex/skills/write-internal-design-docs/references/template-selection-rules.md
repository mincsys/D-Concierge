# 内部設計テンプレートの選定ルール

## 1. 基本方針

- テンプレートの実体は `assets/03_内部設計` を正とする。
- ユーザがフォルダ構成を指定している場合は、その構成へ必要テンプレートだけを当て込む。
- 文書の採用可否は `必須` `任意` `汎用ひな型` の区分で判断する。
- 内部設計では外部設計の要約や再掲をせず、内部実装の判断だけを書く。

## 2. 区分ごとの扱い

| 区分 | 扱い |
| --- | --- |
| 必須 | 原則として採用する。`アーキテクチャ設計` と `ディレクトリ構成` が該当する |
| 任意 | 要件、構成、ユーザ指定のいずれかで必要なときだけ採用する |
| 汎用ひな型 | assets 上に保持し、最終成果物へは具体名へ複製して採用する |

## 3. 必須文書

| 文書 | 採用理由 |
| --- | --- |
| `01_アーキテクチャ設計/アーキテクチャ設計.md` | 内部実装としての構成要素一覧と、読み手が理解しやすい粒度の関連図で、責務分割、関係性、契約境界、副作用境界を決めるため |
| `01_アーキテクチャ設計/ディレクトリ構成.md` | source-controlled な管理対象ディレクトリをルートから末端まで定義するため |

## 4. 任意文書の採用基準

| 文書 | 採用する条件 |
| --- | --- |
| `02_クラス設計/クラス一覧.md` | 非フレームワーク由来の主要クラスが複数あり、責務分割を明示したい |
| `02_クラス設計/〇〇クラス設計.md` | ソースコードディレクトリ構成に合わせて各クラスの詳細設計を起こしたい |
| `03_内部IF設計/IF一覧.md` | モジュール間 API、イベント、ジョブ境界など複数の内部IFを一覧管理したい |
| `03_内部IF設計/〇〇IF.md` | 個別の内部IF契約を明文化したい |
| `04_処理設計/処理一覧.md` | 起点ごとの処理を一覧で管理したい |
| `04_処理設計/〇〇処理設計.md` | 多段処理、例外、トランザクション、副作用を処理単位で明文化したい |
| `05_データ設計/物理データ設計.md` | DB、永続ストア、メッセージ保存領域など物理配置を決める必要がある |
| `06_共通設計/共通設計.md` | ログ、エラー、ID、時刻、認可、設定など横断ルールをまとめたい |
| `07_その他/補助資料.md` | 主文書だけでは誤解が出る補助資料が必要 |

## 5. 汎用ひな型の増やし方

- `〇〇クラス設計.md` は asset 上の唯一のクラス詳細テンプレートとし、成果物ではソースコードディレクトリ構成と同じ階層へ具体名で複製する。
- `〇〇IF.md` は、同一の内部境界や同一の公開口群をまとめて定義するときに増やす。
- `〇〇処理設計.md` は、処理一覧だけでは手順や副作用を表しきれない処理ごとに増やす。
- 汎用ひな型は最終成果物に残さず、具体名へ置き換えた後にのみ採用完了とみなす。

## 6. 採用時の注意

- `クラス設計` を採用する場合は `クラス一覧.md` を入口として先に作る。
- `クラス一覧.md` では、`クラス名` 列が Markdown リンクになっているクラスだけを詳細ファイルへ展開する。
- 詳細ファイルを作るクラスは、`詳細ファイル除外理由` 列に `-` を書く。
- 詳細ファイルを作らないクラスは、`詳細ファイル除外理由` 列に理由を簡潔に書く。
- クラス詳細文書は、ソースコードディレクトリ構成と同じ階層へ `〇〇クラス設計.md` として配置する。
- 同じクラスを複数の詳細ファイルへ重複記載しない。
- `内部IF設計` を採用する場合は `IF一覧.md` を先に作る。
- `処理設計` を採用する場合は `処理一覧.md` を先に作る。
- `04_環境構築` と独立した `テスト方針` 文書は作らない。
