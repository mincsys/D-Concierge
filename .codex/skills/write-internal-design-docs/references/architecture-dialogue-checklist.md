# アーキテクチャ対話チェックリスト

## 1. 使い方

- `アーキテクチャ設計` と `ディレクトリ構成` を決める前に、この観点で未確定事項を整理する。
- 明確に読み取れる事項は再確認せず、複数解釈がある論点だけを推奨案付きで提示する。

## 2. アーキテクチャ判断

- 実装単位は何で分けるか。レイヤ、モジュール、app、package、feature のどれを主軸にするか。
- どの責務をどの単位が所有するか。画面制御、業務ルール、永続化、外部通信、集計、認可などをどう分けるか。
- 依存方向をどう固定するか。逆依存を許す境界、許さない境界はどこか。
- 公開口は何か。同期 API、イベント、ジョブ、CLI、バッチ、共有ライブラリのどれでつなぐか。
- 状態をどこが所有するか。永続データ、セッション、キャッシュ、一時ファイル、メモリ保持をどこへ置くか。

## 3. 契約と副作用

- 境界をまたぐ入力と出力をどの型で表すか。
- `事前条件 / 事後条件 / 不変条件` をどの粒度で書くか。
- DB 書込、外部API 呼出、ファイル操作、時刻取得、ID 発番などの副作用をどこへ隔離するか。
- 例外をどこで正規化し、どこで呼出元へ返すか。
- 部分成功、再試行、冪等性、ロールバックをどの境界で扱うか。

## 4. テスト容易性

- 単体テストの対象単位は何か。ユーザ指定がなければ各クラスの公開メソッド単位を既定にする。
- 結合テストの対象単位は何か。ユーザ指定がなければ処理単位を既定にする。
- 純粋ロジックをどこへ寄せるか。
- 単体テスト対象単位を成立させるために、どの依存を注入するか。
- fake / stub へ差し替える境界はどこか。
- 結合テスト対象単位を成立させるために、どの境界を結合対象として残すか。
- 生成時刻、相関ID、乱数、外部応答をどう抽象化するか。
- テスト方針自体は後工程で設計する前提とし、ここでは対象単位とその対象単位を成立させる設計上の境界だけを合意する。

## 5. ディレクトリ構成

- source-controlled として管理するルート配下ディレクトリは何か。
- 中間ディレクトリと末端ディレクトリの責務をどう分けるか。
- app、module、layer、feature のどれをフォルダ構成の主軸にするか。
- 生成物ディレクトリ、キャッシュディレクトリ、ローカル専用ディレクトリを何として除外するか。
- テスト資材、fixture、共通契約、共通部品をどこへ置くか。
