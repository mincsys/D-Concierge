# Write Internal Design Docs

## Overview

外部設計書を主資料として、日本語の内部設計書群を既定では `docs/03_内部設計` 配下へ作成する。`assets/03_内部設計` にある実テンプレートを出発点にし、まずアーキテクチャ設計とディレクトリ構成をユーザと詳細に合意したうえで、必要な内部設計文書だけを複製、具体化、整合確認する。

## Workflow

### 1. 入力資料を読む

- 先に `docs/` 配下とリポジトリ内の指示文書を読む。
- 内部設計では `docs/02_外部設計` を主資料として扱う。
- 既存の `src/` がある場合は補助資料として現状把握に使い、外部設計やユーザ合意と衝突する箇所は固定せず質問対象にする。
- ユーザが指定したフォルダ構成、ファイル名、文書粒度を最優先で守る。
- 既存の内部設計書がある場合は、用語、ID体系、章立て、表記を合わせる。
- 内部設計では `docs/02_外部設計` の再掲をせず、実装判断だけを書く前提で読み分ける。

### 2. 明確に読み取れる確定事項を先に固定する

- 要件定義書、外部設計書、既存内部設計、ユーザ指定から一意に確定できる事項は、そのまま確定事項として扱う。
- 文書間で矛盾がなく、契約上も解釈の余地がない事項は再確認しない。
- 確定事項と未確定事項を混ぜず、先に確定事項の一覧を頭の中で整理する。
- 既存文書、設定、補助資料としての `src/` から一意に判断した事項は、後続の合意結果一覧で `リポジトリ事実として確定` として扱えるよう根拠を控える。

### 3. アーキテクチャ設計とディレクトリ構成を先に合意する

- 未確定事項は少なくとも `アーキテクチャ判断` `ディレクトリ構成判断` `契約確認` `異常系` `データ整合性` `文書採用可否` に分類する。
- `references/question-checklist.md` を使い、アーキテクチャ、ディレクトリ構成、任意文書採用、契約・副作用、作成対象ファイルを詳細観点まで確認する。
- `アーキテクチャ設計` と `ディレクトリ構成` は必須文書であり、この 2 つは推奨案、代替案、懸念、判断が必要な理由を付けてユーザへ提示する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 選択肢を提示する場合は、採用時の影響が明確に異なるものだけを Markdown の番号付きリストで示し、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ `1. 選択肢名（推奨）` のように明示し、推奨理由を簡潔に添える。
- 選択肢に `など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- この 2 文書が全面合意されるまで、他の内部設計本文は起票しない。
- `アーキテクチャ設計` では、外部設計のソフトウェア構成やシステム構成の再掲ではなく、内部実装としての構成要素一覧表と構成要素関連図を中心に、責務分割、関係性、公開口、副作用境界、契約境界、テスト容易性を扱う。
- アーキテクチャ相談では、単体テストと結合テストの対象単位もユーザと合意する。
- ユーザ指定がない場合の既定は、単体テストを「各クラスの公開メソッド単位」、結合テストを「処理単位」とする。
- この合意結果を踏まえて、依存性の注入、純粋ロジックと副作用の分離、差し替え境界の切り方を提案する。
- ただし、テスト方針やテスト対象単位は内部設計書本文には書かず、後工程のテスト方針設計へ委ねる。
- 構成要素一覧は上位要素の抜粋ではなく、定義した全階層を列挙する表として書く。
- 構成要素関連図は、`第1階層関連図` `第2階層関連図` `第3階層関連図` のように階層深さごとに分けて書く。
- 各関連図では `subgraph` による階層表現と、ラベル付き矢印による関係表現を使う。
- `第N階層関連図` では、対象階層だけでなく、ルートからその階層に至る親階層をすべて `subgraph` で含める。
- 構成要素一覧と関連図で使う構成要素名は完全一致させる。
- `ディレクトリ構成` では、source-controlled な管理対象ディレクトリをルートから末端まで省略なく定義する。

### 4. 任意文書の採用可否を決める

- `アーキテクチャ設計` と `ディレクトリ構成` 以外はすべて任意文書として扱う。
- `クラス設計` `内部IF設計` `処理設計` `物理データ設計` `共通設計` `補助資料` は、文書単位でユーザに作成有無を確認してから採用する。
- `クラス設計` を採用する場合は `クラス一覧.md` を入口にし、詳細設計は `〇〇クラス設計.md` を具体名へ複製して使う。
- 詳細文書は、ソースコードディレクトリ構成と同じ階層へ展開する。
- たとえばソースコード側で `backend/application/services/Foo.py` `backend/application/services/Bar.py` のような構成を採るなら、クラス設計側では `docs/03_内部設計/02_クラス設計/backend/application/services/Fooクラス設計.md` `Barクラス設計.md` のように置く。
- `クラス一覧.md` 自体は `docs/03_内部設計/02_クラス設計/` 直下へ置き、各クラス名から対応する詳細文書へリンクする。
- つまり、クラス設計の成果物構造は「入口文書としての `クラス一覧.md` 1 冊 + ソースコードディレクトリ構成に追随した複数の `〇〇クラス設計.md`」になる。
- `内部IF設計` を採用する場合は `IF一覧.md` を入口にし、`〇〇IF.md` を具体名へ複製して使う。
- `処理設計` を採用する場合は `処理一覧.md` を入口にし、`〇〇処理設計.md` を具体名へ複製して使う。

### 5. 本文作成直前に合意状況を再確認する

- 内部設計本文を作成する直前に、`references/question-checklist.md` の項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理する。
- 合意結果一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイル、設定、補助資料から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 未合意項目がある場合は、内部設計本文の作成へ進まない。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。

### 6. assets から必要テンプレートを選ぶ

- `assets/03_内部設計/テンプレート索引.md` を先に読む。
- `必須` は原則採用する。
- `任意` は要件、構成、ユーザ指定のいずれかで必要なときだけ採用する。
- `汎用ひな型` は最終成果物へそのまま置かず、具体名へ複製して使う。
- `〇〇クラス設計.md` は、成果物ではソースコードディレクトリ構成と同じ階層へ具体名で複製して使う。
- たとえば assets 上の `assets/03_内部設計/02_クラス設計/〇〇クラス設計.md` は 1 本だけだが、成果物では `backend/application/services/Fooクラス設計.md` のように具体名と具体パスへ複製して使う。
- どの文書を採るかの判断は [references/template-selection-rules.md](references/template-selection-rules.md) を参照する。

### 7. テンプレートを具体文書へ仕上げる

- 必要なテンプレートを既定では `docs/03_内部設計` 配下へ複製してから編集する。
- 文書先頭の `【template-guidance】` にある `章構成` で、番号付き見出しの `【必須】` と `【任意】` を確認する。
- 各章や下位見出し直下の `【template-guidance】` を読み、本文を要件と合意内容に合わせて具体化する。
- 最終成果物には `【template-guidance】` ブロック、未置換の `〇〇`、不要な任意章を残さない。
- クラス設計の `事前条件 / 事後条件 / 不変条件` は主要クラスの公開責務として守る契約を書く。
- クラス設計の `公開メソッド` の `入力` `出力` は、日本語で意味が分かる説明を書く。
- クラス設計の `事前条件 / 事後条件 / 不変条件` も、型名やシグネチャ断片ではなく極力日本語で意味が分かる説明を書く。
- クラス設計の `事前条件 / 事後条件 / 不変条件` は、抽象語だけで終えず実装やテストへ直接落とし込める具体度で書く。
- `クラス一覧.md` では、詳細設計を書くクラスの `クラス名` を詳細文書への Markdown リンクにし、`詳細ファイル除外理由` 列には `-` を書く。
- 詳細設計を書かないクラスは `クラス名` をリンクにせず、`詳細ファイル除外理由` 列に理由を書く。
- たとえば `クラス一覧.md` から `backend/application/services/Fooクラス設計.md` へリンクする場合は、`[Fooクラス](backend/application/services/Fooクラス設計.md)` のように書く。
- 処理設計の `事前条件 / 事後条件 / 不変条件` は処理全体としての開始条件、成功保証、処理中に破ってはならない条件を書く。

### 8. 文書間の整合を揃える

- アーキテクチャ設計、ディレクトリ構成、クラス設計、内部IF設計、処理設計、物理データ設計、共通設計で名称と粒度を揃える。
- 外部設計書と対応する機能、画面、IF、データ要素がある場合は、ID と用語を合わせる。
- クラス詳細ファイルに同じクラスを二重記載しない。
- クラス一覧と詳細文書で、ディレクトリパスとクラス名の表記を一致させる。
- 最後に [references/review-checklist.md](references/review-checklist.md) で横断確認する。

## Clarification Rules

- 明確に読み取れる事項は質問せず、そのまま確定事項として扱う。
- 未確定事項を推測だけで埋めず、契約として確認すべき論点は必ずユーザへ提示する。
- ユーザへの提示は、質問列挙ではなく論点整理を優先する。
- 合意前に行うのは、確定事項の整理、未確定論点の整理、推奨案の提示までとする。
- `アーキテクチャ設計` と `ディレクトリ構成` の合意前に、他文書の本文起票を始めない。
- 内部設計本文の起票は、必須文書と採用する任意文書に関する主要論点の合意後、合意結果一覧を提示し、ユーザから明確な作成指示を受けてから始める。
- 合意できていない前提を本文へ持ち込む場合は、ユーザが明示的にその進め方を指示したときに限る。

## Template Rules

- テンプレート本文は完全抽象で扱い、特定プロジェクト固有の技術名、ベンダ名、レイヤ名を固定しすぎない。
- 各テンプレートの `【template-guidance】` ブロックは次の規約で読む。
- 文書先頭のブロックは `【template-guidance】<br>` で始まり、`文書区分` `使う場面` `削除条件` `章構成` を記載し、`【/template-guidance】<br>` で閉じる。
- 章本文側のブロックは `【template-guidance】<br>` で始まり、その章や下位見出しで何を書くか、何と混ぜないか、どの文書を参照するかなどを自由文で記載し、`【/template-guidance】<br>` で閉じる。
- 章本文側のブロックは `##` `###` `####` など必要な任意の見出し階層の直下へ置ける。
- 親章に総論 guidance を置いてよいが、実際の記述指示はできるだけ最下位の実記述見出しへ寄せる。
- 文書先頭の `章構成` は番号付き見出しだけを Markdown リストで並べ、各項目に `【必須】` または `【任意】` を付ける。
- `章構成` のリスト項目には `<br>` を付けず、最後のリスト項目の直後に空行を 1 行入れてから `【/template-guidance】<br>` を置く。
- `章構成` の見出し番号は `1.` `4.1.` `4.2.1.` のように末尾ピリオド付きで書く。
- 文書先頭のブロックは章の採用要否を示し、各章のブロックはその章に何を書くかと何を書かないかを示す。
- `文書区分`: `必須` `任意` `汎用ひな型` のいずれか
- `使う場面`: そのテンプレートを採用する条件
- `削除条件`: 不採用時の扱い、または最終成果物からガイダンスブロックを外す条件
- `〇〇クラス設計.md` は assets 上の汎用ひな型であり、成果物ではソースコードディレクトリ構成と同じ階層へ具体名で複製する。
- クラス設計の配置は、`assets` 側でネスト済みディレクトリ雛形を持つのではなく、成果物生成時に `ディレクトリ構成.md` で合意したソースコードディレクトリ構成へ合わせて展開する。
- たとえば `docs/03_内部設計/02_クラス設計/` 配下では、`クラス一覧.md` は直下、`Fooクラス設計.md` のような詳細文書は `backend/application/services/` など対応ディレクトリ配下へ置く。

## Writing Rules

- 文書は日本語で書く。
- 各文書は原則として `文書の目的` `前提` を持たせ、以降は文書種別に応じた設計章を置く。
- 検討経緯、会話ログ、差分説明ではなく、完成済みの正式文書として書く。
- 外部設計のソフトウェア構成、画面仕様、IF 仕様、論理データ設計をそのまま写さず、内部実装として追加で決めた構成要素の責務分割、構成要素同士の関係、契約、副作用境界、永続化方針だけを書く。
- `04_環境構築` や独立した `テスト方針` 文書は作らない。
- テスト容易性を支える責務分割、DI 境界、副作用隔離は書いてよいが、テスト方針、テスト対象単位、テストケース一覧は本文へ書かない。
