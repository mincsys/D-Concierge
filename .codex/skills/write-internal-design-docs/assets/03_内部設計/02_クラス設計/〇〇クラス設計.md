# 〇〇クラス設計

## 1. 文書の目的
【template-guidance】<br>
- ここには、本書が単一クラスの詳細設計を書く文書であることを書く。
- 対象クラスの責務、不変条件、公開面を定義する文書であることが読めるようにする。
- 他クラスの詳細や利用手順には広げない。

【/template-guidance】<br>

本書は、〇〇クラスの責務、不変条件、公開プロパティ、公開メソッド、および公開イベントを定義することを目的とする。

## 2. 前提
【template-guidance】<br>
- ここには、`クラス一覧.md` に載せた対象クラスであることを書く。
- ソースコードディレクトリ構成と同じ階層へ配置する文書であることを必要に応じて補足してよい。
- 同じクラスを複数文書へ重複記載しないこともここで明記する。

【/template-guidance】<br>

- 本書の対象は `クラス一覧.md` で詳細設計対象とした〇〇クラスのみとする。
- 同じクラスを他のクラス設計文書へ重複記載しない。

## 3. 責務
【template-guidance】<br>
- そのクラスが担う責務だけを箇条書きで書く。
- 他クラスへ委ねる責務や、単なる利用手順は混ぜない。
- 責務が 1 つもないように見える場合は、そのクラスを詳細設計対象にすべきか再確認する。

【/template-guidance】<br>

- 〇〇

## 4. 不変条件
【template-guidance】<br>
- 処理中を通じて破ってはならない整合条件を書く。
- 事前条件や事後条件の言い換えではなく、常に守るべき性質を書く。
- 型名やシグネチャ断片の列挙ではなく、極力日本語で意味が分かる説明を書く。
- 実装やテストへそのまま落とし込めるレベルまで具体的に書く。
- 数式で表現した方が明確な場合は `$...$` を使ってよい。
- 不変条件がない場合は、この章は残し、本文に `-` を書く。

【/template-guidance】<br>

- 〇〇

- 公開プロパティがない場合は、この章ごと削除する。

## 6. 公開メソッド
【template-guidance】<br>
- 外部へ公開するメソッドだけを書く。
- private helper やフレームワーク規約どおりの暗黙メソッドは列挙しない。
- `入力` と `出力` は型名やシグネチャ断片ではなく、日本語で意味が分かる説明を書く。
- `入力` は「何を渡すか」を簡潔に書き、`〜を受け取る` のような冗長な文末にはしない。
- `出力` は「何が返るか / 何が結果になるか」を簡潔に書き、`〜が得られる` のような冗長な文末にはしない。
- `事前条件` は、そのメソッドを呼び出してよい前提条件を書く。入力値の妥当性、依存状態、呼出順の前提を書く。
- `事後条件` は、正常終了時に保証される結果だけを書く。戻り値、状態変化、通知、永続化結果などを書く。
- `事前条件` `事後条件` は型名やシグネチャ断片の列挙ではなく、極力日本語で意味が分かる説明を書く。
- `事前条件` は、テストで成立 / 不成立を判定できる具体性で書く。
- `事後条件` は、戻り値、状態変化、副作用の有無を検証できる具体性で書く。
- `事前条件` `事後条件` とも、抽象語だけで終えず実装判断やテスト条件へそのまま展開できる書き方にする。
- `事前条件` `事後条件` が複数ある場合は、1 セル内で `<br>` を使って複数行に並べる。
- Markdown 表のセル内で改行したい場合も `\n` ではなく `<br>` を使う。
- 数式で表現した方が明確な場合は `$...$` を使ってよい。
- `事前条件` `事後条件` が存在しない場合は `-` を書く。
- 公開メソッドがない場合は、この章ごと削除する。
