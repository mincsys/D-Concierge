# ディレクトリ構成

## 1. 文書の目的
【template-guidance】<br>
- ここには、本書が何のために全ディレクトリを定義するかを書く。
- ディレクトリ構成の記載対象は、原則としてソースコード、テストコード、実装に直接関係する設定資産・開発支援資産のディレクトリとする。
- 要件定義、外部設計、内部設計、設計メモ、運用メモなどは原則として記載対象外とする。
- ユーザが明示した場合だけ、実装に直接関係する設定資産や実行資産を例外的に含めてよい。
- ファイル単位の実装説明にはしない。

【/template-guidance】<br>

本書は、〇〇システムの内部設計における管理対象ディレクトリを、ルートから末端まで省略なく定義することを目的とする。

## 2. 前提
【template-guidance】<br>
- ここでは、source-controlled な対象だけを書くこと、ソースコード系ディレクトリを基本対象にすること、管理対象外の生成物をどこまで除外するかを明記する。
- `.venv/` `node_modules/` `dist/` `coverage/` などの生成物やローカル専用ディレクトリはここで対象外として扱う。
- 設計書やメモを対象外にする場合は、ここで記載対象にしない方針を明記する。
- 開発環境構築手順は書かない。

【/template-guidance】<br>

- 本書に記載するのは source-controlled な管理対象ディレクトリのみとする。
- 記載対象は、原則としてソースコード、テストコード、実装に直接関係する設定資産・開発支援資産のディレクトリとする。
- 要件定義、外部設計、内部設計、設計メモ、運用メモなどは記載対象外とする。
- 生成物ディレクトリ、キャッシュディレクトリ、ローカル一時ディレクトリは管理対象外とする。
- 代表例として `.venv/` `node_modules/` `dist/` `coverage/` を含む。

## 3. 管理対象ディレクトリツリー
【template-guidance】<br>
- ここでは、対象範囲内のディレクトリをルートから末端まで省略なく列挙する。
- 各ディレクトリ行の右側に `# ...` で責務を書く。
- `#` の位置は同じ列に揃える。
- 途中階層の省略記法や、存在しない想定ディレクトリは書かない。
- 責務説明を別表として二重管理しない。

【/template-guidance】<br>

```text
.
└── src/              # 〇〇システムのソースコード
    ├── app/          # 〇〇
    │   └── 〇〇/       # 〇〇
    └── tests/        # テストコード
```

## 4. 命名と所有ルール
【template-guidance】<br>
- ここには、ディレクトリ命名規則、層 / モジュールの並べ順、テスト配置規則などを必要に応じて書く。
- 実装詳細や個別ファイル名の羅列にはしない。
