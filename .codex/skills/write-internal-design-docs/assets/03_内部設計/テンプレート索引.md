# テンプレート索引

## 1. 使い方

- この索引を起点に、要件定義書、外部設計書、ユーザ指定に合わせて必要なテンプレートだけを選ぶ。
- `必須` は原則採用する。
- `任意` は該当条件があるときだけ採用する。
- `汎用ひな型` は具体名へ複製して使い、そのまま成果物へ残さない。
- 内部設計では外部設計の再掲を避け、内部実装として追加で決める責務分割、構成要素同士の関係、契約、副作用境界、物理配置だけを書く。

## 2. テンプレート一覧

| 文書パス | 区分 | 採用条件 | 関連文書 |
| --- | --- | --- | --- |
| `01_アーキテクチャ設計/アーキテクチャ設計.md` | 必須 | 内部実装としての構成要素一覧と、読み手が理解しやすい粒度の関連図で責務分割と関係性を定義するとき | `01_アーキテクチャ設計/ディレクトリ構成.md` |
| `01_アーキテクチャ設計/ディレクトリ構成.md` | 必須 | source-controlled な全ディレクトリを定義するとき | `01_アーキテクチャ設計/アーキテクチャ設計.md` |
| `02_クラス設計/クラス一覧.md` | 任意 | 非フレームワーク由来の主要クラスを一覧化し、詳細文書の配置を決めるとき | `02_クラス設計/〇〇クラス設計.md` |
| `02_クラス設計/〇〇クラス設計.md` | 汎用ひな型 | ソースコードディレクトリ構成に合わせて各クラスの詳細設計を起こすとき | `02_クラス設計/クラス一覧.md` |

| `04_処理設計/処理一覧.md` | 任意 | 主要処理を一覧で管理するとき | `04_処理設計/〇〇処理設計.md` |
| `04_処理設計/〇〇処理設計.md` | 汎用ひな型 | 個別処理の手順、副作用、例外を定義するとき | `04_処理設計/処理一覧.md` |
| `05_データ設計/物理データ設計.md` | 任意 | DB や永続ストアの物理設計が必要なとき | 04_処理設計、06_共通設計 |

| `07_その他/補助資料.md` | 任意 | 主文書だけでは誤解が出る補助資料が必要なとき | 関連する主文書 |

## 3. 採用時の注意

- `【template-guidance】` ブロックは作業ガイドであり、最終成果物では削除する。
- `〇〇` などのプレースホルダは必ず置換する。
- `〇〇クラス設計.md` は成果物へそのまま残さず、ソースコードディレクトリ構成と同じ階層へ具体名で複製する。
- `IF一覧.md` `処理一覧.md` `クラス一覧.md` を採用した場合は、対応する詳細文書と整合させる。
