# 単体テスト方針レビュー観点

## 文書範囲

- 成果物が `docs/04_テスト/02_単体テスト/単体テスト方針.md` の 1 ファイルに限定されているか。
- 単体テストケース一覧、テスト結果一覧、結合テスト方針が混在していないか。
- 日本のウォーターフォール開発向けの正式文書として読めるか。

## 質問合意

- `【質問】` に相当する未確定項目が本文に残っていないか。
- 質問が 1 つずつ行われ、合意内容が整理されているか。
- 選択肢に `など`、`等`、`その他いろいろ` のような曖昧な表現が使われていないか。
- 選択肢が `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のような番号付きリストになっているか。
- ユーザが番号だけで回答できることを質問文で明記しているか。
- 選択肢番号が質問ごとに `1` から始まっているか。
- 推奨選択肢が `1. 選択肢名（推奨）` のように 1 つだけ明示されているか。
- 推奨理由が、選択肢本文または質問説明の中で簡潔に説明されているか。
- 複数の選択肢が同時に推奨扱いになっていないか。
- 本文作成直前に、全質問項目が `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで確認されているか。
- 本文作成直前に、質問表の全質問項目と合意結果がユーザへ表示されているか。
- 合意結果一覧に `質問項目`、`合意結果`、`状態`、`根拠` が含まれているか。
- `根拠` に、合意結果を本文へ反映してよいと判断した理由が書かれているか。
- `合意済み` の根拠に、ユーザがどの回答で合意したかが書かれているか。
- `リポジトリ事実として確定` の根拠に、どのファイルや設定から一意に判断したかが書かれているか。
- `未合意` の根拠に、何が未回答または未確定かが書かれているか。
- 合意結果一覧の表示後、ユーザから明確な作成指示を受けてから本文作成へ進んでいるか。
- `確認しました`、`OKです`、`問題ありません` だけで本文作成へ進んでいないか。
- 明確な作成指示がない場合に、合意結果一覧の確認後も本文作成を開始せず待機しているか。
- テスト対象範囲、テスト対象単位、テスト対象外範囲がユーザ合意に基づいているか。
- カバレッジ目標値、利用ツール、実行コマンドがユーザ合意またはリポジトリ事実に基づいているか。
- 成果物・証跡方針が独立章としてあり、ユーザ合意に基づいているか。

## 観点網羅

- 正常系、異常系、境界値、分岐、条件組合せ、例外処理、戻り値が含まれているか。
- 業務ロジック、入力値検証、データ変換、計算処理など、対象システムに必要な追加観点が含まれているか。
- 副作用境界の扱いが単体テストと結合テストで切り分けられているか。
- `docs/03_内部設計/02_クラス設計/` がある場合、クラス設計書を読み取っているか。
- 各クラス設計書の `事前条件`、`事後条件`、`不変条件` が単体テスト観点へ反映されているか。
- 手動確認を前提にした観点が混在していないか。
- 自動テストコードで実行できる確認だけが単体テスト観点として書かれているか。

## 網羅基準

- 分岐網羅を基本とする方針が書かれているか。
- カバレッジ目標値と対象指標が明確か。
- 目標未達時の対応と例外扱いが明確か。

## 環境・コマンド

- ローカル、共通開発環境、CI の扱いが明確か。
- テストフレームワーク、モックライブラリ、カバレッジ取得ツール、レポート出力方法が明確か。
- テスト実行コマンドが実行可能な形で書かれているか。

## TDD 方針

- リポジトリ規約に従い、Red、Green、Refactor の順序が明記されているか。
- ソースコード変更時のテスト実施タイミングが明確か。
- カバレッジ未達時のテスト追加と再実行が明確か。

## 成果物・証跡方針

- 正式成果物に含めるものが明確か。
- テスト実行ログ、カバレッジレポート、CI 実行結果の扱いが明確か。
- 証跡の保存場所、命名、再テスト時の更新方針が明確か。

## 表現

- 文書が断定形で書かれているか。
- 差分説明、検討メモ、会話ログ、旧仕様の痕跡が残っていないか。
- `【template-guidance】` ブロック、`【質問】`、未置換プレースホルダが最終成果物に残っていないか。
- 既存 docs と用語、文体、リンク表記がそろっているか。
