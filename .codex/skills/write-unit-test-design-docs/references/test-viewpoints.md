# 単体テスト観点整理

## 共通観点

- 正常系: 代表的な入力で期待する戻り値、状態、出力になること。
- 異常系: 不正入力、前提不成立、依存先失敗時に定義どおりの例外またはエラー結果になること。
- 境界値: 最小値、最大値、空、閾値の前後、日付境界、件数境界を確認すること。
- 分岐: 条件分岐ごとの処理経路を確認すること。
- 条件組合せ: 複数条件で結果が変わる場合に、代表的な組合せを確認すること。
- 例外処理: 例外発生時の戻り値、後続処理抑止、再送、ログ、変換結果を確認すること。
- 戻り値: 値、型、構造、順序、欠落時の扱いを確認すること。
- 事前条件: 対象単位を呼び出す前に満たすべき入力、状態、依存先条件を確認すること。
- 事後条件: 正常終了または異常終了後に保証すべき戻り値、状態、出力、副作用依頼を確認すること。
- 不変条件: 処理前後や分岐をまたいで維持すべき業務制約、内部状態、データ整合性を確認すること。
- クラス設計の事前条件: クラス設計書で定義された公開責務の開始条件を満たす場合に対象単位を呼び出せること。
- クラス設計の事後条件: クラス設計書で定義された公開責務の成功保証または失敗時保証が満たされること。
- クラス設計の不変条件: クラス設計書で定義された公開責務として維持すべき条件が処理前後で保たれること。
- 自動実行可能性: 自動テストコードで再現できる確認だけを単体テスト観点として扱うこと。

## 業務ロジック

- 業務ルール、権限制約、状態遷移、保存可否を確認する。
- 利用者権限、所有者制約、ステータス遷移、保存可否を確認する。
- 仕様上の推定値、概算値、丸め、欠落値の扱いを確認する。

## 入力値検証

- 必須、形式、桁数、範囲、列挙値、重複、相関チェックを確認する。
- エラー時のメッセージ識別子、フィールド対応、処理中断を確認する。
- 入力正規化がある場合は、前後空白、大小文字、日付形式、数値形式を確認する。

## データ変換

- DTO、read model、domain object、ORM model、API payload の相互変換を確認する。
- 欠落、null、空配列、順序、単位変換、タイムゾーン変換を確認する。
- 外部 API 応答の構造化検証では、必須項目欠落と型不一致を確認する。

## 計算処理

- 合計、平均、丸め、単位換算、期間集計、栄養素別集計を確認する。
- 0、負数、小数、上限値、精度誤差の扱いを確認する。
- 日次、週次、月跨ぎ、タイムゾーン境界がある場合は境界値として確認する。

## 排他制御

- 単体テストでは、ロックやトランザクションそのものより、排他判断を行う純粋ロジックを確認する。
- 実 DB の競合、デッドロック、トランザクション分離レベルは結合テストまたは別工程へ委ねる。

## リトライ処理

- リトライ対象例外、非対象例外、最大試行回数、待機方針、最終失敗時の結果を確認する。
- 実際の待機時間は差し替え、単体テストでは判断と呼出回数を確認する。

## 非同期処理

- 状態遷移、完了時処理、失敗時処理、キャンセル、タイムアウトを確認する。
- タイマー、キュー、バックグラウンド実行基盤は差し替え、単体テストでは制御ロジックを確認する。

## UI 部品

- props、初期表示、条件表示、入力イベント、無効状態、エラー表示を確認する。
- 画面遷移、API 通信、ブラウザ実機差分は、必要に応じて結合テストまたは E2E テストへ委ねる。

## API 境界

- 単体テストでは、入力検証、認証状態の分岐、ユースケース呼出、レスポンス変換を確認する。
- ルーティング、ミドルウェア、実 DB を含む HTTP 経由確認は結合テストへ委ねる。

## 副作用境界

- DB、外部 API、ファイル、画像変換、時刻、ログ、乱数は差し替え可能にする。
- 単体テストでは、差し替えた依存先への依頼内容、呼出有無、失敗時の扱いを確認する。
- 自動テストコードで再現できない確認は、単体テスト方針本文の確認観点として書かない。
