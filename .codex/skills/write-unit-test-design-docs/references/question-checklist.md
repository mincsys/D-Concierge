# 単体テスト方針 質問チェックリスト

## 使い方

- リポジトリや既存文書から一意に分かる内容は質問しない。
- 本文作成前に、以下の未確定項目をユーザへ提示し、合意してから `単体テスト方針.md` へ反映する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 各質問では、確認が必要な理由、影響する章、未確定のまま進めた場合に曖昧になる内容を説明する。
- 選択肢を提示する場合は、`など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- 選択肢は、採用時の影響が明確に異なるものだけにする。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 推奨選択肢は、リポジトリ事実、既存方針、ウォーターフォール文書としての安全性を基準に選ぶ。
- 設計書本文を作成する直前に、全質問項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理し、未合意項目がある場合は本文作成へ進まない。
- 質問回答がすべて得られた場合、質問表にある全質問項目と合意結果を一覧化してユーザへ表示する。
- 合意結果一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `根拠` には、その合意結果を本文へ反映してよいと判断した理由を書く。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイルや設定から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで `単体テスト方針.md` の本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。
- 明確な作成指示がない場合は、合意結果一覧の確認後に待機し、本文作成を開始しない。

## 1. 対象範囲

- どのシステム、サブシステム、機能、モジュールを単体テスト方針の対象にするか。
- backend、frontend、共通モジュール、モック、管理画面を含めるか。
- 初期リリース対象外や後続工程へ委ねる範囲があるか。

## 2. テスト対象単位

- 何を単体テストの 1 単位とみなすか。
- 関数、メソッド、クラス、モジュール、コンポーネント、hook、バッチ処理、API 単位のいずれを採用するか。
- アーキテクチャ層ごとに単位を変えるか。
- 副作用を持つ処理は、依存先を差し替えて単体テストするか、結合テストへ委ねるか。

## 3. テスト対象外範囲

- テスト対象単位で決めた範囲の中で単体テスト対象外とするケースは何か。
- 外部 API、DB、ファイル、ブラウザ実機などをどう扱うか。
- 自動テストコードで実行できない確認を、単体テスト方針本文の確認観点へ混ぜない方針でよいか。
- 対象外理由を証跡や設計書へ残す必要があるか。

## 4. 網羅基準

- 分岐網羅を採用するか。
- カバレッジ目標値を何%にするか。
- 行、分岐、関数、ステートメントなど、どの指標を採用するか。
- 目標未達を許容する例外条件と承認方法をどうするか。

## 5. テスト環境方針

- 単体テストを実行する環境は何か。
- ローカル環境、共通開発環境、継続的インテグレーション環境の役割分担は何か。
- テストフレームワークは何を使うか。
- モックライブラリは何を使うか。
- カバレッジ取得ツールは何を使うか。
- レポート出力方法は何を使うか。

## 6. テスト実施手順

- テストコードをいつ作成するか。
- ソースコード変更時にどの範囲のテストを実行するか。
- カバレッジ目標未達時にどのように再実施するか。
- 標準のテスト実行コマンドは何か。
- CI で実行するコマンドとローカルで実行するコマンドを分けるか。

## 7. 成果物・証跡方針

- 正式成果物に含めるものは何か。
- テストコードを正式成果物に含めるか。
- テスト実行ログを保存するか。
- カバレッジレポートを保存するか。
- CI 実行結果を証跡とするか。
- 証跡の保存場所はどこか。
- 証跡の命名規則は何か。
- 証跡の保存期間や世代管理を定めるか。
- 不具合修正時および再テスト時に証跡を更新するか。

## 8. 文書完成条件

- `単体テスト方針.md` 以外の成果物を作らない前提でよいか。
- 個別テストケース一覧、テスト結果一覧、結合テスト方針を対象外としてよいか。
- `【質問】` に相当する未合意事項が本文に残っていないか。
