# 単体テスト方針

## 1. 文書の目的
【template-guidance】<br>
- 本書が単体テストの方針、基準、実施ルールを定義する文書であることを書く。
- 個別テストケース一覧やテスト結果一覧ではないことが分かるようにする。

【/template-guidance】<br>

本書は、対象システムの単体テストにおける対象範囲、テスト対象単位、テスト観点、網羅基準、実施環境、実施手順および成果物・証跡の扱いを定義することを目的とする。

## 2. 前提
【template-guidance】<br>
- 参照する要件定義書、外部設計書、内部設計書、開発標準、ユーザ合意事項を書く。
- どのシステム、サブシステム、機能、モジュールを対象にするかはユーザ合意に基づいて書く。
- 実装手順や詳細な環境構築手順は書かない。

【/template-guidance】<br>

- 本書は、要件定義書、外部設計書、内部設計書および開発標準を前提とする。
- 本書の対象範囲は、ユーザと合意した単体テスト対象に限定する。
- 単体テストは、実装変更時に TDD の Red、Green、Refactor の順序で作成および実行する。

## 3. テスト対象単位
【template-guidance】<br>
- 何を単体テストの 1 単位とみなすかを書く。
- 関数、メソッド、クラス、モジュール、バッチ処理、API などから、システム構成に合う単位をユーザ合意に基づいて定義する。
- 言語、フレームワーク、アーキテクチャ層によって単位を分ける場合は、層ごとに書く。

【/template-guidance】<br>

| 区分 | テスト対象単位 | 補足 |
| --- | --- | --- |
| アプリケーション層 | ユーザと合意した単位 | 依存先を差し替え、対象単位の分岐、戻り値、例外を確認する |
| ドメイン層 | ユーザと合意した単位 | 純粋ロジック、不変条件、境界値を確認する |
| インフラストラクチャ層 | ユーザと合意した単位 | 外部副作用を直接伴う確認は、必要に応じて結合テストへ委ねる |
| フロントエンド | ユーザと合意した単位 | 表示、入力、イベント、状態変化を確認する |

## 4. テスト対象外範囲
【template-guidance】<br>
- 単体テスト対象単位で決めた範囲のうち、例外的に対象外にする内容のみを記述する。
- 単体テスト対象外とするケース、実施困難な箇所、自動テストできない箇所の代替策を書く。
- 対象外とする理由と、どの工程や手段で確認するかを合わせて書く。

【/template-guidance】<br>

| 対象外範囲 | 対象外理由 | 代替確認方法 |
| --- | --- | --- |
| ユーザと合意した対象外範囲 | ユーザと合意した理由 | ユーザと合意した代替確認方法 |

## 5. テスト観点
【template-guidance】<br>
- 単体テストで必ず確認する共通観点を書く。
- 機能種別ごとの追加観点は `references/test-viewpoints.md` を参考に、対象システムに必要なものだけを書く。

【/template-guidance】<br>

単体テストでは、少なくとも以下の観点を確認する。

| 観点 | 確認内容 |
| --- | --- |
| 正常系 | 想定される入力に対して、期待する戻り値、状態、出力が得られること |
| 異常系 | 不正な入力、前提不成立、依存先失敗に対して、定義された例外またはエラー結果となること |
| 境界値 | 最小値、最大値、空値、閾値の前後で期待どおりに判定されること |
| 分岐 | 条件分岐ごとに期待する処理経路が実行されること |
| 条件組合せ | 複数条件の組合せにより結果が変わる場合に、代表的な組合せを確認すること |
| 例外処理 | 例外発生時の戻り値、再送、ログ、後続処理抑止が方針どおりであること |
| 戻り値 | 戻り値の値、型、構造、順序、欠落時の扱いが設計どおりであること |
| 事前条件 | 対象単位を呼び出す前に満たすべき入力、状態、依存先条件が設計どおりであること |
| 事後条件 | 正常終了または異常終了後に保証すべき戻り値、状態、出力、副作用依頼が設計どおりであること |
| 不変条件 | 処理前後や分岐をまたいで維持すべき業務制約、内部状態、データ整合性が保たれること |

## 6. 網羅基準
【template-guidance】<br>
- 分岐網羅を採用する方針を書く。
- カバレッジ目標値はユーザ合意に基づいて書く。
- カバレッジ目標を満たせない対象がある場合の例外扱いを書く。

【/template-guidance】<br>

- 単体テストの網羅基準は分岐網羅を基本とする。
- カバレッジ目標値は、ユーザと合意した値を適用する。
- カバレッジ目標未達の場合は、未達理由を確認し、必要なテストコードを追加して再実行する。

## 7. テスト環境方針
【template-guidance】<br>
- ローカル環境、共通開発環境、継続的インテグレーション環境の扱いを書く。
- テストフレームワーク、モックライブラリ、カバレッジ取得ツール、レポート出力方法をユーザ合意に基づいて書く。

【/template-guidance】<br>

| 項目 | 方針 |
| --- | --- |
| 実行環境 | ユーザと合意した環境 |
| テストフレームワーク | ユーザと合意したフレームワーク |
| モックライブラリ | ユーザと合意したライブラリ |
| カバレッジ取得ツール | ユーザと合意したツール |
| レポート出力方法 | ユーザと合意した方法 |

## 8. テストコード記述方針
【template-guidance】<br>
- テストコードの命名、配置、docstring、モック、テストデータの扱いを書く。
- リポジトリ規約に TDD がある場合は、Red、Green、Refactor の順序と矛盾しないようにする。

【/template-guidance】<br>

- テストコードは、対象コードの責務とテスト対象単位に対応する場所へ配置する。
- 1 つのテストでは、原則として 1 つの観点と確認内容を明確にする。
- すべてのテストコードの docstring に以下を記載する。

```text
観点：確認対象の責務や振る舞いが分かる文章を記載する
確認：検証する結果を具体的に記載する
```

- 外部 API、DB、ファイル、時刻、乱数などの副作用は、単体テストでは原則として差し替える。
- テストデータは、確認内容が分かる最小限のデータとする。

## 9. テスト実施手順
【template-guidance】<br>
- テストコード作成タイミング、テスト実施タイミング、再実施条件、テスト実行コマンドを書く。
- テスト実行コマンドはユーザ合意またはリポジトリで確認したコマンドに基づいて書く。

【/template-guidance】<br>

- テストコードは、実装対象の変更前に失敗するテストを作成し、Red、Green、Refactor の順に整備する。
- ソースコードを作成または編集した場合は、関連する単体テストを実施する。
- カバレッジ目標未達時は、テストコードを追加して再実施する。
- テスト実行コマンドは、ユーザと合意したコマンドを使用する。

## 10. 成果物・証跡方針
【template-guidance】<br>
- 正式成果物に含めるもの、テスト実行ログ、カバレッジレポート、CI 実行結果、保存場所、命名、再テスト時の更新方針を書く。
- この章は必ずユーザに質問し、合意した内容だけを書く。

【/template-guidance】<br>

| 項目 | 方針 |
| --- | --- |
| 正式成果物 | ユーザと合意した成果物 |
| テスト実行ログ | ユーザと合意した扱い |
| カバレッジレポート | ユーザと合意した扱い |
| CI 実行結果 | ユーザと合意した扱い |
| 証跡の保存場所 | ユーザと合意した保存場所 |
| 証跡の命名 | ユーザと合意した命名 |
| 再テスト時の更新方針 | ユーザと合意した更新方針 |

## 11. 留意事項
【template-guidance】<br>
- 制約上の注意、後続工程に委ねる事項、個別テスト設計で注意すべき事項を書く。
- 検討メモや会話ログは書かない。

【/template-guidance】<br>

- 単体テスト方針に記載しない個別テストケースは、後続のテスト設計またはテストコードで管理する。
