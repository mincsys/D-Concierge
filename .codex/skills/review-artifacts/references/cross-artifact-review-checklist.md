# Cross Artifact Review Checklist

## 追跡性

- [ ] 要件から設計、実装、テスト、証跡までの追跡が途切れていないか。
- [ ] ID が定義されている成果物では、機能ID、画面ID、IFID、テストケースIDなどを使って追跡できるか。ID が定義されていない成果物では、名称、見出し、ファイル名、関連文書名で一貫して追跡できるか。
- [ ] 要件や設計に対応する実装、テスト、証跡が欠けていないか。
- [ ] 実装、テスト、証跡だけに存在し、要件や設計に根拠を持たない仕様、機能、例外、判定がないか。

## 成果物間の同期

- [ ] docs、src、tests、evidence のどれか一方だけが更新され、他が古いままになっていないか。
- [ ] 同じ名称、ID、状態、エラー、設定値、判定語が成果物間で別の意味に使われていないか。
- [ ] 設定ファイル、環境構築資材、CI、README、開発標準が、実装やテストで要求する実行方法と矛盾していないか。
- [ ] テスト結果や証跡が、対象成果物の最新状態ではなく古い状態を根拠にしていないか。

## 判定と保留

- [ ] `未定`、`TBD`、`TBC`、`保留`、`OK` などの判定語が、成果物間で正式な意味定義なしに使われていないか。
- [ ] 未実施、部分確認、保留、不合格が、別成果物で合格または完了として扱われていないか。
- [ ] 証跡、テスト仕様、実装、文書のいずれかだけで完了扱いになっていないか。

## 修正方針の判断

- [ ] 修正方針が特定成果物だけに偏らず、正本として直すべき成果物を根拠から選べているか。
- [ ] 複数成果物の修正が必要な場合は、どの順序で直すべきかが分かる方針になっているか。
- [ ] 判断にユーザ合意が必要な場合は、判断が必要な論点と、判断後に更新する成果物を具体的に書けるか。
