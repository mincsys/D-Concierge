# Reading Order

## 基本順序

1. `AGENTS.md` または同等のプロジェクト指示を読む。
2. ユーザが指定したレビュー対象を読む。
3. レビュー対象の前提になる要件、設計、テスト方針、開発標準、設定を読む。
4. レビュー対象を作成した可能性が高い `.codex/skills/` 配下の `SKILL.md` を読む。
5. 文書成果物では、対応する作成元スキルの `references/review-checklist.md` を読む。
6. 質問合意、開始ゲート、未合意事項の残存を確認する必要がある場合だけ、対応する作成元スキルの `references/question-checklist.md` を読む。
7. 実際の指摘保存先を読み、同一趣旨の指摘がないか確認する。指摘保存先の指定がない場合は `.issue/review/` を読む。

## 対象別の追加読込

- `docs/` をレビューする場合は、前工程と後工程の文書を読む。
- `src/` をレビューする場合は、対応する要件、外部設計、内部設計、テスト方針、関連テストを読む。
- テストコードをレビューする場合は、対応するテスト方針、対象実装、証跡を読む。
- 証跡をレビューする場合は、対応するテスト方針、テスト仕様、実行コマンド、対象ケースを読む。
- 設定ファイルをレビューする場合は、設定仕様、環境構築書、実装側の読込処理、関連テストを読む。

## 読み方

- 作成元スキルは成果物の作成基準として扱う。
- 作成元スキルの改善提案は行わない。
- 関連資料間で食い違いがある場合は、どの成果物を直すべきかを `修正方針` でゼロベースに判断する。
