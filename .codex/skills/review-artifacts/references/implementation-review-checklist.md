# Implementation Review Checklist

## 仕様・設計との照合

- [ ] 実装が要件、外部設計、内部設計、テスト方針、開発標準と整合しているか。
- [ ] ディレクトリ構成が設計書や開発標準と一致しているか。
- [ ] ファイル構成、ファイル名、配置先が設計書や開発標準と一致しているか。
- [ ] 外部インターフェース、永続化、ファイル、設定、ログ、エラー、状態名などが設計された契約と一致しているか。
- [ ] 設計書にない状態、入出力項目、設定値、永続化項目、操作導線、利用者向け文言が追加されていないか。
- [ ] 実装が正しく、仕様書側が古い可能性がある場合は、仕様書側の修正方針を書く。
- [ ] 対象成果物に存在しない技術要素や実行形態を前提にして指摘していないか。

## 責務分割と依存方向

- [ ] 層、機能、モジュール、コンポーネントの責務が設計と一致しているか。
- [ ] 副作用を持つ処理が定義済みの境界へ閉じているか。
- [ ] 業務判断やドメイン判断が、表示層、入出力層、インフラ層、設定読込など不適切な場所へ漏れていないか。
- [ ] 純粋ロジックが永続化、通信、ファイル、時刻、ID、外部プロセスなどの副作用へ直接依存していないか。
- [ ] 上位層が下位層の実装詳細、テスト用実体、内部データ形式へ直接依存していないか。
- [ ] 依存方向、公開範囲、再利用単位が設計や開発標準と矛盾していないか。
- [ ] 不要な依存関係、未使用コード、到達不能コード、暫定的な分岐、デバッグ用処理が残っていないか。

## 型と構造化データ

- [ ] 使用言語、フレームワーク、開発標準に照らして、構造化データが意味のある型、データ構造、スキーマ、列挙値で表現されているか。
- [ ] 広すぎる型、未検証の動的データ、説明できない型変換、暗黙のデータ形状に依存していないか。
- [ ] 外部境界、永続化境界、設定境界、表示境界、処理内部のデータ構造が必要に応じて分かれているか。
- [ ] ID、状態、種別、payload、メタデータが文字列や汎用コンテナだけに埋もれず、意味のある表現になっているか。
- [ ] 入力値の検証、正規化、変換が境界で行われ、内部処理が未検証データを前提にしていないか。

## エラー・ログ・セキュリティ

- [ ] 利用者向けメッセージと内部調査用情報が分離されているか。
- [ ] 内部パス、秘密情報、スタックトレース、認証情報、個人情報などが利用者向け出力、外部応答、ログ、証跡へ露出していないか。
- [ ] URL、ポート、ファイルパス、認証情報、環境名、外部サービス名などが不適切にハードコードされていないか。
- [ ] エラー分類、追跡情報、状態更新、後続処理の継続または抑止が設計と一致しているか。
- [ ] ファイルパス、URL、コンテンツ種別、マークアップ、コマンド、外部入力などの検証が境界で行われているか。
- [ ] 認証、認可、入力制限、出力制御、監査記録が対象システムの設計と整合しているか。

## 実行時制御

- [ ] 時刻、乱数、ID、並行処理、リトライ、タイムアウト、リソース解放の扱いが仕様、設計、テストと整合しているか。

## コメントと説明

- [ ] コメントや docstring は、仕様上の意図、契約、前提条件、非自明な制約の説明に限定されているか。
- [ ] 作業都合、環境制約、暫定理由、レビュー対応理由、カバレッジ都合、旧仕様説明、コードの単なる言い換えがコメントや docstring に混入していないか。
- [ ] 業務上の非自明な前提、セキュリティ制約、外部仕様制約、特定テストデータが必要な理由、mock/fake/stub の境界理由、並行実行や時刻・ID・通信の注意点など、保守に必要な説明は残されているか。

## テストとの対応

- [ ] 実装に対応する単体テスト、結合テスト、必要な上位テスト仕様があるか。
- [ ] 事前条件、事後条件、不変条件、異常系、境界値がテスト対象になっているか。
- [ ] 実装変更に対して証跡やテスト方針が古くなっていないか。
- [ ] テストが実装詳細だけを固定せず、仕様上の振る舞いや契約を検証しているか。
- [ ] レビュー対象に該当しないテスト種別や証跡形式を必須前提にしていないか。

## 修正方針の判断

- [ ] 実装が設計から外れている場合は、実装と関連テストを直す方針を書く。
- [ ] 設計が実装済みの正しい振る舞いを表せていない場合は、設計とテスト仕様を直す方針を書く。
- [ ] テストや証跡だけが古い場合は、実装を直す前提にせず、テスト仕様、テストコード、証跡の修正方針を書く。
- [ ] 判断にユーザ合意が必要な場合は、どの成果物を確定根拠にするかを修正方針に書く。
- [ ] 特定のアプリ、言語、通信方式、画面構成、保存方式に依存した判断ではなく、レビュー対象の仕様と設計に基づいて指摘する。
