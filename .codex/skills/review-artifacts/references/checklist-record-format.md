# Checklist Record Format

作業用 checklist の各項目には、検証結果と根拠を必ず項目直下へ記録する。`- [x]` は合格ではなく、証拠付きでその項目の検証処理が完了したことを表す。

## 共通ルール

- `検証結果` は `指摘なし`、`指摘あり`、`対象外`、`判断不能` のいずれかにする。
- `確認根拠` はファイル名やパスだけの列挙にせず、どの成果物のどの内容を確認して何を根拠に判定したかを文章で説明する。
- `検証結果: 指摘なし` は、対象項目を確認したうえで指摘が不要と判断できる場合だけ使う。
- `検証結果: 指摘あり` は、対象項目について指摘が必要な場合に使い、作成済みまたは既存の issue ファイルパスを `指摘:` に必ず記録する。
- 1 項目に複数 issue が対応する場合は、1 つの `指摘:` の下にすべて列挙する。
- `検証結果: 対象外` の場合は、`理由:` を必ず記録する。
- `検証結果: 判断不能` の場合は、`不足根拠:` を必ず記録する。

## 指摘なし

```markdown
- [x] ディレクトリ構成が設計書や開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `src/backend/` の層別ディレクトリと `docs/03_内部設計/.../ディレクトリ構成.md` の責務定義を照合し、設計で定義された層と同じ単位でファイルが配置されていることを確認した。
```

## 指摘あり

```markdown
- [x] ディレクトリ構成が設計書や開発標準と一致しているか。
  - 検証結果: 指摘あり
  - 確認根拠: `src/backend/` の実ファイル配置では主要処理が `__init__.py` に集中しており、`docs/03_内部設計/.../ディレクトリ構成.md` で示された責務別ファイル構成と一致しないことを確認した。
  - 指摘: `.issue/review/.../__init__pyに主要実装が集中...md`
```

## 複数指摘

1 項目で複数の独立した問題が見つかった場合は、問題を 1 つの issue に無理にまとめない。issue は独立した修正単位ごとに分け、checklist 項目には対応する issue をすべて記録する。

```markdown
- [x] 外部インターフェース、永続化、ファイル、設定、ログ、エラー、状態名などが設計された契約と一致しているか。
  - 検証結果: 指摘あり
  - 確認根拠: `docs/02_外部設計/.../画面バックエンドAPI IF.md` の応答項目、`docs/02_外部設計/.../ログ設計.md` のログ記録対象、`src/backend/presentation/rest/chat.py` と `src/backend/infrastructure/trace_log/` の実装を照合し、API 契約差異とログ契約差異という独立した問題があることを確認した。
  - 指摘:
    - `.issue/review/..._APIとSSEの回答契約が外部IFと一致しない.md`
    - `.issue/review/..._トレースログが正常処理と既存ファイルを記録対象にしている.md`
```

## 対象外

```markdown
- [x] スクリーンショットがテストケースIDと対応しているか。
  - 検証結果: 対象外
  - 確認根拠: レビュー対象ファイル一覧に画面証跡が含まれておらず、`docs/04_テスト/` のうち今回確認すべき成果物が画面確認ではないことを確認した。
  - 理由: レビュー対象に画面証跡が含まれないため。
```

## 判断不能

```markdown
- [x] 証跡に対象コミットが残っているか。
  - 検証結果: 判断不能
  - 確認根拠: 提出された evidence 一覧を確認したが、対象コミットや対象差分を示す記録が含まれていないため、証跡がどの変更後に取得されたものか判断できなかった。
  - 不足根拠: 証跡ファイルが未提出のため。
```
