# Finding File Format

## 保存先

指摘は指摘保存先へ 1 指摘 1 ファイルで保存する。指摘保存先は、ユーザまたは呼び出し元スキルが明示した場合はその場所を使い、指定がない場合は `.issue/review/` を使う。

ファイル名は次の形式にする。

```text
<指摘保存先>/YYYY-MM-DD_HH-MM-SS_<指摘タイトル>.md
```

## ファイル名ルール

- 日時は作成時刻を `YYYY-MM-DD_HH-MM-SS` 形式で記載する。
- 指摘タイトルは簡潔な日本語にする。
- ファイル名では空白、スラッシュ、制御文字を `_` に置換する。
- 同じ秒に複数指摘を作る場合は、末尾に `_2`、`_3` のように連番を付ける。

## 章構成

指摘ファイルは次の章構成に固定する。

```markdown
# 指摘タイトル

## 重要度

High / Medium / Low

## レビュー対象

- `path/to/review-target`

## 指摘内容

何が問題かを具体的に書く。

## 根拠

どの成果物、どの記述、どの実装、どのテスト、どの証跡を根拠にしたかを書く。

## 影響

放置すると、利用者、開発者、運用、テスト、保守にどのような問題が起きるかを書く。

## 修正方針

ゼロベースで、どの成果物を直すべきかを書く。
レビュー対象を必ず修正する前提にしない。
```

## 指摘の分割

- issue は独立した修正単位ごとに 1 ファイルで作る。
- 同じ checklist 項目から複数の独立した問題が見つかった場合でも、1 つの issue に混ぜない。
- 構造問題、仕様契約問題、テスト不足、証跡不足、運用上の保留は、修正対象と完了条件が異なる場合は別 issue にする。
- 同じ根本問題が複数 checklist 項目に関係する場合は、重複 issue を作らず、既存 issue のパスを各 checklist 項目に記録する。

## 重要度

| 重要度 | 基準 |
| --- | --- |
| High | 要件不達、重大な仕様乖離、セキュリティ、データ破壊、主要導線の失敗、テスト完了判定を壊す問題 |
| Medium | 一部機能、特定条件、テスト網羅、証跡、文書整合に影響する問題 |
| Low | 誤読を招く表現、軽微な命名揺れ、保守性を下げるが直ちに動作や判定を壊さない問題 |

## 書き方

- `指摘内容` は、問題のある成果物と内容を一文で把握できるように書く。
- `根拠` は、レビュー対象と参照元の両方を示す。
- `影響` は、想定される実害を書く。
- `修正方針` は、仕様、設計、実装、テスト、証跡、参照元文書のどれを直すべきかを判断して書く。
- 判断できない場合は、判断が必要な論点と、判断後に更新する成果物を具体的に書く。
