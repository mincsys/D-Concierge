# Documentation Review Checklist

## 基本方針

- [ ] 文書レビューでは、対象文書を作成した可能性が高い `.codex/skills/` を作成基準として読む。
- [ ] 作成元スキルの `references/review-checklist.md` を、その文書種別の主チェックリストとして扱う。
- [ ] `references/question-checklist.md` は、質問合意、未合意事項、開始ゲート、合意結果一覧の扱いを確認する必要がある場合に読む。
- [ ] `review-artifacts` 独自の観点は、文書種別をまたぐ横断整合に限定する。

## 作成元スキル checklist の確認

- [ ] 成果物範囲が作成元スキルの指定と一致しているか。
- [ ] 作成元スキルが指定した必須文書、任意文書、レビュー対象外文書の扱いと一致しているか。
- [ ] `references/review-checklist.md` の観点が本文へ反映されているか。
- [ ] 明確な作成開始指示が必要なスキルでは、その開始ゲートを満たす前提で作られた成果物か。

## 横断整合

- [ ] 要件、外部設計、内部設計、テスト設計、開発標準の用語が一致しているか。
- [ ] 機能ID、画面ID、IF ID、テストID、ファイル名、章番号、リンク先に揺れがないか。
- [ ] ID が定義されている成果物では、機能ID、画面ID、IFID、テストケースIDなどを使って要件、設計、テスト、実装の追跡ができるか。ID が定義されていない成果物では、名称、見出し、ファイル名、関連文書名で一貫して追跡できるか。
- [ ] 要件や設計に対応する後工程成果物が欠けていないか。後工程だけに存在する孤立した仕様、機能、テストがないか。
- [ ] 前工程で定義された範囲指定と除外範囲が、後工程で矛盾なく扱われているか。
- [ ] 文書間で状態名、エラー名、設定項目、DB項目、API URL、SSEイベント、ログ項目が食い違っていないか。
- [ ] 文書内の図、表、本文が同じ仕様を説明しているか。

## 正式文書表現

- [ ] 差分説明、旧仕様の痕跡、会話ログ、検討メモが成果物本文に残っていないか。
- [ ] `【template-guidance】`、`【質問】`、未置換の `〇〇`、汎用ひな型名が残っていないか。
- [ ] `未定`、`TBD`、`TBC`、`保留`、`OK` などの判定語が、正式な意味定義なしに混入していないか。
- [ ] 文書の目的、前提、範囲指定、除外範囲が文書種別に対して自然か。
- [ ] 採用しなかった任意章や任意文書が空のまま残っていないか。
- [ ] Mermaid 図、Markdown 表、コードブロックが本文と矛盾していないか。

## 修正方針の判断

- [ ] 後工程の文書に誤りがある場合は、後工程を直す方針を書く。
- [ ] 前工程の文書が現実の合意や実装に追従していない場合は、前工程側を直す方針を書く。
- [ ] どちらが正か判断できない場合は、判断が必要な論点と、判断後に影響する成果物を具体的に書く。
