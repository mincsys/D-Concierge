# Write Static Test Design Docs

## Overview

要件定義書、設計書、開発標準、既存の静的解析設定を読み、日本のウォーターフォール開発向けの `静的テスト方針.md` を作成する。既定の成果物は `docs/04_テスト/01_静的テスト/静的テスト方針.md` の 1 ファイルだけとし、静的テストはツールによる `lint`、`format check`、`type check` に限定する。このスキルでは、対象プロジェクトの lint、format check、type check 設定ファイルを新規作成または編集しない。

## Workflow

### 1. 入力資料を読む

- 先に `AGENTS.md`、`docs/` 配下、README、既存の静的解析設定ファイル、package manager 設定、CI 設定を読む。
- 既存文書がある場合は、文体、用語、章番号、リンク表記、ID体系を合わせる。
- ユーザが文書名、保存先、章構成、静的テスト種別を指定している場合は、その指示を最優先にする。
- 静的テスト方針では、手動レビュー、ウォークスルー、目視確認を対象にしない。

### 2. 確定事項と質問事項を分ける

- リポジトリ、既存設定、README、CI 設定から一意に読み取れる内容は質問しない。
- 未確定事項は本文作成前にユーザへ質問し、合意してから本文へ反映する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- `references/question-checklist.md` を使い、質問では決定内容、決定が必要な理由、影響する章、未確定のまま進めた場合に曖昧になる内容、選択肢ごとの影響を丁寧に説明する。
- 質問、選択肢、検出対象の列挙では `など`、`等`、`etc.` を使わず、対象を具体名で書く。
- 選択肢は、相互排他的かつ採用時の影響が明確に異なるものだけにする。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 対象範囲は `全てのソースコードとテストコード` を推奨する。
- 静的テスト種別は `lint`、`format check`、`type check` をすべて実施推奨とし、security scan、dependency scan、secret scan、license check は対象外とする。
- 判定基準は `error 1件で不合格`、`warning も不合格に含める` を推奨する。
- 実施タイミングは `ファイルを作成、編集したとき`、失敗時対応は `修正して再テストを実施` を推奨する。
- 設定ファイルは必ずリポジトリで管理する固定方針とし、ユーザへ採否を質問しない。
- 静的テスト方針書と設定ファイル以外の実行ログ、解析レポート、設定変更理由、抑止理由の別ドキュメントは、ユーザから明示依頼がある場合のみ作成する。

### 3. 本文作成直前に合意状況を再確認する

- 設計書本文を作成する直前に、全質問項目についてユーザ合意が取れているかを再確認する。
- 再確認では、各項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理する。
- 質問回答がすべて得られた場合、`references/question-checklist.md` にある全質問項目を一覧化してユーザへ表示する。
- 一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `根拠` には、その合意結果を本文へ反映してよいと判断した理由を書く。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイルや設定から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 未合意項目がある場合は、`静的テスト方針.md` の本文作成へ進まない。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで `静的テスト方針.md` の本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。
- 明確な作成指示がない場合は、合意結果一覧の確認後に待機し、本文作成を開始しない。

### 4. 使用ツールの設定項目を調査する

- lint、format check、type check の使用ツールが決まった後、各ツールの公式ドキュメントを優先して設定可能項目を確認する。
- 公式ドキュメントで不足する場合は、ツール提供元のリファレンス、ツール提供元のリリースノート、ツール提供元の公開リポジトリを確認する。
- 調査した設定項目を `必須設定`、`推奨設定`、`任意設定`、`採用しない設定` に分類する。
- 設定値は推測で固定せず、調査結果を根拠としてユーザへ提示し、合意してから `静的テスト方針.md` へ反映する。

### 5. テンプレートを使う

- 既定テンプレートは `assets/04_テスト/01_静的テスト/静的テスト方針.md` を使う。
- 検出観点の詳細化が必要な場合は `references/static-test-viewpoints.md` を読む。
- 質問が必要な場合は `references/question-checklist.md` を読み、質問の背景、判断理由、選択肢ごとの影響を説明する。
- 章や本文の採否を迷う場合は、ユーザ指定と `references/review-checklist.md` の完成条件を優先する。

### 6. 静的テスト方針へ落とす

- 文書は日本語で、ウォーターフォール開発向けの正式文書として断定形で書く。
- 差分説明、検討メモ、会話ログ、旧仕様の痕跡を書かない。
- `【template-guidance】` ブロック、`【質問】`、未置換プレースホルダを最終成果物に残さない。
- 方針、基準、手順を書く。個別の検出結果一覧、実行ログ、解析レポートは書かない。
- 静的テスト方針書と設定ファイル以外の文書は、ユーザから明示依頼がある場合のみ作成する。
- このスキルの成果物として、静的テスト設定ファイルを新規作成または編集しない。設定ファイルは、既存ファイルまたは別タスクで作成されるファイルをリポジトリ管理対象として扱う。

### 7. 整合確認する

- `references/review-checklist.md` で、文書範囲、対象種別、ツール設定調査、判定基準、成果物・管理対象、正式文書表現を確認する。
- 最終成果物は `docs/04_テスト/01_静的テスト/静的テスト方針.md` だけにする。追加文書が必要に見える場合は、本文内に収めるかユーザへ相談する。
