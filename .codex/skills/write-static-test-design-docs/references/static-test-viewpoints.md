# 静的テスト観点整理

## 前提

- 本資料の対象は `lint`、`format check`、`type check` の 3 種別に限定する。
- security scan、dependency scan、secret scan、license check は対象外とする。
- 実際に採用する検出観点は、使用ツール決定後に公式ドキュメントを調査し、ユーザ合意を得てから決める。

## lint

- コーディング規約違反を検出する。
- 未使用変数を検出する。
- 未使用 import を検出する。
- 到達不能コードを検出する。
- 危険な構文を検出する。
- 非推奨 API 利用を検出する。
- 複雑度の上限超過を検出する。
- 命名規則違反を検出する。
- import 並び順違反を検出する。

## format check

- インデント設定との差分を検出する。
- 改行設定との差分を検出する。
- 空白設定との差分を検出する。
- 行長設定との差分を検出する。
- クォート設定との差分を検出する。
- セミコロン設定との差分を検出する。
- trailing comma 設定との差分を検出する。
- ファイル末尾改行設定との差分を検出する。
- import 並び順設定との差分を検出する。

## type check

- 変数代入時の型不整合を検出する。
- 関数引数の型不整合を検出する。
- 関数戻り値の型不整合を検出する。
- null 参照を検出する。
- undefined 参照を検出する。
- 未定義プロパティ参照を検出する。
- 型注釈不足を検出する。
- ジェネリック型指定不足を検出する。
- 型境界を越えた未検証 payload の受け渡しを検出する。

## 重複観点の扱い

- import 並び順を lint で判定するか format check で判定するかをユーザと合意する。
- 未使用 import を lint で判定するか type check で判定するかをユーザと合意する。
- 型注釈不足を lint で判定するか type check で判定するかをユーザと合意する。
