# 静的テスト方針 質問チェックリスト

## 使い方

- リポジトリや既存文書から一意に分かる内容は質問しない。
- 本文作成前に、以下の未確定項目をユーザへ提示し、合意してから `静的テスト方針.md` へ反映する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 質問では、決定内容、決定が必要な理由、影響する章、未確定のまま進めた場合に曖昧になる内容、選択肢ごとの影響を説明する。
- 選択肢は、相互排他的かつ採用時の影響が明確に異なるものだけにする。
- 質問文、選択肢、検出対象の列挙では `など`、`等`、`etc.` を使わない。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 設計書本文を作成する直前に、全質問項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理し、未合意項目がある場合は本文作成へ進まない。
- 質問回答がすべて得られた場合、質問表にある全質問項目と合意結果を一覧化してユーザへ表示する。
- 合意結果一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `根拠` には、その合意結果を本文へ反映してよいと判断した理由を書く。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイルや設定から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで `静的テスト方針.md` の本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。
- 明確な作成指示がない場合は、合意結果一覧の確認後に待機し、本文作成を開始しない。
- 「設定可能項目を調査するか」は質問しない。使用ツール決定後に公式ドキュメントを優先して必ず調査する。
- 「設定ファイルをリポジトリで管理するか」は質問しない。設定ファイルは必ずリポジトリで管理する。
- 静的テスト方針書と設定ファイル以外の文書は、ユーザから明示依頼がある場合のみ作成する。

## 1. 対象範囲

- どのシステム、リポジトリ、ディレクトリを静的テスト対象にするか。
- ソースコード以外に、設定ファイル、依存定義、CI 定義を含めるか。
- backend、frontend、インフラコード、テストコード、モックコードを含めるか。
- 推奨は、全てのソースコードとテストコードを対象にすること。

## 2. 静的テスト種別

- lint を実施するか。
  - 推奨: 実施する。
- format check を実施するか。
  - 推奨: 実施する。
- type check を実施するか。
  - 推奨: 実施する。
- security scan、dependency scan、secret scan、license check を対象外とするか。
  - 推奨: 対象外とする。

## 3. 使用ツール

- lint に使用するツールは何か。
- format check に使用するツールは何か。
- type check に使用するツールは何か。
- ローカル実行と CI 実行で同じツールを使うか。
- ローカル実行と CI 実行で同じ設定ファイルを使うか。
- ツールのバージョンを固定するか。

## 4. ツール別検出観点

- lint では何を検出対象にするか。
- format check では何を検出対象にするか。
- type check では何を検出対象にするか。
- lint と type check の両方で検出される項目は、どちらを主たる判定元にするか。
- import 並び順を lint と format check のどちらで扱うか。
- 未使用 import を lint と type check のどちらで扱うか。

## 5. ツール設定方針

- 公式ドキュメント調査後に提示された設定項目のうち、どれを有効化するか。
- 公式ドキュメント調査後に提示された設定項目のうち、どれを無効化するか。
- 公式ドキュメント調査後に提示された設定項目のうち、どれを厳格化するか。
- 公式ドキュメント調査後に提示された設定項目のうち、どれを緩和するか。
- プロジェクト共通設定とディレクトリ別設定を分けるか。

## 6. 判定基準

- error が 1 件でもあれば不合格とするか。
  - 推奨: 不合格とする。
- warning を不合格に含めるか。
  - 推奨: 不合格に含める。
- 自動修正可能な指摘も不合格に含めるか。
- 既存指摘を暫定許容するか。

## 7. 除外・抑止

- 自動生成ファイルを除外するか。
- 外部ライブラリを除外するか。
- ビルド成果物を除外するか。
- 配布物を除外するか。
- ignore を許可するか。
- suppress を許可するか。
- 誤検知をどの条件で許容するか。

## 8. 実施手順

- 静的テストをいつ実施するか。
  - 推奨: ファイルを作成、編集したとき。
- 標準実行コマンドは何か。
- 失敗時の修正、再実行の流れをどうするか。
  - 推奨: 修正して再テストを実施する。

## 9. 成果物・管理対象

- 静的テスト方針書と設定ファイル以外の文書を作成しない前提でよいか。
  - 推奨: 作成しない。
- ユーザから明示依頼がある場合に限り、実行ログ、解析レポート、設定変更理由、抑止理由を別途作成する方針でよいか。
  - 推奨: 明示依頼がある場合のみ作成する。

## 10. 文書完成条件

- `静的テスト方針.md` 以外の成果物を作らない前提でよいか。
- 個別の検出結果一覧、実行ログ、解析レポートを対象外としてよいか。
- `【質問】` に相当する未合意事項が本文に残っていないか。
