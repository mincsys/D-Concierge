# 静的テスト方針レビュー観点

## 文書範囲

- 成果物が `docs/04_テスト/01_静的テスト/静的テスト方針.md` の 1 ファイルに限定されているか。
- スキルの成果物として静的テスト設定ファイルを新規作成または編集しない方針があるか。
- 静的テスト方針書と設定ファイル以外の文書作成を既定としていないか。
- 手動レビュー、ウォークスルー、目視確認が静的テストとして扱われていないか。
- 日本のウォーターフォール開発向けの正式文書として読めるか。

## 対象種別

- 静的テスト種別が `lint`、`format check`、`type check` のみに限定されているか。
- security scan、dependency scan、secret scan、license check が対象外として扱われているか。

## 質問合意

- `【質問】` に相当する未確定項目が本文に残っていないか。
- 使用ツール、標準実行コマンド、ツール別検出観点、ツール設定値、除外・抑止条件がユーザ合意に基づいているか。
- 質問が 1 つずつ行われ、合意内容が整理されているか。
- 質問文に、決定内容、決定が必要な理由、影響する章、未確定のまま進めた場合に曖昧になる内容、選択肢ごとの影響を説明する方針があるか。
- 質問文、選択肢、検出対象の列挙で `など`、`等`、`etc.` を使わない方針があるか。
- 選択肢が相互排他的かつ採用時の影響が明確に異なるものだけになっているか。
- 選択肢が `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のような番号付きリストになっているか。
- ユーザが番号だけで回答できることを質問文で明記しているか。
- 選択肢番号が質問ごとに `1` から始まっているか。
- 推奨選択肢が `1. 選択肢名（推奨）` のように 1 つだけ明示されているか。
- 推奨理由が、選択肢本文または質問説明の中で簡潔に説明されているか。
- 複数の選択肢が同時に推奨扱いになっていないか。
- 本文作成直前に、全質問項目が `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで確認されているか。
- 本文作成直前に、質問表の全質問項目と合意結果がユーザへ表示されているか。
- 合意結果一覧に `質問項目`、`合意結果`、`状態`、`根拠` が含まれているか。
- `根拠` に、合意結果を本文へ反映してよいと判断した理由が書かれているか。
- `合意済み` の根拠に、ユーザがどの回答で合意したかが書かれているか。
- `リポジトリ事実として確定` の根拠に、どのファイルや設定から一意に判断したかが書かれているか。
- `未合意` の根拠に、何が未回答または未確定かが書かれているか。
- 合意結果一覧の表示後、ユーザから明確な作成指示を受けてから本文作成へ進んでいるか。
- `確認しました`、`OKです`、`問題ありません` だけで本文作成へ進んでいないか。
- 明確な作成指示がない場合に、合意結果一覧の確認後も本文作成を開始せず待機しているか。

## ツール設定調査

- 使用ツール決定後に公式ドキュメントを優先して設定可能項目を調査する方針があるか。
- 調査結果を `必須設定`、`推奨設定`、`任意設定`、`採用しない設定` に分類する方針があるか。
- 設定値をユーザ合意後に方針書へ反映する方針があるか。

## 判定基準

- error が 1 件でもあれば不合格とする方針になっているか。
- warning を不合格に含める方針になっているか。
- 既存指摘、自動修正可能な指摘、誤検知の扱いが明確か。

## 成果物・管理対象

- 設定ファイルをリポジトリで管理する方針があるか。
- 実行ログ、解析レポート、設定変更理由、抑止理由を既定の別ドキュメントとして作成しない方針があるか。
- ユーザから明示依頼がある場合のみ、実行ログ、解析レポート、設定変更理由、抑止理由を別途作成する方針があるか。

## 表現

- 文書が断定形で書かれているか。
- 差分説明、検討メモ、会話ログ、旧仕様の痕跡が残っていないか。
- `【template-guidance】` ブロック、`【質問】`、未置換プレースホルダが最終成果物に残っていないか。
- 既存 docs と用語、文体、リンク表記がそろっているか。
