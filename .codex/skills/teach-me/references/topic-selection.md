# Topic Selection

## 基本ルール

- ユーザが学習テーマを指定していない場合だけ、まず学習レベルを質問する。
- 質問は一問一答にする。文書回答割合、表現レベル、学習時間などを同時に聞かない。
- 番号だけで回答できる Markdown の番号付きリストを使う。
- 選択肢は質問ごとに `1` から始める。
- 最初の質問にも、機能別候補を出す後続質問にも、`全範囲` を必ず含める。

## 最初の質問

最初は次の候補だけを提示する。機能別候補は最初の質問に混ぜない。

1. 全範囲
2. 要件定義レベル
3. 外部設計レベル
4. 内部設計レベル
5. 実装レベル
6. 自由回答

## 次の質問で機能別候補を出す

- 最初の質問でユーザが `自由回答` 以外を選び、かつ調査済みの主要機能が明確なら、次の一問で機能別候補を提示する。
- 機能別候補の質問にも `全範囲` を必ず入れる。
- 機能別候補は、そのプロジェクトの実装または docs に根拠がある主要能力から作る。例として、データ入力、制御処理、状態管理、設定、外部連携、ファイル処理、通信、UI、バッチ処理、デバイス連携などをプロジェクトに合わせて選ぶ。
- 機能別候補は 2〜6 個程度に絞る。候補が多い場合は、学習レベルに合う代表機能を優先する。
- 機能別候補の質問でも番号だけで回答できるようにし、選択肢は `1` から始める。

例:

```markdown
次に、どの機能または能力を中心に学ぶか選んでください。番号だけで大丈夫です。

1. 全範囲
2. データ入力
3. 制御処理
4. 状態管理
5. 外部連携
6. 自由回答
```

## 初回案内に含めること

- 専門用語が多く難しい場合は、分かりやすい表現にできると伝える。
- 文章で答える問題の割合は指定できると伝える。
- 未指定の場合、文章回答は約 30% にすると伝える。

## テーマ指定済みの場合

- テーマが十分具体的なら、追加質問せずカリキュラム作成へ進む。
- テーマが広すぎる場合だけ、1 問で範囲を絞る。
- テーマが実装に存在しない可能性がある場合は、調査してから候補を出す。
