# コーディング規約

## 1. 文書の目的
【template-guidance】<br>
- 本書が実装者向けのコーディング規約であることを書く。
- 実装フェーズの手順と守るべきルールが分かる文書であることが読めるようにする。
- レビュー手順書や静的テスト方針そのものにはしない。

【/template-guidance】<br>

本書は、対象システムの実装者が、実装開始前の確認事項、標準実装手順、守るべき実装ルール、テスト実装ルールおよび実装完了前の確認事項を把握するためのコーディング規約を定義することを目的とする。

## 2. 前提・参照文書
【template-guidance】<br>
- 本書が前提とする文書を列挙する。
- `docs/` に書かれている内容を重複記載せず、必要に応じて参照する方針を書く。
- 静的テスト方針でカバーされる内容は、本書で重複記載せず、静的テスト方針に従うことを書く。

【/template-guidance】<br>

- 本書は、要件定義書、外部設計書、内部設計書、テスト方針書および開発標準を前提とする。
- 実装前に `docs/03_内部設計/02_クラス設計/` を確認する。
- 静的テストでカバーされる内容は `docs/04_テスト/01_静的テスト/静的テスト方針.md` に従う。
- 単体テストの観点および基準は `docs/04_テスト/02_単体テスト/単体テスト方針.md` を参照する。
- 結合テストの観点および基準は `docs/04_テスト/03_結合テスト/結合テスト方針.md` を参照する。
- 本書では `docs/` 内に既に定義された内容を重複記載せず、必要な箇所で参照する。

## 3. 実装開始前の確認事項
【template-guidance】<br>
- 実装着手前に確認すべき文書、契約条件、対象範囲、影響範囲を書く。
- `事前条件` `事後条件` `不変条件` の確認を必須とする。

【/template-guidance】<br>

- 対象機能に対応する要件定義書、外部設計書、内部設計書を確認する。
- 対象クラスまたは対象関数に対応する `docs/03_内部設計/02_クラス設計/` の記載を確認する。
- 対象クラスまたは対象関数の `事前条件` `事後条件` `不変条件` を把握する。
- 変更対象の影響範囲、関連する実装要素、関連するテストを確認する。

## 4. 標準実装手順
【template-guidance】<br>
- 実装者が従う標準手順を書く。
- TDD、契約条件テスト、単体テスト、結合テスト、リファクタリング、自己確認を順に書く。

【/template-guidance】<br>

1. 対象設計と契約条件を確認する。
2. `事前条件` `事後条件` `不変条件` のテストを先に作成する。
3. TDD の `Red -> Green -> Refactor` の順で実装およびテストを進める。
4. 単体テスト方針および結合テスト方針の観点に従って必要なテストを追加する。
5. カバレッジ目標を満たすことを確認する。
6. 自己確認後にレビュー可能な状態へ整える。

## 5. 全体共通ルール
【template-guidance】<br>
- 実装要素を問わず守る共通ルールを書く。
- 読み取れないルールはユーザ合意に基づいて具体化する。

【/template-guidance】<br>

- 設計で定義した責務分割、依存方向、公開契約を崩さない。
- 設定値、外部境界、副作用境界は、責務に応じた位置へ隔離する。
- 機密情報、認証情報、個人情報をソースコード、コメント、ログへ埋め込まない。
- `docs/` に定義されたルールと矛盾する実装を行わない。

## 7. コメント・docstring ルール
【template-guidance】<br>
- ユーザ指定の必須ルールをそのまま反映する。
- 静的テストで強制される書式ルールは重複記載しない。

【/template-guidance】<br>

- 各クラスと関数単位で、日本語の説明コメント `docstring` を記載する。
- `docstring` には、少なくとも役割、入出力の意味、前提条件、副作用を記載する。
- 関数内の意味のある処理単位ごとに、日本語コメントを記載する。
- コメントは、処理の意図、業務上の意味、分岐理由、境界条件が分かる内容にする。
- コメントに `...を修正` `...を廃止` など以前の実装経緯を残さない。
- コメントは、最初からその実装であったかのような自然な記述に全面的に整合させる。

## 8. テスト実装ルール
【template-guidance】<br>
- TDD、契約条件テスト、単体テスト方針、結合テスト方針、カバレッジ参照を記載する。
- 既存テスト方針と重複する細目は参照に留める。

【/template-guidance】<br>

- テスト実装は TDD を原則とし、`Red -> Green -> Refactor` の順で進める。
- 実装前に、`docs/03_内部設計/02_クラス設計/` の `事前条件` `事後条件` `不変条件` のテストを作成する。
- 単体テストは `docs/04_テスト/02_単体テスト/単体テスト方針.md` の観点に従って追加する。
- 結合テストは `docs/04_テスト/03_結合テスト/結合テスト方針.md` の観点に従って追加する。
- カバレッジ目標値そのものは既存テスト方針を参照し、目標を満たすことを必須とする。

## 9. 実装完了前の確認事項
【template-guidance】<br>
- 実装完了前に実装者が確認すべき項目を書く。
- 静的テスト、単体テスト、結合テスト、契約条件、カバレッジ、不要コード除去を候補にする。

【/template-guidance】<br>

- 静的テスト方針で定義された確認を完了している。
- 追加した契約条件テスト、単体テスト、結合テストが成功している。
- カバレッジ目標を満たしている。
- 不要コード、不要コメント、履歴を感じさせるコメントが残っていない。
- `docs/` の設計内容と実装内容が整合している。
