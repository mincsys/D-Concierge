# Write Coding Standards Docs

## Overview

`docs/` 配下、README、関連設定ファイルを精読し、日本のウォーターフォール開発向けのコーディング規約を作成する。既定の成果物は `docs/05_開発標準/02_コーディング規約/コーディング規約.md` の 1 ファイルだけとし、実装者が読めば実装フェーズの手順と守るべきルールが分かる文書にする。

## Workflow

### 1. 入力資料を読む

- 先に `AGENTS.md`、`docs/` 配下、README、関連設定ファイルを読む。
- **ユーザへ質問する前に `docs/` 内のファイルを精読する。**
- 少なくとも `docs/04_テスト/01_静的テスト/静的テスト方針.md`、`docs/04_テスト/02_単体テスト/単体テスト方針.md`、`docs/04_テスト/03_結合テスト/結合テスト方針.md`、`docs/03_内部設計/02_クラス設計/` を読む。
- 既存文書がある場合は、文体、用語、章番号、リンク表記、ID体系を合わせる。
- ユーザが文書名、保存先、章構成を指定している場合は、その指示を最優先にする。
- 静的テスト方針でカバーされる内容は、本書へ重複記載せず、`静的テスト方針に従う` と整理する。

### 2. 確定事項と質問事項を分ける

- `docs/` や設定ファイルなどから一意に読み取れる内容は質問しない。
- `docs/` や設定ファイルなどから読み取れない全てのルールについて、ユーザへ推奨案を提示して合意を取る。
- 未確定事項は本文作成前にユーザへ質問し、合意してから本文へ反映する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 各質問では、確認が必要な理由、影響する章、未確定のまま進めた場合に曖昧になる内容を丁寧に説明する。
- 選択肢を提示する場合は、`など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- 選択肢は、採用時の影響が明確に異なるものだけにする。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 推奨選択肢は、リポジトリ事実、既存方針、ウォーターフォール文書としての安全性を基準に選ぶ。
- `references/question-checklist.md` を使い、読み取れないルールだけを確認する。

### 3. 本文作成直前に合意状況を再確認する

- 設計書本文を作成する直前に、全質問項目についてユーザ合意が取れているかを再確認する。
- 再確認では、各項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理する。
- 質問回答がすべて得られた場合、`references/question-checklist.md` にある全質問項目を一覧化してユーザへ表示する。
- 一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `根拠` には、その合意結果を本文へ反映してよいと判断した理由を書く。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイルや設定から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 未合意項目がある場合は、本文作成へ進まない。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。
- 明確な作成指示がない場合は、合意結果一覧の確認後に待機し、本文作成を開始しない。

### 4. テンプレートを選ぶ

- 既定テンプレートは `assets/05_開発標準/02_コーディング規約/コーディング規約.md` を使う。
- 章や本文の採否を迷う場合は、ユーザ指定と `references/review-checklist.md` の完成条件を優先する。
- 実装要素別ルールは、対象プロジェクトに存在する要素だけ採用し、不要な見出しは削除する。

### 5. コーディング規約へ落とす

- 文書は日本語で、ウォーターフォール開発向けの正式文書として断定形で書く。
- 差分説明、検討メモ、会話ログ、旧仕様の痕跡を書かない。
- `【template-guidance】` ブロック、`【質問】`、未置換プレースホルダを最終成果物に残さない。
- 成果物は `docs/05_開発標準/02_コーディング規約/コーディング規約.md` のみとする。
- 静的テスト方針でカバーされる内容は重複記載せず、`docs/04_テスト/01_静的テスト/静的テスト方針.md` に従うと書く。
- `docs/03_内部設計/02_クラス設計/` の `事前条件` `事後条件` `不変条件` のテストを実装前に作ることを必須ルールとして書く。
- テスト実装は TDD とし、`Red -> Green -> Refactor` を守ることを必須ルールとして書く。
- `docs/04_テスト/02_単体テスト/単体テスト方針.md` と `docs/04_テスト/03_結合テスト/結合テスト方針.md` の観点に従ってテストを追加することを必須ルールとして書く。
- カバレッジ目標値そのものは重複記載せず、既存テスト方針を参照し、目標を満たすことだけを書く。
- 各クラスと関数単位で、日本語の説明コメント `docstring` を書くことを必須ルールとして書く。
- 関数内の意味のある処理単位で、日本語コメントを書くことを必須ルールとして書く。
- コメントに `...を修正` `...を廃止` など以前の実装経緯を残さず、最初からその実装であったかのような自然な記述に整合させることを必須ルールとして書く。

### 6. 整合確認する

- `references/review-checklist.md` で、文書範囲、質問合意、参照文書整合、静的テストとの切り分け、実装手順、コメント・docstring、テスト実装、例外運用、正式文書表現を確認する。
- 最終成果物は `docs/05_開発標準/02_コーディング規約/コーディング規約.md` だけにする。
