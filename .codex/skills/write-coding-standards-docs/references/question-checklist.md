# コーディング規約 質問チェックリスト

## 使い方

- ユーザへ質問する前に `docs/` 内のファイルを精読する。
- `docs/` や設定ファイルから一意に分かる内容は質問しない。
- `docs/` や設定ファイルなどから読み取れない全てのルールについて、ユーザへ推奨案を提示して合意を取る。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 各質問では、確認が必要な理由、影響する章、未確定のまま進めた場合に曖昧になる内容を説明する。
- 選択肢を提示する場合は、`など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- 選択肢は、採用時の影響が明確に異なるものだけにする。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 推奨選択肢は、リポジトリ事実、既存方針、ウォーターフォール文書としての安全性を基準に選ぶ。
- 設計書本文を作成する直前に、全質問項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理し、未合意項目がある場合は本文作成へ進まない。
- 質問回答がすべて得られた場合、質問表にある全質問項目と合意結果を一覧化してユーザへ表示する。
- 合意結果一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `根拠` には、その合意結果を本文へ反映してよいと判断した理由を書く。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイルや設定から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。

## 1. 規約を分ける単位

- `全体共通 / 言語固有 / 実装要素固有` のどの粒度で分けるか。

## 2. 対象に含める実装要素

- アプリケーション本体、UI、データアクセス、外部連携、バッチ・ジョブ、CLI・スクリプト、インフラ設定のうちどこまでを対象にするか。

## 3. 全体共通ルールの範囲

- 責務分割、依存方向、例外処理、設定値参照、副作用境界、ログ出力のうちどこまでを共通ルールとして固定するか。

## 4. 言語固有ルールの範囲

- 型付け、例外、戻り値、非同期、ファイル構成、禁止事項をどこまで言語別に書くか。

## 5. データアクセス実装ルール

- DB、ファイル、永続化、キャッシュ、マイグレーション、更新境界をどう統一するか。

## 6. 外部連携実装ルール

- 外部 API、メッセージング、通知、ストレージ、外部プロセス呼び出しをどう統一するか。

## 7. UI 実装ルール

- UI を持つ場合のみ、表示責務、入力処理、状態管理、遷移やイベント処理をどう統一するか。

## 8. バッチ・ジョブ実装ルール

- 定期実行、再実行、安全性、失敗時の扱いをどう統一するか。

## 9. CLI・スクリプト実装ルール

- 引数、終了コード、標準出力、標準エラー出力、設定値読み込みをどう統一するか。

## 10. 例外処理ルール

- 握りつぶし、変換、再送出、利用者向けメッセージ、ログ出力の基準をどうするか。

## 11. ログ出力ルール

- 何を出すか、何を出さないか、機密情報の扱いをどうするか。

## 12. 設定値・環境変数の扱い

- 参照位置、直参照の可否、既定値の扱いをどうするか。

## 13. テストコード配置ルール

- 本体コードとの配置関係、単体テストと結合テストの分け方をどうするか。

## 14. テストダブルの扱い

- mock、stub、fake、fixture、factory をどこでどう使うか。

## 15. `docstring` の粒度

- 最低限何を書くか。推奨は `役割 / 入出力の意味 / 前提 / 副作用` とする。

## 16. 関数内コメントの粒度

- どこまでを意味のある処理単位とみなすか。

## 17. 実装完了前の自己確認項目

- 静的テスト通過、単体テスト追加、結合テスト追加、契約条件確認、カバレッジ確認、不要コード削除のどこまで必須にするか。

## 18. 例外運用

- 規約から外れる場合の承認者、記録先、理由記載方法をどうするか。
- 例外運用が不要なら 10 章を削除する。
