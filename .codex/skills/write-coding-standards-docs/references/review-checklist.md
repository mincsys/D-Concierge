# コーディング規約レビュー観点

## 文書範囲

- [ ] 成果物が `docs/05_開発標準/02_コーディング規約/コーディング規約.md` の 1 ファイルに限定されているか。
- [ ] 実装者向けの文書として読めるか。
- [ ] 実装フェーズの手順と守るべきルールが 1 文書で分かるか。

## テンプレート準拠

- [ ] 採用した assets 配下のテンプレートと、成果物の章構成、見出し階層、章番号、見出し名が一致しているか。
- [ ] 採用した assets 配下のテンプレートに定義された必須章、必須表、表の列、記入欄が、成果物で削除、統合、改名、順序変更されていないか。
- [ ] 残した必須章、必須表、表の列、記入欄が、空欄、未記入、プレースホルダのまま残らず、成果物として必要な内容で埋められているか。
- [ ] 実装ルールの区分が `全体共通 / 言語固有 / 実装要素固有` の構成で整理されているか。

## 参照文書整合

- [ ] `docs/04_テスト/01_静的テスト/静的テスト方針.md` への参照があるか。
- [ ] `docs/04_テスト/02_単体テスト/単体テスト方針.md` への参照があるか。
- [ ] `docs/04_テスト/03_結合テスト/結合テスト方針.md` への参照があるか。
- [ ] `docs/03_内部設計/02_クラス設計/` への参照があるか。
- [ ] `docs/` に定義済みの内容を不必要に重複記載していないか。

## 静的テストとの切り分け

- [ ] 静的テスト方針でカバーされる内容を本書へ重複記載していないか。
- [ ] 静的テストでカバーされる内容について、`静的テスト方針に従う` と整理されているか。

## 実装手順

- [ ] 実装開始前に、対象設計と契約条件を確認する手順があるか。
- [ ] カバレッジ目標値は参照に留め、目標を満たすことだけが書かれているか。

## 実装ルール

- [ ] Web 前提の区分に固定されていないか。
- [ ] 対象プロジェクトに存在しない実装要素の小見出しが不要に残っていないか。

## コメント・docstring

- [ ] 各クラスと関数単位で、日本語の説明コメント `docstring` を書くルールがあるか。
- [ ] 関数内の意味のある処理単位で、日本語コメントを書くルールがあるか。
- [ ] `...を修正` `...を廃止` など履歴を感じさせるコメントを残さないルールがあるか。
- [ ] コメントを最初からその実装であったかのように整合させるルールがあるか。

## 例外運用

- [ ] 例外運用が必要な場合に 10 章へ記載されているか。
- [ ] 例外運用が不要な場合に 10 章が削除されているか。

## 考慮漏れ

- [ ] 型、例外、ログ、コメント、設定、ファイル、外部入力、認証情報、テストコードの扱いが漏れていないか。
- [ ] セキュリティ、個人情報、秘密情報、パス、スタックトレース、ログ出力の禁止・許可ルールが明確か。
- [ ] レイヤ責務、依存方向、副作用境界、命名、ディレクトリ配置のルールが実装者向けに足りているか。
- [ ] 実装前の `事前条件` `事後条件` `不変条件` のテスト作成、TDD の `Red -> Green -> Refactor`、単体テスト方針・結合テスト方針に従うテスト追加、レビュー指摘対応、仕様書更新、issue 記録の実装時ルールが漏れていないか。
- [ ] 既存 docs と重複せず、実装者が判断に迷う箇所だけを規約化できているか。

## 表現

- [ ] 文書が断定形で書かれているか。
- [ ] 差分説明、検討メモ、会話ログ、旧仕様の痕跡が残っていないか。
- [ ] `【質問】` が最終成果物に残っていないか。
- [ ] 既存 docs と用語、文体、リンク表記がそろっているか。
