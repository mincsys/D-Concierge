# 〇〇フロー

## 1. 文書の目的
【template-guidance】<br>
必須: 対象業務の手順と判断分岐を定義する目的を書く。<br>
任意: 関連する業務IDや機能IDを書いてよい。<br>
書かない: 実装方式やAPIの詳細。<br>
【/template-guidance】<br>

本書は、〇〇業務の処理手順と例外を定義することを目的とする。

## 2. 前提
【template-guidance】<br>
必須: 認証、入力済み情報、権限などの前提を書く。<br>
任意: 外部サービスや他システム前提を書いてよい。<br>
書かない: 未決定候補の比較。<br>
【/template-guidance】<br>

- 実施主体は必要な権限を持つ。

## 3. 業務フロー概要
【template-guidance】<br>
必須: 原則 `sequenceDiagram` で主体間の時系列を図示し、図の先頭に `autonumber` を入れる。`actor` と `participant` を明示し、参加者は利用者または管理者、画面、アプリ、DB、外部サービスなどの責務単位で分ける。やり取り、内部処理、分岐は `->>`、自己メッセージ、`alt` / `else`、必要に応じて `Note` で表す。図の参加者名、メッセージ、分岐条件は `4. 手順詳細` と `5. 例外処理` の語彙に合わせる。<br>
任意: 1図で視点が混ざる場合のみ `3.1` `3.2` のように節を分けて複数のシーケンス図を書く。時系列の主体間やり取りとして表せない場合に限り `flowchart` を使ってよい。ラベル改行には `<br>` を使う。<br>
書かない: 全データ項目の説明、画面遷移図やサイトマップのような図、`\n` を使った改行。<br>
【/template-guidance】<br>

```mermaid
sequenceDiagram
    autonumber
    actor U as 実施主体
    participant I as 入力画面
    participant S as 〇〇システム
    participant D as データ保存先または外部サービス

    U->>I: 〇〇を開始する
    I->>S: 〇〇処理を依頼する
    S->>S: 入力内容を確認する
    alt 正常に処理できる場合
        S->>D: 必要な処理を実行する
        D-->>S: 処理結果を返却する
        S-->>I: 正常結果を返却する
        I-->>U: 結果を表示する
    else 異常が発生した場合
        S-->>I: エラー結果を返却する
        I-->>U: エラーメッセージを表示する
    end
```

任意: 入力、出力、利用画面を列追加してよい。<br>
書かない: 画面レイアウト詳細。<br>

| --- | --- | --- |

## 5. 例外処理
【template-guidance】<br>
必須: 代表的な異常系と業務上の扱いを書く。<br>
任意: 回復方法や再実行条件を追記してよい。<br>
書かない: ログ出力レベルの詳細。<br>
【/template-guidance】<br>

| 事象 | 業務上の扱い |
| --- | --- |
| 入力不備 | 再入力を求める |
| 外部連携失敗 | 失敗通知を行い処理を終了する |
