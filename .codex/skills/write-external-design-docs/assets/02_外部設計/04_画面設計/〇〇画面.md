# 〇〇画面

## 1. 文書の目的
【template-guidance】<br>
必須: 対象画面の外部仕様を定義する目的を書く。<br>
任意: 画面ID、関連機能ID、主要な利用場面を書いてよい。<br>
書かない: 実装部品名、スタイル定義、画面作成の経緯。<br>
【/template-guidance】<br>

本書は、〇〇画面の外部仕様を定義することを目的とする。

## 2. 前提
【template-guidance】<br>
必須: 認証状態、到達条件、利用者区分、参照権限、必要データの有無などの前提を書く。<br>
任意: 利用端末、権限制約、表示対象データの条件を追記してよい。<br>
書かない: 未確定な遷移案や実装都合の内部制約。<br>
【/template-guidance】<br>

- 対象利用者は必要な権限を持つ。

## 3. 画面レイアウト
【template-guidance】<br>
必須: 個別画面文書と同階層の `04_画面設計/assets` 配下に該当画像がある場合は、その画像を Markdown 画像で引用する。該当画像がない場合は `レイアウト画像なし` と記載する。<br>
任意: レイアウト図の見方や、複数状態がある場合の補足を書いてよい。画像がない場合のみ、補助的に図表や `mermaid` を併記してよい。<br>
書かない: CSS、HTML構造、実装部品名、画像編集手順、存在しない画像の仮置き。<br>

## 4. 項目一覧
【template-guidance】<br>
必須: 画面を領域単位に分けて、`4.1. 〇〇領域` の小節ごとに表を置く。各表の列は `項目名` `機能詳細` `種別` `初期値` `備考` に固定する。<br>
任意: 領域は必要数だけ増やしてよい。入力欄、表示欄、ボタン、切替、表、カード、ナビゲーションなどを同じ構成で扱ってよい。<br>
書かない: 画像を見れば分かる位置や見た目だけの説明、HTML要素名、CSSクラス名。<br>
【/template-guidance】<br>

### 4.1. 〇〇領域
【template-guidance】<br>
必須: その領域に含まれる項目を表で整理する。`機能詳細` には表示内容だけでなく、表示フォーマット、単位、桁数、丸め、未設定時表示、選択肢、初期選択、入力時のバリデーション、表示条件、並び順、繰り返し条件を必要に応じて書く。<br>
任意: 関連するメッセージID、他画面との整合条件、列挙値の意味を補足してよい。<br>
書かない: 実装コード、内部 API 名、画像だけで判断できる位置説明。<br>
【/template-guidance】<br>

| 項目名 | 機能詳細 | 種別 | 初期値 | 備考 |
| --- | --- | --- | --- | --- |
| 画面タイトル | 画面名称を表示する。必要に応じて表示文言、改行条件、未設定時の扱いを書く。 | ラベル | `〇〇` | 固定表示とする。 |
| 一覧または入力欄 | 表示対象または入力対象を扱う。必要に応じて表示形式、入力形式、必須条件、選択肢、桁数、単位、並び順、バリデーション条件を書く。 | テキストボックス、ドロップダウン、表、カードなど | - | 補足が不要な場合は `-` を記載する。 |
| メッセージ領域 | エラー、警告、通知を表示する。表示条件、メッセージ種別、表示優先順位を必要に応じて書く。 | ラベル | - | 関連メッセージIDがある場合のみ記載する。 |

書かない: 表の列追加や画面ごとの独自列。<br>

## 5. イベント一覧
【template-guidance】<br>
必須: 画面で発生する主要イベントを `5.1. 初期表示時` `5.2. 〇〇時` の小節ごとに整理し、番号付き手順で書く。`5.1. 初期表示時` は原則必須とする。<br>
任意: 複雑な分岐や状態遷移は、必要に応じて表や `mermaid` を補助的に使ってよい。<br>
書かない: クラス名、関数名、内部 API 呼出順、詳細設計レベルの実装手順。<br>
【/template-guidance】<br>

### 5.1. 初期表示時
【template-guidance】<br>
必須: 認証確認、表示対象の特定、データ取得、権限確認、初期値設定、表示更新、メッセージ表示の扱いを外部設計の粒度で書く。入力を伴う画面では、初期値設定やバリデーション前提も書く。<br>
任意: データなし時、権限不正時、セッション切れ時の扱いを追記してよい。<br>
書かない: SQL、ORM、画面部品の描画実装。<br>
【/template-guidance】<br>

1. 利用者の認証状態を確認し、未認証の場合はログイン画面へ遷移させる。
2. 表示対象または初期表示条件を判定し、必要なデータを取得する。
3. 取得結果に対して参照権限または表示条件を確認し、不正時はメッセージ表示や遷移を行う。
4. 各領域へ初期表示値を設定し、必要な項目を表示する。
5. データなし時や注意事項がある場合は、所定のメッセージまたは補足表示を行う。

### 5.2. 〇〇時
【template-guidance】<br>
必須: ボタン押下、画面切替、検索、登録、更新、削除、入力確定などのイベントごとに、入力値確認、バリデーション、分岐、メッセージ表示、画面更新、遷移の流れを書く。<br>
任意: 確認ダイアログ、再入力、リトライ、例外系を追記してよい。<br>
書かない: 内部例外スタックやコードレベルの戻り値処理。<br>
【/template-guidance】<br>

1. 利用者が〇〇を実行したことを検知する。
2. 必要な入力値や選択状態を確認し、バリデーションエラーがある場合は対象項目へメッセージを表示して同一画面に留める。
3. 入力内容が妥当な場合は、対象データの参照、更新、登録、削除、または画面切替を行う。
4. 処理結果に応じて画面表示を更新し、必要に応じて完了メッセージまたはエラーメッセージを表示する。
5. 遷移が必要な場合は、所定の画面へ遷移する。
