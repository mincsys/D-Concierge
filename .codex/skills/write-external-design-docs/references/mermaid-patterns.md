# Mermaid 図種対応表

## 1. 基本方針

- 図は本文の補助として使い、図だけで仕様を完結させない。
- 文書の目的に対して最も単純な図種を選ぶ。
- 用語は本文と一致させ、内部実装名を出さない。

## 2. 文書と図種の対応

| 文書種別 | 推奨図種 | 主な用途 |
| --- | --- | --- |
| システム概要 | `flowchart` | 利用者、システム、外部サービスの関係 |
| ハードウェア構成 | `flowchart` | 端末、ネットワーク機器、サーバの接続 |
| ソフトウェア構成 | `flowchart` | フロント、バック、DB、外部サービスの関係 |
| ネットワーク構成 | `flowchart` | 通信経路、境界、接続方向 |
| 業務フロー | 原則 `sequenceDiagram`、例外的に `flowchart` | 主体間の時系列、画面とアプリのやり取り、外部サービス連携、分岐を含む業務手順 |
| 画面一覧 | `flowchart` | 画面遷移 |
| 論理データ設計 | `erDiagram` | エンティティと関連 |
| 〇〇IF | `flowchart` | IF 相手とシステムの関係、主要な授受情報の俯瞰 |
| 非機能設計 | `flowchart` | 性能目標や制約の補助説明 |

## 3. 避けること

- 1つの図に複数視点を混在させること
- 置換前の `〇〇` を図に残すこと
- 本文と別の用語を図で使うこと
- `autonumber` なしのシーケンス図を既定にすること
- `actor` や `participant` を省略して責務境界を曖昧にすること
- 改行に `\n` を使うこと（代わりに`<br>`を使って改行する）
