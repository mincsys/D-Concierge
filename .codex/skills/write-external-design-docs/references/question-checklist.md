# 外部設計 質問チェックリスト

## 使い方

- リポジトリ、既存文書、ユーザ指定から一意に分かる内容は質問しない。
- 外部設計作成時点では要件定義書を主資料として扱い、モックがある場合は画面、導線、項目、表示文言の補助資料として扱う。
- 要件定義書とモックが衝突する場合は、どちらを優先するかを衝突箇所ごとに質問する。
- 内部設計書、テスト設計書、実装が存在する場合でも、外部設計作成の前提資料として存在を必須にしない。
- 未確定事項は本文作成前にユーザへ提示し、合意してから外部設計書へ反映する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 各質問では、確認が必要な理由、影響する文書、未確定のまま進めた場合に曖昧になる内容を説明する。
- 選択肢を提示する場合は、採用時の影響が明確に異なるものだけにする。
- 選択肢を提示する場合は、必ず Markdown の番号付きリストで `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のように書く。
- 質問文では、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ明示する。
- 推奨選択肢は番号付きリスト内で `1. 選択肢名（推奨）` のように表記する。
- 推奨理由は、選択肢本文または質問説明の中で簡潔に説明する。
- 複数の選択肢を同時に推奨扱いにしない。
- 選択肢に `など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- 推奨選択肢は、要件定義書、既存文書、ユーザ指定、ウォーターフォール文書としての契約安全性を基準に選ぶ。
- 設計書本文を作成する直前に、全質問項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理し、未合意項目がある場合は本文作成へ進まない。
- 質問回答とリポジトリ事実の整理がそろった場合、質問表にある全質問項目と合意結果を一覧化してユーザへ表示する。
- 合意結果一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイルや設定から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。

## 1. システム構成

- システムの全体像、利用者、構成要素、外部連携先をどう定義するか。
- ハードウェア、ソフトウェア、ネットワークのうち、要件定義書から確定できない範囲をどう扱うか。
- ローカル実行、サーバ実行、外部サービス接続、利用端末、通信境界の前提をどう置くか。

## 2. 業務設計

- 業務単位、業務主体、開始契機、終了条件をどう分けるか。
- 個別業務フローをどの業務に対して作成するか。
- 正常系、例外系、途中終了、再実行、取り消しを業務上どう扱うか。

## 3. 機能設計

- 機能ID、機能名、入出力、関連画面、関連データをどう整理するか。
- 作成、参照、更新、削除、実行、キャンセル、再試行の可否をどう定義するか。
- 対象外機能、将来構想、初期リリース範囲をどう分けるか。

## 4. 画面設計

- 画面一覧へ載せる画面、ダイアログ、ビューア、共通部品をどう分けるか。
- 個別画面設計をどの画面に対して作成するか。
- モックがある場合、画面構成、導線、項目名、表示文言を外部設計へどう反映するか。
- 空データ、入力エラー、処理中、失敗、タイムアウト、制約超過時の表示をどう扱うか。

## 5. 論理データ設計

- エンティティ、主要属性、関連、保持方針をどう定義するか。
- 保存対象、保存しない情報、履歴化、削除時の扱いをどう分けるか。
- 個人情報、認証情報、添付ファイル、外部サービス由来データを保持するか。

## 6. 外部インターフェース設計

- IF 相手、通信方向、入力条件、出力条件、エラー時処理をどう定義するか。
- API、CLI、設定ファイル、外部サービス、ファイル入出力をどの単位で文書化するか。
- 認証、再試行、タイムアウト、部分成功、通信失敗、応答形式不正をどう扱うか。

## 7. 非機能設計

- 性能、信頼性、操作性、セキュリティ、運用、保守、移行のうち、要件定義書から確定できない項目をどう扱うか。
- 同時利用、データ量、処理時間、ブラウザ、端末、外部サービス停止時の前提をどう置くか。
- ログ、監視、バックアップ、復旧、証跡、保守作業の方針をどう定義するか。

## 8. 共通設計

- ログ分類、ログ出力条件、ログ保持方針をどう定義するか。
- エラーメッセージの ID、表示文言、発生条件、利用者向け説明をどう整理するか。
- 用語、列挙値、状態名、権限名、画面表示名を文書間でどう統一するか。

## 9. 契約論点

- 権限、責務分担、対象外範囲、削除可否、更新可否をどう定義するか。
- 異常系、失敗時、部分成功、再試行、保存可否をどう扱うか。
- 空データ、入力制約超過、外部IFエラー、タイムアウト、キャンセル、復旧不能時の扱いをどう定義するか。
- 要件定義書、モック、既存外部設計、ユーザ指定の間に矛盾がある場合、どれを正として直すか。

## 10. 作成対象ファイル

- `assets/02_外部設計/テンプレート索引.md` の必須文書をすべて採用するか。
- 任意文書を採用するか。
- `〇〇画面.md`、`〇〇フロー.md`、`〇〇IF.md` をどの具体名で作成するか。
- 同じ設計内容を複数文書へ重複記載しない前提でよいか。
