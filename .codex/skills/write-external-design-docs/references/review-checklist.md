# 外部設計書レビュー観点

## 要件トレース

- 要件定義書の業務要件、機能要件、画面要件、データ要件、外部連携要件、非機能要件に対応する文書があるか。
- 要件にある利用者区分、対象範囲、制約が設計書へ反映されているか。

## 文書構成

- ユーザ指定の素案フォーマットを守っているか。
- 追加資料がある場合でも `09_その他/` に限定されているか。
- 各文書に `文書の目的` `前提` があり、文書種別に応じた設計章がそろっているか。
- assets 上の `汎用ひな型` が具体名へ置き換えられ、成果物へそのまま残っていないか。

## 質問合意

- `references/question-checklist.md` の項目が、`合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理されているか。
- 質問が 1 つずつ行われ、合意内容が整理されているか。
- 選択肢が `1. 選択肢名`、`2. 選択肢名`、`3. 選択肢名` のような番号付きリストになっているか。
- ユーザが番号だけで回答できることを質問文で明記しているか。
- 選択肢番号が質問ごとに `1` から始まっているか。
- 推奨選択肢が `1. 選択肢名（推奨）` のように 1 つだけ明示されているか。
- 推奨理由が、選択肢本文または質問説明の中で簡潔に説明されているか。
- 複数の選択肢が同時に推奨扱いになっていないか。
- 選択肢に `など`、`等`、`その他いろいろ` のような曖昧な表現が使われていないか。
- 本文作成直前に、合意結果一覧がユーザへ表示されているか。
- 合意結果一覧に `質問項目`、`合意結果`、`状態`、`根拠` が含まれているか。
- 合意結果一覧の表示後、ユーザから明確な作成指示を受けてから本文作成へ進んでいるか。
- `確認しました`、`OKです`、`問題ありません` だけで本文作成へ進んでいないか。
- 未合意の契約論点が本文へ持ち込まれていないか。

## 横断整合

- 業務一覧、機能一覧、画面一覧、外部IF、論理データで名称と粒度が一致しているか。
- 画面一覧の各画面に個別画面設計があるか。
- 主要画面に対応するAPIまたはIFが定義されているか。
- 主要機能に必要なエンティティや保持項目が論理データ設計にあるか。
- 論理データ設計に、現行要件の業務、機能、画面、外部IF、非機能要件から利用目的を説明できないエンティティや属性が含まれていないか。
- `将来使う可能性がある`、`あると便利`、`一般的には持つ` だけのデータ、履歴、メタ情報が含まれていないか。
- 固定値や列挙値が文書ごとに食い違っていないか。

## 表現

- 日本語の正式文書として自然な文体になっているか。
- 差分説明、旧仕様の痕跡、検討メモが残っていないか。
- `mermaid` 図が本文の説明と矛盾していないか。
- `【template-guidance】` ブロックが最終成果物に残っていないか。
- `〇〇` や `外部サービスA` などの未置換プレースホルダが残っていないか。

## 留意事項

- 要件定義書を主資料として扱い、モックがある場合は画面、導線、項目、表示文言の補助資料として扱っているか。
- 要件定義書、既存外部設計、モック、ユーザ指定から明確に読み取れる事項だけを確定事項として扱っているか。
- 要件定義書とモックが衝突する箇所を、ユーザ合意なしで固定していないか。
- 内部設計書、テスト設計書、実装の存在を外部設計作成の必須前提として扱っていないか。
- 契約確認が必要な論点を未確認のまま固定していないか。
- 異常系や矛盾点をユーザ合意なしで解釈していないか。
- 推測した前提がある場合、ユーザの明示指示に基づくものであり、関連文書で一貫して扱われているか。
- 設計の制約や前提が非機能設計やシステム構成へ落ちているか。
- 採用しなかった任意文書や任意章が空のまま成果物へ残っていないか。
