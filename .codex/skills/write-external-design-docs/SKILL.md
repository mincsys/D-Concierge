# Write External Design Docs

## Overview

要件定義書から、日本語の外部設計書群を既定では `docs/02_外部設計` 配下へ作成する。`assets/02_外部設計` にある実テンプレートを出発点にし、必要な文書だけを複製、具体化、整合確認する。外部設計作成時点では要件定義書を主資料とし、モックがある場合は画面、導線、項目、表示文言の補助資料として扱う。

## Workflow

### 1. 入力資料を読む

- 先に `docs/` 配下とリポジトリ内の指示文書を読む。
- 外部設計作成時点では、要件定義書を主資料として扱う。
- モックがある場合は、画面構成、導線、項目名、表示文言を読み取り、要件定義書と衝突する箇所は固定せず質問対象にする。
- 内部設計書、テスト設計書、実装が存在する場合でも、外部設計作成の前提資料として存在を必須にしない。
- ユーザが指定したフォルダ構成、ファイル名、文書粒度を最優先で守る。
- 既存の外部設計書がある場合は、用語、ID体系、章立て、表記を合わせる。

### 2. 明確に読み取れる確定事項を先に固定する

- 要件定義書、既存外部設計、ユーザ指定から一意に確定できる事項は、そのまま確定事項として扱う。
- 文書間で矛盾がなく、契約上も解釈の余地がない事項は再確認しない。
- 確定事項と未確定事項を混ぜず、先に確定事項の一覧を頭の中で整理する。
- 既存文書や設定から一意に判断した事項は、後続の合意結果一覧で `リポジトリ事実として確定` として扱えるよう根拠を控える。

### 3. 未確定論点を提案付きで合意する

- 未確定事項は少なくとも `契約確認` `異常系` `疑問点・矛盾点` `文書間整合に関わる論点` に分類する。
- `references/question-checklist.md` を使い、テンプレート対応論点、契約論点、モック論点、作成対象ファイルを確認する。
- 次のような事項は、要件から明確でない限り確認対象にする。
  - 仕様として複数解釈が成立する事項
  - 異常系、失敗時、部分成功、再試行、保存可否などの扱い
  - 権限、責務分担、対象外範囲、削除・更新可否
  - 外部IFの入力条件、出力条件、エラー時処理
  - 画面や帳票での空データ時、失敗時、制約超過時の挙動
  - 文書間の矛盾、用語不一致、ID体系不一致、列挙値の不一致
- 質問は単発で投げず、論点ごとに `現状の読み取り` `未確定理由` `推奨案` `必要な判断` を付けて提示する。
- 質問はまとめて行わず、必ず 1 つずつ順番に行う。
- 選択肢を提示する場合は、採用時の影響が明確に異なるものだけを Markdown の番号付きリストで示し、ユーザが番号だけで回答できることを明記する。
- 選択肢番号は質問ごとに `1` から始め、複数質問をまたいで連番にしない。
- 選択肢を提示する場合は、必ず推奨選択肢を 1 つだけ `1. 選択肢名（推奨）` のように明示し、推奨理由を簡潔に添える。
- 選択肢に `など`、`等`、`その他いろいろ` のような曖昧な表現を使わない。
- 矛盾を見つけた場合は、矛盾している文書や記述箇所を示し、解消案を提案する。
- 複数文書に波及する論点の回答がそろうまで、外部設計本文の起票を開始しない。
- 設計中に新しい矛盾や未確定事項が出た場合は、その場で起票を止め、再度合意を取ってから続ける。

### 4. 本文作成直前に合意状況を再確認する

- 外部設計本文を作成する直前に、`references/question-checklist.md` の項目を `合意済み`、`未合意`、`リポジトリ事実として確定` のいずれかで整理する。
- 合意結果一覧には `質問項目`、`合意結果`、`状態`、`根拠` を含める。
- `合意済み` の根拠には、ユーザがどの回答で合意したかを書く。
- `リポジトリ事実として確定` の根拠には、どのファイルや設定から一意に判断したかを書く。
- `未合意` の根拠には、何が未回答または未確定かを書く。
- 未合意項目がある場合は、外部設計本文の作成へ進まない。
- 合意結果一覧をユーザへ表示した後、ユーザから明確な作成指示を受けるまで本文作成へ進まない。
- 明確な作成指示の例は、`作成してください`、`この内容で作成してください`、`本文作成に進んでください` とする。
- `確認しました`、`OKです`、`問題ありません` だけでは、作成開始指示とは扱わない。

### 5. assets から必要テンプレートを選ぶ

- `assets/02_外部設計/テンプレート索引.md` を先に読む。
- `必須` は原則採用する。
- `任意` は要件またはユーザ指定で必要なときだけ採用する。
- `汎用ひな型` は最終成果物へそのまま置かず、具体名へ複製して使う。
- `〇〇IF.md` は IF 相手ごとに 1 文書を作る汎用ひな型であり、1 文書内に複数 IF を含めてよい。
- どの文書を採るかの判断は [references/template-selection-rules.md](references/template-selection-rules.md) を参照する。

### 6. テンプレートを具体文書へ仕上げる

- 必要なテンプレートを既定では `docs/02_外部設計` 配下へ複製してから編集する。
- 文書先頭の `【template-guidance】` にある `章構成` で、番号付き見出しの `【必須】` と `【任意】` を確認する。
- 各章の `【template-guidance】` にある `必須` `任意` `書かない` を読み、本文を要件に合わせて具体化する。
- 最終成果物には `【template-guidance】` ブロック、未置換の `〇〇`、不要な任意章を残さない。
- `mermaid` 図が必要な文書では、図を本文の補助として挿入する。

### 7. 文書間の整合を揃える

- 業務、機能、画面、IF、論理データ、非機能で名称と粒度を揃える。
- 画面一覧と個別画面設計、業務一覧と業務フロー、個別 IF 文書と画面・機能の関係を対応付ける。
- 最後に [references/review-checklist.md](references/review-checklist.md) で横断確認する。

## Clarification Rules

- 明確に読み取れる事項は質問せず、そのまま確定事項として扱う。
- 未確定事項を推測だけで埋めず、契約として確認すべき論点は必ずユーザへ提示する。
- ユーザへの提示は、質問列挙ではなく論点整理を優先する。
- 合意前に行うのは、確定事項の整理、未確定論点の整理、推奨案の提示までとする。
- 外部設計本文の起票は、複数文書に波及する主要論点の合意後、合意結果一覧を提示し、ユーザから明確な作成指示を受けてから始める。
- 合意できていない前提を本文へ持ち込む場合は、ユーザが明示的にその進め方を指示したときに限る。

## Template Rules

- テンプレート本文は完全抽象で扱い、特定プロジェクト固有の技術名やベンダ名を持ち込まない。
- 各テンプレートの `【template-guidance】` ブロックは次の規約で読む。
- 文書先頭のブロックは `【template-guidance】<br>` で始まり、`文書区分` `使う場面` `削除条件` `章構成` を記載し、`【/template-guidance】<br>` で閉じる。
- 文書先頭の `章構成` は番号付き見出しだけを Markdown リストで並べ、各項目に `【必須】` または `【任意】` を付ける。
- `章構成` のリスト項目には `<br>` を付けず、最後のリスト項目の直後に空行を 1 行入れてから `【/template-guidance】<br>` を置く。
- `章構成` の見出し番号は `1.` `4.1.` `4.2.1.` のように末尾ピリオド付きで書く。
- `章構成` には `IF-01` のような非番号小節を含めない。
- 文書先頭のブロックは章の採用要否を示す。
- `文書区分`: `必須` `任意` `汎用ひな型` のいずれか
- `使う場面`: そのテンプレートを採用する条件
- `削除条件`: 不採用時の扱い、または最終成果物からガイダンスブロックを外す条件
- `【必須】`: 文書先頭の `章構成` において通常採用する番号付き見出し
- `【任意】`: 文書先頭の `章構成` において条件付きで採用する番号付き見出し。不要なら章ごと削除する
- 章本文側のブロックは `【template-guidance】<br>` で始まり、その章や下位見出しで何を書くべきか、何を書くべきでないか、なにを任意で書いてもよいか、その他必要事項などを自由文で記載し、`【/template-guidance】<br>` で閉じる。
- 章本文側のブロックは `##` `###` `####` など必要な任意の見出し階層の直下へ置ける。
- 親章に総論 guidance を置いてよいが、実際の記述指示はできるだけ最下位の実記述見出しへ寄せる。
- `〇〇画面.md` `〇〇フロー.md` `〇〇IF.md` は assets 上の汎用ひな型であり、そのまま成果物へ置かない。

## Writing Rules

- 文書は日本語で書く。
- 各文書は原則として `文書の目的` `前提` を持たせ、以降は文書種別に応じた設計章を置く。
- 検討経緯、会話ログ、差分説明ではなく、完成済みの正式文書として書く。
- `mermaid` の図種選定は [references/mermaid-patterns.md](references/mermaid-patterns.md) を参照する。
