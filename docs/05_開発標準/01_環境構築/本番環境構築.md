# 本番環境構築

## 1. 文書の目的

本書は、D-Concierge MVPの本番環境を構築し、社内LANで実運用できる状態にするための環境構成、構築手順、起動確認、監視、ログ、バックアップおよび障害時対応の入口を定義することを目的とする。

本書では、本番環境の構築パターンとしてDocker Compose版、Linux非Docker版、Windows非Docker版を定義する。

## 2. 前提

- 本番環境は社内LANでの小規模利用を対象とする。
- 本番環境は単一ホスト構成とし、Web画面、REST API、SSE、codex exec、PostgreSQLを同一ホスト上で動作させる。
- 利用者端末からPostgreSQLへ直接接続しない。
- `config.example.yaml` はGit管理対象の設定ひな形とし、本番環境ではこれをコピーして `config.yaml` を作成する。
- `config.yaml` は本番環境固有値と秘密値を含むためGit管理対象外とする。
- Codex CLIの認証情報は本番ホストまたは本番コンテナ実行環境で事前に準備し、Git管理対象外とする。
- DB migrationはAlembicで実行し、共有1ユーザはSQLで手動投入する。
- バックアップは任意とし、実施有無、取得頻度、保存先、保存期間は運用者が決定する。
- 本番環境で使用するDBパスワード、Codex認証情報、サーバアカウント情報、個人情報は本書に記載しない。

## 3. 本番環境構成

### 3.1. 構成パターン

| パターン | 用途 | 実行基盤 | 起動管理 |
| --- | --- | --- | --- |
| Docker Compose版 | 社内配布の標準構成 | `infra/Dockerfile`、`infra/compose.yml` | Docker Compose |
| Linux非Docker版 | Linux単一ホストへ直接配置する構成 | ホスト上のPython、Node.js、PostgreSQL、Codex CLI | systemd |
| Windows非Docker版 | Windows単一ホストへ直接配置する構成 | ホスト上のPython、Node.js、PostgreSQL、Codex CLI | タスクスケジューラ |

### 3.2. 構成要素

| 構成要素 | 用途 | 配置先 | 備考 |
| --- | --- | --- | --- |
| フロントエンド静的ファイル | Web画面表示 | アプリケーション配信領域 | `src/frontend` のビルド成果物 |
| バックエンド | REST、SSE、静的配信、実行制御 | アプリケーションホスト | FastAPI、Uvicornで起動する |
| PostgreSQL | 履歴、実行状態、回答、参照元、成果物メタ情報の保存 | アプリケーションホスト | 利用者端末へ直接公開しない |
| 生成用Codex CLI | 回答生成 | アプリケーションホストまたはアプリコンテナ | 認証済み状態で使用する |
| 検証用Codex CLI | 参照元検証 | アプリケーションホストまたはアプリコンテナ | 認証済み状態で使用する |
| 共有データソース | Codexの検索・分析対象 | `codex/readonly` | 利用者は直接参照しない |
| 保存済みCodex成果物 | 検証済み回答が参照する成果物本体 | `codex/saved_artifacts` | API経由で配信する |
| トレースログ | 異常系障害調査用YAMLログ | `logs/trace` | 利用者画面へ表示しない |
| アプリ共通タイムゾーン | 運用者向け日時の基準 | `app.timezone` | IANA timezone名で指定し、MVP標準は `Asia/Tokyo` とする |

## 5. 環境変数・機密情報

| 項目 | 管理方法 | 用途 | 備考 |
| --- | --- | --- | --- |
| `config.example.yaml` | Git管理 | 本番設定ひな形 | 秘密値を含めない |
| `config.yaml` | 本番ホスト上で管理 | 本番アプリ設定 | Git管理しない |
| `database.url` | `config.yaml` | アプリケーション実行時のDB接続先 | DBパスワードを含む場合はファイル権限で保護する |
| `D_CONCIERGE_DATABASE_URL` | migration実行時の環境変数 | AlembicのDB接続先 | 恒久設定ではなく実行時だけ指定する |
| `POSTGRES_DB` | Docker Compose環境変数 | PostgreSQL DB名 | Docker Compose版で使用する |
| `POSTGRES_USER` | Docker Compose環境変数 | PostgreSQLユーザ | Docker Compose版で使用する |
| `POSTGRES_PASSWORD` | Docker Compose環境変数 | PostgreSQLパスワード | 本番値をGit管理しない |
| `APP_PORT` | Docker Compose環境変数 | アプリケーション公開ポート | 未指定時は `8000` |
| Codex CLI認証情報 | 各ホストで事前ログイン・事前配置 | `codex exec` 実行 | Git管理しない |
| `UV_CACHE_DIR` | OS環境変数またはサービス環境変数 | uvキャッシュ保存先 | Linuxでは `/tmp/uv-cache`、Windowsでは `%TEMP%\uv-cache` |

## 6. インフラ準備

### 6.1. Docker Compose版

本番ホストにDockerとDocker Composeを導入し、リポジトリを配置する。

設定ファイルと永続領域を準備する。

```bash
cp config.example.yaml config.yaml
mkdir -p logs/trace codex/saved_artifacts codex/sessions codex/sessions_validator
```

`config.yaml` の `database.url` は、Docker Compose内部のPostgreSQLへ接続する値へ変更する。

```yaml
database:
  url: "postgresql+psycopg://d_concierge:<本番DBパスワード>@postgres:5432/d_concierge"
```

Docker Compose実行時には本番DBパスワードを環境変数で指定する。

```bash
export POSTGRES_DB='d_concierge'
export POSTGRES_USER='d_concierge'
export POSTGRES_PASSWORD='<本番DBパスワード>'
export APP_PORT='8000'
```

Docker Compose版では、アプリコンテナ内にCodex CLIを含め、認証情報と用途別Codex資産はホスト側の `codex/` を `/app/codex` へマウントして利用する。運用者は、ホスト側の `codex/.codex` と `codex/.codex_validator` を認証済み状態にする。

### 6.2. Linux非Docker版

本番ホストにPython 3.12、uv、Node.js 24、npm、PostgreSQL、psql、Codex CLIを導入し、リポジトリを配置する。

アプリケーション配置先を `/opt/d-concierge` とする。

```bash
id -u d-concierge >/dev/null 2>&1 || sudo useradd --system --home /opt/d-concierge --shell /usr/sbin/nologin d-concierge
sudo mkdir -p /opt/d-concierge
sudo chown -R d-concierge:d-concierge /opt/d-concierge
```

設定ファイルと永続領域を準備する。

```bash
cp config.example.yaml config.yaml
mkdir -p logs/trace codex/saved_artifacts codex/sessions codex/sessions_validator
```

PostgreSQLに本番DBと本番ユーザを作成し、`config.yaml` の `database.url` を本番DB接続先へ変更する。

### 6.3. Windows非Docker版

本番ホストにPython 3.12、uv、Node.js 24、npm、PostgreSQL、psql、Codex CLIを導入し、リポジトリを配置する。

アプリケーション配置先を `C:\D-Concierge` とする。

```powershell
New-Item -ItemType Directory -Force C:\D-Concierge
```

設定ファイルと永続領域を準備する。

```powershell
Copy-Item config.example.yaml config.yaml
New-Item -ItemType Directory -Force logs\trace
New-Item -ItemType Directory -Force codex\saved_artifacts
New-Item -ItemType Directory -Force codex\sessions
New-Item -ItemType Directory -Force codex\sessions_validator
```

検証用Codexへ生成成果物を提示する処理ではシンボリックリンクを使用する。Windows非Docker版では、タスク実行ユーザにsymlink作成権限を付与するか、開発者モードと同等の設定を有効にする。symlink作成に失敗する環境は環境不備として扱い、アプリ側でコピーへフォールバックしない。

PostgreSQLに本番DBと本番ユーザを作成し、`config.yaml` の `database.url` を本番DB接続先へ変更する。

## 7. アプリケーション配置

### 7.1. Docker Compose版

本番イメージをビルドする。

```bash
docker compose -f infra/compose.yml build app
```

### 7.2. Linux非Docker版

Python依存関係を同期する。

```bash
UV_CACHE_DIR=/tmp/uv-cache uv sync --frozen --no-dev
```

フロントエンドをビルドし、バックエンドの静的配信領域へ配置する。

```bash
cd src/frontend
npm ci
npm run build
cd ../..
rm -rf src/backend/app/static/dist
mkdir -p src/backend/app/static
cp -R src/frontend/dist src/backend/app/static/dist
```

### 7.3. Windows非Docker版

Python依存関係を同期する。

```powershell
$env:UV_CACHE_DIR = "$env:TEMP\uv-cache"
uv sync --frozen --no-dev
```

フロントエンドをビルドし、バックエンドの静的配信領域へ配置する。

```powershell
Set-Location src/frontend
npm ci
npm run build
Set-Location ../..
Remove-Item -Recurse -Force src/backend/app/static/dist -ErrorAction SilentlyContinue
New-Item -ItemType Directory -Force src/backend/app/static
Copy-Item -Recurse src/frontend/dist src/backend/app/static/dist
```

## 8. 初期データ・マイグレーション

### 8.1. Docker Compose版

PostgreSQLを起動する。

```bash
docker compose -f infra/compose.yml up -d postgres
```

DB migrationを実行する。

```bash
docker compose -f infra/compose.yml run --rm \
  -e D_CONCIERGE_DATABASE_URL='postgresql+psycopg://d_concierge:<本番DBパスワード>@postgres:5432/d_concierge' \
  app .venv/bin/alembic upgrade head
```

共有1ユーザを投入する。

```bash
docker compose -f infra/compose.yml exec -T postgres psql \
  -U d_concierge \
  -d d_concierge <<'SQL'
INSERT INTO local_users (id, display_name, is_active)
VALUES ('00000000-0000-0000-0000-000000000001', '共有利用者', true)
ON CONFLICT (id) DO UPDATE
SET display_name = EXCLUDED.display_name,
    is_active = true;
SQL
```

### 8.2. Linux非Docker版

DB migrationを実行する。

```bash
D_CONCIERGE_DATABASE_URL='postgresql+psycopg://d_concierge:<本番DBパスワード>@127.0.0.1:5432/d_concierge' \
UV_CACHE_DIR=/tmp/uv-cache \
.venv/bin/alembic upgrade head
```

共有1ユーザを投入する。

```bash
psql 'postgresql://d_concierge:<本番DBパスワード>@127.0.0.1:5432/d_concierge' <<'SQL'
INSERT INTO local_users (id, display_name, is_active)
VALUES ('00000000-0000-0000-0000-000000000001', '共有利用者', true)
ON CONFLICT (id) DO UPDATE
SET display_name = EXCLUDED.display_name,
    is_active = true;
SQL
```

### 8.3. Windows非Docker版

DB migrationを実行する。

```powershell
$env:D_CONCIERGE_DATABASE_URL = "postgresql+psycopg://d_concierge:<本番DBパスワード>@127.0.0.1:5432/d_concierge"
$env:UV_CACHE_DIR = "$env:TEMP\uv-cache"
.\.venv\Scripts\alembic.exe upgrade head
```

共有1ユーザを投入する。

```powershell
@"
INSERT INTO local_users (id, display_name, is_active)
VALUES ('00000000-0000-0000-0000-000000000001', '共有利用者', true)
ON CONFLICT (id) DO UPDATE
SET display_name = EXCLUDED.display_name,
    is_active = true;
"@ | psql "postgresql://d_concierge:<本番DBパスワード>@127.0.0.1:5432/d_concierge"
```

## 9. 起動・停止手順

### 9.1. Docker Compose版

起動する。

```bash
docker compose -f infra/compose.yml up -d --build app
```

停止する。

```bash
docker compose -f infra/compose.yml down
```

アプリケーションだけを再起動する。

```bash
docker compose -f infra/compose.yml restart app
```

### 9.2. Linux非Docker版

systemdサービスを作成する。

```bash
sudo tee /etc/systemd/system/d-concierge.service >/dev/null <<'UNIT'
[Unit]
Description=D-Concierge
After=network.target postgresql.service

[Service]
Type=simple
User=d-concierge
WorkingDirectory=/opt/d-concierge
Environment=UV_CACHE_DIR=/tmp/uv-cache
ExecStart=/opt/d-concierge/.venv/bin/uvicorn backend.main:app --app-dir src --host 0.0.0.0 --port 8000
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
UNIT
```

起動する。

```bash
sudo systemctl daemon-reload
sudo systemctl enable d-concierge
sudo systemctl start d-concierge
```

停止する。

```bash
sudo systemctl stop d-concierge
```

再起動する。

```bash
sudo systemctl restart d-concierge
```

### 9.3. Windows非Docker版

タスクスケジューラに起動タスクを登録する。作業ディレクトリは `C:\D-Concierge` とする。

```powershell
$Action = New-ScheduledTaskAction `
  -Execute "powershell.exe" `
  -Argument "-NoProfile -ExecutionPolicy Bypass -Command `"Set-Location C:\D-Concierge; `$env:UV_CACHE_DIR=`"`$env:TEMP\uv-cache`"; .\.venv\Scripts\uvicorn.exe backend.main:app --app-dir src --host 0.0.0.0 --port 8000`""
$Trigger = New-ScheduledTaskTrigger -AtStartup
Register-ScheduledTask -TaskName "D-Concierge" -Action $Action -Trigger $Trigger -RunLevel Highest
```

起動する。

```powershell
Start-ScheduledTask -TaskName "D-Concierge"
```

停止する。

```powershell
Stop-ScheduledTask -TaskName "D-Concierge"
```

タスクを削除する。

```powershell
Unregister-ScheduledTask -TaskName "D-Concierge" -Confirm:$false
```

## 11. 監視・ログ・バックアップ

### 11.1. 監視・ログ確認

| パターン | 確認対象 | 確認コマンド |
| --- | --- | --- |
| Docker Compose版 | アプリケーションログ | `docker compose -f infra/compose.yml logs app` |
| Docker Compose版 | PostgreSQLログ | `docker compose -f infra/compose.yml logs postgres` |
| Linux非Docker版 | systemdログ | `journalctl -u d-concierge` |
| Windows非Docker版 | タスク状態 | `Get-ScheduledTask -TaskName "D-Concierge"` |
| 共通 | トレースログ | `logs/trace/<yyyy-MM-dd>/` 配下のYAMLファイル |
| 共通 | DB容量 | PostgreSQLのDBサイズ |
| 共通 | 成果物容量 | `codex/saved_artifacts` の使用量 |
| 共通 | トレースログ容量 | `logs/trace` の使用量 |

容量確認は運用者が定期的に実施する。トレースログは `app.timezone` 基準の日付で、`trace_log.retention_days` の保存期間を過ぎた日付ディレクトリをアプリケーション起動時に削除し、`trace_log.max_files_per_day` の設定に従ってアプリケーション起動ごとの同日保存件数を制限する。チャット履歴、DB、保存済みCodex成果物の自動削除機能はMVPでは提供しない。

### 11.2. 任意バックアップ

バックアップ実施有無、頻度、保存先、保存期間は運用者が決定する。バックアップを行う場合は、PostgreSQL、保存済みCodex成果物、トレースログ、`config.yaml` を対象にする。

Docker Compose版のPostgreSQLバックアップ例:

```bash
mkdir -p backup
docker compose -f infra/compose.yml exec -T postgres pg_dump \
  -U d_concierge \
  -d d_concierge > backup/d_concierge_$(date +%Y%m%d).sql
```

Linux非Docker版のPostgreSQLバックアップ例:

```bash
mkdir -p backup
pg_dump 'postgresql://d_concierge:<本番DBパスワード>@127.0.0.1:5432/d_concierge' \
  > backup/d_concierge_$(date +%Y%m%d).sql
```

Windows非Docker版のPostgreSQLバックアップ例:

```powershell
New-Item -ItemType Directory -Force backup
pg_dump "postgresql://d_concierge:<本番DBパスワード>@127.0.0.1:5432/d_concierge" `
  | Out-File -Encoding utf8 "backup\d_concierge_$(Get-Date -Format yyyyMMdd).sql"
```

ファイル領域のバックアップ対象:

| 対象 | 内容 |
| --- | --- |
| `codex/saved_artifacts` | 保存済みCodex成果物 |
| `logs/trace` | 障害調査用トレースログ |
| `config.yaml` | 本番設定 |

Linuxでは `tar`、Windowsでは `Compress-Archive` を使用して、対象ディレクトリを任意のバックアップ保存先へ退避する。

### 11.3. 任意バックアップの自動化

Docker Compose版とLinux非Docker版では、運用者判断で `cron` または `systemd timer` によりバックアップコマンドを定期実行する。Windows非Docker版では、運用者判断でタスクスケジューラによりバックアップコマンドを定期実行する。

自動化する場合も、バックアップ結果の確認と復元確認は運用者が実施する。

## 12. 障害時対応

| 事象 | 初動 | 確認対象 | エスカレーション |
| --- | --- | --- | --- |
| 画面に接続できない | アプリケーションプロセスを確認し、必要に応じて再起動する | Dockerログ、systemdログ、タスク状態、ポート公開状態 | 運用者 |
| APIがエラーを返す | トレースログとアプリケーションログを確認する | `logs/trace`、アプリケーションログ、DB接続 | 運用者 |
| DB接続に失敗する | PostgreSQLの稼働状態と `config.yaml` のDB接続先を確認する | PostgreSQLログ、DBユーザ、DBパスワード、ネットワーク | 運用者 |
| codex execが失敗する | Codex CLIの認証状態と実行環境を確認する | Codex CLI、`codex/.codex`、`codex/.codex_validator`、トレースログ | 運用者 |
| 容量不足が発生する | DB、保存済み成果物、トレースログの使用量を確認する | PostgreSQL、`codex/saved_artifacts`、`logs/trace` | 運用者 |
| データ復旧が必要 | 任意バックアップの有無を確認し、復元対象を特定する | DBバックアップ、ファイルバックアップ、`config.yaml` | 運用者 |

再起動コマンド:

| パターン | コマンド |
| --- | --- |
| Docker Compose版 | `docker compose -f infra/compose.yml restart app` |
| Linux非Docker版 | `sudo systemctl restart d-concierge` |
| Windows非Docker版 | `Stop-ScheduledTask -TaskName "D-Concierge"`、`Start-ScheduledTask -TaskName "D-Concierge"` |
