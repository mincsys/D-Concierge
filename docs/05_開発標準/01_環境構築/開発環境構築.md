# 開発環境構築

## 1. 文書の目的

本書は、D-Conciergeの開発環境を構築し、開発者が実装、確認、静的テスト、単体テスト、結合テストを開始できる状態にするための手順を定義することを目的とする。

本書は開発環境を対象とし、本番運用の常駐管理、バックアップ、監視、障害対応は `本番環境構築.md` で扱う。

## 2. 前提

- 本書は、要件定義書、外部設計書、内部設計書、テスト設計書、`pyproject.toml`、`uv.lock`、`src/frontend/package.json`、`src/frontend/package-lock.json`、`infra/compose.yml`、`config.example.yaml` を前提とする。
- 開発環境はLinuxとWindowsを対象とする。
- Python依存関係は `uv` と `.venv` で管理する。
- Node.js依存関係は `npm` と `src/frontend/package-lock.json` で管理する。
- 開発用PostgreSQLはDocker Composeで起動することを標準とする。
- `config.example.yaml` はGit管理対象の設定ひな形とし、開発者はこれをコピーして `config.yaml` を作成する。
- `config.yaml` は環境依存値を含むためGit管理対象外とする。
- Codex実行コンテナで使用する認証情報、用途別Codexホーム、作業領域はGit管理対象外とし、各開発者がローカル環境で準備する。
- バックエンドを起動するユーザは、Codex実行コンテナを起動できるDocker実行権限を持つ。
- sudoまたは管理者権限が必要なツール導入は、開発者が所属環境の標準手順に従って実施する。

## 3. 必要なツール

| 区分 | ツール | バージョン | 用途 |
| --- | --- | --- | --- |

| Node.js | Node.js | 24系 | React/Viteフロントエンドの開発実行 |
| Node.js依存管理 | npm | 11系 | フロントエンド依存関係同期、npm scripts実行 |

## 4. 環境変数・設定ファイル

| 項目 | 設定先 | 用途 | 設定方針 |
| --- | --- | --- | --- |
| `config.example.yaml` | リポジトリルート | `config.yaml` 作成元 | Git管理対象とする |
| `config.yaml` | リポジトリルート | アプリケーション実行設定 | `config.example.yaml` からコピーし、開発者ごとの値へ変更する |
| `app.timezone` | `config.yaml` | 運用者向け日時の基準 | IANA timezone名で指定し、開発標準は `Asia/Tokyo` とする |
| `D_CONCIERGE_DATABASE_URL` | シェル環境変数 | Alembic migration実行時のDB接続先 | migration実行時だけ設定する |
| `D_CONCIERGE_TEST_DATABASE_URL` | シェル環境変数 | バックエンド結合テストのPostgreSQL接続先 | 未指定時は `postgres-test` の既定接続先を使用する |
| `UV_CACHE_DIR` | シェル環境変数 | uvキャッシュ保存先 | Linuxでは `/tmp/uv-cache`、Windowsでは `%TEMP%\uv-cache` を使用する |
| `VITE_BACKEND_TARGET` | シェル環境変数 | Vite開発サーバから実バックエンドへの転送先 | 未指定時は `http://127.0.0.1:8000` を使用する |
| `codex_docker.image` | `config.yaml` | Codex実行コンテナのDockerイメージ | 開発標準は `codex-python-runner:latest` とする |
| `codex_docker.workspace_dir` | `config.yaml` | コンテナ内Codex作業ディレクトリ | 開発標準は `/workspace` とする |
| `codex_docker.codex_home_dir` | `config.yaml` | コンテナ内Codexホームディレクトリ | 開発標準は `/home/codex/.codex` とする |
| `codex_docker.codex_api_key` | `config.yaml` | Codex実行コンテナへ渡す `CODEX_API_KEY` | APIキー認証を使う場合に設定し、Codexホーム側認証を使う場合は空文字を指定できる |
| `CODEX_API_KEY` | シェル環境変数 | 手動のCodex実行コンテナ確認 | `infra/codex_docker/scripts` の手動確認で使用する |
| `POSTGRES_DB` | Docker Compose環境変数 | 開発用PostgreSQLのDB名 | 未指定時は `d_concierge` を使用する |
| `POSTGRES_USER` | Docker Compose環境変数 | 開発用PostgreSQLのユーザ名 | 未指定時は `d_concierge` を使用する |
| `POSTGRES_PASSWORD` | Docker Compose環境変数 | 開発用PostgreSQLのパスワード | 開発環境では既定値を使用できる |
| `POSTGRES_PORT` | Docker Compose環境変数 | 開発用PostgreSQLのホスト公開ポート | 未指定時は `5432` を使用する |
| `POSTGRES_TEST_DB` | Docker Compose環境変数 | 結合テスト用PostgreSQLのDB名 | 未指定時は `d_concierge_test` を使用する |
| `POSTGRES_TEST_USER` | Docker Compose環境変数 | 結合テスト用PostgreSQLのユーザ名 | 未指定時は `d_concierge_test` を使用する |
| `POSTGRES_TEST_PASSWORD` | Docker Compose環境変数 | 結合テスト用PostgreSQLのパスワード | 未指定時は `d_concierge_test` を使用する |
| `POSTGRES_TEST_PORT` | Docker Compose環境変数 | 結合テスト用PostgreSQLのホスト公開ポート | 未指定時は `55432` を使用する |

## 5. 依存サービスの準備

| サービス | 起動方法 | 接続情報 | 備考 |
| --- | --- | --- | --- |
| 開発用PostgreSQL | `docker compose -f infra/compose.yml up -d postgres` | `127.0.0.1:5432` | 開発中の通常DBとして使用する |
| 結合テスト用PostgreSQL | `docker compose -f infra/compose.yml up -d postgres-test` | `127.0.0.1:55432` | バックエンド結合テストで使用する |
| Codex実行用Dockerイメージ | `docker build -t codex-python-runner:latest -f infra/codex_docker/Dockerfile infra/codex_docker` | ローカルDocker | 回答生成・回答検証で使用する |
| 生成用Codexホーム | 開発者が事前準備 | `codex/.codex` | `AGENTS.md`、Skills、認証情報を配置し、Git管理しない |
| 検証用Codexホーム | 開発者が事前準備 | `codex/.codex_validator` | `AGENTS.md`、Skills、認証情報を配置し、Git管理しない |
| 共有データソース | 開発者が配置 | `codex/data_source` | Codex実行コンテナへ読み取り専用でマウントする |

## 6. アプリケーションのセットアップ

### 6.1. Linux

リポジトリルートで以下を実行する。

```bash
cp config.example.yaml config.yaml
mkdir -p logs/trace codex/data_source codex/.codex codex/.codex_validator codex/sessions codex/sessions_validator codex/saved_artifacts
UV_CACHE_DIR=/tmp/uv-cache uv sync --python 3.14 --all-groups
cd src/frontend
npm ci
cd ../..
```

Codex実行用Dockerイメージをビルドする。

```bash
docker build -t codex-python-runner:latest -f infra/codex_docker/Dockerfile infra/codex_docker
```

`config.yaml` のCodex Docker設定は以下を標準とする。

```yaml
codex_docker:
  image: "codex-python-runner:latest"
  workspace_dir: "/workspace"
  codex_home_dir: "/home/codex/.codex"
  codex_api_key: ""
```

開発用PostgreSQLを起動する。

```bash
docker compose -f infra/compose.yml up -d postgres
```

初期スキーマを作成する。

```bash
D_CONCIERGE_DATABASE_URL='postgresql+psycopg://d_concierge:d_concierge@127.0.0.1:5432/d_concierge' \
UV_CACHE_DIR=/tmp/uv-cache \
uv run alembic upgrade head
```

### 6.2. Windows

PowerShellを開き、リポジトリルートで以下を実行する。

```powershell
Copy-Item config.example.yaml config.yaml
New-Item -ItemType Directory -Force logs\trace
New-Item -ItemType Directory -Force codex\data_source
New-Item -ItemType Directory -Force codex\.codex
New-Item -ItemType Directory -Force codex\.codex_validator
New-Item -ItemType Directory -Force codex\sessions
New-Item -ItemType Directory -Force codex\sessions_validator
New-Item -ItemType Directory -Force codex\saved_artifacts
$env:UV_CACHE_DIR = "$env:TEMP\uv-cache"
uv sync --python 3.14 --all-groups
Set-Location src/frontend
npm ci
Set-Location ../..
```

Codex実行用Dockerイメージをビルドする。

```powershell
docker build -t codex-python-runner:latest -f infra/codex_docker/Dockerfile infra/codex_docker
```

`config.yaml` のCodex Docker設定は以下を標準とする。

```yaml
codex_docker:
  image: "codex-python-runner:latest"
  workspace_dir: "/workspace"
  codex_home_dir: "/home/codex/.codex"
  codex_api_key: ""
```

開発用PostgreSQLを起動する。

```powershell
docker compose -f infra/compose.yml up -d postgres
```

初期スキーマを作成する。

```powershell
$env:D_CONCIERGE_DATABASE_URL = "postgresql+psycopg://d_concierge:d_concierge@127.0.0.1:5432/d_concierge"
$env:UV_CACHE_DIR = "$env:TEMP\uv-cache"
uv run alembic upgrade head
```

### 6.3. Codex実行資産の準備

生成用Codexホームと検証用Codexホームには、それぞれの用途に応じた `AGENTS.md`、Skills、認証情報を配置する。`codex_docker.codex_api_key` にAPIキーを設定する場合は、Codex実行コンテナへ `CODEX_API_KEY` として渡される。空文字の場合は、Codexホーム配下の認証情報を使用する。

検証用Codexへ生成成果物を提示する処理では、生成用セッションの成果物領域を検証用Codex実行コンテナへ読み取り専用でマウントする。検証時に生成用成果物領域が存在しない場合は、成果物領域のマウントを行わない。

## 7. 起動・停止手順

### 7.1. Linux起動手順

バックエンドを起動する。

```bash
UV_CACHE_DIR=/tmp/uv-cache \
uv run uvicorn backend.main:app --app-dir src --host 127.0.0.1 --port 8000 --reload --reload-dir src/backend
```

別ターミナルでフロントエンドを起動する。

```bash
cd src/frontend
npm run dev
```

フロントエンドだけをモックデータで確認する場合は以下を実行する。

```bash
cd src/frontend
npm run dev:mock
```

### 7.2. Windows起動手順

PowerShellでバックエンドを起動する。

```powershell
$env:UV_CACHE_DIR = "$env:TEMP\uv-cache"
uv run uvicorn backend.main:app --app-dir src --host 127.0.0.1 --port 8000 --reload --reload-dir src/backend
```

別PowerShellでフロントエンドを起動する。

```powershell
Set-Location src/frontend
npm run dev
```

フロントエンドだけをモックデータで確認する場合は以下を実行する。

```powershell
Set-Location src/frontend
npm run dev:mock
```

### 7.3. 停止手順

バックエンドとフロントエンドは、起動したターミナルで `Ctrl+C` により停止する。

PostgreSQLを停止する。

```bash
docker compose -f infra/compose.yml stop postgres
```

結合テスト用PostgreSQLを停止する。

```bash
docker compose -f infra/compose.yml stop postgres-test
```
