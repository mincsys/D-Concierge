# ログ設計

## 1. 文書の目的

本書は、D-Conciergeで保存する障害調査用トレースログの対象、出力方針、保持方針、秘匿方針を定義することを目的とする。

## 2. 前提

- 本システムで保存対象とするログは障害調査用トレースログに限定する。
- トレースログは1異常1YAMLファイルとして保存する。
- アクセスログ、操作監査ログ、通常システムログの保存機能は対象外である。
- トレースログは利用者向け画面に表示しない。
- トレースログは設定された保存期間とアプリケーション起動ごとの同日最大保存件数に従って保存する。

## 3. ログ分類

| ログID | ログ名 | 保存対象 | 保存先 | 用途 |
| --- | --- | --- | --- | --- |
| LOG-01 | トレースログ | 回答生成失敗、最大試行回数到達、タイムアウト、PDF読み取り失敗、設定不備、API/SSE例外、起動時回復失敗 | 設定されたファイル保存先 | 障害調査 |

## 4. ファイル形式

トレースログは整形済みYAML形式で保存する。1ファイルを1件のトレースログレコードとし、各ファイルは独立したYAMLドキュメントとして扱えること。

1異常1YAMLファイル形式を採用する理由は次の通りである。

- 1件ごとのスタックトレースや検証コメントをYAMLブロック形式で読みやすく保存できる。
- ファイル単位で調査対象を共有しやすい。
- 同一時刻に複数異常が発生しても連番ファイルで分離できる。
- 巨大な単一ログファイルとして破損範囲が広がることを避けやすい。

## 5. 出力タイミング

| タイミング | 内容 |
| --- | --- |
| 回答生成失敗 | 生成用codex exec起動失敗、異常終了、出力不正を記録する。 |
| 最大試行回数到達 | 形式検証や参照元検証の不合格が上限に到達し、画面へエラーを返す場合に最後の検証情報だけを記録する。 |
| タイムアウト | タイムアウト発生段階と実行状態を記録する。 |
| PDF読み取り失敗 | 回答候補の参照元PDFが存在するがPDFとして読み取れない場合に記録する。 |
| 設定不備 | 必須設定不足、不正なパス設定、タイムアウト設定不正を記録する。 |
| API/SSE例外 | REST APIまたはSSE処理中の例外を記録する。 |
| 起動時回復失敗 | 未完了runの再登録や終端整合に失敗した場合に記録する。 |

REST APIまたはSSEの共通例外ハンドラでは、`AppError.trace=True` のエラーと予期しない例外だけを記録する。入力不正、対象なし、表示不可、キャンセル不可、通常の競合は利用者向け応答へ変換するが、トレースログには記録しない。

## 6. 保存項目

| 項目 | 内容 |
| --- | --- |
| 発生日時 | トレースログを出力した日時。 |
| trace_id | トレースログを識別するID。 |
| chat_id | 対象チャットID。 |
| run_id | 対象チャット実行処理ID。 |
| user_id | 共有1ユーザのID。 |
| stage | 発生段階。 |
| error_type | エラー分類。 |
| retry_count | 再生成回数。 |
| run_state | 実行状態。 |
| os_name | 実行OS名。 |
| runner_type | 実行制御区分。 |
| process_result | codex execの終了結果。 |
| validation_failure_reason | 最大試行回数到達時の最後の検証失敗理由。 |
| validation_comment | 最大試行回数到達時の最後の検証コメント。 |
| stacktrace | 捕捉した例外のスタックトレース。 |
| message | 調査用メッセージ。 |

## 7. 文字列の扱い

トレースログは開発者向け調査ログであるため、APIキー、環境変数、秘密情報、絶対パス、内部ディレクトリ、codex execの生イベント、利用者指示全文、回答全文を不必要にマスクしない。巨大な文字列だけを次の上限で切り詰める。

- `message`、`validation_comment`、`request_validation_errors` などの自由記述は64KiB。
- `stacktrace` は1MiB。
- 改行を含む文字列はYAMLのブロック形式で保存し、改行を `\n` エスケープとして表示しない。
- 切り詰めた文字列の末尾には `...<truncated>` を付ける。

## 8. 保持方針

- トレースログの標準保存期間は90日とする。
- トレースログの発生日時、日付ディレクトリ、ファイル名、保存期間判定は `app.timezone` のタイムゾーン基準で扱う。
- 保存期間は `trace_log.retention_days` で管理し、`trace_log.retention_days=90` の場合は90日前ちょうどの日付ディレクトリを保持し、91日前以前の日付ディレクトリを削除対象にする。
- 削除対象は `<trace_log.dir>/<yyyy-MM-dd>/` 形式の日付ディレクトリに限定する。日付形式でないディレクトリ、通常ファイル、現在形式外のファイルは削除しない。
- 保存期間を過ぎた日付ディレクトリはアプリケーション起動時に削除する。削除失敗はアプリケーション処理を止めない。
- アプリケーション起動ごとの同日最大保存件数は `trace_log.max_files_per_day` で管理し、標準は1000件とする。
- 同日最大保存件数はアプリケーション起動後に書き込んだ件数だけで判定し、起動前から存在する同日ログファイル数は数えない。
- 上限到達後は当日中の追加トレースログを保存しない。上限到達を示すトレースログも保存しない。
- 日付が変わった場合は起動中でも同日保存件数のカウンタを0に戻し、保存を再開する。

## 9. 参照方針

- 本システムではトレースログ閲覧画面を提供しない。
- 運用者はサーバ側でトレースログファイルを確認する。
- 利用者向けエラーにはトレースログの内容、内部パス、スタックトレースを含めない。
