# ログ設計

## 1. 文書の目的

本書は、D-Concierge MVPで保存する障害調査用トレースログの対象、出力方針、保持方針、秘匿方針を定義することを目的とする。

## 2. 前提

- MVPで保存対象とするログは障害調査用トレースログに限定する。
- トレースログはJSONL形式のファイルとして保存する。
- アクセスログ、操作監査ログ、通常システムログの保存機能はMVP対象外である。
- トレースログは利用者向け画面に表示しない。
- チャット履歴と同様、MVPではトレースログを無期限保持する。

## 3. ログ分類

| ログID | ログ名 | 保存対象 | 保存先 | 用途 |
| --- | --- | --- | --- | --- |
| LOG-01 | トレースログ | 回答生成失敗、検証失敗、キャンセル、タイムアウト、参照元表示失敗、設定不備 | 設定されたファイル保存先 | 障害調査 |

## 4. ファイル形式

トレースログはJSONL形式で保存する。1行を1件のトレースログレコードとし、各行は独立したJSONオブジェクトとして扱えること。

JSONL形式を採用する理由は次の通りである。

- エラー発生時に追記しやすい。
- 行単位で検索、抽出、集計しやすい。
- 後続のログ収集や解析へ転用しやすい。
- 巨大な単一JSONファイルとして破損範囲が広がることを避けやすい。

## 5. 出力タイミング

| タイミング | 内容 |
| --- | --- |
| 回答生成失敗 | 生成用codex exec起動失敗、異常終了、出力不正を記録する。 |
| 検証失敗 | 形式検証失敗、参照元検証失敗、再試行上限超過を記録する。 |
| キャンセル | キャンセル要求、キャンセル結果、キャンセル失敗を記録する。 |
| タイムアウト | タイムアウト発生段階と実行状態を記録する。 |
| 参照元表示失敗 | 参照元データ取得失敗や不正な参照位置を記録する。 |
| 設定不備 | 必須設定不足、不正なパス設定、タイムアウト設定不正を記録する。 |

## 6. 保存項目

| 項目 | 内容 |
| --- | --- |
| 発生日時 | トレースログを出力した日時。 |
| trace_id | トレースログを識別するID。 |
| chat_id | 対象チャットID。 |
| run_id | 対象チャット実行処理ID。 |
| user_id | 共有1ユーザのID。 |
| stage | 発生段階。 |
| error_class | エラー分類。 |
| retry_count | 再生成回数。 |
| run_state | 実行状態。 |
| os_name | 実行OS名。 |
| runner_type | 実行制御区分。 |
| process_result | codex execの終了結果。 |
| message | 調査用の要約メッセージ。 |

## 7. 保存しない情報

次の情報は保存しない、またはマスクする。

- APIキー。
- 環境変数全文。
- 秘密情報。
- OS依存の絶対パス。
- 不要な内部ディレクトリ。
- codex execの生イベント全文。
- 巨大な回答全文。
- 個人情報や業務上の機密情報を含む可能性がある全文データ。

## 8. 保持方針

- トレースログはMVPでは無期限保持する。
- 自動削除機能と手動削除機能はMVP対象外とする。
- 容量確認は運用上の留意事項として扱う。

## 9. 参照方針

- MVPではトレースログ閲覧画面を提供しない。
- 運用者はサーバ側でトレースログファイルを確認する。
- 利用者向けエラーにはトレースログの内容、内部パス、スタックトレースを含めない。
