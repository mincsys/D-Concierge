# ユーザログイン・アカウント管理

## 目的

本メモは、ユーザログイン、アカウント登録、ログインセッション、設定ダイアログ、アカウント削除の詳細を整理する。

要件定義書では認証・アカウント管理の要求を簡潔に記載し、画面導線、入力制約、Cookie属性、削除再試行などの詳細は本メモを外部設計・内部設計の入力とする。

## 対象範囲

今回追加する対象範囲は次の通りである。

- アカウント登録。
- ログイン。
- ログアウト。
- ログイン状態の長期保持。
- 未ログイン時のアクセス制御。
- ログイン中ユーザ表示。
- 設定ダイアログ。
- ユーザ名変更。
- パスワード変更。
- アカウント削除。

ロール、管理者画面、管理者によるユーザ管理、パスワード再設定は対象外とする。ロールと管理者機能の将来構想は、`将来構想/ロール・管理者機能.md` に分離する。

## 認証方式

認証方式は、D-Concierge内のローカル認証とする。外部IdP連携は行わない。

利用者は、ユーザIDとパスワードでログインする。

## ユーザ情報

ユーザは、少なくとも次の情報を持つ。

| 項目 | 内容 |
| --- | --- |
| ユーザID | ログイン識別子。登録後は変更できない。 |
| ユーザ名 | 画面表示名。登録後に変更できる。 |
| パスワード | ログイン認証に用いる秘密情報。登録後に変更できる。 |
| ユーザ状態 | 通常利用可能、削除中など、ログイン可否と削除処理を判断する状態。 |

ユーザIDは一意であり、重複登録できない。削除中ユーザのユーザIDは、物理削除が完了するまで再利用できない。物理削除完了後は、同じユーザIDで新規登録できる。

## 入力制約

### ユーザID

ユーザIDの制約は次の通りである。

- 1文字以上30文字以内。
- 半角英数字、`_`、`-` のみを許可する。
- 先頭と末尾は半角英数字とし、`_`、`-` では始めたり終えたりできない。
- 登録済みまたは削除中のユーザIDと重複できない。

### ユーザ名

ユーザ名の制約は次の通りである。

- 1文字以上30文字以内。
- 文字種の制限は設けない。

### パスワード

パスワードの制約は次の通りである。

- 5文字以上30文字以内。
- 半角の英字、数字、記号を許可する。
- 英字、数字、記号の組み合わせ条件は設けない。

## 画面入力と送信

ログイン画面、アカウント登録画面、ユーザ名変更フォーム、パスワード変更フォームは、ボタン押下時だけAPI送信する。Enterキー押下だけでは送信しない。

画面側の入力制限は、入力欄の `maxLength=30` による最大文字数制限に留める。必須、最小文字数、文字種、ユーザID重複、現在パスワード照合、パスワード確認一致の検証はAPI側で行い、画面はAPI応答の入力項目別エラーを各入力欄の近くに表示する。

アカウント登録画面、ユーザ名変更フォーム、パスワード変更フォームでは、入力制約の概要を対象入力欄のプレースホルダに表示する。ログイン画面と、パスワード変更フォームの現在のパスワード欄には、入力制約プレースホルダを表示しない。

ユーザID、ユーザ名、パスワード、パスワード確認、新しいパスワード確認など、API項目名と画面モデル名が異なる項目は、画面側でAPI項目名へ変換して送信する。

ブラウザの認証情報保存を妨げないよう、ログイン画面とアカウント登録画面の入力欄には用途に応じた `autocomplete` を設定する。ユーザ名変更フォームは、ログイン認証情報の保存対象ではないため `autocomplete` を設定しない。

パスワード入力欄には表示・非表示を切り替えるアイコンボタンを置く。

## アカウント登録

未ログイン状態で保護対象画面へアクセスした場合は、ログイン画面を表示する。ログイン画面にはアカウント登録画面へのリンクを表示する。

ログイン画面は `/login`、アカウント登録画面は `/register` で表示する。ログイン画面とアカウント登録画面は保護対象画面に含めず、ログイン状態に関わらず直接表示できる。

アカウント登録画面では、次の項目を入力する。

- ユーザID。
- ユーザ名。
- パスワード。
- パスワード確認。

登録ボタン押下時に、ユーザID、ユーザ名、パスワード、パスワード確認の入力制約と、ユーザIDの重複を確認する。入力中またはフォーカスアウト時のユーザID重複確認は行わない。

複数の入力エラーがある場合は、各入力項目の近くにすべて同時表示する。ユーザIDが既に使用されている場合は、重複していることを明示する。

登録成功後は、自動的にログインして開始画面へ遷移する。ブラウザの認証情報保存はブラウザ機能に委ねる。

登録画面には、ログイン画面へ戻るリンクを表示する。

## ログイン

ログイン画面では、ユーザIDとパスワードを入力する。

ログイン成功後は、常に開始画面へ遷移する。ログイン前にアクセスしようとしていた画面へ戻す制御は行わない。

ログイン失敗時は、ユーザID不存在とパスワード不一致を区別して表示する。失敗回数によるアカウントロックは設けない。

## ログイン状態

ログインセッションは、ブラウザごとに独立して保持する。同じユーザが複数ブラウザ、複数端末で同時ログインできる。

各ブラウザでは、明示的なログアウト、アカウント削除、強制失効がない限りログイン状態を維持する。

セッション切れまたは未ログイン状態で保護対象APIにアクセスした場合、画面は表示中の内容を破棄してログイン画面へ遷移する。

## Cookie

ログイン状態は、バックエンドが発行するログインセッションをブラウザのCookieで保持して実現する。

ログインセッションCookieの属性は次の通りである。

| 属性 | 方針 |
| --- | --- |
| `HttpOnly` | 付与する。画面側JavaScriptからCookie値を直接扱わない。 |
| `SameSite` | `Lax` とする。 |
| `Secure` | 今回はHTTP通信を前提とするため必須にしない。HTTPS対応時に見直す。 |
| `Max-Age` | 400日、すなわち `34560000` 秒とする。 |

画面側はCookie値を直接参照せず、認証状態確認APIでログイン中ユーザを取得する。

CSRF対策は、`HttpOnly` Cookieと `SameSite=Lax` により行う。CSRFトークンは持たず、DBにも保存しない。

## API配置

アカウント登録、ログイン、ログアウト、認証状態確認、ユーザ名変更、パスワード変更、アカウント削除のAPIは、既存の画面バックエンドAPIと同じ `/api` 配下に置く。

APIの配置と正常応答は次の通りである。

| 操作 | メソッド | パス | 正常応答 |
| --- | --- | --- | --- |
| 認証状態確認 | GET | `/api/auth/me` | `200`、`{ user: { user_id, user_name } }` |
| アカウント登録 | POST | `/api/auth/register` | `200`、`{ user: { user_id, user_name } }` |
| ログイン | POST | `/api/auth/login` | `200`、`{ user: { user_id, user_name } }` |
| ログアウト | POST | `/api/auth/logout` | `204`、レスポンスボディなし |
| ユーザ名変更 | PATCH | `/api/account/name` | `200`、`{ user: { user_id, user_name } }` |
| パスワード変更 | PATCH | `/api/account/password` | `204`、レスポンスボディなし |
| アカウント削除 | DELETE | `/api/account` | `202`、`{ account_state: "deleting" }` |

ログイン画面とアカウント登録画面で利用する `/api/auth/login` と `/api/auth/register` は未ログインでも呼び出せる。その他の `/api` 配下APIはログイン状態を必要とする。

入力エラーと認証エラーの応答は、次の形式を基本とする。

```json
{
  "error": "validation_error",
  "message": "入力内容を確認してください。",
  "field_errors": {
    "user_id": "このユーザIDは既に使用されています。"
  }
}
```

未ログインまたはセッション切れの場合は、`401` とし、利用者向けメッセージとして `ログインしてください。` を返す。

## ログイン中ユーザ表示

メイン画面では、左下の領域にログイン中ユーザを表示する。

表示内容は次の通りである。

- ユーザIDの先頭1文字をアイコン表示する。
- アイコンの右にユーザ名を表示する。

ユーザ名変更成功後は、左下のユーザ表示をすぐに新しいユーザ名へ更新する。

## 設定ダイアログ

左下のユーザ表示領域を選択すると、画面中央に設定ダイアログを表示する。

設定ダイアログは左側にメニュー、右側にコンテンツ領域を持つ。左メニューには `アカウント` のみを表示する。

右側のアカウント領域には、次の項目を一覧表示する。

- ユーザ名変更。
- パスワード変更。
- ログアウト。
- アカウント削除。

ユーザ名変更とパスワード変更は、選択後に個別フォームへ遷移する。各個別フォームでは、コンテンツ領域のヘッダー左側に戻るアイコンを表示し、押下するとアカウント一覧へ戻る。

ログアウトとアカウント削除は個別フォームを作らず、アカウント領域にボタンを表示する。ボタン押下時に確認ダイアログを表示する。

ログイン画面、登録画面、設定ダイアログは、既存UIと同じ shadcn/ui と Tailwind CSS のトーンに合わせる。

## ユーザ名変更

ユーザ名変更フォームでは、新しいユーザ名だけを入力する。現在のパスワードによる本人確認は行わない。

入力エラーは、入力欄の近くに表示する。変更成功時は成功メッセージを表示せず、設定ダイアログ内の表示更新と左下のユーザ名更新で結果を示す。

## パスワード変更

パスワード変更フォームでは、次の項目を入力する。

- 現在のパスワード。
- 新しいパスワード。
- 新しいパスワード確認。

現在のパスワード照合に成功し、新しいパスワードが入力制約を満たし、新しいパスワード確認と一致した場合に変更できる。

入力エラーは、入力欄の近くに表示する。変更成功時は成功メッセージを表示しない。

パスワード変更後も、現在のブラウザおよび他ブラウザ・端末のログイン状態は維持する。

## ログアウト

ログアウトは、設定ダイアログのアカウント領域から行う。

ログアウトボタン押下時に確認ダイアログを表示する。確認後、現在のログインセッションを終了し、ログイン画面へ遷移する。

ログアウト確認ダイアログのタイトルは `ログアウトしますか？` とする。本文は表示しない。確認ボタンの表示は `ログアウト` とし、通常ボタンとする。

## アカウント削除

アカウント削除は、設定ダイアログのアカウント領域から行う。

アカウント削除ボタン押下時に確認ダイアログを表示する。確認ダイアログのタイトルは `アカウントを完全に削除しますか？` とする。本文は `この操作は取り消せません。アカウントに紐づけられている全てのデータが完全に削除されます。` とする。確認ボタンの表示は `削除する` とし、危険操作として赤系で表示する。

アカウント削除要求を受け付けた時点で、対象ユーザを削除中または操作不可状態にし、対象ユーザの全ログインセッションを失効する。操作中のブラウザはログイン画面へ遷移する。削除中ユーザはログイン不可、通常操作不可とする。

未完了のチャット実行処理がある場合は、キャンセル要求を行う。未完了処理が残っている間は、作業領域削除、保存済み成果物削除、DB削除へ進まず、削除中状態を維持する。

未完了処理がなくなった後、対象ユーザに紐づく次のデータを削除する。

- ユーザ情報。
- ログインセッション。
- チャット履歴。
- ユーザ指示。
- チャット実行処理。
- 中間メッセージ。
- 回答。
- 参照元。
- 添付画像。
- Codex成果物メタ情報。
- 保存済みCodex成果物実体。
- 生成用・検証用作業領域。

トレースログはアカウント削除の対象外とし、ログ設計の保存期間に従って保持する。

## アカウント削除の再試行

アカウント削除は、既存のチャット削除と同じく、削除要求受付と物理削除を分ける。

物理削除が途中失敗した場合は、対象ユーザを削除中または操作不可状態のまま維持し、通常利用可能な状態へ復帰させない。失敗内容は内部ログへ記録する。

アプリ起動時は、削除中ユーザを取得し、アカウント削除ジョブへ再登録する。再登録に失敗した場合も、対象ユーザを通常利用可能な状態へ戻さず、次回起動または次回削除処理で再試行できる状態を維持する。

ファイルまたはディレクトリが存在しない場合は、再試行時の冪等性として削除済み扱いにする。

## 操作結果メッセージ

アカウント登録、ログイン、ログアウト、ユーザ名変更、パスワード変更、アカウント削除では、失敗時だけ画面上にメッセージを表示する。

成功時は、画面遷移、ダイアログ終了、ユーザ名表示更新などの状態変化で結果を示す。

## パスワード再設定

パスワード再設定機能は提供しない。ログインできない場合は、運用者がDB等で対応する。

## 将来拡張

HTTPS通信対応は将来拡張とする。HTTPS対応時は、ログインセッションCookieへ `Secure` 属性を付与する方針を再検討する。

## フロントエンドモックでの扱い

フロントエンドモックでは、backend_mock内にアカウント状態、ログインセッション、ユーザ別チャット履歴を保持する。

開発サーバ起動直後の初回認証状態確認でログインセッションCookieがない場合は、`demo-user`、ユーザ名 `デモユーザ`、パスワード `password` のデモユーザとしてログイン済み状態を作る。これは既存のモック画面を起動直後に確認しやすくするためのモック専用仕様である。ログアウトまたはアカウント削除後は自動ログインを再発行せず、ログイン画面を表示する。

デモユーザには既存のモックチャット履歴を紐づける。アカウント登録で作成した新規ユーザは、登録直後はチャット履歴を持たない。

フロントエンドモックのアカウント削除は、ユーザ、ログインセッション、ユーザ別チャット履歴を即時にモック状態から削除する。削除後は同じユーザIDで再登録できる。
