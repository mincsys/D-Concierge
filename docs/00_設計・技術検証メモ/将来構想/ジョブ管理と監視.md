# ジョブ管理と監視

## 目的

本メモは、将来構想要件定義から分離したジョブ状態、実行キュー、タイムアウト、再実行、管理者向け監視、通知、エラー詳細の設計メモを整理する。

## ジョブ状態

将来構想では、1回のチャット実行をジョブとして管理する。

ジョブは、生成用codex exec、固定検証、検証用codex exec、再生成を含む一連の処理を表す。

ジョブ状態:

| 状態 | 説明 |
| --- | --- |
| queued | 実行待ち。 |
| running_generation | 生成用codex execを実行中。 |
| running_fixed_validation | 固定検証を実行中。 |
| running_reference_validation | 検証用codex execを実行中。 |
| retrying | 検証失敗後、再生成を実行中。 |
| succeeded | 検証済み回答の生成に成功。 |
| failed | 再試行上限超過または復旧不能エラーで失敗。 |
| timed_out | 実行時間上限を超過。 |

MVPでは、チャット実行の状態管理は最小限でよい。将来構想では、ジョブ状態を永続化し、画面や管理機能から確認できるようにする。

## 実行キュー

将来構想では、codex execの同時実行数を制御する。

必要な要件:

- 同時実行数の上限を設定できる。
- 上限を超えたチャット実行は `queued` として待機する。
- キュー内のジョブは受付順に処理することを標準とする。
- ジョブごとに作成日時、開始日時、終了日時を記録する。
- キュー待ちが長い場合、利用者に待機中であることを表示する。

設定例:

```yaml
jobs:
  max_concurrent_jobs: 2
  queue_timeout_seconds: 600
  execution_timeout_seconds: 900
```

この例では、同時に実行するcodex exec関連ジョブは最大2件である。3件目以降はキューで待機する。待機時間が600秒を超える場合はタイムアウト扱いにできる。

## キャンセル

キャンセルはMVPで提供する利用者操作として扱う。将来構想では、この機能を管理者操作、やり直し対象、運用集計へ広げない。

将来構想のジョブ管理では、完了、失敗、タイムアウト、滞留を主な運用確認対象にする。

## タイムアウト

タイムアウト対象:

- キュー待ち時間。
- 生成用codex exec実行時間。
- 固定検証時間。
- 検証用codex exec実行時間。
- 再生成を含むジョブ全体の実行時間。

タイムアウト時は、ジョブ状態を `timed_out` にする。利用者には、処理時間が上限を超えたため回答を表示できなかったことを示す。

## 再実行

再実行の対象:

- `failed`
- `timed_out`

再実行時は、元のユーザ指示本文、アプリ設定、利用者IDを引き継ぐ。過去の失敗ログは上書きせず、新しいジョブとして記録する。

再実行の例:

1. 利用者がユーザ指示を送信する。
2. 参照元検証で失敗し、再試行上限を超える。
3. ジョブ状態が `failed` になる。
4. 利用者が履歴から再実行を選ぶ。
5. 新しいジョブIDで同じユーザ指示を再実行する。

## 監視対象

監視対象:

- ジョブ数。
- 実行中ジョブ数。
- キュー待ちジョブ数。
- 成功件数。
- 失敗件数。
- タイムアウト件数。
- 平均処理時間。
- codex exec起動失敗。
- 固定検証エラー。
- 参照元検証エラー。

利用状況分析は将来構想に含めない。そのため、ユーザ別の利用頻度ランキング、人気ユーザ指示、部門別集計などは対象外とする。

## 管理者向けジョブ一覧

表示項目:

- ジョブID
- チャットID
- 実行ユーザ
- 状態
- 作成日時
- 開始日時
- 終了日時
- 経過時間
- 再試行回数
- エラー分類

ジョブ一覧では、ユーザ指示本文や回答本文の全文表示は必須にしない。運用ポリシーによっては、個人情報や業務情報を含む可能性があるためである。

## エラー詳細

管理者は、失敗したジョブのエラー詳細を確認できる。

確認できる情報:

- 失敗段階
  - 生成用codex exec起動
  - 生成用codex exec実行
  - 固定検証
  - 検証用codex exec起動
  - 検証用codex exec実行
  - 再試行上限超過
- エラー分類
- エラーメッセージ
- 再試行回数
- 発生日時

管理者向けエラー詳細は、トレースログそのものの無制限表示ではなく、必要範囲の要約と分類を表示する。

秘密情報、絶対パス、環境変数、APIキー、不要な全文データは管理者画面にも表示しない。必要な場合は、サーバログ側で権限を制限して確認する。

## 通知

通知対象の例:

- codex execの連続起動失敗。
- タイムアウトの連続発生。
- キュー滞留の長期化。
- 固定検証エラーの多発。
- 検証用codex execの連続失敗。

通知先や通知方式は、アプリケーション設定で指定できるようにする。

MVPでは通知は必須ではない。将来構想では、少なくともログまたは管理画面上で重大エラーを認識できることを必須とする。

## ジョブ管理の実装メモ

ジョブ管理では、PostgreSQL上のジョブ情報を正とする。

FastAPIは、利用者からチャット実行要求を受け取るとジョブを作成し、codex execプロセスの状態とジョブ状態を同期する。

保存する主な情報:

- ジョブID。
- チャットID。
- 実行ユーザID。
- ジョブ状態。
- codex execの実行段階。
- 再試行回数。
- 作成日時、開始日時、終了日時。
- エラー分類。

同時実行数制御はFastAPI側で行う。小規模組織向けの初期構成では、外部キューサービスを必須にしない。必要になった場合に、バックグラウンドワーカーをDocker Compose構成へ追加できる余地を残す。

## 監視の実装メモ

監視は、まずFastAPIのサーバログ、PostgreSQL上のジョブ情報、管理者画面で確認できる範囲を必須とする。

外部監視サービス、メトリクス基盤、ログ集約基盤との連携は必須にしない。社内や小さな組織での運用では、導入・保守コストを抑えるためである。

ただし、将来必要になった場合に備え、FastAPIはジョブ状態、エラー分類、処理時間を構造化して記録する。
