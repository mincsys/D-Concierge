# D-Concierge 将来構想要件定義

## 1. 位置付け

本書は、MVP要件定義から追加・修正される将来構想の要件だけを定義する。

将来構想では、D-Conciergeを個人または小規模利用のチャット基盤から、組織で継続利用できる業務アプリケーションへ拡張する。

主な拡張軸は次の3つである。

- ログインを前提とした組織利用。
- Codex execのジョブ管理と監視による運用安定化。
- チャット履歴の検索性と整理性の強化。

将来構想に含めないものは、次の通りである。

- PDF表示とHTML参照元表示の切替。
- 回答品質評価機能。
- スキーマバージョン管理。
- 利用状況分析。
- 利用者によるデータソース追加。
- 管理画面からのデータソース更新。

データソースはMVPと同じく、開発者がアプリケーションごとに配置・設定する。

## 2. 組織利用

### 2.1 ログイン必須化

将来構想では、アプリケーション利用時にログインを必須とする。

ログイン後、利用者は自分のチャット履歴だけを閲覧・検索できる。別ユーザのチャット履歴は表示しない。

MVPからの変更点:

| 項目 | MVP | 将来構想 |
| --- | --- | --- |
| 利用者識別 | ローカル利用者ID単位 | ログインユーザ単位 |
| 履歴保存 | ローカル利用者ごとに分離 | ログインユーザごとに分離 |
| 画面表示 | 利用者名表示は任意 | ログインユーザ名を表示 |
| アクセス制御 | 必須ではない | 未ログイン利用を禁止 |

### 2.2 ユーザ情報

将来構想では、少なくとも次のユーザ情報を保持する。

- ユーザID
- 表示名
- メールアドレスまたはログイン識別子
- ロール
- 作成日時
- 最終ログイン日時

ロールは、MVPには存在しない新しい概念である。

MVPからの追加ロール:

| ロール | 説明 |
| --- | --- |
| 一般ユーザ | チャットの作成、回答閲覧、自分の履歴検索を行う。 |
| 管理者 | 運用状態、ジョブ状態、エラー情報を確認する。 |

管理者は、データソースそのものを画面から追加・更新しない。データソースの配置や `AGENTS.md`・Skillsの差し替えは、引き続き開発者または運用者がアプリケーション設定として行う。

### 2.3 チャット履歴の分離

将来構想では、チャット履歴をユーザごとに分離する。

一般ユーザが閲覧できる履歴:

- 自分が作成したチャット。
- 自分が実行した質問。
- 自分の質問に対する回答。
- 自分のチャットに紐づく中間メッセージ。
- 自分のチャットに紐づく検証結果。

一般ユーザが閲覧できない履歴:

- 他ユーザのチャット。
- 他ユーザの質問文。
- 他ユーザの回答。
- 他ユーザの参照元リンク。

管理者は、運用調査に必要な範囲でジョブ状態やエラー状態を確認できる。ただし、管理者が他ユーザの回答本文を閲覧できるかどうかは、アプリケーションの運用ポリシーで制御できるようにする。

### 2.4 セッション管理

将来構想では、ログインセッションを管理する。

必要な要件:

- 一定時間操作がない場合、セッションを期限切れにする。
- セッション期限切れ後のAPIリクエストは拒否する。
- ログアウト操作を提供する。
- 未ログイン状態でチャット画面へアクセスした場合、ログイン画面へ遷移する。

セッション期限の値は設定可能とする。

## 3. ジョブ管理

### 3.1 ジョブの概念

将来構想では、1回のチャット実行をジョブとして管理する。

ジョブは、生成用Codex exec、固定検証、検証用Codex exec、再生成を含む一連の処理を表す。

ジョブ状態:

| 状態 | 説明 |
| --- | --- |
| queued | 実行待ち。 |
| running_generation | 生成用Codex execを実行中。 |
| running_fixed_validation | 固定検証を実行中。 |
| running_reference_validation | 検証用Codex execを実行中。 |
| retrying | 検証失敗後、再生成を実行中。 |
| succeeded | 検証済み回答の生成に成功。 |
| failed | 再試行上限超過または復旧不能エラーで失敗。 |
| canceled | 利用者または管理者がキャンセル。 |
| timed_out | 実行時間上限を超過。 |

MVPでは、チャット実行の状態管理は最小限でよい。将来構想では、ジョブ状態を永続化し、画面や管理機能から確認できるようにする。

### 3.2 実行キュー

将来構想では、Codex execの同時実行数を制御する。

必要な要件:

- 同時実行数の上限を設定できる。
- 上限を超えたチャット実行は `queued` として待機する。
- キュー内のジョブは受付順に処理することを標準とする。
- ジョブごとに作成日時、開始日時、終了日時を記録する。
- キュー待ちが長い場合、利用者に待機中であることを表示する。

設定例:

```yaml
jobs:
  max_concurrent_jobs: 2
  queue_timeout_seconds: 600
  execution_timeout_seconds: 900
```

この例では、同時に実行するCodex exec関連ジョブは最大2件である。3件目以降はキューで待機する。待機時間が600秒を超える場合はタイムアウト扱いにできる。

### 3.3 キャンセル

将来構想では、実行中ジョブのキャンセルを提供する。

キャンセルできる利用者:

- 一般ユーザは、自分が開始したジョブをキャンセルできる。
- 管理者は、任意の実行中ジョブをキャンセルできる。

キャンセル時の処理:

- 実行中のCodex execプロセスへ終了要求を送る。
- プロセス終了を確認する。
- ジョブ状態を `canceled` に更新する。
- チャット画面には、回答生成がキャンセルされたことを表示する。
- 部分的な回答JSONは最終回答として表示しない。

### 3.4 タイムアウト

将来構想では、ジョブ全体と各処理段階にタイムアウトを設定できる。

タイムアウト対象:

- キュー待ち時間。
- 生成用Codex exec実行時間。
- 固定検証時間。
- 検証用Codex exec実行時間。
- 再生成を含むジョブ全体の実行時間。

タイムアウト時は、ジョブ状態を `timed_out` にする。利用者には、処理時間が上限を超えたため回答を表示できなかったことを示す。

### 3.5 再実行

将来構想では、失敗またはタイムアウトしたジョブを再実行できる。

再実行の対象:

- `failed`
- `timed_out`
- `canceled`

再実行時は、元の質問文、アプリ設定、利用者IDを引き継ぐ。過去の失敗ログは上書きせず、新しいジョブとして記録する。

再実行の例:

1. 利用者が質問する。
2. 参照元検証で失敗し、再試行上限を超える。
3. ジョブ状態が `failed` になる。
4. 利用者が履歴から再実行を選ぶ。
5. 新しいジョブIDで同じ質問を再実行する。

## 4. 監視

### 4.1 監視対象

将来構想では、運用者または管理者がシステム状態を確認できるようにする。

監視対象:

- ジョブ数。
- 実行中ジョブ数。
- キュー待ちジョブ数。
- 成功件数。
- 失敗件数。
- タイムアウト件数。
- キャンセル件数。
- 平均処理時間。
- Codex exec起動失敗。
- 固定検証エラー。
- 参照元検証エラー。

利用状況分析は将来構想に含めない。そのため、ユーザ別の利用頻度ランキング、人気質問、部門別集計などは対象外とする。

### 4.2 管理者向けジョブ一覧

管理者は、ジョブ一覧を確認できる。

表示項目:

- ジョブID
- チャットID
- 実行ユーザ
- 状態
- 作成日時
- 開始日時
- 終了日時
- 経過時間
- 再試行回数
- エラー分類

ジョブ一覧では、質問文や回答本文の全文表示は必須にしない。運用ポリシーによっては、個人情報や業務情報を含む可能性があるためである。

### 4.3 エラー詳細

管理者は、失敗したジョブのエラー詳細を確認できる。

確認できる情報:

- 失敗段階
  - 生成用Codex exec起動
  - 生成用Codex exec実行
  - 固定検証
  - 検証用Codex exec起動
  - 検証用Codex exec実行
  - 再試行上限超過
- エラー分類
- エラーメッセージ
- 再試行回数
- 発生日時

秘密情報、絶対パス、環境変数、APIキーは管理者画面にも表示しない。必要な場合は、サーバログ側で権限を制限して確認する。

### 4.4 通知

将来構想では、重大な運用エラーを通知できる。

通知対象の例:

- Codex execの連続起動失敗。
- タイムアウトの連続発生。
- キュー滞留の長期化。
- 固定検証エラーの多発。
- 検証用Codex execの連続失敗。

通知先や通知方式は、アプリケーション設定で指定できるようにする。

MVPでは通知は必須ではない。将来構想では、少なくともログまたは管理画面上で重大エラーを認識できることを必須とする。

## 5. 履歴強化

### 5.1 履歴全文検索

将来構想では、チャット履歴の全文検索を強化する。

検索対象:

- チャットタイトル。
- 利用者の質問文。
- 回答本文。
- 参照元ラベル。
- タグ。

検索対象外:

- 秘密情報を含む可能性がある内部ログ。
- Codex execの生JSONL全体。
- 管理者向けエラー詳細。

検索結果には、該当チャットのタイトル、更新日時、ヒットした本文の短い抜粋を表示する。

### 5.2 タグ

将来構想では、チャット履歴にタグを付けられる。

タグの用途:

- 後で見返すチャットを分類する。
- 業務テーマごとに履歴を整理する。
- 検索条件として使う。

タグ操作:

- タグを追加する。
- タグを削除する。
- タグで履歴を絞り込む。

タグ例:

- `要件定義`
- `障害調査`
- `センサ異常`
- `設計確認`
- `レビュー用`

タグはユーザごとに管理する。他ユーザのタグは表示しない。

### 5.3 ピン留め

将来構想では、重要なチャット履歴をピン留めできる。

ピン留めされたチャットは、履歴一覧の上部に表示する。

ピン留めの用途:

- 繰り返し参照する回答をすぐ開く。
- 調査中のチャットを見失わないようにする。
- 作業の起点となるチャットを固定する。

### 5.4 タイトル編集

将来構想では、チャットタイトルを利用者が編集できる。

MVPでは、タイトルは最初の質問文から自動生成する。将来構想では、利用者が業務上わかりやすい名前へ変更できるようにする。

タイトル編集の例:

- 自動生成タイトル: `要件定義を成功させるポイント`
- 編集後タイトル: `IPA資料から整理した要件定義成功ポイント`

タイトル編集は、チャット作成者だけが行える。

## 6. 将来構想の画面追加

### 6.1 ログイン画面

ログイン必須化に伴い、ログイン画面を追加する。

必要な表示:

- アプリケーション名。
- ログインID入力欄。
- パスワード入力欄または認証方式に応じた入力欄。
- ログインボタン。
- ログイン失敗時のメッセージ。

### 6.2 ジョブ状態表示

チャット画面では、MVPの中間メッセージ表示に加えて、ジョブ状態を明示できるようにする。

表示例:

- `実行待ち`
- `回答生成中`
- `回答検証中`
- `再生成中`
- `完了`
- `失敗`
- `キャンセル済み`
- `タイムアウト`

### 6.3 管理者画面

将来構想では、管理者向け画面を追加する。

管理者画面で扱うもの:

- ジョブ一覧。
- ジョブ詳細。
- エラー詳細。
- 実行中ジョブのキャンセル。
- 監視情報。

管理者画面で扱わないもの:

- データソースの追加。
- PDFやCSVのアップロード。
- `AGENTS.md` の編集。
- Skillsの編集。
- 回答品質評価。
- 利用状況分析。

## 7. 技術要件の追加

将来構想では、MVPで採用した `FastAPI + React/Vite + PostgreSQL` 構成を維持する。Next.jsなど別のフルスタック構成へ置き換えるのではなく、FastAPI側に認証、ジョブ管理、監視機能を追加する。

### 7.1 認証・セッション管理

ログイン必須化に伴い、FastAPIが認証APIとセッション管理を担当する。

PostgreSQLには、少なくとも次の情報を保存する。

- ユーザ。
- ロール。
- ログインセッション。
- ユーザごとのチャット履歴。
- ユーザごとのタグ、ピン留め、タイトル編集情報。

React/Vite側は、ログイン状態を確認し、未ログイン時はログイン画面を表示する。認証状態の最終判定はFastAPI側で行い、APIアクセス時に毎回確認する。

### 7.2 ジョブ管理

ジョブ管理では、PostgreSQL上のジョブ情報を正とする。

FastAPIは、利用者からチャット実行リクエストを受け取るとジョブを作成し、Codex execプロセスの状態とジョブ状態を同期する。

保存する主な情報:

- ジョブID。
- チャットID。
- 実行ユーザID。
- ジョブ状態。
- Codex execの実行段階。
- 再試行回数。
- 作成日時、開始日時、終了日時。
- エラー分類。

同時実行数制御はFastAPI側で行う。小規模組織向けの初期構成では、外部キューサービスを必須にしない。必要になった場合に、バックグラウンドワーカーをDocker Compose構成へ追加できる余地を残す。

### 7.3 監視

監視は、まずFastAPIのサーバログ、PostgreSQL上のジョブ情報、管理者画面で確認できる範囲を必須とする。

外部監視サービス、メトリクス基盤、ログ集約基盤との連携は必須にしない。社内や小さな組織での運用では、導入・保守コストを抑えるためである。

ただし、将来必要になった場合に備え、FastAPIはジョブ状態、エラー分類、処理時間を構造化して記録する。

### 7.4 Docker Compose構成の拡張

将来構想では、MVPのDocker Compose構成を次のように拡張できる。

- アプリケーションコンテナ。
- PostgreSQLコンテナ。
- 必要に応じたバックグラウンドワーカー。
- 必要に応じたバックアップ用ジョブ。

開発時は引き続きDockerを必須にしない。開発者はFastAPIとReact/Viteをローカル起動し、PostgreSQLだけDocker ComposeまたはローカルDBを選べる。

## 8. 設定追加

将来構想では、MVPの設定ファイルに次の項目を追加する。

設定例:

```yaml
auth:
  required: true
  session_timeout_minutes: 60

database:
  url: "postgresql+psycopg://d_concierge:password@db:5432/d_concierge"

jobs:
  max_concurrent_jobs: 2
  queue_timeout_seconds: 600
  execution_timeout_seconds: 900
  allow_user_cancel: true
  allow_admin_cancel: true

monitoring:
  enabled: true
  notify_on_consecutive_failures: 3

history:
  enable_tags: true
  enable_pin: true
  enable_title_edit: true

deployment:
  development:
    require_docker: false
  production:
    recommended: "docker_compose"
    optional_worker: true
```

設定値は、アプリケーションごとに変更できる。

## 9. 受け入れ条件

将来構想は、次の条件を満たすと受け入れ可能である。

- 未ログイン状態ではチャット画面を利用できない。
- ログインユーザごとにチャット履歴が分離される。
- 一般ユーザは自分の履歴だけを閲覧できる。
- 管理者ロールを持つユーザが管理者画面へアクセスできる。
- チャット実行がジョブとして永続化される。
- ジョブ状態が `queued` から完了または失敗まで追跡できる。
- 同時実行数上限を超えたジョブがキューで待機する。
- 実行中ジョブをキャンセルできる。
- タイムアウト時にジョブ状態が `timed_out` になる。
- 失敗、キャンセル、タイムアウトしたジョブを新しいジョブとして再実行できる。
- 管理者がジョブ一覧とエラー詳細を確認できる。
- チャット履歴を全文検索できる。
- チャット履歴にタグを付けられる。
- チャット履歴をピン留めできる。
- チャットタイトルを編集できる。
- 認証、ジョブ管理、監視がMVPの `FastAPI + React/Vite + PostgreSQL` 構成を維持して実現できる。
- 社内配布・本番運用時にDocker Composeで起動でき、開発時にはDockerを必須にしない。
- PDF/HTML参照元表示切替、回答品質評価、スキーマバージョン管理、利用状況分析が要件に含まれていない。
