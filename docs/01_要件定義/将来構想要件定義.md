# D-Concierge 将来構想要件定義

## 1. 文書の目的

本書は、D-Conciergeの将来拡張として追加する将来構想の要件を定義する。

本書では、将来拡張で必要となる業務要求、機能要求、画面要求、データ要求、外部連携要求、非機能要求、運用制約を定義する。ジョブ状態名、設定ファイル例、データ保存方式、監視基盤、コンテナ構成などの詳細は、後続設計または `docs/00_設計・技術検証メモ/` に委ねる。

## 2. システム概要

### 2.1 背景

本システムでは、D-Conciergeを個人または小規模利用のユーザ指示応答・分析チャットアプリとして提供する。将来構想では、組織で継続利用できる業務アプリケーションへ拡張する。

組織利用では、ログインユーザごとの履歴分離を前提に、実行状態管理、管理者による監視、履歴整理、入力拡張が必要になる。

### 2.2 システム化の目的

将来構想では、次の拡張を目的とする。

- チャット実行をジョブとして管理し、状態確認、タイムアウト、再実行を可能にする。
- 管理者が運用状態とエラー状態を確認できるようにする。
- ロールにより通常利用者と管理者向け機能を区別できるようにする。
- チャット履歴を検索・整理しやすくする。
- ユーザ指示に画像ファイルを添付できるようにする。

### 2.3 対象利用者

将来構想の対象利用者は次の通りである。

| 区分 | 役割 |
| --- | --- |
| 一般ユーザ | チャットを利用し、自分の履歴を閲覧・検索・整理する。 |
| 管理者 | ジョブ状態、エラー状態、監視情報を確認する。 |
| 運用者 | アプリケーション設定、データソース、実行環境、バックアップ、通知先を管理する。 |
| 開発者 | 用途別アプリに必要な生成指示、検証指示、Skills、参照元ビューア、出力契約を追加・保守する。 |

## 3. 開発対象範囲

### 3.1 将来拡張対象

将来構想で追加する範囲は次の通りである。

- ロールによる利用者区分。
- 管理者向け機能のアクセス制御。
- チャット実行のジョブ管理。
- 実行待ち、実行中、完了、失敗、タイムアウトの状態管理。
- 失敗、タイムアウトしたジョブの再実行。
- 管理者向けのジョブ一覧、ジョブ詳細、エラー詳細、監視情報。
- 重大な運用エラーの通知。
- チャット履歴検索。
- ピン留め、タイトル編集による履歴整理。
- ユーザ指示送信時の画像ファイル添付。

### 3.2 将来構想の対象外

将来構想に含めない範囲は次の通りである。

- 利用者によるデータソース追加。
- 管理画面からのデータソース更新。
- データソース追加または検索対象登録を目的としたPDF、CSV、その他ファイルのアップロード。
- 生成指示やSkillsの画面編集。
- 回答品質評価。
- スキーマバージョン管理。
- 利用状況分析。
- 複数種類の参照元ビューアを利用者が画面上で切り替える機能。
- 本システムで提供する利用者キャンセル機能の管理者向け拡張。

データソースは本システムと同じく、開発者または運用者がアプリケーションごとに配置・設定する。

## 4. 業務要件

### 4.1 組織利用業務

一般ユーザは、チャット画面を利用する。一般ユーザは必要に応じてユーザ指示に画像ファイルを添付し、自分のチャット履歴だけを閲覧・検索・整理できる。他ユーザの履歴、ユーザ指示、回答、参照元は閲覧できない。

管理者は、運用調査に必要な範囲でジョブ状態とエラー状態を確認できる。ただし、管理者が他ユーザの回答本文を閲覧できるかどうかは、運用ポリシーで制御できること。

### 4.2 ジョブ管理業務

システムは、1回のチャット実行をジョブとして管理する。ジョブには、回答生成、形式検証、参照元検証、再生成を含む一連の処理を対応付ける。

一般ユーザは、自分が開始したジョブの状態を確認できる。管理者は、システム全体のジョブ状態を確認できる。

### 4.3 監視業務

管理者は、運用状態を確認し、失敗や滞留が発生しているジョブを把握できる。重大な運用エラーが発生した場合、管理者または運用者が認識できること。

### 4.4 履歴整理業務

一般ユーザは、過去のチャットを検索し、ピン留め、タイトル編集により業務上見返しやすく整理できる。

## 5. 機能要件

### 5.1 ロール管理

- システムは、一般ユーザと管理者を区別できること。
- システムは、管理者向け機能を管理者ロールに限定できること。
- システムは、一般ユーザが管理者向け監視機能へアクセスすることを防止できること。

### 5.2 ジョブ管理

- システムは、チャット実行をジョブとして永続的に管理できること。
- システムは、ジョブの受付、実行、検証、再生成、完了、失敗、タイムアウトを追跡できること。
- システムは、同時に実行するジョブ数を制御できること。
- システムは、同時実行数を超えるジョブを待機させられること。
- システムは、ジョブの作成日時、開始日時、終了日時、実行ユーザ、実行状態、再試行回数、エラー分類を記録できること。

### 5.3 タイムアウト

- システムは、ジョブ全体または処理段階に対してタイムアウトを設定できること。
- システムは、タイムアウトしたジョブを失敗とは区別して扱えること。
- システムは、タイムアウト時に利用者へ回答を表示できなかった理由を示すこと。

### 5.4 再実行

- システムは、失敗、タイムアウトしたジョブを再実行できること。
- システムは、再実行時に元のユーザ指示本文、アプリケーション設定、実行ユーザを引き継げること。
- システムは、再実行を過去ジョブの上書きではなく、新しいジョブとして記録できること。

### 5.5 管理者向け監視

- 管理者は、ジョブ一覧を確認できること。
- 管理者は、ジョブ詳細とエラー詳細を確認できること。
- 管理者は、実行中ジョブ数、待機中ジョブ数、成功件数、失敗件数、タイムアウト件数を確認できること。
- システムは、重大な運用エラーを管理者または運用者が認識できる形で通知または表示できること。
- 管理者画面には、秘密情報、絶対パス、環境変数、認証情報、不要な全文データを表示しないこと。
- 管理者向けエラー詳細は、トレースログ全量ではなく、必要範囲の要約と分類を表示すること。

### 5.6 履歴強化

- システムは、チャット履歴を検索できること。
- チャット履歴検索の対象は、チャットタイトル、ユーザ指示本文、回答本文、参照元ラベルとすること。
- システムは、重要なチャット履歴をピン留めできること。
- システムは、チャットタイトルを利用者が編集できること。
- ピン留め、タイトル編集はユーザごとに管理すること。

### 5.7 画像ファイル添付

- システムは、利用者がユーザ指示に画像ファイルを添付できること。
- システムは、添付画像を当該チャット実行の入力として回答生成に利用できること。
- システムは、添付画像をデータソースまたは後続の検索対象として登録しないこと。
- システムは、添付画像をチャット実行と紐づけて保存し、履歴再表示時にも確認できること。
- システムは、許可されていないファイル種別の添付を拒否できること。

## 6. 画面要件

### 6.1 チャット画面の拡張

チャット画面は、本システムのチャット機能に加えて、ジョブ状態表示、再実行操作、履歴整理操作、画像ファイル添付操作を提供する。

主な追加表示・操作は次の通りである。

- ジョブ状態。
- 再実行操作。
- ファイル添付ボタン。
- 添付画像プレビュー。
- 添付画像の解除操作。
- ユーザ指示表示内の添付画像。
- 過去履歴再表示時の添付画像。
- ピン留め操作。
- タイトル編集操作。

### 6.2 管理者画面

管理者画面は、管理者がジョブ状態、エラー状態、監視情報を確認するための画面である。

主な表示・操作は次の通りである。

- ジョブ一覧。
- ジョブ詳細。
- エラー詳細。
- 監視指標。
- 重大エラーの表示。

管理者画面では、データソース追加、データソース登録を目的としたファイルアップロード、生成指示の編集、Skillsの編集、回答品質評価、利用状況分析を扱わない。

## 7. データ要件

将来構想で追加または強化する主要データは次の通りである。

| データ | 内容 |
| --- | --- |
| ロール | 一般ユーザ、管理者などの権限区分。 |
| ジョブ | チャット実行単位、実行状態、時刻、再試行、エラー分類。 |
| 監視情報 | ジョブ件数、失敗件数、滞留、処理時間、重大エラー。 |
| ピン留め | ユーザが重要な履歴を固定表示するための情報。 |
| タイトル編集 | ユーザが変更したチャットタイトル。 |
| 添付画像 | チャット実行に紐づく利用者添付画像。 |

将来構想でも、チャット履歴、回答、参照元、添付画像はユーザ単位で分離する。

## 8. 外部連携要件

### 8.1 通知連携

システムは、重大な運用エラーを管理者または運用者に通知できること。通知先や通知方式はアプリケーションごとに設定できること。

### 8.2 データソース連携

将来構想でも、データソースの追加・更新は開発者または運用者が行う。一般ユーザや管理者が画面からデータソースを追加・更新する外部連携は対象外とする。

## 9. 非機能要件

### 9.1 セキュリティ

- 一般ユーザは他ユーザの履歴、ジョブ、ピン留め、タイトル編集情報へアクセスできないこと。
- 管理者機能は管理者ロールに限定すること。
- 管理者画面にも秘密情報を表示しないこと。
- 添付画像は、許可された利用者と対象チャットからだけ参照できること。

### 9.2 操作性

- 一般ユーザは、チャット利用中にジョブ状態を確認できること。
- ジョブ状態は、利用者が現在の処理状況を理解できる表現で表示すること。
- 再実行、履歴整理操作は、誤操作を防ぎつつ簡単に実行できること。

### 9.3 信頼性

- ジョブ状態は、画面表示と実際の実行状態が矛盾しないよう管理すること。
- 再実行しても過去ジョブの記録を失わないこと。
- 障害調査に必要な記録を保存できること。

### 9.4 運用性

- 管理者または運用者が、失敗、滞留、タイムアウト、連続エラーを確認できること。
- 同時実行数やタイムアウトなどの運用上の制約を設定できること。
- 通知先や通知条件を運用に合わせて調整できること。
- バックアップと復元の対象に、ユーザ、履歴、ジョブ、ピン留め、タイトル編集情報、添付画像を含められること。

### 9.5 保守性

- 将来拡張後も、D-Concierge本体は汎用的なユーザ指示応答・分析チャット基盤として維持すること。
- データソース固有の処理は、引き続き用途別の生成指示、検証指示、Skills、出力契約、参照元ビューアへ分離すること。
- ジョブ管理、監視、履歴強化は、本システムのチャット機能を不必要に複雑化させない形で追加できること。

## 10. 運用要件・制約

- 将来構想は、組織内で継続利用される業務アプリケーションを前提とする。
- 管理者は、システム全体の実行状態を確認できること。
- 運用者は、設定、通知、バックアップ、ログ確認を管理できること。
- データソースの追加・更新は、引き続き開発者または運用者の作業とすること。
- 回答品質評価、利用状況分析、スキーマバージョン管理は将来構想の対象外とすること。

## 11. 受け入れ条件

将来構想は、次の条件を満たすと受け入れ可能である。

- 管理者ロールを持つユーザが管理者画面へアクセスできる。
- チャット実行をジョブとして追跡できる。
- 同時実行数上限を超えたジョブを待機させられる。
- タイムアウトしたジョブを識別できる。
- 失敗、タイムアウトしたジョブを新しいジョブとして再実行できる。
- 管理者がジョブ一覧、ジョブ詳細、エラー詳細、監視情報を確認できる。
- 重大な運用エラーを管理者または運用者が認識できる。
- チャットタイトル、ユーザ指示本文、回答本文、参照元ラベルを対象にチャット履歴を検索できる。
- チャット履歴をピン留めできる。
- チャットタイトルを編集できる。
- ユーザ指示に画像ファイルを添付し、回答生成入力として利用できる。
- 添付画像をユーザ指示表示と履歴再表示で確認できる。
- 添付画像がデータソースまたは検索対象として登録されない。
- データソース追加、管理画面からのデータソース更新、回答品質評価、利用状況分析が要件に含まれていない。
