# D-Concierge MVP要件定義

## 1. 文書の目的

本書は、D-ConciergeのMVPで実現する業務要求、機能要求、画面要求、データ要求、外部連携要求、非機能要求、運用制約を定義する。

本書では、D-Conciergeを汎用的な質問応答・分析チャットアプリとして定義する。実装方式、詳細な設定例、内部データ形式、コマンド実行例、ライブラリ選定、ディレクトリ構成の詳細は、後続の外部設計、内部設計、または `docs/00_設計メモ/` に委ねる。

## 2. システム概要

### 2.1 背景

利用者が業務資料、説明書、センサデータ、コードベース、設計書、議事録などを調査する場合、情報源の探索、内容の要約、根拠確認に時間がかかる。D-Conciergeは、利用者が自然文で質問し、システムが設定済みのデータソースを検索・分析して、参照元を明示した回答を返すことで、この調査作業を支援する。

### 2.2 システム化の目的

D-Conciergeは、特定のデータソースに閉じた質問応答・分析チャットアプリを、共通の画面とバックエンド機能で構築できることを目的とする。

MVPでは、開発者がアプリケーションごとにデータソース、生成指示、検証指示、出力契約、参照元ビューアを設定し、利用者にはあらかじめ決められた専門領域に回答するチャットアプリとして提供する。

PDF検索は用途例の一つである。D-Concierge本体は、説明書検索、センサデータ分析、コードベース調査、設計書・議事録検索などにも転用できる共通基盤として扱う。

### 2.3 対象利用者

MVPの対象利用者は次の通りである。

| 区分 | 役割 |
| --- | --- |
| 利用者 | チャット画面から質問し、回答、参照元、中間メッセージ、過去履歴を確認する。 |
| 開発者 | 用途別アプリに必要なデータソース、生成指示、検証指示、出力契約、参照元ビューア、アプリケーション設定を用意する。 |
| 運用者 | アプリケーションの設定、データソース配置、実行環境、ログ確認、バックアップを管理する。MVPでは開発者と兼務してよい。 |

MVPではログインを必須にしない。ただし、システムはローカル利用者を識別し、チャット履歴とCodex execの作業領域を同じ利用者単位で分離できること。

## 3. 開発対象範囲

### 3.1 MVP対象

MVPで実現する範囲は次の通りである。

- 利用者がチャット画面から質問を送信できること。
- システムが設定済みのデータソースを検索・分析し、参照元付きの回答を生成できること。
- 回答生成中の中間メッセージを利用者へ逐次表示できること。
- 回答生成中に、利用者が自分の実行中処理をキャンセルできること。
- 回答を表示する前に、形式検証と参照元検証を実行できること。
- 検証に成功した回答だけをチャット画面に表示できること。
- 回答本文を読みやすい形式で表示できること。
- 回答中の参照元から、設定された参照元ビューアを開けること。
- PDF参照元ビューアを設定した場合は、参照元リンクから該当PDFページを表示できること。
- 過去のチャット履歴を一覧表示し、選択した履歴を再表示できること。
- 障害調査用のトレースログを保存できること。

### 3.2 MVP対象外

MVPで実現しない範囲は次の通りである。

- 利用者によるデータソース選択。
- 利用者によるPDF、CSV、その他データファイルのアップロード。
- 実行中アプリケーション内での生成指示、検証指示、Skills、データソースの切替。
- ログイン、ユーザ権限、組織管理。
- 管理画面からのデータソース更新。
- 複数アプリケーション間でのチャット履歴共有。
- 過去チャットに対する検索機能。
- 回答品質評価、利用状況分析、スキーマバージョン管理。

## 4. 業務要件

### 4.1 業務概要

利用者は、調査したい内容を自然文で入力し、システムから回答を受け取る。回答には参照元が含まれ、利用者は回答内容の根拠を確認できる。

用途例は次の通りである。

| 用途 | 利用者の質問例 |
| --- | --- |
| PDF検索 | 資料に記載された要件定義のポイントを整理してほしい。 |
| 説明書検索 | 機器のエラーコードの原因と対応手順を知りたい。 |
| センサ分析 | 直近期間の異常傾向と原因候補を確認したい。 |
| コード調査 | 機能の実装箇所、設計意図、変更影響を調べたい。 |

### 4.2 業務フロー

MVPの基本業務フローは次の通りである。

1. 利用者はチャット画面を開く。
2. 利用者は質問を入力して送信する。
3. システムは質問を受け付け、回答生成を開始する。
4. システムは処理状況を中間メッセージとして表示する。
5. 利用者は、回答生成中に必要に応じて自分の実行中処理をキャンセルする。
6. キャンセルされた場合、システムは部分回答を最終回答として表示せず、キャンセル済みであることを表示する。
7. キャンセルされない場合、システムは回答候補を検証する。
8. 検証に成功した場合、システムは回答と参照元を表示する。
9. 検証に失敗し、再試行可能な場合、システムは回答を再生成する。
10. 回答を表示できない場合、システムは利用者向けエラーを表示する。
11. 利用者は必要に応じて参照元を開き、根拠を確認する。
12. 利用者は過去履歴から以前のチャットを再表示する。

### 4.3 管理業務

MVPでは専用の管理画面を提供しない。開発者または運用者は、アプリケーション設定、データソース配置、生成指示、検証指示、出力契約、実行環境、ログ、バックアップを管理する。

## 5. 機能要件

### 5.1 チャット実行

- システムは、利用者が入力した質問を受け付けること。
- システムは、空の質問や受付できない入力を拒否し、利用者に修正を促すこと。
- システムは、質問送信後に回答生成を開始すること。
- システムは、回答生成中に同一チャットへの追加送信を制御し、送信操作をキャンセル操作へ切り替えること。
- システムは、利用者が自分の実行中処理をキャンセルできること。
- システムは、キャンセル要求中とキャンセル済みを利用者に示せること。
- システムは、新規チャットでは新しい作業領域を用意し、継続チャットでは既存の作業領域を再利用できること。

### 5.2 回答生成

- システムは、設定された生成指示とSkillsに従って、データソースを検索・分析できること。
- システムは、回答本文と参照元を含む構造化された回答を生成できること。
- システムは、用途別アプリの出力契約に従って回答を扱えること。
- システムは、生成中の処理状況を中間メッセージとして利用者へ表示できること。
- システムは、キャンセルされた実行の部分回答、未検証回答、途中生成物を最終回答として扱わないこと。

### 5.3 回答検証

- システムは、回答を画面に表示する前に形式検証を実行すること。
- システムは、回答の参照元が回答内容を支えているかを検証すること。
- システムは、検証に成功した回答だけを利用者に表示すること。
- システムは、検証失敗時に設定された上限まで再生成を試みること。
- システムは、再生成後も検証に成功しない場合、回答ではなくエラーメッセージを表示すること。

### 5.4 回答表示

- システムは、回答本文をMarkdown相当の読みやすい形式で表示できること。
- システムは、回答中の表、コードブロック、引用、リンクを表示できること。
- システムは、回答中のMermaid記法のブロックを図としてレンダリングできること。
- システムは、回答に図、画像、サニタイズ済みHTMLが含まれる場合、表示可能な形式で表示できること。
- システムは、表示に失敗した一部要素があっても、回答全体の閲覧を可能な範囲で継続すること。

### 5.5 参照元表示

- システムは、回答中の参照元リンクから、参照元種別に対応するビューアを開けること。
- システムは、登録済みの参照元種別だけを表示対象とすること。
- システムは、PDF参照元ビューアを設定した場合、該当PDFの指定ページまたはページ範囲を表示できること。
- システムは、参照元表示に失敗した場合、内部パスを表示せず、利用者向けの失敗メッセージを表示すること。

### 5.6 チャット履歴

- システムは、チャット、利用者メッセージ、回答、中間メッセージ、検証結果、エラー情報を履歴として保存できること。
- システムは、ローカル利用者単位で履歴を分離できること。
- システムは、過去チャットの一覧を表示できること。
- システムは、過去チャットを選択して再表示できること。
- システムは、履歴表示時に保存済みの回答と生成物を再現できること。

### 5.7 エラー処理

- システムは、入力エラー、回答生成失敗、検証失敗、タイムアウト、参照元表示失敗を利用者に分かる形で通知すること。
- システムは、利用者向けエラーに内部構成、絶対パス、秘密情報、スタックトレースを含めないこと。
- システムは、障害調査に必要な範囲でエラー分類、発生時刻、検証結果、再試行回数を記録できること。
- システムは、エラー発生時に障害調査用のトレースログを保存できること。
- システムは、トレースログを利用者向け画面に表示しないこと。

## 6. 画面要件

### 6.1 チャット画面

チャット画面は、利用者が質問し、回答と参照元を確認するための画面である。

主な表示項目は次の通りである。

- アプリケーション名。
- 新しいチャット操作。
- チャット履歴一覧。
- 利用者メッセージ。
- 中間メッセージ。
- 回答本文。
- 参照元リンク。
- 質問入力欄。
- 送信操作。
- キャンセル操作。
- 処理中表示。
- キャンセル要求中表示。
- キャンセル済み表示。
- エラーメッセージ。

主な操作は次の通りである。

- 新しいチャットを開始する。
- 質問を送信する。
- 回答生成中の自分の実行をキャンセルする。
- 過去チャットを選択する。
- 中間メッセージを展開または折りたたむ。
- 参照元を開く。

### 6.2 参照元ビューア

参照元ビューアは、回答の根拠となる情報源を表示するための画面または画面部品である。

主な要件は次の通りである。

- 参照元種別ごとに適切な表示方法を提供すること。
- 参照元の表示名または資料名を確認できること。
- 参照元の指定位置を初期表示できること。
- 表示を閉じてチャット画面に戻れること。
- 読み込みに失敗した場合、利用者向けの失敗メッセージを表示すること。

### 6.3 履歴画面領域

履歴はチャット画面内の一覧領域として表示する。MVPでは独立した履歴管理画面を必須にしない。

主な表示項目は次の通りである。

- チャットタイトル。
- 最終更新日時。
- 実行状態。

チャットタイトルは、最初の質問文から自動生成できること。利用者によるタイトル編集はMVPでは必須にしない。

## 7. データ要件

MVPで扱う主要データは次の通りである。

| データ | 内容 |
| --- | --- |
| ローカル利用者 | ログインなしで履歴と作業領域を分離するための利用者識別情報。 |
| チャット | 一連の質問応答をまとめる単位。 |
| メッセージ | 利用者の質問、システムの回答、中間メッセージ。 |
| 回答 | 回答本文、参照元、表示用生成物との関連。 |
| 参照元 | 回答の根拠となるデータソース種別、表示名、位置情報。 |
| 検証結果 | 形式検証、参照元検証、再試行結果。 |
| エラー情報 | エラー分類、発生時刻、利用者表示用メッセージ、調査用情報。 |
| トレースログ | チャット実行、発生段階、エラー分類、調査用情報を含む内部記録。 |
| アプリケーション設定 | 生成指示、検証指示、出力契約、データソース、参照元ビューア、実行制約。 |
| 生成物 | 回答表示に必要な画像、HTML、CSVなどの出力物。 |
| 実行状態 | 受付、実行中、検証中、キャンセル要求中、キャンセル済み、完了、エラーなどの状態。 |

データは、履歴再表示と障害調査に必要な期間保持できること。保持期間、削除方式、バックアップ方式の詳細は後続設計で定義する。

## 8. 外部連携要件

### 8.1 Codex exec連携

システムは、回答生成と参照元検証のためにCodex execを実行できること。

Codex exec連携の要件は次の通りである。

- 生成用と検証用の実行設定を分離できること。
- 生成用と検証用で別々の指示とSkillsを利用できること。
- 用途別アプリごとに生成指示、検証指示、Skills、作業領域、出力契約を差し替えられること。
- Codex execが出力する中間メッセージを利用者に逐次表示できること。
- 利用者のキャンセル要求に応じて、実行中のCodex exec連携処理をキャンセルできること。
- Codex execの作業領域は、利用者とチャット単位で分離できること。

### 8.2 データソース連携

システムは、開発者または運用者が配置したデータソースを検索・分析対象として扱えること。

MVPでは、利用者が画面からデータソースを追加・更新する機能は提供しない。データソースの配置、更新、構造化、検索用Skillsの準備は、開発者または運用者の責務とする。

### 8.3 設定連携

システムは、アプリケーションごとの設定に基づいて、生成、検証、参照元表示、履歴保存、実行制約を制御できること。

設定項目名、ファイル形式、ディレクトリ構成、具体的な相対パスの扱いは設計メモまたは後続設計で定義する。

## 9. 非機能要件

### 9.1 操作性

- 利用者は、チャット画面を開いてすぐ質問できること。
- 回答生成中であることが利用者に分かること。
- 回答生成中は、送信操作がキャンセル操作へ切り替わったことが分かること。
- 長時間処理でも、処理が止まっているのか進行中なのかを判断できること。
- 参照元確認からチャット画面へ戻りやすいこと。

### 9.2 応答性

- システムは、質問送信後、短時間で処理開始状態または中間メッセージを表示すること。
- システムは、回答生成が長時間化する場合に備え、タイムアウトを扱えること。
- システムは、キャンセル要求後、短時間でキャンセル要求中またはキャンセル済みであることを表示すること。
- MVPでは同時実行数を小さく保ってよい。ただし、実行中処理を識別し、状態を管理できること。

### 9.3 信頼性

- システムは、検証に成功していない回答を最終回答として表示しないこと。
- システムは、キャンセルされた実行の部分回答、未検証回答、途中生成物を最終回答として表示または履歴再表示に採用しないこと。
- システムは、再試行上限を超えた場合、回答表示を中止してエラー表示に切り替えること。
- システムは、履歴表示時に過去回答の内容を再現できること。
- システムは、回答表示用生成物が後から変更されても、履歴表示の内容が不意に変わらないように扱えること。

### 9.4 セキュリティ

- システムは、Codex execの作業範囲をアプリケーションが許可した範囲に限定すること。
- システムは、参照元や生成物のパスが許可範囲外を指す場合に拒否できること。
- システムは、ディレクトリトラバーサルを拒否できること。
- システムは、HTMLを表示する場合に安全化すること。
- システムは、任意のスクリプト実行を許可しないこと。
- システムは、中間メッセージ、回答、エラー、ログに秘密情報を不用意に表示または保存しないこと。
- システムは、トレースログに秘密情報や不要な全文データを保存しないこと。

### 9.5 互換性

- システムは、WindowsとLinuxの両方で利用できること。
- システムは、OS差異を利用者のチャット操作、キャンセル操作、履歴表示へ露出しないこと。
- システムは、OSごとのパス表現の違いにより、参照元表示、生成物表示、パス検証が不正確にならないこと。

### 9.6 保守性

- 共通の画面とバックエンド機能は、特定のデータソース固有の処理に依存しないこと。
- 用途別の差分は、生成指示、検証指示、Skills、出力契約、参照元ビューア、設定、データソースへ分離できること。
- PDF検索以外の用途へ転用できること。

### 9.7 監査性

- システムは、障害調査に必要な実行記録を保存できること。
- 実行記録には、チャット、実行開始・終了時刻、終了状態、検証結果、再試行回数、エラー分類を含められること。
- 実行記録には、秘密情報や不要な全文データを保存しないこと。
- システムは、回答生成失敗、検証失敗、キャンセル、タイムアウトなどの障害調査に必要なトレースログを保存できること。
- トレースログは、利用者向け表示ではなく運用者の障害調査に利用すること。

## 10. 運用要件・制約

- MVPは、社内または小規模組織での利用を前提とする。
- MVPは、Windows環境とLinux環境のどちらでも開発・利用できることを前提とする。
- 開発環境では、開発者が画面、バックエンド、データ保存先を個別に起動・確認できること。
- 運用環境では、アプリケーションとデータ保存先を少ない手順で起動・停止できること。
- チャット履歴と検証結果をバックアップできること。
- アプリケーション設定、生成指示、検証指示、Skills、作業領域、共有データソースはアプリケーション単位で差し替えられること。
- 共有データソースはセッションごとに複製しないこと。
- 生成物は、履歴表示に必要な内容を保存済みデータとして扱えること。

## 11. 受け入れ条件

MVPは、次の条件を満たすと受け入れ可能である。

- 利用者がチャット画面から質問を送信できる。
- 質問送信後、送信操作がキャンセル操作へ切り替わる。
- 回答生成中に、利用者が自分の実行中処理をキャンセルできる。
- キャンセル後、部分回答、未検証回答、途中生成物が最終回答として表示されない。
- キャンセル要求中またはキャンセル済みであることを画面に表示できる。
- システムが設定済みの生成指示とSkillsに従って回答を生成できる。
- 回答生成中の中間メッセージを表示できる。
- 回答完了後、中間メッセージを折りたたみ、必要に応じて再表示できる。
- 最終回答に対して形式検証を実行できる。
- 参照元検証を実行できる。
- 検証失敗時、設定上限まで再生成を試みることができる。
- 検証成功後にだけ回答を表示できる。
- 回答本文、表、コードブロック、Mermaid記法のブロック、図、画像、サニタイズ済みHTMLを表示できる。
- 設定された参照元種別に対応する参照元ビューアを開ける。
- PDF参照元ビューアを設定した場合は、参照元リンクから該当PDFページを表示できる。
- 過去チャット履歴を一覧表示できる。
- 未対応の参照元種別を最終表示前に拒否できる。
- エラー時に、内部パスや秘密情報を含まない利用者向けメッセージを表示できる。
- 回答生成失敗、検証失敗、キャンセル、タイムアウト時に、障害調査用のトレースログを保存できる。
- Windows環境とLinux環境の両方で、回答生成、キャンセル、履歴表示、参照元表示が利用できる。
