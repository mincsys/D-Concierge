# D-Concierge 要件定義

## 1. 文書の目的

本書は、D-Conciergeで実現する業務要求、機能要求、画面要求、データ要求、外部連携要求、非機能要求、運用制約を定義する。

本書では、D-Conciergeを汎用的なユーザ指示応答・分析チャットアプリとして定義する。実装方式、詳細な設定例、内部データ形式、コマンド実行例、ライブラリ選定、ディレクトリ構成、認証方式の詳細は、後続の外部設計または内部設計に委ねる。

## 2. システム概要

### 2.1 背景

利用者が業務資料、説明書、センサデータ、コードベース、設計書、議事録などを調査する場合、情報源の探索、内容の要約、根拠確認に時間がかかる。D-Conciergeは、利用者が自然文でユーザ指示を送信し、システムが設定済みのデータソースを検索・分析して、参照元を明示した回答を返すことで、この調査作業を支援する。

### 2.2 システム化の目的

D-Conciergeは、特定のデータソースに閉じたユーザ指示応答・分析チャットアプリを、共通の画面とバックエンド機能で構築できることを目的とする。

本システムでは、開発者がアプリケーションごとにデータソース、生成用・検証用ホームディレクトリ、出力契約、参照元ビューア実装を用意し、利用者にはあらかじめ決められた専門領域に回答するチャットアプリとして提供する。

PDF検索は用途例の一つである。D-Concierge本体は、説明書検索、センサデータ分析、コードベース調査、設計書・議事録検索などにも転用できる共通基盤として扱う。

### 2.3 対象利用者

本システムの対象利用者は次の通りである。

| 区分 | 役割 |
| --- | --- |
| 利用者 | アカウント登録またはログイン後、チャット画面からユーザ指示を送信し、回答、参照元、中間メッセージ、過去履歴を確認する。 |
| 開発者 | 用途別アプリに必要なデータソース、生成用・検証用ホームディレクトリ、出力契約、参照元ビューア実装、アプリケーション設定を用意する。 |
| 運用者 | アプリケーションの設定、データソース配置、実行環境、ログ確認、バックアップを管理する。本システムでは開発者と兼務してよい。 |

本システムではログインを必須とする。システムはログインユーザごとにチャット履歴とCodex実行の作業領域を分離する。ロール、管理者画面、管理者によるユーザ管理は本システムの対象外とする。

## 3. 開発対象範囲

### 3.1 対象

本システムで実現する範囲は次の通りである。

- 利用者がチャット画面からユーザ指示を送信できること。
- 利用者が開始画面からユーザ指示を入力し、設定された入力候補を利用できること。
- 利用者がアカウントを登録し、ログインできること。
- 利用者がログアウトできること。
- 利用者がユーザ名とパスワードを変更できること。
- 利用者が自分のアカウントを削除できること。
- ログインユーザごとにチャット履歴と作業領域を分離できること。
- システムが設定済みのデータソースを検索・分析し、参照元付きの回答を生成できること。
- 回答生成中の中間メッセージを利用者へ逐次表示できること。
- 回答生成中に、利用者が画面で実行中の処理をキャンセルできること。
- 回答を表示する前に、固定検証処理とCodex検証処理を含む回答検証処理を実行できること。
- 検証に成功した回答だけをチャット画面に表示できること。
- 回答本文を読みやすい形式で表示できること。
- 回答中の参照元から、参照元種別に対応する実装済み参照元ビューアを開けること。
- PDF参照元ビューアを実装済みビューアとして組み込む場合は、参照元リンクから該当PDFページを表示できること。
- 生成用・検証用のCodex実行をDockerコンテナ内で行い、許可された範囲でデータソース参照と成果物作成を行えること。
- 過去のチャット履歴を一覧表示し、選択した履歴を再表示できること。
- 過去のチャット履歴から、チャット一式を削除できること。
- チャット履歴一覧を含むサイドバーを折りたたみ、必要に応じて展開できること。
- 障害調査用のトレースログを保存できること。

### 3.2 対象外

本システムで実現しない範囲は次の通りである。

- 利用者によるデータソース選択。
- 利用者によるPDF、CSV、その他データファイルのアップロード。
- 実行中アプリケーション内での生成用・検証用ホームディレクトリ、データソースの切替。
- ロール、管理者画面、管理者によるユーザ管理、組織管理。
- パスワード再設定。
- 管理画面からのデータソース更新。
- 複数アプリケーション間でのチャット履歴共有。
- 過去チャットに対する検索機能。
- 回答品質評価、利用状況分析、スキーマバージョン管理。

## 4. 業務要件

### 4.1 業務概要

利用者は、調査したい内容を自然文で入力し、システムから回答を受け取る。回答には参照元が含まれ、利用者は回答内容の根拠を確認できる。

用途例は次の通りである。

| 用途 | 利用者のユーザ指示例 |
| --- | --- |
| PDF検索 | 資料に記載された要件定義のポイントを整理してほしい。 |
| 説明書検索 | 機器のエラーコードの原因と対応手順を知りたい。 |
| センサ分析 | 直近期間の異常傾向と原因候補を確認したい。 |
| コード調査 | 機能の実装箇所、設計意図、変更影響を調べたい。 |

### 4.2 業務フロー

本システムの基本業務フローは次の通りである。

1. 利用者はアカウント登録またはログインを行う。
2. 利用者はチャット画面を開く。
3. 利用者はユーザ指示を入力して送信する。
4. システムはユーザ指示を受け付け、回答生成を開始する。
5. システムは処理状況を中間メッセージとして表示する。
6. 利用者は、回答生成中に必要に応じて画面で実行中の処理をキャンセルする。
7. キャンセルされた場合、システムは部分回答を最終回答として表示せず、キャンセル済みであることを表示する。
8. キャンセルされない場合、システムは回答候補を検証する。
9. 検証に成功した場合、システムは回答と参照元を表示する。
10. 検証に失敗し、再試行可能な場合、システムは回答を再生成する。
11. 回答を表示できない場合、システムは利用者向けエラーを表示する。
12. 利用者は必要に応じて参照元を開き、根拠を確認する。
13. 利用者は過去履歴から以前のチャットを再表示する。
14. 利用者は不要になった過去チャットを選択し、チャット一式を削除する。
15. 利用者は必要に応じてユーザ名変更、パスワード変更、ログアウト、アカウント削除を行う。

### 4.3 管理業務

本システムでは専用の管理画面を提供しない。開発者または運用者は、アプリケーション設定、データソース配置、生成用・検証用ホームディレクトリ、出力契約、実行環境、ログ、バックアップを管理する。

参照元ビューア実装は、開発者が用途別アプリに組み込む。

## 5. 機能要件

### 5.1 認証・アカウント管理

- システムは、未ログイン状態の利用者にログイン画面を表示できること。
- システムは、ログイン状態に関わらずログイン画面とアカウント登録画面を直接表示できること。
- システムは、利用者がユーザID、ユーザ名、パスワードを登録できること。
- システムは、登録済みのユーザIDとパスワードで利用者を認証できること。
- システムは、ユーザIDの形式と重複を検証できること。
- システムは、ログイン状態を長期保持できること。
- システムは、ログイン中の利用者を画面上で識別できること。
- システムは、未ログイン状態で保護対象機能を利用させないこと。
- システムは、利用者がログアウトできること。
- システムは、利用者が自分のユーザ名を変更できること。
- システムは、利用者が現在のパスワードを確認した上でパスワードを変更できること。
- システムは、利用者が自分のアカウントを削除できること。
- システムは、アカウント削除時に対象ユーザに紐づくチャット履歴、作業領域、保存済みCodex成果物、ログインセッションを削除対象にできること。
- システムは、アカウント削除処理が途中失敗した場合、削除中の利用者を通常利用へ復帰させず、後続の再試行対象として扱えること。

### 5.2 チャット実行

- システムは、開始画面を表示し、利用者がすぐにユーザ指示を入力できること。
- システムは、アプリケーション設定に基づいて入力候補を表示できること。
- システムは、利用者が選択した入力候補を入力欄に反映できること。
- システムは、利用者が入力したユーザ指示を受け付けること。
- システムは、空のユーザ指示や受付できない入力を拒否し、利用者に修正を促すこと。
- システムは、ユーザ指示送信後に回答生成を開始すること。
- システムは、同一チャットに未完了のチャット実行処理がある間、追加のユーザ指示を受け付けないこと。
- 未完了のチャット実行処理は、受付、実行中、検証中、キャンセル要求中のいずれかの状態にある処理とする。
- システムは、回答生成中に送信操作をキャンセル操作へ切り替えること。
- システムは、利用者が画面で実行中の処理をキャンセルできること。
- システムは、キャンセル要求中とキャンセル済みを利用者に示せること。
- システムは、対象チャットの最新処理が完了、キャンセル済み、エラー、タイムアウトのいずれかになった後に、継続指示を受け付けられること。
- システムは、新規チャットでは新しい作業領域を用意し、継続チャットでは既存の作業領域を再利用できること。

### 5.3 回答生成

- システムは、生成用ホームディレクトリの `AGENTS.md` とSkillsに従って、データソースを検索・分析できること。
- システムは、回答本文と参照元を含む構造化された回答を生成できること。
- システムは、用途別アプリの出力契約に従って回答を扱えること。
- システムは、生成、検証、再生成の処理状況を中間メッセージとして利用者へ表示できること。
- システムは、キャンセルされた実行の部分回答、未検証回答、途中生成物を最終回答として扱わないこと。

### 5.4 回答検証

- システムは、回答を画面に表示する前に固定検証処理を実行すること。
- システムは、Codex検証処理で生成用Codexの回答を検証すること。
- システムは、検証に成功した回答だけを利用者に表示すること。
- システムは、検証失敗時に設定された上限まで再生成を試みること。
- システムは、再生成後も検証に成功しない場合、回答ではなくエラーメッセージを表示すること。

### 5.5 回答表示

- システムは、回答本文をMarkdownとして読みやすい形式で表示できること。
- システムは、回答中の表、コードブロック、引用、リンクを表示できること。
- システムは、回答中のMermaid記法のブロックを図としてレンダリングできること。
- システムは、回答に図、画像、サニタイズ済みHTMLが含まれる場合、表示可能な形式で表示できること。
- システムは、回答中の画像を拡大、縮小、パン操作で確認できること。
- システムは、表示に失敗した一部要素があっても、回答全体の閲覧を可能な範囲で継続すること。

### 5.6 参照元表示

- システムは、回答中の参照元リンクから、参照元種別に対応する実装済み参照元ビューアを開けること。
- システムは、登録済みの参照元種別だけを表示対象とすること。
- システムは、PDF参照元ビューアを実装済みビューアとして組み込む場合、該当PDFの指定ページまたはページ範囲を表示できること。
- システムは、参照元表示に失敗した場合、内部パスを表示せず、利用者向けの失敗メッセージを表示すること。

### 5.7 チャット履歴

- システムは、チャット、利用者メッセージ、回答、中間メッセージ、実行状態、エラー情報を履歴として保存できること。
- システムは、ログインユーザごとにチャット履歴を保存し、チャット単位で履歴を再表示できること。
- システムは、過去チャットの一覧を表示できること。
- システムは、過去チャットを選択して再表示できること。
- システムは、履歴表示時に保存済みの回答と生成物を再現できること。
- システムは、履歴項目から削除操作を開始できること。
- システムは、削除操作により、チャット、利用者メッセージ、回答、中間メッセージ、実行状態、エラー情報、参照元、Codex成果物メタ情報を含むチャット一式を削除できること。
- システムは、削除操作により、当該チャットに紐づく生成用作業ディレクトリ、検証用作業ディレクトリ、保存済みCodex成果物実体を削除できること。
- システムは、実行中チャットに対する削除操作を受け付けた場合、キャンセル要求を行い、実行が終了した後に削除できること。
- システムは、削除したチャット一式を履歴一覧、履歴再表示、成果物配信の対象外にできること。

### 5.8 エラー処理

- システムは、入力エラー、回答生成失敗、検証失敗、タイムアウト、参照元表示失敗を利用者に分かる形で通知すること。
- システムは、利用者向けエラーに内部構成、絶対パス、秘密情報、スタックトレースを含めないこと。
- システムは、障害調査に必要な範囲でエラー分類、発生時刻、検証失敗理由、再試行回数をトレースログへ記録できること。
- システムは、エラー発生時に障害調査用のトレースログを保存できること。
- システムは、トレースログを利用者向け画面に表示しないこと。

## 6. 画面要件

### 6.1 ログイン画面

ログイン画面は、利用者が認証してアプリケーションを利用開始するための画面である。
ログイン画面は、ログイン状態に関わらず直接表示できる。

主な表示項目は次の通りである。

- ロゴ。
- アプリケーション名。
- ユーザID入力欄。
- パスワード入力欄。
- ログイン操作。
- アカウント登録画面への導線。
- ログイン失敗時のメッセージ。

### 6.2 アカウント登録画面

アカウント登録画面は、利用者が自分のアカウントを登録するための画面である。
アカウント登録画面は、ログイン状態に関わらず直接表示できる。

主な表示項目は次の通りである。

- ロゴ。
- アプリケーション名。
- ユーザID入力欄。
- ユーザ名入力欄。
- パスワード入力欄。
- パスワード確認入力欄。
- 登録操作。
- ログイン画面へ戻る導線。
- 入力エラーメッセージ。

### 6.3 開始画面

開始画面は、利用者が新しいユーザ指示を開始するための画面である。

主な表示項目は次の通りである。

- アプリケーション名。
- ウェルカムメッセージ。
- 補足案内文。
- 入力欄。
- 送信操作。
- 入力候補チップ。
- ログイン中ユーザ表示。

主な操作は次の通りである。

- ユーザ指示を入力する。
- 入力候補チップを選択し、入力欄へ反映する。
- ユーザ指示を送信する。
- 設定ダイアログを開く。

入力候補チップは、アプリケーション設定に基づいて表示内容を変更できること。

### 6.4 チャット画面

チャット画面は、利用者がユーザ指示を送信し、回答と参照元を確認するための画面である。

主な表示項目は次の通りである。

- アプリケーション名。
- 新しいチャット操作。
- チャット履歴一覧。
- 利用者メッセージ。
- 中間メッセージ。
- 回答本文。
- 参照元リンク。
- 入力欄。
- 送信操作。
- キャンセル操作。
- サイドバー折りたたみ・展開操作。
- 処理中表示。
- キャンセル要求中表示。
- キャンセル済み表示。
- エラーメッセージ。
- ログイン中ユーザ表示。

主な操作は次の通りである。

- 新しいチャットを開始する。
- ユーザ指示を送信する。
- 回答生成中に画面で実行中の処理をキャンセルする。
- 過去チャットを選択する。
- 過去チャットを削除する。
- サイドバーを折りたたむ、または展開する。
- 中間メッセージを展開または折りたたむ。
- 回答内画像を拡大表示する。
- 参照元を開く。
- 設定ダイアログを開く。

回答本文では、Markdown、表、コードブロック、画像、Mermaid図、サニタイズ済みHTMLを表示できること。回答内画像は、必要に応じて拡大表示し、表示範囲を操作できること。

### 6.5 参照元ビューア

参照元ビューアは、回答の根拠となる情報源を表示するための画面または画面部品である。

主な要件は次の通りである。

- 参照元種別ごとに適切な表示方法を提供すること。
- 参照元の表示名または資料名を確認できること。
- 参照元の指定位置を初期表示できること。
- 表示を閉じてチャット画面に戻れること。
- 読み込みに失敗した場合、利用者向けの失敗メッセージを表示すること。

### 6.6 履歴画面領域

履歴はチャット画面内の一覧領域として表示する。本システムでは独立した履歴管理画面を必須にしない。

履歴一覧で利用者に表示する文字情報は、チャットタイトルとする。チャットタイトルは、最初のユーザ指示本文から自動生成できること。利用者によるタイトル編集は本システムでは必須にしない。

履歴一覧は最終更新日時の降順で表示できること。実行状態は、履歴詳細の再表示、回答中チャットへの復帰、キャンセル済み・エラー・タイムアウト時の状態表示に利用できること。履歴項目から削除操作を開始し、不要なチャット一式を削除できること。

### 6.7 設定ダイアログ

設定ダイアログは、ログイン中の利用者がアカウント操作を行うための画面部品である。

主な表示項目は次の通りである。

- アカウントメニュー。
- ユーザID表示。
- ユーザ名変更操作。
- パスワード変更操作。
- ログアウト操作。
- アカウント削除操作。
- 入力エラーメッセージ。
- 確認ダイアログ。

主な操作は次の通りである。

- ユーザ名を変更する。
- パスワードを変更する。
- ログアウトする。
- アカウントを削除する。

## 7. データ要件

本システムで扱う主要データは次の通りである。

| データ | 内容 |
| --- | --- |
| ユーザ | ログイン利用者を識別し、ユーザ名、認証情報、利用状態を管理する情報。 |
| ログインセッション | ブラウザごとのログイン状態と有効期限を管理する情報。ログアウト、アカウント削除、有効期限切れでは削除対象とする。 |
| チャット | 一連のユーザ指示応答をまとめる単位。 |
| ユーザ指示 | 利用者が送信した1回分の自然文入力。 |
| 中間メッセージ | 回答生成や検証の途中状況を利用者へ示すメッセージ。 |
| 回答 | 回答本文、参照元、表示用生成物との関連。 |
| 参照元 | 回答の根拠となるデータソース種別、表示名、位置情報。 |
| エラー情報 | エラー分類、発生時刻、利用者表示用メッセージ、調査用情報。 |
| トレースログ | チャット実行、発生段階、エラー分類、調査用情報を含む内部記録。 |
| アプリケーション設定 | UI設定、生成用・検証用ホームディレクトリ、出力契約、データソース、実行制約。 |
| 生成物 | 回答表示に必要な画像、HTML、CSVなどの出力物。 |
| セッション作業領域 | 生成用・検証用のCodex実行がチャット単位で利用する作業ディレクトリ。 |
| 保存済みCodex成果物実体 | 検証済み回答が参照する画像、HTML、CSVなどの保存済みファイル。 |
| 実行状態 | 受付、実行中、検証中、キャンセル要求中、キャンセル済み、完了、エラーなどの状態。 |

データは、削除操作が行われない限り、履歴再表示と障害調査に必要な期間保持できること。ユーザ、ログインセッション、チャット履歴、保存済み回答、参照元、Codex成果物は履歴再表示と認証に必要なデータとして保持し、トレースログは障害調査用として保存期間とアプリケーション起動ごとの同日最大保存件数を設定値で管理できること。標準ではトレースログ保存期間を90日、アプリケーション起動ごとの同日最大保存件数を1000件とする。トレースログは保存期間を過ぎた日付ディレクトリを削除対象とし、同日上限到達後は当日中の追加保存を抑止できること。

利用者がチャットを削除した場合、チャット、利用者メッセージ、回答、中間メッセージ、実行状態、エラー情報、参照元、Codex成果物メタ情報を含むDB上のチャット一式を履歴一覧と履歴再表示の対象外にできること。あわせて、`generator.workdir/<user-id>/<session-id>` の生成用作業ディレクトリ、`validator.workdir/<user-id>/<session-id>` の検証用作業ディレクトリ、当該チャットの保存済みCodex成果物メタ情報に対応する `generator.saved_artifacts_dir` 配下の成果物実体と保存先ディレクトリを削除できること。共有データソース本体はチャット削除の対象外とする。トレースログはチャット削除の対象外とし、既存の保存期間と同日最大保存件数のルールに従って保持すること。

利用者が自分のアカウントを削除した場合、対象ユーザに紐づくユーザ情報、ログインセッション、チャット履歴、ユーザ指示、実行状態、中間メッセージ、回答、参照元、Codex成果物メタ情報、保存済みCodex成果物実体、生成用・検証用作業領域を削除対象にできること。トレースログはアカウント削除の対象外とし、既存の保存期間と同日最大保存件数のルールに従って保持すること。

## 8. 外部連携要件

### 8.1 Codex連携

システムは、回答生成のために生成用Codexを、回答検証のために検証用CodexをDockerコンテナ内で実行できること。

Codex連携の要件は次の通りである。

- 生成用と検証用の実行設定を分離できること。
- 生成用と検証用で、各ホームディレクトリに含まれる別々の指示とSkillsを利用できること。
- Codex認証情報をアプリケーションごとに設定できること。
- 用途別アプリごとに生成用・検証用ホームディレクトリ、作業領域、出力契約を差し替えられること。
- Codexが出力する中間メッセージを利用者に逐次表示できること。
- 利用者のキャンセル要求に応じて、実行中のCodex連携処理をキャンセルできること。
- Codex実行時にアクセスできる範囲を、アプリケーションが許可したデータソースと成果物領域に限定できること。
- Codexの作業領域は、ログインユーザ配下でチャット単位に分離できること。

### 8.2 データソース連携

システムは、開発者または運用者が配置したデータソースを検索・分析対象として扱えること。

本システムでは、利用者が画面からデータソースを追加・更新する機能は提供しない。データソースの配置、更新、構造化、検索用Skillsの準備は、開発者または運用者の責務とする。

### 8.3 設定連携

システムは、アプリケーションごとの設定に基づいて、生成、検証、参照元表示、履歴保存、実行制約、入力候補表示を制御できること。

設定項目名、ファイル形式、ディレクトリ構成、具体的な相対パスの扱いは設計メモまたは後続設計で定義する。

## 9. 非機能要件

### 9.1 操作性

- 利用者は、ログイン後すぐユーザ指示を送信できること。
- 利用者は、アカウント登録、ログイン、ログアウト、ユーザ名変更、パスワード変更、アカウント削除を画面上で行えること。
- 回答生成中であることが利用者に分かること。
- 回答生成中は、送信操作がキャンセル操作へ切り替わったことが分かること。
- 長時間処理でも、処理が止まっているのか進行中なのかを判断できること。
- 参照元確認からチャット画面へ戻りやすいこと。

### 9.2 応答性

- システムは、ユーザ指示送信後、短時間で処理開始状態と中間メッセージを表示すること。
- システムは、回答生成が長時間化する場合に備え、タイムアウトを扱えること。
- システムは、キャンセル要求後、短時間でキャンセル要求中またはキャンセル済みであることを表示すること。
- 本システムでは同時実行数を小さく保ってよい。ただし、実行中処理を識別し、状態を管理できること。

### 9.3 信頼性

- システムは、検証に成功していない回答を最終回答として表示しないこと。
- システムは、キャンセルされた実行の部分回答、未検証回答、途中生成物を最終回答として表示または履歴再表示に採用しないこと。
- システムは、再試行上限を超えた場合、回答表示を中止してエラー表示に切り替えること。
- システムは、履歴表示時に過去回答の内容を再現できること。
- システムは、回答表示用生成物が後から変更されても、履歴表示の内容が不意に変わらないように扱えること。

### 9.4 セキュリティ

- システムは、未ログイン状態で保護対象機能を利用させないこと。
- システムは、ログインユーザが他ユーザのチャット履歴、参照元、Codex成果物、作業領域へアクセスすることを防止できること。
- システムは、認証情報を安全に扱うこと。
- システムは、Codex実行をDockerコンテナ内に閉じ、作業範囲をアプリケーションが許可した範囲に限定すること。
- システムは、参照元や生成物のパスが許可範囲外を指す場合に拒否できること。
- システムは、ディレクトリトラバーサルを拒否できること。
- システムは、HTMLを表示する場合に安全化すること。
- システムは、任意のスクリプト実行を許可しないこと。
- システムは、中間メッセージ、回答、エラー、ログに秘密情報を不用意に表示または保存しないこと。
- システムは、画面、SSE、利用者向けエラーに秘密情報や不要な全文データを表示しないこと。トレースログは開発者向け障害調査情報として扱い、巨大な文字列だけを上限長で切り詰めること。

### 9.5 互換性

- システムは、WindowsとLinuxの両方で利用できること。
- システムは、WindowsとLinuxの各環境でDockerを利用できることを前提にCodexを実行できること。
- システムは、OS差異を利用者のチャット操作、キャンセル操作、履歴表示へ露出しないこと。
- システムは、OSごとのパス表現の違いにより、参照元表示、生成物表示、パス検証が不正確にならないこと。

### 9.6 保守性

- 共通の画面とバックエンド機能は、特定のデータソース固有の処理に依存しないこと。
- 用途別の差分は、生成用・検証用ホームディレクトリ、出力契約、参照元ビューア実装、設定、データソースへ分離できること。
- PDF検索以外の用途へ転用できること。

### 9.7 監査性

- システムは、障害調査に必要な実行記録を保存できること。
- 実行記録には、チャット、実行開始・終了時刻、終了状態、エラー分類を含められること。
- トレースログには、障害調査に必要な検証失敗理由、再試行回数、失敗段階を含められること。
- 実行記録には、秘密情報や不要な全文データを保存しないこと。
- システムは、回答生成失敗、検証上限到達、キャンセル失敗、タイムアウトなどの障害調査に必要なトレースログを保存できること。
- トレースログは、利用者向け表示ではなく運用者の障害調査に利用すること。

## 10. 運用要件・制約

- 本システムは、社内または小規模組織での利用を前提とする。
- 本システムは、Windows環境とLinux環境のどちらでも開発・利用できることを前提とする。
- 開発環境では、開発者が画面、バックエンド、データ保存先を個別に起動・確認できること。
- 運用環境では、アプリケーションとデータ保存先を少ない手順で起動・停止できること。
- 運用環境では、Codex実行に利用するDocker実行環境を用意できること。
- チャット履歴、保存済み回答、参照元、Codex成果物をバックアップできること。
- ユーザ情報とログインセッションをバックアップできること。
- アプリケーション設定、生成用・検証用ホームディレクトリ、作業領域、共有データソースはアプリケーション単位で差し替えられること。
- 共有データソースはセッションごとに複製しないこと。
- 生成物は、履歴表示に必要な内容を保存済みデータとして扱えること。

## 11. 受け入れ条件

本システムは、次の条件を満たすと受け入れ可能である。

- 未ログイン状態では保護対象画面を利用できない。
- 利用者がアカウントを登録できる。
- 利用者が登録済みのユーザIDとパスワードでログインできる。
- ログイン成功後、開始画面を表示できる。
- ログイン状態を長期保持できる。
- ログイン中の利用者を画面上で識別できる。
- 利用者がログアウトできる。
- 利用者がユーザ名を変更できる。
- 利用者が現在のパスワードを確認した上でパスワードを変更できる。
- 利用者が自分のアカウントを削除できる。
- アカウント削除後、対象ユーザのログインセッションとユーザに紐づくデータが通常利用対象外になる。
- 利用者がチャット画面からユーザ指示を送信できる。
- 開始画面に設定された入力候補チップを表示できる。
- 入力候補チップを選択すると、入力欄へ反映できる。
- ユーザ指示送信後、送信操作がキャンセル操作へ切り替わる。
- 同一チャットに未完了のチャット実行処理がある間、追加のユーザ指示を受け付けない。
- 回答生成中に、利用者が画面で実行中の処理をキャンセルできる。
- キャンセル後、部分回答、未検証回答、途中生成物が最終回答として表示されない。
- キャンセル要求中またはキャンセル済みであることを画面に表示できる。
- システムが生成用ホームディレクトリの `AGENTS.md` とSkillsに従って回答を生成できる。
- 生成用・検証用のCodex実行をDockerコンテナ内で実行できる。
- Codex実行時にアクセスできる範囲を、許可されたデータソースと成果物領域に限定できる。
- 回答生成中の中間メッセージを表示できる。
- 回答完了後、中間メッセージを折りたたみ、必要に応じて再表示できる。
- 最終回答に対して固定検証処理を実行できる。
- Codex検証処理で生成用Codexの回答を検証できる。
- 検証失敗時、設定上限まで再生成を試みることができる。
- 検証成功後にだけ回答を表示できる。
- チャット履歴一覧を含むサイドバーを折りたたみ、展開できる。
- 回答本文、表、コードブロック、Mermaid記法のブロック、図、画像、サニタイズ済みHTMLを表示できる。
- 回答本文内の画像を拡大表示し、拡大、縮小、パン操作で確認できる。
- 実装済み参照元ビューアが対応する参照元種別を開ける。
- PDF参照元ビューアを実装済みビューアとして組み込む場合は、参照元リンクから該当PDFページを表示できる。
- 過去チャット履歴を一覧表示できる。
- 過去チャット履歴からチャット一式、当該セッションの作業ディレクトリ、保存済みCodex成果物実体を削除できる。
- 未対応の参照元種別を最終表示前に拒否できる。
- エラー時に、内部パスや秘密情報を含まない利用者向けメッセージを表示できる。
- 回答生成失敗、検証上限到達、キャンセル失敗、タイムアウト時に、障害調査用のトレースログを保存できる。
- Windows環境とLinux環境の両方で、Dockerを利用した回答生成、キャンセル、履歴表示、参照元表示が利用できる。
