# 回答スキーマと表示レンダリング

## 目的

本メモは、回答JSON、参照元表現、回答本文内データの内部処理メモを整理する。

画面で表示できる要素や参照元ビューアの画面仕様は外部設計を正とし、本メモではスキーマと変換処理の補足だけを扱う。

## 回答JSONスキーマの考え方

出力スキーマは、生成用codex execの最終出力契約であり、バックエンドの固定検証でも同じスキーマを使う。

内部処理で前提にする考え方:

- 回答本文はMarkdownとして扱える文字列である。
- 回答は複数のブロックに分割できる。
- 各回答ブロックは、0件以上の参照元を持つ。
- 参照元は `source_type` で種類を示し、スキーマ側の `const` で許可値を固定する。
- 参照元の具体的な位置情報は `locator` に格納する。
- `locator` の形は、`codex.output_schema` が指すスキーマで定義する。
- 参照元の表示用 `label` は生成用codex execの出力対象にせず、バックエンドの整形処理で `locator` から算出してフロントエンドへ渡す。
- 画像やHTMLなどの回答内データは、回答本文またはCodex成果物として扱う。

## 参照元JSONの例

PDFページを参照元とする例:

```json
{
  "source_type": "pdf",
  "locator": {
    "path": "raw/pdf/ユーザのための要件定義ガイド 第2版.pdf",
    "start_page": 42,
    "end_page": 45
  }
}
```

コード行範囲を参照元とする例:

```json
{
  "source_type": "code",
  "locator": {
    "path": "src/auth/session.ts",
    "start_line": 120,
    "end_line": 168
  }
}
```

PDF検索アプリを構成する場合の設定例では、`source_type` は `pdf` のみを対応必須とする。コード調査アプリなど別用途のアプリでは、`codex.output_schema` を差し替えて `source_type` と `locator` の形を変える。

表示用 `label` は、PDFでは `locator.path` のファイル名から拡張子を除いた資料名、コードでは `locator.path` そのものとしてバックエンドが算出する。ページ範囲や行範囲は `start_page`、`end_page`、`start_line`、`end_line` から別途扱う。

## 回答本文の例

回答本文の例:

```markdown
要件定義を成功させるポイントは、以下の5点です。

1. 目的と背景を共有する
2. 利用者視点で要求を具体化する
3. 優先順位とスコープを明確にする
4. 検証可能な形で要求を定義する
5. 継続的な見直しと合意形成を行う

参照元は各項目のリンクから確認できます。
```

回答本文にMermaidを含める場合の例:

````markdown
```mermaid
flowchart LR
  A["ユーザ指示"] --> B["検索"]
  B --> C["分析"]
  C --> D["回答"]
```
````

回答本文にHTMLを含める場合の例:

```html
<table>
  <thead>
    <tr><th>項目</th><th>説明</th></tr>
  </thead>
  <tbody>
    <tr><td>目的共有</td><td>関係者間の認識を合わせる</td></tr>
  </tbody>
</table>
```

## 参照元変換メモ

バックエンドは回答JSONの `source_type` と `locator` をもとに参照元を保存し、参照元IDを画面へ返す。

内部変換例:

| source_type | locatorの例 | 変換後の扱い |
| --- | --- | --- |
| `pdf` | `path`, `start_page`, `end_page` | PDF参照元として保存し、参照元IDを回答側へ紐づける。 |
| `code` | `path`, `start_line`, `end_line` | コード参照元として保存し、参照元IDを回答側へ紐づける。 |

指定スキーマに適合しない `source_type` は固定検証でエラーとし、再生成対象にする。

参照元リンクの表示文言は、保存済み参照元の表示ラベルと位置情報から生成する。

表示文言の例:

```markdown
要件定義では目的・背景の共有が重要です。
[ユーザのための要件定義ガイド PDF p.42-45]
```

## 回答内HTMLのサニタイズ候補

HTMLサニタイズで許可する要素の候補:

- `table`
- `thead`
- `tbody`
- `tr`
- `th`
- `td`
- `p`
- `ul`
- `ol`
- `li`
- `strong`
- `em`
- `code`
- `pre`
- `a`

禁止する要素・属性の候補:

- `script`
- `iframe`
- `object`
- `embed`
- イベント属性全般
- `javascript:` URL
- 外部通信や任意コード実行につながる属性
