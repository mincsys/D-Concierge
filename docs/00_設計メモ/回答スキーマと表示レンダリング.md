# 回答スキーマと表示レンダリング

## 目的

本メモは、Codex出力スキーマそのものの検討メモ、回答本文サンプル、MVP外の参照元種別の検討材料を整理する。

## 回答JSONスキーマの考え方

出力スキーマは、生成用codex execの最終出力契約であり、バックエンドの固定検証でも同じスキーマを使う。

スキーマ検討時の前提:

- 回答本文はMarkdownとして扱える文字列である。
- 回答は複数のブロックに分割できる。
- 各回答ブロックは、0件以上の参照元を持つ。
- 参照元は `source_type` で種類を示し、スキーマ側の `const` で許可値を固定する。
- 参照元の具体的な位置情報は `locator` に格納する。
- `locator` の形は、`codex.output_schema` が指すスキーマで定義する。
- 画像やHTMLなどの回答内データは、回答本文またはCodex成果物として扱う。

## 参照元JSONの例

PDFページを参照元とする例:

```json
{
  "source_type": "pdf",
  "locator": {
    "path": "raw/pdf/ユーザのための要件定義ガイド 第2版.pdf",
    "start_page": 42,
    "end_page": 45
  }
}
```

コード行範囲を参照元とする例:

```json
{
  "source_type": "code",
  "locator": {
    "path": "src/auth/session.ts",
    "start_line": 120,
    "end_line": 168
  }
}
```

PDF検索アプリを構成する場合の設定例では、`source_type` は `pdf` のみを対応必須とする。コード調査アプリなど別用途のアプリでは、`codex.output_schema` を差し替えて `source_type` と `locator` の形を変える。

## 回答本文の例

回答本文の例:

```markdown
要件定義を成功させるポイントは、以下の5点です。

1. 目的と背景を共有する
2. 利用者視点で要求を具体化する
3. 優先順位とスコープを明確にする
4. 検証可能な形で要求を定義する
5. 継続的な見直しと合意形成を行う

参照元は各項目のリンクから確認できます。
```

回答本文にMermaidを含める場合の例:

````markdown
```mermaid
flowchart LR
  A["ユーザ指示"] --> B["検索"]
  B --> C["分析"]
  C --> D["回答"]
```
````

回答本文にHTMLを含める場合の例:

```html
<table>
  <thead>
    <tr><th>項目</th><th>説明</th></tr>
  </thead>
  <tbody>
    <tr><td>目的共有</td><td>関係者間の認識を合わせる</td></tr>
  </tbody>
</table>
```
