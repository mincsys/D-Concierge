# 技術構成と実行方式

## 目的

本メモは、D-Conciergeの内部設計で参照する技術選定理由、開発時の起動例、配布時の実装メモを整理する。

外部設計に定義済みの構成、API、設定項目、画面仕様は本メモでは扱わない。

## 技術選定メモ

Next.jsは採用しない。D-Conciergeは外部公開Webサイトではなく、社内や小さな組織で使う業務支援ツールであるため、SSRや外部公開向けのフルスタック機能よりも、バックエンド側へプロセス制御と運用責務を集約する単純さを優先する。

画面はSPAとして実装し、開発時は画面開発サーバとバックエンドを分けて起動できるようにする。運用時はビルド済みSPAをバックエンドから配信する構成に寄せる。

## 開発時の起動例

開発時にDockerの利用は必須ではない。

バックエンドを直接起動する例:

```bash
uv run fastapi dev src/backend/main.py
```

フロントエンドを開発サーバで起動する例:

```bash
npm run dev
```

PostgreSQLだけDocker Composeで起動する場合の例:

```bash
docker compose up db
```

この分け方により、画面開発ではViteの高速な開発サーバを使い、バックエンド開発では例外やログを直接確認できる。

## 社内配布時の実装メモ

社内配布では、画面ビルド成果物をバックエンドの静的配信対象へ配置する。

起動例:

```bash
docker compose up -d
```

Docker Composeは、社内サーバへ配布するときに環境差を減らすための手段である。開発者の端末で常にDockerを使うことを要求するものではない。

## PDF検索アプリの資産配置例

PDF検索アプリを構成する場合の設定例では、次の既存資産を利用する。

- `codex/.codex/AGENTS.md`
- `codex/.codex/skills/custom/doc-html-finder/SKILL.md`
- `codex/.codex/skills/custom/doc-html-keyword-search/SKILL.md`
- `codex/.codex_validator/AGENTS.md`
- `codex/readonly/raw/pdf`
- `codex/readonly/raw/meta`
- `codex/readonly/html`
- `codex/sessions`
- `codex/sessions_validator`
- `codex/saved_artifacts`
- `codex/output_json_schema/pdf-reference-schema.json`
- `codex/output_json_schema/validator_schema.json`

この配置はPDF検索アプリを構成する場合の例であり、汎用チャット基盤は用途別の `AGENTS.md`、Skills、出力スキーマ、データソース配置により振る舞いを変える。
