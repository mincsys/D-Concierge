# 総合テスト方針

## 1. 文書の目的

本書は、D-Concierge MVPの総合テストにおける対象範囲、総合テスト分割単位、テスト観点、Playwright利用方針、手動テスト実施方針、実施環境、成果物・証跡および完了条件を定義することを目的とする。

本書は総合テストの方針を定義する文書であり、個別のテストケースとケースごとの実施結果は `テスト仕様・結果/チャット実行テスト.md`、`テスト仕様・結果/キャンセルテスト.md`、`テスト仕様・結果/履歴再表示テスト.md` に記載する。

## 2. 前提

- 本書は、要件定義書、外部設計書、内部設計書、静的テスト方針、単体テスト方針、結合テスト方針およびユーザ合意事項を前提とする。
- 総合テストは、利用者がWeb画面から業務を完了できることを確認する。
- 総合テストは手動テストを基本とし、ChromeのPlaywrightで検査できる画面操作、表示確認、画面遷移、スクリーンショット取得はPlaywrightで実施する。
- 実Codex CLIを使用し、回答生成、回答検証、参照元表示、Codex成果物表示までをMVP全体の業務シナリオとして確認する。
- 画面から観測できる異常系は、検証用設定、API失敗状態、SSE切断、短いタイムアウト、参照元データ欠損、Codex成果物欠損を用いて確認する。
- 利用者はDBに事前登録された共有1ユーザでログイン済みとして扱う。
- 静的テスト、単体テスト、結合テストで確認するツール解析、クラス・モジュール単位の分岐、API単位の契約、Fake/Stubによる内部連携は本書の確認前提とする。
- 高負荷、多数同時利用、長時間稼働、本番相当インフラ、ブラウザ互換性、セキュリティ侵入検査、Codex回答品質の網羅、全PDFパターンの網羅は、総合テストの合格判定には含めない。

## 3. 総合テスト対象範囲

| 対象区分 | 対象範囲 | 確認目的 |
| --- | --- | --- |
| Web画面 | 開始画面、チャット画面、参照元ビューア | 利用者がユーザ指示送信、回答確認、参照元確認、キャンセル、履歴再表示を画面から完了できることを確認する |
| バックエンドAPI | アプリ設定取得、新規チャット開始、継続指示、履歴一覧取得、履歴詳細取得、キャンセル要求、参照元データ取得、Codex成果物取得 | 画面操作を起点としてREST APIの結果が画面表示と業務状態へ反映されることを確認する |
| SSE | 実行状態、中間メッセージ、最終回答、エラー、キャンセル結果の配信 | 回答生成中の状態変化が利用者に分かる形で表示され、終端状態が画面へ反映されることを確認する |
| PostgreSQL | チャット、チャット実行処理、ユーザ指示、中間メッセージ、回答、参照元、Codex成果物情報 | 画面操作後に保存済み内容を履歴から再表示できることを確認する |
| 実Codex CLI | 生成用codex exec、検証用codex exec | 実Codexを含む回答生成・回答検証の一連の流れが業務上利用できることを確認する |
| 共有データソース | 検証用参照PDF | 回答内の参照元リンクからPDF参照元ビューアで根拠を確認できることを確認する |
| Codex成果物保存領域 | 検証済み回答が参照する保存済みCodex成果物 | 回答Markdown内の `/api/artifacts/{artifact_id}` URLから成果物を表示できることを確認する |
| 利用者向け異常表示 | 入力不正、受付失敗、SSE切断、生成失敗、検証失敗、タイムアウト、参照元表示失敗、Codex成果物表示失敗 | 内部パス、秘密情報、スタックトレースを表示せず、利用者が次の操作を判断できることを確認する |

## 4. 総合テスト分割単位

総合テストは、外部設計の業務一覧で定義された業務シナリオ単位で分割する。

| 分割単位 | 作成するテスト仕様・結果ファイル | 対象 |
| --- | --- | --- |
| BIZ-01 チャット実行処理 | `テスト仕様・結果/チャット実行テスト.md` | 開始画面またはチャット画面でユーザ指示を送信し、検証済み回答、エラー、キャンセル済み、タイムアウトのいずれかに到達する業務 |
| BIZ-02 キャンセル | `テスト仕様・結果/キャンセルテスト.md` | 回答生成中または検証中のチャット実行処理を利用者がキャンセルする業務 |
| BIZ-03 履歴再表示 | `テスト仕様・結果/履歴再表示テスト.md` | チャット履歴一覧から過去チャットを選択し、保存済み内容を再表示する業務 |

## 5. 総合テスト観点

総合テストでは、少なくとも以下の観点を確認する。

| 観点 | 確認内容 |
| --- | --- |
| 業務シナリオ | 利用者が開始画面、チャット画面、参照元ビューアを通じて、チャット実行、キャンセル、履歴再表示を完了できること |
| 画面遷移 | 開始画面、チャット画面、参照元ビューア、履歴選択、新しいチャット操作、ビューアを閉じる操作が設計どおりであること |
| 入力検証 | 空白だけのユーザ指示、未完了runがあるチャットへの追加指示、受付失敗時の表示が設計どおりであること |
| 実行状態表示 | 受付、実行中、検証中、キャンセル要求中、キャンセル済み、完了、エラー、タイムアウトが利用者に分かる形で表示されること |
| SSE表示 | 接続直後の状態通知、状態変化、中間メッセージ、最終回答、エラー、キャンセル結果、意図しない切断が画面へ反映されること |
| 回答表示 | 検証済み回答だけがMarkdown、表、コードブロック、画像、Mermaid、サニタイズ済みHTMLとして表示されること |
| 参照元表示 | 回答内の参照元リンクから参照元ビューアを開き、PDF名、ページ番号、ページ範囲、PDF本体を確認できること |
| Codex成果物表示 | 回答Markdown内の `/api/artifacts/{artifact_id}` URLにより、保存済みCodex成果物を回答内要素として表示できること |
| キャンセル | 受付、実行中、検証中のキャンセル要求、キャンセル中表示、キャンセル済み表示、未検証回答の非表示が設計どおりであること |
| 履歴再表示 | 完了、キャンセル済み、エラー、タイムアウト、継続中のチャット実行処理を履歴から再表示できること |
| 異常系 | API失敗、SSE接続失敗、SSE途中切断、生成失敗、検証失敗上限、タイムアウト、参照元取得失敗、Codex成果物取得失敗が設計どおり表示されること |
| メッセージ表示 | 利用者向けメッセージがエラーメッセージ設計に沿い、内部情報を含まないこと |
| ブラウザ表示 | Chromeで主要画面のレイアウト、入力、ボタン、リンク、スクロール、折りたたみ、ビューア操作が利用可能であること |
| 手動妥当性 | 実Codexの回答がユーザ指示に対応しており、参照元が回答根拠として自然に確認できること |

## 6. Playwright 利用方針

| 区分 | 方針 |
| --- | --- |
| Playwrightで検査する範囲 | Chromeで実行できる画面操作、フォーム入力、ボタン操作、画面遷移、履歴選択、SSE状態表示、中間メッセージ表示、回答表示、参照元ビューア表示、Codex成果物表示、利用者向けエラー表示、スクリーンショット取得をPlaywrightで実施する |
| Playwrightで発生させる異常系 | API失敗、SSE接続失敗、SSE途中切断、短いタイムアウト、参照元データ欠損、Codex成果物欠損、キャンセル連打、画面遷移によるSSE購読解除を、検証用環境または検証用状態で発生させる |
| Playwrightで検査しない範囲 | 実Codexの回答内容が業務的に妥当か、参照元が回答根拠として自然か、文言に利用者視点の違和感がないかの判断は手動確認で実施する |
| Playwrightで検査できない範囲 | 実Codex内部の推論過程、多数同時利用、長時間稼働、本番ネットワーク差分、Chrome以外のブラウザ差分、脆弱性診断、全PDF種別の網羅はPlaywrightによる総合テストの検査範囲に含めない |
| 証跡 | 必要最小限のスクリーンショットのみを保存する。動画、Playwright trace、Playwright詳細レポート、詳細実行ログは保存しない |

## 7. 手動テスト実施方針

| 項目 | 方針 |
| --- | --- |
| 操作 | Playwrightで自動操作できる範囲はPlaywrightから操作し、Playwrightで安定実施できない確認だけを人が操作する |
| 目視確認 | 実Codexの回答内容、参照元の根拠妥当性、画面文言、PDF表示の自然さ、Markdown/HTML/Mermaid表示の読みやすさを確認する |
| 証跡取得 | 合格判定に必要な画面だけをスクリーンショットとして保存する |
| 結果記入 | 各ケースの実施結果、実施日、実施者、証跡、備考を `テスト仕様・結果/*.md` に記入する |
| 不具合記録 | 不合格または保留のケースは備考へ不具合識別子または確認事項を記録し、修正後に同じケースを再実施する |

## 8. テスト環境方針

| 項目 | 方針 |
| --- | --- |
| 実施環境 | ローカル検証環境を使用する |
| データベース | Dockerコンテナで起動するPostgreSQLを使用する |
| バックエンド | ローカル検証環境でバックエンドアプリケーションを起動する |
| フロントエンド | ローカル検証環境でフロントエンドアプリケーションを起動する |
| Codex | 実Codex CLIを使用し、生成用codex execと検証用codex execを実行できる状態にする |
| Playwright実行環境 | ローカル検証環境からChromeブラウザを起動する |
| 利用ブラウザ | Chromeを使用する |
| 共有データソース | 検証用の参照PDFを使用し、利用者が画面からデータソースを追加または切り替えない |
| 異常状態の作成 | 検証用設定、ネットワーク遮断、短いタイムアウト、参照元データ欠損、Codex成果物欠損、API失敗状態を利用し、画面から観測できる異常系を再現する |
| 環境差分の扱い | Windows/Linux差分、本番プロキシ、証明書、バックアップ、リストア、監視、ログローテーションは総合テストの合格判定に含めない |

## 9. 成果物・証跡方針

| 項目 | 方針 |
| --- | --- |
| 文書成果物 | `docs/04_テスト/04_総合テスト/総合テスト方針.md`、`docs/04_テスト/04_総合テスト/テスト仕様・結果/チャット実行テスト.md`、`docs/04_テスト/04_総合テスト/テスト仕様・結果/キャンセルテスト.md`、`docs/04_テスト/04_総合テスト/テスト仕様・結果/履歴再表示テスト.md` を正式成果物として管理する |
| スクリーンショット | 必要最小限のスクリーンショットを `docs/04_テスト/04_総合テスト/evidence/` 配下へ保存する |
| スクリーンショット命名 | ケースIDを含むファイル名とし、どのケースの証跡かをテスト仕様・結果ファイルから追跡できるようにする |
| 動画 | 保存しない |
| Playwright trace | 保存しない |
| Playwright詳細レポート | 保存しない |
| 詳細実行ログ | 保存しない |
| 不具合票 | 別途不具合管理を行う場合は、テスト仕様・結果ファイルの備考に不具合識別子を記録する |
| 再テスト結果 | 修正後に同じケースを再実施し、テスト仕様・結果ファイルの実施結果、実施日、証跡、備考を更新する |
| 証跡の保存期間 | 総合テスト成果物と同じ管理期間で保持する |
| 証跡の更新方針 | 再テスト時は、合格判定に必要な最新スクリーンショットへ差し替える |

## 10. 完了条件

- `テスト仕様・結果/チャット実行テスト.md`、`テスト仕様・結果/キャンセルテスト.md`、`テスト仕様・結果/履歴再表示テスト.md` の全テストケースが実施済みである。
- 全テストケースの実施結果が合格である。
- 未解決の重大または中程度の不具合が残っていない。
- 不合格または保留になったケースは、原因対応後に同じケースで再テストされている。
- 必要最小限のスクリーンショット証跡が `docs/04_テスト/04_総合テスト/evidence/` 配下に保存され、テスト仕様・結果ファイルから参照できる。
- 利用者向け画面に内部パス、秘密情報、スタックトレースが表示されないことを異常系ケースで確認済みである。
- 総合テストで確認しない高負荷、長時間稼働、本番相当インフラ、Chrome以外のブラウザ互換性、脆弱性診断、Codex回答品質の網羅は、総合テスト完了判定の未実施扱いにしない。
