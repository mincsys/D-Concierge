# 単体テスト方針

## 1. 文書の目的

本書は、D-Conciergeの単体テストにおける対象範囲、テスト対象単位、対象外範囲、テスト観点、網羅基準、実施環境、実施手順および成果物・証跡の扱いを定義することを目的とする。

本書は単体テストの方針を定義する文書であり、個別テストケース一覧、テスト結果一覧、結合テスト方針は扱わない。

## 2. 前提

- 本書は、要件定義書、外部設計書、内部設計書、静的テスト方針、開発標準およびユーザ合意事項を前提とする。
- 単体テスト対象は、`src/backend/` と `src/frontend/src/` 配下の本体コードとする。
- `src/frontend/backend_mock/` はフロントエンド単体確認用の開発支援資産であり、単体テスト対象外とする。
- 単体テストコードは、バックエンドは `src/backend/tests/unit/`、フロントエンドは `src/frontend/tests/unit/` へ配置する。
- 単体テストは自動テストコードで実行できる確認だけを対象とし、手動操作、目視確認、Playwrightによるブラウザ確認は対象外とする。
- 実装変更時は、TDDのRed、Green、Refactorの順序で単体テストコードを作成および実行する。
- テスト設定ファイルと依存lock fileは、後続の実装または開発標準整備の工程で本方針に従って作成する。

## 3. テスト対象範囲

| 区分 | 対象範囲 |
| --- | --- |
| バックエンド本体コード | `src/backend/` 配下のPythonソースコード |
| フロントエンド本体コード | `src/frontend/src/` 配下のTypeScript、TSX |
| バックエンド単体テストコード | `src/backend/tests/unit/` 配下のPythonテストコード |
| フロントエンド単体テストコード | `src/frontend/tests/unit/` 配下のTypeScript、TSXテストコード |
| テスト支援コード | `src/backend/tests/support/`、`src/frontend/tests/support/` 配下のFake、fixture、render helper、固定ID、固定時刻 |

テスト支援コードは単体テストを成立させるための補助資産であり、分岐カバレッジ目標の対象には含めない。テスト支援コードに振る舞いを持たせる場合は、本体コードと同じ水準で単純さと可読性を保つ。

## 4. テスト対象単位

| 区分 | テスト対象単位 | 補足 |
| --- | --- | --- |
| バックエンド アプリケーション層 | クラスの公開メソッド、公開関数 | PortをFake、Stub、Mockへ差し替え、ユースケースの分岐、戻り値、状態更新依頼、例外処理を確認する |
| バックエンド ドメイン層 | クラスの公開メソッド、公開関数 | 副作用を伴わない業務判断、値の不変条件、状態遷移、境界値を確認する |
| バックエンド インフラストラクチャ層 | クラスの公開メソッド、公開関数 | 外部副作用の実体は直接実行せず、入力検証、依存先への依頼内容、失敗時変換を確認する |
| フロントエンド API境界 | モジュールの公開関数 | `/api/...` 通信境界、SSEイベント処理、API応答から画面モデルへの変換を確認する |
| フロントエンド コンポーネント | Reactコンポーネント | props、条件表示、入力イベント、無効状態、エラー表示、公開イベントを確認する |
| フロントエンド 純粋ロジック | モジュールの公開関数 | 表示ラベル生成、回答表示演出、値変換を確認する |

詳細設計の有無ではなく、分岐、変換、検証、状態変更、イベント処理、例外処理を持つクラスまたはモジュールを単体テスト対象とする。型定義のみ、静的アセット、単純な定義、薄いラッパーは単体テスト対象外またはカバレッジ計測対象外とする。

## 5. テスト対象外範囲

| 対象外範囲 | 対象外理由 | 代替確認方法 |
| --- | --- | --- |
| `src/frontend/backend_mock/` | フロントエンド単体確認用の開発支援資産であり、本体コードではないため | フロントエンド単体確認、静的テスト |
| `codex/output_json_schema/` | JSON Schema契約資産であり、クラスまたはモジュールの単体ではないため | 回答スキーマ検証、契約確認 |
| 実DB接続、実トランザクション | 単体テストでは外部副作用の実体を扱わないため | バックエンド結合テスト |
| 実ファイル入出力、実PDF読込、実成果物コピー | ファイルシステム依存により単体テストの再現性が下がるため | バックエンド結合テスト、成果物配信確認 |
| 実codex execプロセス起動 | 外部プロセス、実行時間、環境差分を伴うため | Codex実行結合テスト |
| HTTP/SSEの実通信 | ルーティング、ASGI、EventSourceを含むため | API単位の結合テスト |
| ブラウザ実体、レイアウト目視、スクリーンショット確認 | 単体テストコードだけで完結しないため | 総合テストまたは画面確認 |
| PDF.js、Mermaid、React外部ライブラリそのものの描画品質 | ライブラリ提供機能の実装確認になるため | 結合テスト、総合テスト |
| 型定義のみのファイル | 実行時の振る舞いを持たないため | 静的テスト |
| `src/frontend/src/components/ui/` のshadcn/ui由来の薄いラッパー | アプリ固有の分岐や状態をほとんど持たないため | 静的テスト、利用側コンポーネントの単体テスト |

単体テスト対象外の副作用を持つ依存先は、Fake、Stub、Mock、fixture、`monkeypatch`、`vi.fn`、`vi.stubGlobal` により差し替える。差し替え後の単体テストでは、依頼内容、呼出有無、例外変換、後続処理抑止を確認する。

## 6. テスト観点

単体テストでは、少なくとも以下の観点を確認する。

| 観点 | 確認内容 |
| --- | --- |
| 正常系 | 想定入力に対して期待する戻り値、状態、出力、表示が得られること |
| 異常系 | 不正入力、前提不成立、依存先失敗に対して、定義された例外、エラー結果、利用者向けメッセージになること |
| 境界値 | 空値、最小値、最大値、閾値の前後、ページ範囲、再生成上限、タイムアウト境界が設計どおりに扱われること |
| 分岐 | 条件分岐ごとの処理経路が期待どおりに実行されること |
| 条件組合せ | 複数条件で結果が変わる場合に、代表的な組合せで結果が変化すること |
| 例外処理 | 例外発生時の戻り値、状態、ログ依頼、後続処理抑止が設計どおりであること |
| 戻り値 | 値、型、構造、順序、欠落時の扱い、snake_caseからcamelCaseへの変換が設計どおりであること |
| 事前条件 | クラス設計またはモジュール設計の公開メソッド・公開関数に定義された開始条件を満たす場合に呼び出せること |
| 事後条件 | 正常終了または異常終了後に保証すべき戻り値、状態、出力、副作用依頼が満たされること |
| 不変条件 | 処理前後や分岐をまたいで維持すべき業務制約、内部状態、データ整合性が保たれること |

### 6.1. バックエンド固有観点

| 区分 | 確認内容 |
| --- | --- |
| ユースケース | Repository、CodexRunner、ArtifactStore、ReferenceStore、RunEventPublisher、TraceLoggerを差し替え、状態遷移、保存依頼、SSE配信依頼、トレースログ依頼を確認する |
| 実行制御 | `受付`、`実行中`、`検証中`、`キャンセル要求中`、終端状態の遷移可否、再生成時の `検証中` から `実行中` への復帰、全体deadline超過後の起動抑止を確認する |
| 回答検証 | JSON構文、スキーマ、空回答、`source_type=pdf`、ページ範囲、許可パス、危険HTML、`start_page` / `end_page` から `page_start` / `page_end` への正規化を確認する |
| JSONL解析 | `thread.started`、`item.completed.agent_message`、`turn.completed`、`turn.failed`、`error`、未知イベント、解析不能行、`payload.kind="progress"` の中間表示、`payload.kind="final"` の中間表示抑止を確認する |
| パス安全性 | 許可範囲内パス、許可範囲外パス、ディレクトリトラバーサル、保存済み成果物URL置換の判定を確認する |
| Repository実装 | DB実体は使わず、状態条件付き更新の入力、未完了run競合時の例外変換、Codex側resume用ID保存依頼を確認する |
| トレースログ | trace_id、run ID、chat ID、stage、例外種別、runner種別、再生成回数、検証失敗理由、キャンセル、タイムアウトのマスク済み出力依頼を確認する |

### 6.2. フロントエンド固有観点

| 区分 | 確認内容 |
| --- | --- |
| API境界 | `/api/...` だけへ通信すること、`src/frontend/backend_mock/` を直接importしないこと、API応答を画面モデルへ変換すること |
| SSE処理 | `state`、`message`、`answer`、`error`、`canceled` の順序処理、終端イベント後のEventSource終了、旧ストリーム化後の通知抑止を確認する |
| チャット画面 | 開始画面とチャットスレッドの切替、実行状態反映、中間メッセージ反映、回答反映、キャンセル中run ID解除、対象runだけの置換を確認する |
| 入力部品 | 空白だけの指示抑止、送信可否、キャンセル可否、textareaの状態変化、公開イベント呼出を確認する |
| Markdown表示 | 許可HTML、禁止HTML、イベント属性、`javascript:` URL、同一オリジンかつ `/api/artifacts/` 配下画像だけの表示を確認する |
| Mermaid表示 | Mermaidコードブロックの委譲、通常コードブロックとの切替、描画失敗時の表示、拡大表示導線を確認する |
| 参照元表示 | PDF参照元リンク、参照元ダイアログ開閉、ページ範囲ラベル、選択中参照元がない場合の非表示を確認する |
| 純粋表示ロジック | 単一ページと範囲ページのラベル生成、回答チャンク化、入力オブジェクト非破壊を確認する |

## 7. 網羅基準

| 項目 | 基準 |
| --- | --- |
| 採用指標 | 分岐カバレッジ |
| 目標値 | バックエンド本体コード、フロントエンド本体コードそれぞれ95%以上 |
| 判定単位 | バックエンド全体、フロントエンド全体 |
| 判定対象外指標 | 行カバレッジ、ステートメントカバレッジ、関数カバレッジ |
| 目標未達時 | 不足している分岐のテストコードを追加し、同じコマンドで再実行する |

分岐カバレッジ95%未満の場合、単体テストは不合格とする。証跡ファイルには分岐カバレッジだけを出力し、行カバレッジ、ステートメントカバレッジ、関数カバレッジは出力しない。

カバレッジ計測対象外にできる範囲は、型定義のみ、静的アセット、CSS、薄いエントリポイント、shadcn/ui由来の薄いラッパーに限定する。計測除外を行う場合は、設定ファイルに対象パスと除外理由を確認できる形で最小範囲だけ記載する。

## 8. テスト環境方針

| 項目 | バックエンド | フロントエンド |
| --- | --- | --- |
| 実行環境 | ローカル開発環境 | ローカル開発環境 |
| テストフレームワーク | pytest | Vitest |
| コンポーネントテスト | 対象外 | React Testing Library |
| DOM環境 | 対象外 | jsdom |
| Mock/Fake | pytest fixture、monkeypatch、unittest.mock、必要なFake実装 | `vi.fn`、`vi.stubGlobal`、React Testing Libraryのrender helper |
| カバレッジ取得 | pytest-cov | Vitest V8 coverage |
| レポート形式 | 分岐カバレッジだけを記載したテキスト | 分岐カバレッジだけを記載したテキスト |
| 依存管理 | `uv`、`uv.lock` | `npm`、`src/frontend/package-lock.json` |

Pythonの単体テストは、`UV_CACHE_DIR=/tmp/uv-cache` を指定して `uv` 経由で実行する。フロントエンドの単体テストは、`src/frontend/package.json` のコマンド定義で管理する。

CIでの単体テスト実行は必須としない。単体テストの証跡は、ローカル実行で更新する `docs/04_テスト/02_単体テスト/evidence/` 配下のテキストファイルを正とする。

## 9. テストコード記述方針

- テストコードは、対象コードの責務とテスト対象単位に対応する `src/backend/tests/unit/` または `src/frontend/tests/unit/` 配下へ配置する。
- バックエンドは `test_*.py` を使用し、対象ソースのディレクトリ構造に対応させる。
- フロントエンドは `*.test.ts` または `*.test.tsx` を使用し、対象ソースのディレクトリ構造に対応させる。
- 1つのテストでは、原則として1つの観点と確認内容を明確にする。
- テスト名は、対象責務、入力条件、期待結果が分かる表現にする。
- 外部API、DB、ファイル、Codex exec、時刻、ID発番、EventSource、clipboardは、単体テストでは原則として差し替える。
- テストデータは、確認内容が分かる最小限のデータとする。
- `Any`、広すぎる `object`、安易な `cast(...)`、TypeScriptの `any` に依存するテストデータは作成しない。
- `src/frontend/backend_mock/` の固定データ、API代替コード、参照元、Codex成果物を単体テストから直接importしない。

すべての単体テストコードには、Pythonではdocstring、TypeScript/TSXでは対象テストの直前コメントまたは `test` / `it` の説明文字列として、以下の2項目を記載する。

```text
観点：確認対象の責務や振る舞いが分かる文章を記載する
確認：検証する結果を具体的に記載する
```

## 10. テスト実施手順

1. 実装対象の責務、事前条件、事後条件、不変条件を内部設計から確認する。
2. 実装前に失敗する単体テストを作成し、Redを確認する。
3. 実装を追加または修正し、対象単体テストを成功させてGreenを確認する。
4. 重複、命名、型、依存境界を整理し、Refactor後に同じ単体テストを再実行する。
5. 対象領域の分岐カバレッジを取得する。
6. 分岐カバレッジ95%未満の場合、不足している分岐の単体テストを追加して再実行する。
7. 合格後、`docs/04_テスト/02_単体テスト/evidence/` 配下の分岐カバレッジテキストを更新する。

標準の単体テスト実行コマンドは以下とする。

```bash
UV_CACHE_DIR=/tmp/uv-cache uv run pytest src/backend/tests/unit
```

```bash
cd src/frontend
npm run test:unit
```

`npm run test:unit` は、Vitest、React Testing Library、jsdomを使って `src/frontend/tests/unit/` を実行するコマンドとして定義する。

分岐カバレッジ証跡更新時は、pytest-covまたはVitest V8 coverageの機械可読出力から分岐指標だけを抽出するコマンドを、テスト設定またはpackage managerのコマンド定義で管理する。証跡更新コマンドは、行カバレッジ、ステートメントカバレッジ、関数カバレッジを標準出力または証跡ファイルへ出力しない。

## 11. 成果物・証跡方針

| 区分 | 扱い |
| --- | --- |
| 単体テスト方針書 | `docs/04_テスト/02_単体テスト/単体テスト方針.md` を正式成果物として管理する |
| バックエンド単体テストコード | `src/backend/tests/unit/` 配下を正式成果物として管理する |
| フロントエンド単体テストコード | `src/frontend/tests/unit/` 配下を正式成果物として管理する |
| テスト支援コード | `src/backend/tests/support/`、`src/frontend/tests/support/` 配下を管理対象とする |
| テスト設定ファイル | `pyproject.toml`、`src/frontend/vitest.config.*`、`src/frontend/package.json` を管理対象とする |
| 依存lock file | `uv.lock`、`src/frontend/package-lock.json` を管理対象とする |
| バックエンド分岐カバレッジ証跡 | `docs/04_テスト/02_単体テスト/evidence/backend-unit-coverage.txt` を最新結果で上書きする |
| フロントエンド分岐カバレッジ証跡 | `docs/04_テスト/02_単体テスト/evidence/frontend-unit-coverage.txt` を最新結果で上書きする |
| テスト実行ログ | 正式成果物として保存しない |
| 行カバレッジレポート | 作成しない |
| CI実行結果 | 単体テストの必須証跡としない |

分岐カバレッジ証跡ファイルには、対象、実行日時、実行コマンド、分岐カバレッジ、総分岐数、通過分岐数、判定結果を記載する。再テスト時は同じファイル名で最新結果へ上書きする。

証跡ファイルには、以下の項目だけを `key=value` 形式で記載する。

| 項目 | 記載内容 |
| --- | --- |
| `target` | `backend` または `frontend` |
| `executed_at` | テスト実行日時 |
| `command` | 実行した単体テストコマンド |
| `branch_coverage` | 分岐カバレッジの割合 |
| `covered_branches` | 通過した分岐数 |
| `total_branches` | 計測対象の総分岐数 |
| `result` | `pass` または `fail` |

## 12. 留意事項

- 本書は単体テスト方針を定義する文書であり、個別テストケースの粒度、入力値、期待値はテストコードで管理する。
- 単体テストで確認できない実DB接続、実HTTP/SSE通信、実codex exec、実ブラウザ表示は、結合テストまたは総合テストで扱う。
- 外部ライブラリそのものの正しさは単体テスト対象にしない。アプリケーション側がライブラリへ渡す入力、受け取る結果、失敗時の扱いを確認する。
- 分岐カバレッジ95%を満たすために、価値の低いテストを量産しない。設計上の分岐、異常系、境界値、不変条件を優先して確認する。
- テストコードが実装詳細に密結合し、リファクタリングを阻害する場合は、公開責務、公開メソッド、公開関数、公開イベントの観点へ戻してテストを整理する。
