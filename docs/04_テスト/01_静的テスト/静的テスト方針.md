# 静的テスト方針

## 1. 文書の目的

本書は、D-Conciergeの静的テストにおける対象範囲、対象外範囲、使用ツール、検出観点、設定方針、判定基準、実施手順および成果物・管理対象を定義することを目的とする。

本書で扱う静的テストは、ツールによる `lint`、`format check`、`type check` に限定する。手動レビュー、ウォークスルー、目視確認は本書の対象外とする。

## 2. 前提

- 本書は、要件定義書、外部設計書、内部設計書、開発標準およびユーザ合意事項を前提とする。
- 静的テスト対象は、`src/` 配下のソースコード、テストコード、`src/frontend/backend_mock/` とする。
- `codex/output_json_schema/` は静的テスト対象外とする。
- 静的テスト種別は `lint`、`format check`、`type check` の 3 種別とする。
- security scan、dependency scan、secret scan、license check は本書の対象外とする。
- 静的テスト設定ファイルと依存lock fileは、リポジトリ管理対象とする。
- 静的テスト方針書以外の実行ログ、解析レポート、設定変更理由、抑止理由の別文書は作成しない。
- Python と Node.js の具体バージョンは、後続の環境構築書と依存lock fileに従う。

## 3. 静的テスト対象範囲

| 区分 | 対象範囲 |
| --- | --- |
| バックエンドソースコード | `src/backend/` 配下のPythonソースコード |
| バックエンドテストコード | `src/backend/tests/` 配下のPythonテストコード |
| フロントエンドソースコード | `src/frontend/src/` 配下のTypeScript、TSX、CSS、静的アセット参照コード |
| フロントエンドテストコード | `src/frontend/tests/` 配下のTypeScript、TSXテストコード |
| フロントエンド単体確認用モック | `src/frontend/backend_mock/` 配下のAPI代替コード、固定データ、参照元、Codex成果物 |
| フロントエンド実行設定 | `src/frontend/vite.config.ts`、`src/frontend/tsconfig.json`、`src/frontend/package.json` |
| 静的テスト設定ファイル | `pyproject.toml`、`src/frontend/eslint.config.*`、`src/frontend/prettier.config.*`、`src/frontend/.prettierrc*` |
| 依存lock file | `uv.lock`、`src/frontend/package-lock.json` |

## 4. 静的テスト対象外範囲

| 対象外範囲 | 対象外理由 | 代替確認方法 |
| --- | --- | --- |
| `codex/output_json_schema/` | JSON Schema契約資産であり、`lint`、`format check`、`type check` の主対象ではないため | 回答スキーマ検証、契約テスト、JSON Schemaメタスキーマ検証で確認する |
| `codex/.codex/`、`codex/.codex_validator/` | codex exec実行環境資産であり、アプリケーションソースコードではないため | codex exec連携確認で確認する |
| `src/frontend/node_modules/` | 外部依存ライブラリであり、プロジェクト側で静的解析しないため | 依存解決時のlock fileで管理する |
| `src/frontend/dist/` | ビルド成果物であり、ソースコードから再生成されるため | ビルド処理で再生成する |
| `src/frontend/tsconfig.tsbuildinfo` | TypeScriptの増分ビルド生成物であり、ソースコードから再生成されるため | TypeScript実行時に再生成する |
| `.venv/` | Python仮想環境であり、プロジェクト側で静的解析しないため | `uv.lock` と環境構築手順で再現する |
| `.uv-cache/`、`/tmp/uv-cache` | uvのキャッシュ領域であり、プロジェクト側で静的解析しないため | 依存解決時に再生成する |
| `coverage/` | テストカバレッジ生成物であり、静的テスト入力ではないため | テスト実行時に再生成する |
| `.ruff_cache/`、`.mypy_cache/` | 静的解析ツールのキャッシュ領域であり、静的テスト入力ではないため | 静的テスト実行時に再生成する |
| 手動レビュー、ウォークスルー、目視確認 | ツール解析ではないため | 後続のレビュー工程、単体テスト、結合テスト、総合テストで扱う |

## 5. 静的テスト種別

| 種別 | 採用方針 | 目的 |
| --- | --- | --- |
| lint | 採用する | コーディング規約違反、危険な構文、未使用要素、import整理違反、非推奨API利用を検出する |
| format check | 採用する | フォーマット設定との差分、空白設定との差分、改行設定との差分、ファイル末尾改行設定との差分を検出する |
| type check | 採用する | 型不整合、未定義参照、型境界の破れ、明示的な `Any`、暗黙的な `any`、危険な `any` 由来操作を検出する |

## 6. 使用ツール方針

| 対象 | 種別 | 使用ツール | 設定ファイル | 標準実行コマンド |
| --- | --- | --- | --- | --- |
| バックエンド | lint | Ruff | `pyproject.toml` | `UV_CACHE_DIR=/tmp/uv-cache uv run ruff check src/backend` |
| バックエンド | format check | Ruff formatter | `pyproject.toml` | `UV_CACHE_DIR=/tmp/uv-cache uv run ruff format --check src/backend` |
| バックエンド | type check | mypy | `pyproject.toml` | `UV_CACHE_DIR=/tmp/uv-cache uv run mypy src/backend` |
| フロントエンド | lint | ESLint | `src/frontend/eslint.config.*` | `cd src/frontend && npm run lint` |
| フロントエンド | format check | Prettier | `src/frontend/prettier.config.*` または `src/frontend/.prettierrc*` | `cd src/frontend && npm run format:check` |
| フロントエンド | type check | TypeScript Compiler | `src/frontend/tsconfig.json` | `cd src/frontend && npm run typecheck` |

全体実行入口は、スクリプトファイルを増やさず、package managerのコマンド定義で管理する。CIでは同じ設定ファイルと同じ判定基準を用いて静的テストを実施する。

## 7. ツール別検出観点

| 対象 | ツール | 検出観点 |
| --- | --- | --- |
| バックエンド | Ruff | pycodestyle系違反、Pyflakes系違反、import整理違反、flake8-bugbear系違反、pyupgrade系違反、未使用import、未使用変数、危険な構文 |
| バックエンド | Ruff formatter | インデント、空白、改行、クォート、行長、ファイル末尾改行 |
| バックエンド | mypy | 型不整合、未定義参照、戻り値型不整合、引数型不整合、ジェネリック型指定不足、明示的な `Any`、冗長な `cast(...)` |
| フロントエンド | ESLint | React/TypeScript推奨ルール違反、未使用import、未使用変数、hooks利用違反、明示的な `any`、危険な `any` 由来の代入、呼び出し、メンバーアクセス、戻り値 |
| フロントエンド | Prettier | インデント、空白、改行、クォート、セミコロン、trailing comma、行長、ファイル末尾改行 |
| フロントエンド | TypeScript Compiler | 型不整合、未定義参照、null/undefined境界の破れ、モジュール解決失敗、暗黙的な `any` |

import並び順と未使用importは `lint` の判定対象とする。型境界の破れは `type check` の判定対象とし、ESLintの型情報利用ルールで補助する。

## 8. ツール設定方針

| 種別 | 設定分類 | 設定項目 | 採用方針 |
| --- | --- | --- | --- |
| lint | 必須設定 | Ruffの `E`、`F`、`I`、`B`、`UP` 系ルール | 採用する |
| lint | 必須設定 | ESLintのReact推奨ルール、TypeScript推奨ルール、hooks推奨ルール | 採用する |
| lint | 必須設定 | `@typescript-eslint/no-explicit-any` | `error` として採用する |
| lint | 必須設定 | `@typescript-eslint/no-unsafe-assignment`、`@typescript-eslint/no-unsafe-member-access`、`@typescript-eslint/no-unsafe-call`、`@typescript-eslint/no-unsafe-return` | `error` として採用する |
| format check | 必須設定 | Ruff formatter | 採用する |
| format check | 必須設定 | Prettier | 採用する |
| type check | 必須設定 | mypy `strict` 相当 | 採用する |
| type check | 必須設定 | mypy `disallow_any_explicit` | 採用する |
| type check | 必須設定 | mypy `disallow_any_generics` | 採用する |
| type check | 必須設定 | mypy `warn_redundant_casts` | 採用する |
| type check | 必須設定 | TypeScript `strict` | 採用する |
| type check | 必須設定 | TypeScript `noImplicitAny` | 採用する |
| 共通 | 必須設定 | 除外対象 | 生成物、依存ライブラリ、キャッシュ領域に限定する |
| 共通 | 推奨設定 | ツールバージョン | Python側は `uv.lock`、フロントエンド側は `src/frontend/package-lock.json` で固定する |
| 共通 | 採用しない設定 | ソースコード内の広範なignore、suppress、per-file ignore | 採用しない |

Pythonでは `Any` に逃がさず、`TypedDict`、dataclass、Protocol、具体的な型エイリアスを使う。広すぎる `object` と必要性を説明できない `cast(...)` は原則禁止とする。ただし、`object` の使用そのものと危険な `cast(...)` の一般判定は mypy だけでは完全検出できないため、コーディング規約、レビュー、後続のlint拡張で補完する。TypeScriptでは `any` に逃がさず、`unknown` と型ガードで境界を絞り込む。

## 9. 判定基準

| 項目 | 判定基準 |
| --- | --- |
| error | 1 件でも検出された場合は不合格とする |
| warning | 1 件でも検出された場合は不合格とする |
| 自動修正可能な指摘 | 1 件でも検出された場合は不合格とする |
| 既存指摘 | 暫定許容せず、不合格とする |
| 明示的な `Any` / `any` | 1 件でも検出された場合は不合格とする |
| 暗黙的な `any` | 1 件でも検出された場合は不合格とする |
| 危険な `any` 由来操作 | 1 件でも検出された場合は不合格とする |
| 設定ファイル不備 | 静的テストが実行できない場合は不合格とする |
| lock file不整合 | 依存定義とlock fileが一致しない場合は不合格とする |

静的テストで不合格となった場合は、対象箇所を修正し、同じコマンドで再テストする。指摘を残したまま完了扱いにしない。

## 10. 除外・抑止方針

| 項目 | 方針 |
| --- | --- |
| 除外対象 | 生成物、外部依存ライブラリ、キャッシュ領域、静的テスト種別の主対象ではない契約資産に限定する |
| ignore | 原則禁止とする |
| suppress | 原則禁止とする |
| per-file ignore | 原則禁止とする |
| 誤検知許容 | 原則禁止とする |
| 例外登録 | 必要な場合は設定ファイルに理由付きで最小範囲のみ登録する |
| ファイル内抑止コメント | 原則禁止とする |

例外登録は、ツール仕様上の誤検知、外部ライブラリ型定義の不足、生成物境界の明示が必要な場合に限る。例外登録を行う場合も、対象パス、対象ルール、理由、解消条件を設定ファイル上で確認できる状態にする。

## 11. 実行環境方針

| 環境 | 方針 |
| --- | --- |
| ローカル環境 | ファイルを作成または編集したときに、対象領域の静的テストを実施する |
| CI環境 | ローカル環境と同じ設定ファイル、同じlock file、同じ判定基準で静的テストを必須実施する |
| Python実行環境 | `uv` と `uv.lock` で依存関係を固定し、`UV_CACHE_DIR=/tmp/uv-cache` を指定して実行する |
| Node.js実行環境 | `npm` と `src/frontend/package-lock.json` で依存関係を固定する |
| バージョン方針 | Python と Node.js の具体バージョンは、後続の環境構築書と依存lock fileに従う |
| 設定ファイル | ローカル環境とCI環境で同じ設定ファイルを参照する |

CI定義ファイルは本書の成果物ではない。CI定義を作成する工程では、本書の標準実行コマンドと判定基準をそのまま反映する。

## 12. 静的テスト実施手順

1. ファイルを作成または編集した場合、該当する対象領域の静的テストを実行する。
2. バックエンドを作成または編集した場合、次のコマンドを実行する。

```bash
UV_CACHE_DIR=/tmp/uv-cache uv run ruff check src/backend
UV_CACHE_DIR=/tmp/uv-cache uv run ruff format --check src/backend
UV_CACHE_DIR=/tmp/uv-cache uv run mypy src/backend
```

3. フロントエンドを作成または編集した場合、次のコマンドを実行する。

```bash
cd src/frontend
npm run lint
npm run format:check
npm run typecheck
```

4. 静的テストが不合格となった場合、指摘箇所を修正する。
5. 修正後、同じコマンドで再テストする。
6. すべての対象コマンドが合格した状態を、静的テスト完了条件とする。

全体実行入口を定義する場合は、スクリプトファイルを追加せず、package managerのコマンド定義で管理する。

## 14. 留意事項

- 本書は静的テスト方針を定義する文書であり、個別の検出結果一覧を扱わない。
- 静的テスト設定ファイルの具体値は、本方針で定義した採用方針、除外方針、判定基準に従う。
- `codex/output_json_schema/` のJSON Schema検証は、静的テストではなく契約確認またはテスト工程で扱う。
- `Any` / `any` を必要とする外部境界では、値を `unknown` または具体的なpayload型として受け取り、検証後に利用する。
- 広すぎる `object` と必要性を説明できない `cast(...)` は、静的テストだけで完全検出できる前提にしない。実装時の型設計、コーディング規約、レビューで補完する。
