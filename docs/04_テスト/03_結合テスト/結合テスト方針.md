# 結合テスト方針

## 1. 文書の目的

本書は、D-Conciergeの結合テストにおける対象範囲、結合テスト対象単位、対象外範囲、テスト観点、網羅基準、テストデータ、実施環境、実施手順、成果物・証跡および完了条件を定義することを目的とする。

本書は結合テストの方針を定義する文書であり、個別テストケース一覧、テストデータ一覧、テスト結果一覧、静的テスト方針、単体テスト方針、総合テスト方針は扱わない。

## 2. 前提

- 本書は、要件定義書、外部設計書、内部設計書、静的テスト方針、単体テスト方針、開発標準およびユーザ合意事項を前提とする。
- 結合テストは、テストコードを作成して自動実行する。
- 結合テストで確認する対象は、自動テストコードで再現できるものに限定する。
- 手動操作、目視確認、Playwrightによるブラウザ確認、実Codex実行、実外部サービス確認は本書の対象外とする。
- バックエンド結合テストは、REST/SSE API単位を主単位とする。
- `起動時実行回復処理` はAPI起点ではないが、未完了runの回復・終端整合に関わる重要処理として結合テスト対象に含める。
- フロントエンド結合テストは、`src/frontend/tests/integration/` で、ページ、feature、API境界、SSEイベント反映の連携を確認する。
- 文書成果物は `docs/04_テスト/03_結合テスト/結合テスト方針.md` のみとする。

## 3. 結合テスト対象範囲

| 対象区分 | 対象範囲 | 確認目的 |
| --- | --- | --- |
| バックエンド REST API | 外部IF設計で定義するREST API全件 | HTTP契約、入力検証、DB永続化、エラー変換、トレースログ出力の連携を確認する |
| バックエンド SSE API | `GET /api/chats/{chat_id}/runs/{run_id}/sse` | 現在状態通知、イベント配信、終端イベント、対象runなしの扱いを確認する |
| バックエンド 起動時回復 | `起動時実行回復処理` | `accepted` runの再登録、`running` / `validating` runのエラー終端、`cancel_requested` runのキャンセル済み整合を確認する |
| バックエンド DB連携 | Repository、SQLAlchemy ORM、migration適用後のPostgreSQLテストDB | 登録、更新、参照、制約違反、トランザクション、部分UNIQUE制約の連携を確認する |
| バックエンド ファイル配信 | 参照元PDF取得、Codex成果物配信で使う一時テストファイル | 許可範囲判定、MIMEタイプ、存在しないファイル、許可外パスの扱いを確認する |
| バックエンド Codex境界 | CodexRunnerのFake/Stub応答 | 実Codexを起動せず、生成成功、検証失敗、再生成、異常終了、タイムアウト、キャンセルの上位連携を確認する |
| フロントエンド ページ連携 | `ChatPage` を起点とする画面状態統合 | 開始画面、チャットスレッド、履歴、回答、中間メッセージ、参照元ビューアの連携を確認する |
| フロントエンド API境界連携 | `features/*/api/` と画面・featureの連携 | fetch/EventSourceのテスト応答が画面状態、表示、エラー、キャンセルへ反映されることを確認する |

バックエンドREST APIの対象は以下とする。

| IF ID | API | 確認対象 |
| --- | --- | --- |
| IF-SB-01 | `GET /api/app-config` | アプリ設定取得 |
| IF-SB-02 | `POST /api/chats/start` | 新規チャット開始 |
| IF-SB-03 | `POST /api/chats/{chat_id}/runs` | 継続指示受付 |
| IF-SB-04 | `GET /api/chat-histories` | 履歴一覧取得 |
| IF-SB-05 | `GET /api/chats/{chat_id}` | 履歴詳細取得 |
| IF-SB-07 | `POST /api/chats/{chat_id}/runs/{run_id}/cancel` | キャンセル要求 |
| IF-SB-08 | `GET /api/artifacts/{artifact_id}` | Codex成果物取得 |
| IF-SB-09 | `GET /api/references/{reference_id}` | 参照元データ取得 |

## 4. 結合テスト対象単位

| 区分 | 結合テスト対象単位 | 結合する要素 | 補足 |
| --- | --- | --- | --- |
| バックエンド REST | REST API 1エンドポイント | presentation、application、domain、infrastructure、PostgreSQL、トレースログ | HTTPレスポンスとDB状態を主な観測点とする |
| バックエンド SSE | SSE API 1エンドポイント | presentation/sse、RunEventBroker、ChatRepository、SSE payload変換 | 初期状態通知とpublish後イベントを観測点とする |
| バックエンド 実行連携 | 受付APIから到達する実行処理 | RunExecutionDispatcher、ExecuteChatRunUseCase、ValidateAnswerUseCase、Repository、SSE配信、Fake CodexRunner | 内部実行処理を単独の主単位にせず、API受付後のDB状態、SSEイベント、履歴詳細APIで観測する |
| バックエンド 起動時回復 | アプリ起動時の回復処理 | RunExecutionDispatcher、ChatRepository、PostgreSQL、トレースログ | API起点ではない補助対象として扱う |
| フロントエンド 画面連携 | `ChatPage` 起点の主要feature連携 | page、layout、chat feature、answer-rendering、reference-viewer、chatApi | 実バックエンドへ接続せず、fetch/EventSourceをテスト応答へ差し替える |

## 5. 結合テスト対象外範囲

| 対象外範囲 | 対象外理由 |
| --- | --- |
| 実Codex実行 | 外部プロセス、実行時間、環境差分が大きく、結合テストの再現性を下げるため |
| 実ブラウザ、Playwright、スクリーンショット確認 | 結合テストはテストコードによる自動確認に限定し、ブラウザ実機差分は総合テストで扱うため |
| フロントエンドから実バックエンドへの接続 | jsdom上の結合テストではfetch/EventSourceを固定し、実配線確認は総合テストで扱うため |
| 本番相当データ、手動投入データ | 再現性、独立性、機微情報管理を損なうため |
| 実外部サービス、課金を伴うサービス、社外ネットワーク | 自動実行の安定性と費用管理を損なうため |
| 本番相当負荷、性能負荷、長時間耐久 | 結合テストでは機能連携を確認し、性能は別工程で扱うため |
| 運用手順確認、障害復旧手順確認 | 手順確認であり、自動テストコードの確認対象ではないため |
| `src/frontend/backend_mock/` の直接import | UIコンポーネントはAPI境界を経由する方針であり、モック資産への直接依存を避けるため |
| `codex/output_json_schema/` 単体のメタスキーマ検証 | JSON Schema契約資産そのものの検証であり、結合テストの主対象ではないため |

## 6. 結合テスト観点

結合テストでは、少なくとも以下の共通観点を確認する。

| 観点 | 確認内容 |
| --- | --- |
| 層間連携 | presentation、application、domain、infrastructure、DB、ファイル、SSE、API境界が設計どおりに連携すること |
| IF契約 | URL、HTTPメソッド、リクエスト、レスポンス、ステータスコード、SSEイベント名、payload、エラー形式が外部IF設計どおりであること |
| DB永続化 | 登録、更新、参照、制約違反、未完了run一意制約、履歴再表示用データ保存が設計どおりであること |
| トランザクション | 成功時の確定、失敗時のrollback、dispatcher登録失敗時の状態整合、個別DB操作失敗時の扱いが設計どおりであること |
| エラー伝播 | 入力不正、対象なし、競合、パス拒否、DB失敗、CodexRunner Fake失敗がHTTP/SSEエラーとトレースログへ設計どおりに変換されること |
| ログ | trace_id、chat_id、run_id、処理段階、例外分類、検証失敗理由、キャンセル、タイムアウトが必要な場面で記録されること |
| タイムアウト | `execution_deadline_at`、残り秒数、CodexRunner Fakeのタイムアウト、後続処理抑止が設計どおりに連携すること |
| キャンセル | 受付、実行中、検証中、終端状態それぞれのキャンセル受付、状態更新、SSE配信が設計どおりであること |
| 処理設計の事前条件 | 各処理設計書で定義された開始条件を満たす場合に処理を開始できること |
| 処理設計の事後条件 | 各処理設計書で定義された成功保証または失敗時保証が満たされること |
| 処理設計の不変条件 | 各処理設計書で定義された処理中に破ってはならない条件が維持されること |

### 6.1. バックエンド処理別観点

| 処理 | 結合テスト観点 |
| --- | --- |
| アプリ設定取得 | 内部設定を画面へ返さず、歓迎メッセージと入力候補だけを返すこと |
| 新規チャット開始 | チャット、初回指示、runが同一受付で保存され、SSE URLと `accepted` 状態が返ること |
| 継続指示受付 | 既存チャットへの指示とrun保存、未完了run競合のHTTP 409、対象チャットなしのHTTP 404を確認すること |
| チャット実行 | API受付後にFake CodexRunner経由で `running`、`validating`、`completed`、`error`、`timed_out`、`canceled` の連携を確認すること |
| 回答検証・再生成 | 固定検証、参照元検証、再生成指示、再生成上限到達、`page_start` / `page_end` 正規化をAPI経由の結果として確認すること |
| SSE購読 | 接続直後の現在状態、`state`、`message`、`answer`、`error`、`canceled` の配信契約を確認すること |
| キャンセル | キャンセル可能状態、キャンセル不可状態、受付状態の即時 `canceled`、実行中のFake終了要求結果を確認すること |
| 履歴一覧取得 | 更新日時順、最新run状態、タイトル、空履歴、DB失敗時のHTTP 500を確認すること |
| 履歴詳細取得 | 指示、run、中間メッセージ、回答、参照元が保存済み内容から整形されること |
| 参照元PDF取得 | 許可範囲内PDFだけを `application/pdf` で返し、パストラバーサルと対象なしを拒否すること |
| Codex成果物配信 | 採用済み成果物だけを保存済みMIMEタイプで返し、未採用ファイルと許可外MIMEを拒否すること |
| 起動時実行回復 | `accepted` の再登録、`running` / `validating` の `error` 終端、`cancel_requested` の `canceled` 終端、終端済みrun不変を確認すること |

### 6.2. フロントエンド連携観点

| 区分 | 確認内容 |
| --- | --- |
| 初期表示 | `ChatPage` がアプリ設定、履歴一覧、直近チャット詳細を取得し、開始画面またはチャット画面へ反映すること |
| 新規チャット開始 | 入力値が `/api/chats/start` のpayloadになり、受付応答後にチャット詳細取得とSSE購読へつながること |
| 継続指示 | 既存チャットへの追加指示がAPI境界を通り、対象runだけが画面状態に追加されること |
| SSE反映 | `state`、`message`、`answer`、`error`、`canceled` がチャットスレッド、中間メッセージ、回答、エラー表示へ反映されること |
| 履歴再表示 | 履歴選択時に詳細取得結果を表示し、継続中runがある場合はSSE再接続へ進むこと |
| キャンセル | キャンセルAPI応答とSSE終端イベントにより、キャンセル中表示が解除されること |
| 参照元表示 | 回答内参照元リンクから参照元ビューアが開き、参照元取得URLをAPI境界経由で扱うこと |
| エラー表示 | API失敗、SSE失敗、旧ストリーム化、意図しない切断が画面状態へ設計どおり反映されること |

## 7. 網羅基準

| 項目 | 基準 |
| --- | --- |
| 主基準 | 設計単位の網羅 |
| バックエンド主基準 | REST API全件、SSE API、起動時実行回復処理 |
| フロントエンド主基準 | `ChatPage` 起点の主要feature連携 |
| 補助基準 | 分岐カバレッジ |
| 目標値 | バックエンド結合テスト、フロントエンド結合テストそれぞれ80%以上 |
| 判定対象外指標 | 行カバレッジ、ステートメントカバレッジ、関数カバレッジ |

結合テストの完了判定では、設計単位の網羅を主基準とし、分岐カバレッジ80%以上を補助基準として扱う。分岐カバレッジだけで結合テストの完了を判断しない。

網羅基準を満たさない場合は、不足しているAPI、処理、画面連携、分岐のテストコードを追加し、同じ環境で再実行する。対象外とする場合は、本書の対象外範囲に該当することを確認できる状態にする。

## 8. テストデータ方針

| 項目 | 方針 |
| --- | --- |
| テストデータ作成方法 | テストごとにfixture/factoryで独立データを作成する |
| DB初期化 | migration適用後、テストごとにPostgreSQLテストDBを既知状態へ初期化する |
| Codex応答 | CodexRunner Fake/Stubで、生成成功、検証失敗、再生成、異常終了、タイムアウト、キャンセル結果を固定する |
| 時刻固定 | RuntimeProviderまたはfixtureで現在時刻を固定する |
| ID固定 | ID発番をfixtureで固定し、レスポンス、DB、SSE、ログの対応を確認できるようにする |
| ファイル | 参照元PDFとCodex成果物はテスト用一時ファイルとして作成し、テスト終了時に削除する |
| フロントエンドAPI応答 | テストごとにfetch/EventSource応答を固定し、テスト間で状態を共有しない |
| 機微情報 | 本番データ、個人情報、秘密情報、絶対パスをテストデータに含めない |

テストデータは、テストの意図を説明できる最小限のデータとする。テスト同士は実行順序に依存せず、並列実行してもDB状態、Fake応答、画面状態が汚染されないようにする。

## 9. テスト環境方針

| 項目 | 方針 |
| --- | --- |
| 実行環境 | ローカル開発環境を中心とする |
| バックエンドテストフレームワーク | pytest |
| バックエンドDB | Dockerコンテナで起動するPostgreSQLテストDB |
| DB構築 | `postgres-test` 起動後、migrationを適用してから結合テストを実行する |
| バックエンドAPI実行 | FastAPIアプリをテスト用設定で起動し、REST/SSE境界をテストコードから呼び出す |
| 実Codex | 起動しない。CodexRunner Fake/Stubで代替する |
| フロントエンドテストフレームワーク | Vitest |
| フロントエンドDOM環境 | jsdom |
| フロントエンドコンポーネント確認 | React Testing Library |
| フロントエンド通信 | fetch/EventSourceをテスト内で差し替える |
| CI | 必須実行対象にしない |

Docker PostgreSQLは、結合テスト用サービス名を `postgres-test` とする。Docker Compose定義を作成する工程では、`postgres-test` が本番DBとは別のDB名、ユーザ、ポート、永続化領域を使うようにする。

## 10. テストコード記述方針

- 結合テストコードは、バックエンドは `src/backend/tests/integration/`、フロントエンドは `src/frontend/tests/integration/` へ配置する。
- バックエンドは `test_*.py` を使用し、REST/SSE APIまたは起動時回復処理に対応するファイル構成にする。
- フロントエンドは `*.test.ts` または `*.test.tsx` を使用し、主要feature連携に対応するファイル構成にする。
- 1つのテストでは、原則として1つの結合観点と確認内容を明確にする。
- 実Codex、実ブラウザ、実バックエンド接続、本番データに依存するテストコードを作成しない。
- `Any`、広すぎる `object`、安易な `cast(...)`、TypeScriptの `any` に依存するテストデータは作成しない。
- `src/frontend/backend_mock/` の固定データ、API代替コード、参照元、Codex成果物をフロントエンド結合テストから直接importしない。
- テストごとに必要なDB、ファイル、Fake応答、時刻、IDを初期化し、別テストの実行順序に依存しないようにする。

すべての結合テストコードには、Pythonではdocstring、TypeScript/TSXでは対象テストの直前コメントまたは `test` / `it` の説明文字列として、以下の2項目を記載する。

```text
観点：確認対象の責務や振る舞いが分かる文章を記載する
確認：検証する結果を具体的に記載する
```

## 11. テスト実施手順

1. Docker ComposeでPostgreSQLテストDBを起動する。
2. バックエンドのmigrationをPostgreSQLテストDBへ適用する。
3. バックエンド結合テストを実行する。
4. フロントエンド結合テストを実行する。
5. 失敗時は、API応答、SSEイベント、DB状態、Fake応答、テスト用ファイル、トレースログ、画面状態を確認する。
6. 修正後、PostgreSQLテストDBを既知状態へ初期化して同じテストを再実行する。
7. 合格後、`docs/04_テスト/03_結合テスト/evidence/` 配下のテキスト証跡を更新する。

標準の結合テスト実行コマンドは以下とする。

```bash
docker compose up -d postgres-test
UV_CACHE_DIR=/tmp/uv-cache uv run pytest src/backend/tests/integration
```

```bash
cd src/frontend
npm run test:integration
```

`npm run test:integration` は、Vitest、React Testing Library、jsdomを使って `src/frontend/tests/integration/` を実行するコマンドとして定義する。

分岐カバレッジ証跡更新時は、pytest-covまたはVitest V8 coverageの機械可読出力から分岐指標だけを抽出するコマンドを、テスト設定またはpackage managerのコマンド定義で管理する。証跡更新コマンドは、行カバレッジ、ステートメントカバレッジ、関数カバレッジを標準出力または証跡ファイルへ出力しない。

## 12. 成果物・証跡方針

| 区分 | 扱い |
| --- | --- |
| 結合テスト方針書 | `docs/04_テスト/03_結合テスト/結合テスト方針.md` を正式成果物として管理する |
| バックエンド結合テストコード | `src/backend/tests/integration/` 配下を正式成果物として管理する |
| フロントエンド結合テストコード | `src/frontend/tests/integration/` 配下を正式成果物として管理する |
| テスト支援コード | `src/backend/tests/support/`、`src/frontend/tests/support/` 配下を管理対象とする |
| テスト設定ファイル | `pyproject.toml`、Docker Compose設定、`src/frontend/vitest.config.*`、`src/frontend/package.json` を管理対象とする |
| 依存lock file | `uv.lock`、`src/frontend/package-lock.json` を管理対象とする |
| バックエンド結合テスト証跡 | `docs/04_テスト/03_結合テスト/evidence/backend-integration-evidence.txt` を最新結果で上書きする |
| フロントエンド結合テスト証跡 | `docs/04_テスト/03_結合テスト/evidence/frontend-integration-evidence.txt` を最新結果で上書きする |
| JUnit XML | 正式成果物として保存しない |
| 詳細実行ログ | 正式成果物として保存しない |
| DB初期化ログ | 正式成果物として保存しない |
| 行カバレッジレポート | 作成しない |
| CI実行結果 | 結合テストの必須証跡としない |

結合テスト証跡ファイルには、対象、実行日時、実行コマンド、設計単位の網羅結果、分岐カバレッジ、総分岐数、通過分岐数、テスト件数、失敗件数、判定結果を記載する。再テスト時は同じファイル名で最新結果へ上書きする。

証跡ファイルには、以下の項目だけを `key=value` 形式で記載する。

| 項目 | 記載内容 |
| --- | --- |
| `target` | `backend` または `frontend` |
| `executed_at` | テスト実行日時 |
| `command` | 実行した結合テストコマンド |
| `design_coverage` | 対象設計単位の網羅結果 |
| `branch_coverage` | 分岐カバレッジの割合 |
| `covered_branches` | 通過した分岐数 |
| `total_branches` | 計測対象の総分岐数 |
| `tests` | 実行したテスト件数 |
| `failures` | 失敗したテスト件数 |
| `result` | `pass` または `fail` |

## 13. 完了条件

結合テストは、以下をすべて満たした場合に完了とする。

- バックエンドのREST API全件、SSE API、`起動時実行回復処理` が結合テストで確認されている。
- フロントエンドの `ChatPage` 起点の主要feature連携が結合テストで確認されている。
- 結合テストコードがすべて成功している。
- バックエンド結合テストとフロントエンド結合テストの分岐カバレッジがそれぞれ80%以上である。
- `docs/04_テスト/03_結合テスト/evidence/` 配下のバックエンド・フロントエンド証跡が最新結果へ更新されている。
- 未確認項目がない。
- 未解決不具合がない。
- 結合テスト対象外とした項目が本書の対象外範囲と矛盾していない。

未確認項目、失敗テスト、分岐カバレッジ80%未達、証跡未保存、未解決不具合が1件でもある場合、結合テストは未完了とする。

## 14. 留意事項

- 本書は結合テスト方針を定義する文書であり、個別テストケースの入力値、期待値、実行結果はテストコードと証跡で管理する。
- 結合テストでは実Codexを起動しないため、Codex CLIそのものの挙動、OSレベルのプロセス制御、実JSONL出力の互換性は別工程で確認する。
- フロントエンド結合テストでは実バックエンドへ接続しないため、ブラウザと実バックエンドの配線、CORS、ネットワーク、実SSE接続の体感差分は総合テストで確認する。
- Docker PostgreSQLのDB名、ユーザ、ポート、永続化領域は本番環境と分離する。
- 結合テストが実装詳細に密結合する場合は、外部IF、処理設計、画面連携の観点へ戻してテストを整理する。
