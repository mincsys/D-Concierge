# 物理データ設計

## 1. 文書の目的

本書は、D-Concierge MVPで使用する永続ストアの物理配置、項目、制約、索引を定義することを目的とする。

## 2. 前提

- 関連文書: `docs/02_外部設計/05_論理データ設計/論理データ設計.md`、`docs/03_内部設計/04_処理設計/処理一覧.md`、`docs/03_内部設計/06_共通設計/共通設計.md`
- RDBMSはPostgreSQLを使用する。
- テーブル定義とマイグレーションは `src/backend/infrastructure/database/migrations/` で管理する。
- ORMモデルは `src/backend/infrastructure/database/models/base.py`、`chat.py`、`answer.py` に置き、application層へSQLAlchemyモデルを返さない。
- DBには論理データと保存済みファイルのメタ情報を保持し、参照元PDF実体、Codex成果物実体、トレースログ実体はファイルとして保持する。
- Codex出力スキーマの参照ページは `start_page` / `end_page` だが、DBへ保存するlocatorは正規化後の `page_start` / `page_end` とする。
- Codex出力スキーマの参照元pathは `readonly/` から始まるCodex作業領域上の相対パスだが、DBへ保存するlocatorの `path` は `readonly/` を除いた共有データソースルートからの相対パスとする。

| ストレージ名 | 種別 | 用途 |
| --- | --- | --- |
| `local_users` | PostgreSQLテーブル | MVPで共有するローカル利用者を保持する。 |
| `chats` | PostgreSQLテーブル | チャット単位、履歴タイトル、Codex側resume用IDを保持する。 |
| `chat_runs` | PostgreSQLテーブル | ユーザ指示1回ごとの実行状態、開始/終了日時、利用者向けメッセージを保持する。 |
| `user_instructions` | PostgreSQLテーブル | runに対応する利用者指示本文を保持する。 |
| `intermediate_messages` | PostgreSQLテーブル | 履歴再表示用の中間メッセージを保持する。 |
| `answer_blocks` | PostgreSQLテーブル | 完了runに紐づき、Codex出力の `answers` 単位で検証済み回答ブロック本文と表示順を保持する。 |
| `references` | PostgreSQLテーブル | 回答ブロックに紐付くPDF参照元メタ情報とlocatorを保持する。 |
| `artifacts` | PostgreSQLテーブル | 回答ブロックに紐付く保存済みCodex成果物メタ情報を保持する。 |

## 4. 物理定義

### 4.1. `local_users`

| 項目 | 内容 |
| --- | --- |
| 物理名 | `local_users` |
| 用途 | 共有1ユーザの識別情報を保持する。 |

| カラム名 | 型 | NULL可否 | 制約 | 説明 |
| --- | --- | --- | --- | --- |
| `id` | `uuid` | 否 | PK | ローカル利用者ID。 |
| `display_name` | `varchar(100)` | 否 |  | 画面表示名。 |
| `is_active` | `boolean` | 否 | DEFAULT `true` | 有効状態。 |

### 4.2. `chats`

| 項目 | 内容 |
| --- | --- |
| 物理名 | `chats` |
| 用途 | 一連の指示応答をまとめるチャット単位を保持する。 |

| カラム名 | 型 | NULL可否 | 制約 | 説明 |
| --- | --- | --- | --- | --- |
| `id` | `uuid` | 否 | PK | チャットID。 |
| `local_user_id` | `uuid` | 否 | FK -> `local_users.id` | 所有するローカル利用者ID。 |

### 4.3. `chat_runs`

| 項目 | 内容 |

## 5. 制約と索引

| 対象 | 種別 | 内容 |
| --- | --- | --- |
| `chats.local_user_id` | FK | `local_users.id` を参照し、利用者削除時は制限する。 |
| `chats.session_id` | UNIQUE | Codex作業領域をチャット単位で一意にする。 |
| `chats(local_user_id, updated_at desc)` | INDEX | 履歴一覧を利用者別、更新日時降順で取得する。 |
| `chat_runs.chat_id` | FK | `chats.id` を参照し、チャット削除時は関連runを削除する。 |
| `chat_runs.state` | CHECK | 定義済み8状態だけを許可する。 |
| `chat_runs(chat_id, started_at, id)` | INDEX | チャット詳細でrunを開始日時とIDの安定順に取得する。 |
| `chat_runs(chat_id, state)` | INDEX | 未完了run確認に使用する。 |
| `chat_runs_one_unfinished_per_chat` | 部分UNIQUE INDEX | `chat_id` を対象に、`state IN ('受付','実行中','検証中','キャンセル要求中')` の行を同一チャットで1件だけ許可する。 |
| `user_instructions.run_id` | UNIQUE/FK | runとユーザ指示を1対1にする。 |
| `intermediate_messages(run_id, created_at, id)` | INDEX | run内の中間メッセージを作成日時とIDの安定順で取得する。 |
| `answer_blocks.run_id` | FK | `chat_runs.id` を参照し、run削除時は回答ブロックを削除する。 |
| `answer_blocks(run_id, position)` | UNIQUE | 回答ブロックのrun内表示順を一意にし、表示順で取得する。 |
| `references.answer_block_id` | FK | `answer_blocks.id` を参照し、回答ブロック削除時は参照元を削除する。 |
| `references(answer_block_id, position)` | UNIQUE | 参照元の回答ブロック内表示順を一意にし、表示順で取得する。 |
| `references.locator` | CHECK | `path` が文字列、`page_start` と `page_end` が正の整数、`page_start <= page_end` であることを検証する。`path` はアプリケーション層で共有データソースルートからの相対PDFパスへ正規化済みであることを前提とする。 |
| `artifacts.answer_block_id` | FK | `answer_blocks.id` を参照し、回答ブロック削除時は成果物メタ情報を削除する。 |
| `artifacts.storage_path` | UNIQUE | 同一保存先参照の重複登録を防ぐ。 |

## 6. 保持期間と整合性

- チャット履歴、ユーザ指示、実行状態、中間メッセージ、回答ブロック、参照元、Codex成果物メタ情報はMVPでは無期限保持する。
- 保存済みCodex成果物実体は、対応する `artifacts` レコードが存在する限り保持する。
- セッション内 `artifacts/` は採用前の一時領域であり、キャンセル、エラー、タイムアウト、検証失敗で終端したrunの成果物候補は保存済み成果物へコピーしない。
- トレースログはMVPでは無期限保持するが、DBの参照整合性対象には含めない。
- チャット詳細再表示ではDBを正とし、SSE未受信イベントがあってもDB保存済み状態から復元する。
- DB commit後にSSE配信へ失敗した場合、DB状態を巻き戻さない。
- ファイル保存とDBメタ情報保存の間で失敗した場合は、処理設計に従い未参照ファイルを後続の保守対象とし、回答採用を完了扱いにしない。
