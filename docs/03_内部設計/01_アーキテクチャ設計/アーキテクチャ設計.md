# アーキテクチャ設計

## 1. 文書の目的

本書は、D-Conciergeの内部実装における構成要素の責務分割、依存方向、公開口、副作用境界、契約境界を定義することを目的とする。

本書では、画面、API、DB、codex exec、ファイル保存、トレースログを単なる外部構成として再掲せず、実装内部でどの層とモジュールがどの判断と副作用を所有するかを定義する。

## 2. 前提

- 要件定義書は `docs/01_要件定義/要件定義.md` を正とする。
- 外部設計書は `docs/02_外部設計/` 配下の外部設計を正とする。
- バックエンドは `presentation`、`application`、`domain`、`infrastructure` のレイヤに分割する。
- フロントエンドは `pages`、`features`、`components`、`app` を中心に分割する。
- 参照元ビューアの実装対象はPDFのみとする。
- `src/frontend/backend_mock/` はフロントエンド単体確認用の開発支援資産であり、アプリ本体のクラス・モジュール設計対象から除外する。
- ファイル添付ボタンと表示設定ボタンは将来拡張用の表示部品として扱い、本システムの処理設計および内部IF設計には含めない。

## 5. 留意事項

- `domain` はDB、codex exec、ファイル、時刻、ID発番、ログ出力を直接呼び出さない。
- `application` は `application/ports` を通じて副作用を利用し、状態条件付き更新とトランザクション境界を所有する。
- `presentation` はAPI契約とSSE契約を所有し、業務判断を持たない。API/SSE payloadへの変換では、`domain` が定義するEnumや値オブジェクトを参照してよい。
- `infrastructure` は外部副作用の実装を所有し、業務上の採用可否判断を持たない。
- `shared` はエラー分類とtrace_idのように複数レイヤを接続する横断契約だけを置く。
- 実行状態のAPI/SSE/DB保存値は `RunState.value` を外部仕様値として扱い、各境界で送受信・保存する直前に文字列へ変換する。
- `ChatState` はチャット単位のライフサイクル状態として扱う。
- `RunState` にはチャット単位のライフサイクル状態を含めない。
- 保護対象APIはpresentation層の認証依存関係でログインセッションを検証し、application層のチャット系ユースケースへ認証済みユーザIDを明示的に渡す。
- ログインセッショントークンの生値はCookie境界だけで扱い、DBには照合用ハッシュだけを保持する。
- 削除要求受付後のチャットは履歴一覧、履歴詳細、継続指示、参照元取得、成果物取得の対象外とし、物理削除が失敗した場合も履歴へ復帰させない。
- チャット物理削除は、生成用・検証用のセッション作業領域と保存済み成果物実体だけを削除し、共有データソース、`readonly/` のリンク先実体、トレースログ実体は削除しない。
- アカウント物理削除は、対象ユーザの生成用・検証用作業領域と保存済み成果物実体をユーザ単位で削除し、共有データソース、`readonly/` のリンク先実体、トレースログ実体は削除しない。
- production codeの直接importは層境界に従い、`application` は `infrastructure` の具象実装を生成しない。具象実装の生成とportへの注入は `src/backend/app` と `src/backend/main.py` のcomposition rootで行う。
- `infrastructure` から `application` を参照する場合は `application/ports` に定義された抽象IFとDTOだけを対象にし、UseCaseや補助実装へ直接依存しない。
- PDF以外の参照元種別は本システムの画面実装対象に含めない。
- `src/frontend/backend_mock/` はフロントエンド単体確認用の補助資産として維持するが、UIコンポーネントから直接参照しない。
