# 新規チャット開始処理設計

## 1. 文書の目的

本書は、新規チャット開始処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: 利用者が開始画面で初回指示を送信する。
- 関連文書: `docs/02_外部設計/06_外部インターフェース設計/画面バックエンドAPI IF.md`、`docs/03_内部設計/03_内部IF設計/チャットRepository IF.md`

## 3. 処理概要

- 初回指示を検証し、チャット、ユーザ指示、受付済みrunを作成する。
- 受付結果としてチャットID、run ID、SSE URL、受付状態を返す。
- 実行本体は受付後にチャット実行処理へ引き渡す。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `presentation/rest` | HTTP入力検証、trace_id生成、HTTP応答変換 |
| `StartChatUseCase` | チャット作成とrun受付の調停 |
| `RunStatePolicy` | 初期状態と未完了状態の判断 |
| `ChatRepository` | チャット、指示、runの保存 |
| `Runtime Provider` | チャットID、run ID、時刻の生成 |
| `TraceLogWriter` | 受付開始、成功、例外の記録 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | REST境界でtrace_idを生成し、リクエスト本文をPydanticスキーマで検証する。 |
| 2 | `StartChatUseCase` が指示本文の空白を除去し、空文字を入力不正として扱う。 |
| 3 | Runtime ProviderでチャットID、run ID、session ID、受付日時を生成する。 |
| 4 | Repositoryトランザクション内でチャット、初回指示、runを保存する。 |
| 5 | SSE URLを生成し、受付応答を返す。 |
| 6 | チャット実行処理をrun ID単位で開始できる状態にする。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- 初回指示本文が送信されている。
- DB接続、ID発番、時刻取得、設定読込が利用可能である。

### 6.2. 事後条件

- チャット、ユーザ指示、受付済みrunが同一トランザクションで保存される。
- 応答にはチャットID、run ID、SSE URL、`受付` 状態が含まれる。

### 6.3. 不変条件

- チャットだけ、またはrunだけが保存された部分成功を残さない。
- 実行本体の成否は受付処理のHTTP応答完了を阻害しない。

## 7. 副作用

| 副作用 | 発生箇所 | 補足 |
| --- | --- | --- |
| DB書込 | Repository | チャット、指示、runを保存する |
| ID発番 | Runtime Provider | チャットID、run ID、session ID |
| 時刻取得 | Runtime Provider | 作成日時、更新日時、受付日時 |
| ログ出力 | TraceLogWriter | 受付開始、成功、失敗 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| 指示本文が空 | 入力不正としてHTTP 400へ変換する |
| DB保存失敗 | rollbackし、システムエラーとしてHTTP 500へ変換する |
| ID発番失敗 | rollbackし、システムエラーとして扱う |

## 9. ログ出力

| タイミング | ログ内容 |
| --- | --- |
| 受付開始 | trace_id、処理名、APIパス |
| 保存成功 | trace_id、chat_id、run_id、状態 |
| 例外発生 | trace_id、エラー分類、処理段階 |

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | チャット、指示、run保存直前 |
| 終了点 | 受付済みrun保存完了 |
| ロールバック条件 | 入力不正以外の保存失敗、ID/時刻生成結果の利用不能、DB制約違反 |
