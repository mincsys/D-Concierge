# 認証状態確認処理設計

## 1. 文書の目的

本書は、認証状態確認処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: フロントエンドが保護対象画面表示前に `GET /api/auth/me` を呼び出す。
- ログインセッションはCookieのトークン生値とDBの `token_hash`、`expires_at` で判定する。
- セッション行が存在しない、期限切れ、または対象ユーザが`deleting`の場合は未ログイン扱いにする。

## 3. 処理概要

- REST境界でCookieからログインセッショントークンを取得する。
- `AuthenticateSessionUseCase` がトークン照合用ハッシュを生成し、Repositoryからセッションとユーザを取得する。
- 有効期限切れのセッションは削除し、未ログイン応答へ変換する。
- 有効なセッションの場合はユーザIDとユーザ名を返す。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `account_router` | `GET /api/auth/me` のREST境界 |
| `AuthenticateSessionUseCase` | セッション検証の調停 |
| `SessionTokenProviderPort` | Cookieトークンの照合用ハッシュ生成 |
| `AccountRepositoryPort` | セッション取得、期限切れセッション削除、ユーザ取得 |
| `ClockPort` | 現在時刻取得 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | REST境界でtrace_idを発行し、Cookieからログインセッショントークンを取得する。 |
| 2 | Cookieがない場合は未ログイン応答を返す。 |
| 3 | `AuthenticateSessionUseCase` を呼び出す。 |
| 4 | トークン生値から `token_hash` を生成する。 |
| 5 | Repositoryから `token_hash` に対応するログインセッションとユーザを取得する。 |
| 6 | セッションが存在しない場合は未ログイン応答を返す。 |
| 7 | `expires_at` が現在時刻以前の場合、対象セッションを削除して未ログイン応答を返す。 |
| 8 | ユーザが`deleting`の場合、対象セッションを削除して未ログイン応答を返す。 |
| 9 | ユーザIDとユーザ名をAPI応答へ変換する。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- Cookie名が共通設計で定義済みである。
- Repository、TokenProvider、Clockが注入済みである。

### 6.2. 事後条件

- 有効なセッションの場合、現在ユーザが返る。
- 無効なセッションの場合、未ログイン応答が返る。
- 期限切れまたは`deleting`ユーザのセッションは削除対象になる。

### 6.3. 不変条件

- Cookieトークン生値をDB、ログ、応答へ出さない。
- 認証失敗は通常の未ログインとして扱い、原則としてトレースログへ記録しない。

## 7. 副作用

| 種別 | 対象 | 内容 |
| --- | --- | --- |
| DB削除 | `login_sessions` | 期限切れまたは`deleting`ユーザのセッション行削除 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| Cookieなし | 未ログイン応答を返す |
| セッションなし | 未ログイン応答を返す |
| 期限切れ | 対象セッションを削除し、未ログイン応答を返す |
| ユーザが`deleting` | 対象セッションを削除し、未ログイン応答を返す |
| DB接続失敗 | システムエラーとして返し、トレースログへ記録する |

## 9. ログ出力

通常の未ログイン、期限切れ、`deleting`ユーザによる未ログイン扱いはトレースログへ記録しない。DB接続失敗など予期しない例外だけをtrace_id付きで記録する。

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | セッション取得開始時 |
| 終了点 | セッション確認または期限切れ削除commit時 |
| rollback条件 | DB例外が発生した場合 |
| トランザクション外処理 | HTTP応答変換 |
