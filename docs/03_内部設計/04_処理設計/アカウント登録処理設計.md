# アカウント登録処理設計

## 1. 文書の目的

本書は、アカウント登録処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: 利用者がアカウント登録画面で登録ボタンを押し、フロントエンドが `POST /api/auth/register` を呼び出す。
- 登録成功時は自動ログインし、ログインセッションCookieを設定する。
- 既存ログインセッションCookieが含まれる場合は、既存セッション行を削除して新しいセッションを発行する。

## 3. 処理概要

- REST境界でリクエストJSONをPydanticスキーマへ変換する。
- `RegisterAccountUseCase` が入力規則、ユーザID重複、パスワード確認一致を検証する。
- パスワードをハッシュ化し、ユーザとログインセッションを同一トランザクションで作成する。
- Cookieへ設定するログインセッショントークンを応答へ付与し、現在ユーザを返す。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `account_router` | `POST /api/auth/register` のREST境界 |
| `RegisterAccountUseCase` | 登録処理の調停 |
| `UserIdPolicy`、`UserNamePolicy`、`PasswordPolicy` | 入力規則の判定 |
| `PasswordHasherPort` | パスワードハッシュ生成 |
| `SessionTokenProviderPort` | セッショントークン発行と照合用ハッシュ生成 |
| `AccountRepositoryPort` | ユーザ作成、既存セッション削除、新規セッション作成 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | REST境界でtrace_idを発行し、リクエストJSONを検証する。 |
| 2 | Cookieに既存ログインセッショントークンがあれば取得する。 |
| 3 | `RegisterAccountUseCase` を呼び出す。 |
| 4 | ユーザID、ユーザ名、パスワード、パスワード確認をdomain policyで検証する。 |
| 5 | RepositoryでユーザID重複を確認する。 |
| 6 | パスワードハッシュを生成する。 |
| 7 | 新しいログインセッショントークンを発行し、照合用ハッシュを生成する。 |
| 8 | トランザクション内でユーザを作成し、既存Cookieに対応するセッションがあれば削除し、新規ログインセッションを作成する。 |
| 9 | commit後、Cookie設定とユーザ情報をAPI応答へ変換する。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- ユーザID、ユーザ名、パスワード、パスワード確認がリクエストに含まれている。
- Repository、Hasher、TokenProvider、Clockが注入済みである。

### 6.2. 事後条件

- `users` に`active`状態のユーザが作成される。
- `login_sessions` に新しいセッション行が作成される。
- Cookieへログインセッショントークンが設定される。
- 応答としてユーザIDとユーザ名を返す。

### 6.3. 不変条件

- パスワード生値とCookieトークン生値をDBへ保存しない。
- ユーザ作成とログインセッション作成は同時に成功または失敗する。
- ユーザID重複時はユーザもセッションも作成しない。

## 7. 副作用

| 種別 | 対象 | 内容 |
| --- | --- | --- |
| DB登録 | `users` | 新規ユーザ作成 |
| DB登録 | `login_sessions` | 新規ログインセッション作成 |
| DB削除 | `login_sessions` | 既存Cookieに対応するセッション削除 |
| Cookie設定 | ブラウザ | ログインセッショントークンを設定 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| 入力形式不正 | 項目別入力エラーとして返す |
| ユーザID重複 | `user_id` の項目別入力エラーとして返す |
| パスワード確認不一致 | `password_confirmation` の項目別入力エラーとして返す |
| ハッシュ生成失敗 | システムエラーとして返し、トレースログへ記録する |
| DB保存失敗 | システムエラーとして返し、トレースログへ記録する |

## 9. ログ出力

入力不正とユーザID重複はトレースログへ記録しない。ハッシュ生成失敗、DB保存失敗、予期しない例外をtrace_id付きで記録する。

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | ユーザID重複確認後のDB登録開始時 |
| 終了点 | ユーザ作成、既存セッション削除、新規セッション作成commit時 |
| rollback条件 | ユーザ作成、セッション削除、セッション作成のいずれかでDB例外が発生した場合 |
| トランザクション外処理 | Cookie設定、HTTP応答変換 |
