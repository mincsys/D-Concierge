# 起動時アカウント回復処理設計

## 1. 文書の目的

本書は、起動時アカウント回復処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: FastAPIアプリケーション起動時。
- ログインセッションは有効期限を過ぎた行を起動時に削除する。
- `deleting`ユーザはアカウント物理削除ジョブへ再登録する。

## 3. 処理概要

- Clock portで現在時刻を取得する。
- Repositoryで期限切れログインセッション行を削除する。
- Repositoryで`deleting`ユーザID一覧を取得する。
- `AccountDeletionDispatcher` へ1件ずつ削除ジョブを登録する。
- 登録失敗はトレースログへ記録し、次回起動時に再試行できる状態を維持する。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `app` composition root | 起動時処理の呼出 |
| `ClockPort` | 現在時刻取得 |
| `AccountRepositoryPort` | 期限切れセッション削除、削除中ユーザ一覧取得 |
| `AccountDeletionDispatcher` | アカウント物理削除ジョブ再登録 |
| `TraceLoggerPort` | 起動時回復失敗の記録 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | アプリケーション起動時にtrace_idを生成する。 |
| 2 | Clock portで現在時刻を取得する。 |
| 3 | Repositoryで `expires_at` が現在時刻以前のログインセッション行を削除する。 |
| 4 | Repositoryで`deleting`ユーザID一覧を取得する。 |
| 5 | `AccountDeletionDispatcher` へユーザIDを1件ずつ渡す。 |
| 6 | 登録結果が `registered` または `already_registered` の場合、次のユーザへ進む。 |
| 7 | 登録結果が `failed` の場合、トレースログへ記録して次のユーザへ進む。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- DB接続とTransactionManagerが利用可能である。
- AccountDeletionDispatcherとTraceLoggerが注入済みである。

### 6.2. 事後条件

- 起動時点で期限切れのログインセッション行が削除される。
- `deleting`ユーザがアカウント物理削除ジョブへ登録されるか、次回再試行可能な状態で残る。

### 6.3. 不変条件

- 起動時回復処理はユーザを`deleting`から`active`へ戻さない。
- Dispatcher登録失敗があってもアプリケーション起動全体を停止しない。

## 7. 副作用

| 種別 | 対象 | 内容 |
| --- | --- | --- |
| DB削除 | `login_sessions` | 期限切れセッション行削除 |
| バックグラウンド登録 | `AccountDeletionDispatcher` | `deleting`ユーザの物理削除ジョブ登録 |
| ログ出力 | トレースログ | 起動時回復失敗の調査情報 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| 期限切れセッション削除失敗 | トレースログへ記録し、起動処理は継続する |
| `deleting`ユーザ一覧取得失敗 | トレースログへ記録し、起動処理は継続する |
| 削除ジョブ登録失敗 | トレースログへ記録し、次回起動で再試行できる状態を維持する |

## 9. ログ出力

期限切れセッション削除失敗、削除中ユーザ一覧取得失敗、削除ジョブ登録失敗、予期しない例外を起動時trace_id付きで記録する。

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | 期限切れセッション削除開始時 |
| 終了点 | 期限切れセッション削除commit時 |
| rollback条件 | セッション削除でDB例外が発生した場合 |
| トランザクション外処理 | 削除中ユーザのDispatcher登録、トレースログ出力 |
