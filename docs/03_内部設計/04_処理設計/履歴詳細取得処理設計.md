# 履歴詳細取得処理設計

## 1. 文書の目的

本書は、履歴詳細取得処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: フロントエンドが履歴選択時、開始/継続受付後、起動時の直近チャット表示時にチャット詳細を取得する。
- 本処理は認証済みユーザだけが呼び出せる。
- 関連文書: `docs/03_内部設計/03_内部IF設計/チャットRepositoryIF.md`

## 3. 処理概要

- 認証済みユーザが所有する通常操作対象の指定チャットについて、run、ユーザ指示、中間メッセージ、採用済み回答、参照元を取得し、画面再表示用に返す。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `presentation/rest` | 認証依存関係、HTTP入力検証、trace_id生成、HTTP応答変換 |
| `GetChatDetailUseCase` | 詳細取得の調停 |
| `ChatRepository` | チャット詳細の関連データ取得 |
| `presentation/schemas/api.py` | API応答への変換 |
| `TraceLogWriter` | 例外の記録 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | REST境界でtrace_idを生成し、認証依存関係で現在ユーザを取得する。 |
| 2 | チャットIDを検証する。 |
| 3 | Repositoryで認証済みユーザが所有する通常操作対象のチャット本体と関連runを取得する。 |
| 4 | 各runへユーザ指示、中間メッセージ、回答、参照元を紐付ける。 |
| 5 | runは開始日時とIDの安定順に整え、中間メッセージは作成日時とIDの安定順に整えて、応答スキーマへ変換する。 |
| 6 | チャット詳細を返す。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- チャットIDが指定されている。
- 認証済みユーザIDが確定している。
- DB接続が利用可能である。

### 6.2. 事後条件

- 対象チャットが存在する場合、チャットID、タイトル、run配列が返る。
- 回答が未生成のrunでは回答項目を未設定として返す。
- 対象チャットが`deleting`の場合は詳細を返さない。

### 6.3. 不変条件

- 詳細取得はDBを更新しない。
- 保存済み成果物の実ファイル内容は詳細応答へ含めず、配信用URLだけを含める。
- `deleting`のチャットは、履歴詳細の通常取得対象にしない。

## 7. 副作用

| 副作用 | 発生箇所 | 補足 |
| --- | --- | --- |
| DB読込 | Repository | チャット、run、指示、中間メッセージ、回答、参照元 |
| ログ出力 | TraceLogWriter | 失敗 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| 未ログイン | 未ログイン応答を返す |
| 対象チャットなし | HTTP 404へ変換する |
| 対象チャットが`deleting` | HTTP 409へ変換する |
| DB接続失敗 | HTTP 500へ変換する |
| 関連データ不整合 | データ不整合としてtraceログへ記録し、HTTP 500へ変換する |

## 9. ログ出力

| タイミング | ログ内容 |
| --- | --- |
| 例外発生 | trace_id、chat_id、エラー分類、例外種別、スタックトレース |

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | 読取専用セッション開始時 |
| 終了点 | チャット詳細読取完了 |
| ロールバック条件 | 読取専用のためDB更新rollbackは発生しない |
