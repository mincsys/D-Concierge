# キャンセル処理設計

## 1. 文書の目的

本書は、キャンセル処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: 利用者が未完了runのキャンセルを選択する。
- 関連文書: `docs/03_内部設計/03_内部IF設計/Codex実行IF.md`、`docs/03_内部設計/03_内部IF設計/SSEイベント配信IF.md`

## 3. 処理概要

- 対象runがキャンセル可能状態であることを確認し、`キャンセル要求中` へ更新する。
- 対象runが `受付` の場合はcodex execプロセスが未登録であるため、`CodexRunner.cancel` を呼ばずに `キャンセル済み` へ終端する。
- 対象runが `実行中` または `検証中` の場合はcodex execプロセスへ終了要求を送り、`sent`、`already_exited`、`not_registered` の結果に応じて状態を整合させる。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `CancelChatRunUseCase` | キャンセル受付と終了要求の調停 |
| `RunStatePolicy` | キャンセル可能状態の判定 |
| `ChatRepository` | 状態条件付き更新 |
| `CodexRunner` | 対象プロセスへの終了要求 |
| `RunEventBroker` | `state`、`canceled` イベント配信 |
| `TraceLogWriter` | キャンセル要求と結果の記録 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | REST境界でチャットIDとrun IDを検証する。 |
| 2 | Repositoryで対象runを取得し、キャンセル可能状態か判定する。 |
| 3 | 状態条件付き更新で `キャンセル要求中` へ更新する。 |
| 4 | SSE `state` を配信する。 |
| 5 | 変更前状態が `受付` の場合は、CodexRunnerへ終了要求を送らず、同一ユースケース内で `キャンセル済み` へ更新してSSE `canceled` を配信する。 |
| 6 | 変更前状態が `実行中` または `検証中` の場合は、CodexRunnerへ対象プロセスの終了要求を送る。 |
| 7 | CodexRunnerの結果が `sent` の場合は、実行処理側の終端ハンドリングで `キャンセル済み` へ更新し、SSE `canceled` を配信する。 |
| 8 | CodexRunnerの結果が `already_exited` または `not_registered` の場合は、DB状態とプロセス状態を確認し、回答採用前であれば `キャンセル済み` へ整合させる。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- 対象チャットとrunが存在する。
- 対象runが `受付`、`実行中`、`検証中` のいずれかである。

### 6.2. 事後条件

- キャンセル受付成功時はrunが `キャンセル要求中` になる。ただし変更前状態が `受付` の場合は応答時点で `キャンセル済み` まで進む。
- 実行処理が終了要求を反映した後、runは `キャンセル済み` になる。

### 6.3. 不変条件

- `完了`、`エラー`、`タイムアウト`、`キャンセル済み` のrunはキャンセルしない。
- 終端済みrunへのキャンセル要求は状態を変更しない。
- `受付` のrunに対しては、プロセス未登録を正常系として扱い、`CodexRunner.cancel` を呼ばない。
- `not_registered` は必ずしも異常ではなく、起動前キャンセルまたは起動時回復との競合として状態整合を優先する。

## 7. 副作用

| 副作用 | 発生箇所 | 補足 |
| --- | --- | --- |
| DB読込/更新 | Repository | 状態条件付き更新 |
| プロセス終了要求 | CodexRunner | 対象runが `実行中` または `検証中` の場合だけ実行 |
| SSE配信 | RunEventBroker | 状態変更とキャンセル終端 |
| ログ出力 | TraceLogWriter | 要求、成功、失敗 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| 対象runなし | HTTP 404へ変換する |
| キャンセル不可状態 | HTTP 409へ変換する |
| `受付` のキャンセル | CodexRunnerを呼ばずに `キャンセル済み` へ更新する |
| プロセスが既に終了 | `already_exited` として扱い、DB状態を正として終端状態へ整合させる |
| プロセス未登録 | `not_registered` として扱い、受付前後または起動時回復との競合を確認して状態を整合させる |
| DB更新失敗 | rollbackし、HTTP 500へ変換する |

## 9. ログ出力

| タイミング | ログ内容 |
| --- | --- |
| 要求受付 | trace_id、chat_id、run_id |
| 状態更新 | trace_id、run_id、`キャンセル要求中` |
| 終了要求 | trace_id、run_id、プロセス識別情報、cancel_result |
| 受付キャンセル終端 | trace_id、run_id、`キャンセル済み` |
| 例外発生 | trace_id、エラー分類、処理段階 |

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | run状態を `キャンセル要求中` へ更新する直前 |
| 終了点 | `キャンセル要求中` または `キャンセル済み` への状態更新完了 |
| ロールバック条件 | 状態条件付き更新のDBエラー、対象消失、制約違反 |
