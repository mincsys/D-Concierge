# キャンセル処理設計

## 1. 文書の目的

本書は、キャンセル処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: 利用者が未完了runのキャンセルを選択する。
- 本処理は認証済みユーザだけが呼び出せる。
- 関連文書: `docs/03_内部設計/03_内部IF設計/Codex実行IF.md`、`docs/03_内部設計/03_内部IF設計/SSEイベント配信IF.md`

## 3. 処理概要

- 認証済みユーザが所有する対象runがキャンセル可能状態であることを確認し、`cancel_requested` へ更新する。
- 対象runが `accepted` の場合はcodex execプロセスが未登録であるため、`CodexRunner.cancel` を呼ばずに `canceled` へ終端する。
- 対象runが `running` または `validating` の場合はcodex execプロセスへ終了要求を送り、`sent`、`already_exited`、`not_registered` の結果に応じて状態を整合させる。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `presentation/rest` | 認証依存関係、HTTP入力検証、trace_id生成、HTTP応答変換 |
| `CancelChatRunUseCase` | キャンセル受付と終了要求の調停 |
| `RunStatePolicy` | キャンセル可能状態の判定 |
| `ChatRepository` | 状態条件付き更新 |
| `CodexRunner` | 対象プロセスへの終了要求 |
| `RunEventBroker` | `state`、`canceled` イベント配信 |
| `TraceLogWriter` | 例外の記録 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | REST境界でtrace_idを生成し、認証依存関係で現在ユーザを取得する。 |
| 2 | チャットIDとrun IDを検証する。 |
| 3 | Repositoryで認証済みユーザが所有する対象runを取得し、キャンセル可能状態か判定する。 |
| 4 | 状態条件付き更新で `cancel_requested` へ更新する。 |
| 5 | SSE `state` を配信する。 |
| 6 | 変更前状態が `accepted` の場合は、CodexRunnerへ終了要求を送らず、同一ユースケース内で `canceled` へ更新してSSE `canceled` を配信する。 |
| 7 | 変更前状態が `running` または `validating` の場合は、CodexRunnerへ対象プロセスの終了要求を送る。 |
| 8 | CodexRunnerの結果が `sent` の場合は、実行処理側の終端ハンドリングで `canceled` へ更新し、SSE `canceled` を配信する。 |
| 9 | CodexRunnerの結果が `already_exited` または `not_registered` の場合は、DB状態とプロセス状態を確認し、回答採用前であれば `canceled` へ整合させる。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- 対象チャットとrunが存在する。
- 認証済みユーザIDが確定している。
- 対象チャットが認証済みユーザに紐づいている。
- 対象runが `accepted`、`running`、`validating` のいずれかである。

### 6.2. 事後条件

- キャンセル受付成功時はrunが `cancel_requested` になる。ただし変更前状態が `accepted` の場合は応答時点で `canceled` まで進む。
- 実行処理が終了要求を反映した後、runは `canceled` になる。

### 6.3. 不変条件

- `completed`、`error`、`timed_out`、`canceled` のrunはキャンセルしない。
- 終端済みrunへのキャンセル要求は状態を変更しない。
- `accepted` のrunに対しては、プロセス未登録を正常系として扱い、`CodexRunner.cancel` を呼ばない。
- `not_registered` は必ずしも異常ではなく、起動前キャンセルまたは起動時回復との競合として状態整合を優先する。

## 7. 副作用

| 副作用 | 発生箇所 | 補足 |
| --- | --- | --- |
| DB読込/更新 | Repository | 状態条件付き更新 |
| プロセス終了要求 | CodexRunner | 対象runが `running` または `validating` の場合だけ実行 |
| SSE配信 | RunEventBroker | 状態変更とキャンセル終端 |
| ログ出力 | TraceLogWriter | 失敗 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| 未ログイン | 未ログイン応答を返す |
| 対象runなし | HTTP 404へ変換する |
| キャンセル不可状態 | HTTP 409へ変換する |
| `accepted` のキャンセル | CodexRunnerを呼ばずに `canceled` へ更新する |
| プロセスが既に終了 | `already_exited` として扱い、DB状態を正として終端状態へ整合させる |
| プロセス未登録 | `not_registered` として扱い、受付前後または起動時回復との競合を確認して状態を整合させる |
| DB更新失敗 | rollbackし、HTTP 500へ変換する |

## 9. ログ出力

| タイミング | ログ内容 |
| --- | --- |
| 例外発生 | trace_id、chat_id、run_id、エラー分類、例外種別、スタックトレース |

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | run状態を `cancel_requested` へ更新する直前 |
| 終了点 | `cancel_requested` または `canceled` への状態更新完了 |
| ロールバック条件 | 状態条件付き更新のDBエラー、対象消失、制約違反 |
