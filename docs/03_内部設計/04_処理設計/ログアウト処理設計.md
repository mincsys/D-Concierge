# ログアウト処理設計

## 1. 文書の目的

本書は、ログアウト処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: 利用者が設定ダイアログでログアウトを確定し、フロントエンドが `POST /api/auth/logout` を呼び出す。
- ログアウトは現在のCookieトークンに対応するログインセッション行だけを削除する。
- Cookieの破棄はHTTP応答で行う。

## 3. 処理概要

- REST境界でCookieからログインセッショントークンを取得する。
- `LogoutUseCase` がトークン照合用ハッシュを生成し、対応するセッション行を削除する。
- セッションが存在しない場合もログアウト済みとして扱う。
- HTTP応答でCookieを失効させる。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `account_router` | `POST /api/auth/logout` のREST境界 |
| `LogoutUseCase` | ログアウト処理の調停 |
| `SessionTokenProviderPort` | Cookieトークンの照合用ハッシュ生成 |
| `AccountRepositoryPort` | ログインセッション削除 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | REST境界でtrace_idを発行し、Cookieからログインセッショントークンを取得する。 |
| 2 | Cookieが存在する場合、`LogoutUseCase` を呼び出す。 |
| 3 | トークン生値から `token_hash` を生成する。 |
| 4 | Repositoryで対応するログインセッション行を削除する。 |
| 5 | Cookieが存在しない、またはセッション行が存在しない場合も正常扱いにする。 |
| 6 | HTTP応答でCookieを失効させる。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- RepositoryとTokenProviderが注入済みである。

### 6.2. 事後条件

- 現在Cookieに対応するログインセッション行が存在しない状態になる。
- ブラウザ側Cookieが失効する。

### 6.3. 不変条件

- 他ブラウザや他端末のログインセッションは削除しない。
- ユーザ、チャット、成果物データを変更しない。

## 7. 副作用

| 種別 | 対象 | 内容 |
| --- | --- | --- |
| DB削除 | `login_sessions` | 現在Cookieに対応するセッション行削除 |
| Cookie失効 | ブラウザ | ログインセッションCookieを失効 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| Cookieなし | 正常なログアウト済みとして扱う |
| セッションなし | 正常なログアウト済みとして扱う |
| DB削除失敗 | システムエラーとして返し、トレースログへ記録する |

## 9. ログ出力

Cookieなし、セッションなしはトレースログへ記録しない。DB削除失敗、予期しない例外をtrace_id付きで記録する。

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | セッション削除開始時 |
| 終了点 | セッション削除commit時 |
| rollback条件 | DB例外が発生した場合 |
| トランザクション外処理 | Cookie失効、HTTP応答変換 |
