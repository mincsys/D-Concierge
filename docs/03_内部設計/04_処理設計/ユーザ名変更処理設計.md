# ユーザ名変更処理設計

## 1. 文書の目的

本書は、ユーザ名変更処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: 利用者が設定ダイアログのユーザ名変更画面で変更ボタンを押し、フロントエンドが `PATCH /api/account/name` を呼び出す。
- 本処理は認証済みユーザだけが呼び出せる。
- ユーザIDは変更しない。

## 3. 処理概要

- REST境界の認証依存関係で現在ユーザを確定する。
- `ChangeUserNameUseCase` が新しいユーザ名を検証する。
- 対象ユーザが`active`状態で存在する場合だけユーザ名を更新する。
- 更新後のユーザIDとユーザ名を返す。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `account_router` | `PATCH /api/account/name` のREST境界 |
| `get_authenticated_user` | ログインセッション検証 |
| `ChangeUserNameUseCase` | ユーザ名変更の調停 |
| `UserNamePolicy` | ユーザ名入力規則の判定 |
| `AccountRepositoryPort` | ユーザ名更新 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | REST境界でtrace_idを発行し、認証依存関係で現在ユーザを取得する。 |
| 2 | リクエストJSONを検証する。 |
| 3 | `ChangeUserNameUseCase` を呼び出す。 |
| 4 | 新しいユーザ名を `UserNamePolicy` で検証する。 |
| 5 | Repositoryで認証済みユーザの `user_name` を更新する。 |
| 6 | 更新後ユーザをAPI応答へ変換する。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- 認証済みユーザIDが確定している。
- Repositoryが注入済みである。

### 6.2. 事後条件

- `users.user_name` が更新される。
- 更新後のユーザIDとユーザ名が返る。

### 6.3. 不変条件

- ユーザID、パスワードハッシュ、ログインセッション、チャットデータを変更しない。
- `deleting`ユーザのユーザ名は変更しない。

## 7. 副作用

| 種別 | 対象 | 内容 |
| --- | --- | --- |
| DB更新 | `users` | ユーザ名と更新日時を更新 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| 未ログイン | 未ログイン応答を返す |
| ユーザ名入力不正 | `user_name` の項目別入力エラーとして返す |
| 対象ユーザなし | 未ログインまたは対象なしとして返す |
| 対象ユーザが`deleting` | 未ログインとして返す |
| DB更新失敗 | システムエラーとして返し、トレースログへ記録する |

## 9. ログ出力

入力不正、未ログイン、`deleting`ユーザはトレースログへ記録しない。DB更新失敗、予期しない例外をtrace_id付きで記録する。

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | ユーザ名更新開始時 |
| 終了点 | ユーザ名更新commit時 |
| rollback条件 | DB例外が発生した場合 |
| トランザクション外処理 | HTTP応答変換 |
