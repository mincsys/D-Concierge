# チャット実行処理設計

## 1. 文書の目的

本書は、チャット実行処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: 新規チャット開始処理または継続指示受付処理でrunが受付済みになった後。
- 関連文書: `docs/03_内部設計/03_内部IF設計/Codex実行IF.md`、`docs/03_内部設計/03_内部IF設計/SSEイベント配信IF.md`

## 3. 処理概要

- 受付済みrunを実行中へ更新し、`execution_deadline_at` を保存して生成用codex execを起動する。
- 生成、検証、再生成の節目でシステム固定の中間メッセージをDBへ保存し、SSE `message` として即時配信する。
- JSONLイベントの `item.completed.agent_message.text` をJSONとして解析し、`payload.kind="progress"` の場合だけ `payload.text` をSSE `message` として即時配信する。`payload.kind="final"` は中間メッセージとして配信しない。
- 回答検証・再生成処理の判定結果を受け、採用可能なら採用済み回答、参照元、成果物を保存し、再生成指示ならretry回数を増やして生成へ戻る。
- 完了、エラー、タイムアウト、キャンセルのいずれかで終端する。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `ExecuteChatRunUseCase` | 実行全体の調停 |
| `CodexRunner` | 生成用codex exec起動とJSONLイベント取得 |

| `ValidateAnswerUseCase` | 回答候補の検証と再生成判断 |
| `SaveAdoptedArtifactsUseCase` | 採用済み成果物保存 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | 初回実行では受付済みrunを、再生成では検証中runを状態条件付きで `実行中` に更新し、SSEで状態を通知する。`execution_deadline_at = 初回の実行中遷移時刻 + server.timeout_seconds` は初回実行時だけ保存する。 |
| 2 | 初回生成の場合は `作業を開始します。` を、再生成の場合は `回答を修正します。` を中間メッセージとして保存し、SSE `message` として即時配信する。`作業を開始します。` は同一run内で1回だけ配信する。 |
| 3 | `execution_deadline_at` までの残り秒数を計算し、生成用codex execを起動してJSONLイベントを順次解析する。 |
| 4 | `thread.started.thread_id` を受信した場合は、生成用Codex側resume用IDとしてRepositoryへ保存する。 |
| 5 | `item.completed.agent_message` を受信した場合、`item.text` をJSONとして解析する。`payload.kind="progress"` なら `payload.text` を中間メッセージとして保存し、SSE `message` として即時配信する。`payload.kind="final"` は中間メッセージとして保存・配信しない。 |
| 6 | `turn.completed` を受信した場合、最新の `item.completed.agent_message.text` を最終回答候補としてJSON parse、出力スキーマ検証へ進める。 |
| 7 | 最終回答候補を取得したら `作業が完了しました。` を中間メッセージとして保存し、SSE `message` として即時配信する。 |
| 8 | 最終回答候補取得後に `execution_deadline_at` までの残り秒数を計算し、残り時間がある場合だけrunを `検証中` に更新してSSEで状態を通知する。 |
| 9 | `回答の検証を開始します。` を中間メッセージとして保存し、SSE `message` として即時配信してから、回答検証・再生成処理を実行する。検証用 `thread.started.thread_id` を受信した場合は、検証用Codex側resume用IDとしてRepositoryへ保存する。 |
| 10 | 検証結果が `再生成指示` の場合、retry回数を増やし、生成プロンプトへ検証失敗理由を加えて手順1へ戻る。次の生成用codex exec起動前に手順2で `回答を修正します。` を配信する。 |
| 11 | 検証結果が `採用可能` の場合、`回答の検証を完了しました。` を中間メッセージとして保存し、SSE `message` として即時配信してから、採用済み回答、参照元、成果物メタ情報を保存する。 |
| 12 | runを `完了` に更新し、SSE `answer` を配信する。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- 対象runが `受付` 状態で存在する。
- Codex設定、出力スキーマ、作業領域、タイムアウト設定が利用可能である。
- 対象チャットの生成用/検証用Codex側resume用IDが保存済みの場合、Repositoryから取得できる。

### 6.2. 事後条件

- 正常終了時はrunが `完了` になり、採用済み回答と参照元が保存される。
- 異常終了時はrunが `エラー` または `タイムアウト` になり、利用者向けメッセージが保存される。
- キャンセル済みの場合は回答を採用しない。
- 生成用/検証用Codex側resume用IDを取得した場合は、次回resume用に保存される。

### 6.3. 不変条件

- run終端状態は1つだけであり、終端後に中間メッセージや回答を追加しない。
- DB保存を正とし、SSE配信失敗だけで保存済み状態を巻き戻さない。
- 回答採用前に検証を必ず通す。
- システム固定の中間メッセージは、該当する処理を起動する直前または完了直後に保存・配信する。
- 再生成用の生成用codex exec起動は本処理だけが行い、回答検証・再生成処理は再生成指示を返すだけとする。
- retry回数は処理中カウンタとして扱い、上限到達時はrunを `エラー` へ終端する。
- 各codex execへ渡す `timeout_seconds` は、`execution_deadline_at` から現在時刻を引いた残り秒数である。
- Codex由来の中間メッセージは `payload.kind="progress"` の `payload.text` だけから保存・配信する。
- `payload.kind="final"` の生成結果JSONまたは検証結果JSON、コマンド、標準出力、絶対パス、秘密情報は中間メッセージにも回答にも含めない。

## 7. 副作用

| 副作用 | 発生箇所 | 補足 |
| --- | --- | --- |
| DB更新 | Repository | run状態、`execution_deadline_at`、Codex側resume用ID、中間メッセージ、回答、参照元、成果物 |
| codex exec起動 | CodexRunner | 生成用Codexホームと `codex/sessions/<user-id>/<session-id>` を使用 |
| ファイル保存 | FileArtifactStore | 採用済み成果物のみ |
| SSE配信 | RunEventBroker | state、message、answer、error、canceled |
| ログ出力 | TraceLogWriter | 実行段階、例外、再生成 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| codex exec起動失敗 | runを `エラー` に更新し、SSE `error` を配信する |
| JSONL解析失敗 | runを `エラー` に更新し、traceログへ解析対象を要約記録する |
| `turn.failed` またはプロセス異常終了 | 最終回答候補を採用せず、runを `エラー` に更新する |
| 全体deadline超過 | 新しいcodex execを起動せず、runを `タイムアウト` に更新する |
| codex exec単位のタイムアウト | 内部固定のgrace timeoutでプロセス終了を待ち、runを `タイムアウト` に更新する |
| キャンセル要求検知 | ユーザキャンセル要求、終了要求結果、プロセス終了結果を合わせて判定し、runを `キャンセル済み` に更新する |
| 回答検証失敗かつ再生成不可 | retry回数と検証失敗理由をトレースログへ出し、runを `エラー` に更新する |
| 回答採用処理中の予期しない例外 | `処理中にエラーが発生しました。` を利用者向けメッセージとしてrunを `エラー` に更新し、例外種別と例外要約をトレースログへ記録する |

## 9. ログ出力

| タイミング | ログ内容 |
| --- | --- |
| 実行開始 | trace_id、chat_id、run_id、session_id |
| 状態変更 | trace_id、run_id、変更前状態、変更後状態 |
| deadline設定 | trace_id、run_id、execution_deadline_at、server.timeout_seconds |
| Codexイベント受信 | trace_id、run_id、イベント種別 |
| Codex終了 | trace_id、run_id、Codex終了コード、runner種別、プロセス終了理由、残り秒数 |
| 検証結果 | trace_id、run_id、合否、retry回数、検証失敗理由 |
| 再生成 | trace_id、run_id、retry回数、検証失敗理由 |
| 終端 | trace_id、run_id、終端状態、エラー分類 |

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | 各DB状態更新または保存の直前 |
| 終了点 | 各状態更新、メッセージ保存、回答保存の完了ごと |
| ロールバック条件 | 個別DB操作の失敗時。ただしcodex execやSSE配信はDBトランザクションへ含めない |
