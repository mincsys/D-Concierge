# チャット実行処理設計

## 1. 文書の目的

本書は、チャット実行処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: 新規チャット開始処理または継続指示受付処理でrunが受付済みになった後。
- 関連文書: `docs/03_内部設計/03_内部IF設計/Codex実行IF.md`、`docs/03_内部設計/03_内部IF設計/SSEイベント配信IF.md`

## 3. 処理概要

- 受付済みrunを実行中へ更新し、生成用codex execを起動する。
- JSONLイベントから `pending_agent_message` を管理し、中間メッセージと最終回答候補を誤分類しないように抽出する。
- 回答検証・再生成処理を経て、採用済み回答、参照元、成果物を保存する。
- 完了、エラー、タイムアウト、キャンセルのいずれかで終端する。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `ExecuteChatRunUseCase` | 実行全体の調停 |
| `CodexRunner` | 生成用codex exec起動とJSONLイベント取得 |
| `JsonlEventParser` | JSONLを構造化イベントへ変換 |
| `ValidateAnswerUseCase` | 回答候補の検証と再生成判断 |
| `SaveAdoptedArtifactsUseCase` | 採用済み成果物保存 |
| `RunEventBroker` | SSEイベントpublish |
| `ChatRepository` | run状態、中間メッセージ、回答保存 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | 受付済みrunを状態条件付きで `実行中` に更新し、SSEで状態を通知する。 |
| 2 | 生成用codex execを起動し、JSONLイベントを順次解析する。 |
| 3 | `thread.started.thread_id` を受信した場合は、生成用Codex会話継続IDとして保存する。 |
| 4 | `item.completed` の `agent_message` は `pending_agent_message` として保持し、直ちに中間メッセージとして保存しない。 |
| 5 | 後続イベントで処理継続が確認できた場合、直前の `pending_agent_message` をマスク済み中間メッセージに変換し、SSE `message` として配信する。 |
| 6 | `turn.completed` を受信した場合、現在の `pending_agent_message` を最終回答候補としてJSON parse、出力スキーマ検証、`--output-last-message` 照合へ進める。 |
| 7 | 最終回答候補を取得したらrunを `検証中` に更新し、SSEで状態を通知する。 |
| 8 | 回答検証・再生成処理を実行する。 |
| 9 | 採用済み回答、参照元、成果物メタ情報を保存する。 |
| 10 | runを `完了` に更新し、SSE `answer` を配信する。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- 対象runが `受付` 状態で存在する。
- Codex設定、出力スキーマ、作業領域、タイムアウト設定が利用可能である。

### 6.2. 事後条件

- 正常終了時はrunが `完了` になり、採用済み回答と参照元が保存される。
- 異常終了時はrunが `エラー` または `タイムアウト` になり、利用者向けメッセージが保存される。
- キャンセル済みの場合は回答を採用しない。

### 6.3. 不変条件

- run終端状態は1つだけであり、終端後に中間メッセージや回答を追加しない。
- DB保存を正とし、SSE配信失敗だけで保存済み状態を巻き戻さない。
- 回答採用前に検証を必ず通す。
- `pending_agent_message` は常に最大1件だけ保持し、`turn.failed`、異常終了、キャンセル検知時は破棄する。
- `command_execution` のコマンド、標準出力、絶対パス、秘密情報は中間メッセージにも回答にも含めない。

## 7. 副作用

| 副作用 | 発生箇所 | 補足 |
| --- | --- | --- |
| DB更新 | Repository | run状態、中間メッセージ、回答、参照元、成果物 |
| codex exec起動 | CodexRunner | 生成用Codexホームと `codex/sessions/<user-id>/<session-id>` を使用 |
| ファイル保存 | FileArtifactStore | 採用済み成果物のみ |
| SSE配信 | RunEventBroker | state、message、answer、error、canceled |
| ログ出力 | TraceLogWriter | 実行段階、例外、再生成 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| codex exec起動失敗 | runを `エラー` に更新し、SSE `error` を配信する |
| JSONL解析失敗 | runを `エラー` に更新し、traceログへ解析対象を要約記録する |
| `--output-last-message` 照合不一致 | 最終回答候補を破棄し、runを `エラー` に更新する |
| `turn.failed` またはプロセス異常終了 | `pending_agent_message` を破棄し、runを `エラー` に更新する |
| タイムアウト | プロセス終了要求後、runを `タイムアウト` に更新する |
| キャンセル要求検知 | ユーザキャンセル要求、終了要求結果、プロセス終了結果を合わせて判定し、runを `キャンセル済み` に更新する |
| 回答検証失敗かつ再生成不可 | runを `エラー` に更新する |

## 9. ログ出力

| タイミング | ログ内容 |
| --- | --- |
| 実行開始 | trace_id、chat_id、run_id、session_id |
| 状態変更 | trace_id、run_id、変更前状態、変更後状態 |
| Codexイベント受信 | trace_id、run_id、イベント種別 |
| Codex終了 | trace_id、run_id、Codex終了コード、runner種別、プロセス終了理由 |
| 検証結果 | trace_id、run_id、合否、再生成回数 |
| 終端 | trace_id、run_id、終端状態、エラー分類 |

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | 各DB状態更新または保存の直前 |
| 終了点 | 各状態更新、メッセージ保存、回答保存の完了ごと |
| ロールバック条件 | 個別DB操作の失敗時。ただしcodex execやSSE配信はDBトランザクションへ含めない |
