# アカウント削除受付処理設計

## 1. 文書の目的

本書は、アカウント削除受付処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: 利用者が設定ダイアログでアカウント削除を確定し、フロントエンドが `DELETE /api/account` を呼び出す。
- 本処理は削除要求の受付を担当し、作業領域、保存済み成果物実体、DBデータの物理削除完了を待たない。
- 物理削除は [アカウント物理削除処理設計.md](アカウント物理削除処理設計.md) に従って非同期に実行する。

## 3. 処理概要

- REST境界の認証依存関係で削除対象ユーザを確定する。
- Repositoryトランザクション内で対象ユーザを`deleting`へ更新する。
- 対象ユーザに紐づく全チャットを`deleting`へ更新する。
- 対象ユーザの全ログインセッションを削除する。
- `AccountDeletionDispatcher` へ削除ジョブを登録する。
- HTTP応答でCookieを失効させる。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `account_router` | `DELETE /api/account` のREST境界 |
| `get_authenticated_user` | ログインセッション検証 |
| `DeleteAccountUseCase` | 削除受付の調停 |
| `AccountRepositoryPort` | ユーザ状態更新、チャット状態更新、ログインセッション削除 |
| `AccountDeletionDispatcher` | 物理削除ジョブ登録 |
| `TraceLoggerPort` | 削除ジョブ登録失敗の記録 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | REST境界でtrace_idを発行し、認証依存関係で現在ユーザを取得する。 |
| 2 | `DeleteAccountUseCase` を呼び出す。 |
| 3 | Repositoryトランザクション内で対象ユーザを`deleting`へ更新する。 |
| 4 | 対象ユーザに紐づく全チャットを`deleting`へ更新する。 |
| 5 | 対象ユーザの全ログインセッションを削除する。 |
| 6 | トランザクションcommit後、`AccountDeletionDispatcher` へユーザIDとtrace_idを渡して削除ジョブを登録する。 |
| 7 | 登録結果が `registered` または `already_registered` の場合、削除受付応答を返す。 |
| 8 | 登録結果が `failed` の場合、トレースログへ登録失敗を記録し、削除受付応答を返す。 |
| 9 | HTTP応答でCookieを失効させる。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- 認証済みユーザIDが確定している。
- Repository、AccountDeletionDispatcher、TraceLoggerが注入済みである。

### 6.2. 事後条件

- 対象ユーザは`deleting`として保存される。
- 対象ユーザの全チャットは`deleting`として保存される。
- 対象ユーザの全ログインセッションは削除される。
- 物理削除ジョブが登録済み、登録済み扱い、または起動時再登録で再試行可能な状態になる。
- ブラウザ側Cookieが失効する。

### 6.3. 不変条件

- 削除受付成功後は、物理削除失敗があっても対象ユーザを通常状態へ復帰させない。
- 削除受付処理は、作業領域、保存済み成果物実体、DB関連データの物理削除を実行しない。
- 対象ユーザの認証状態は削除受付後に未ログイン扱いになる。

## 7. 副作用

| 種別 | 対象 | 内容 |
| --- | --- | --- |
| DB更新 | `users` | ユーザ状態を`deleting`へ更新 |
| DB更新 | `chats` | 対象ユーザの全チャット状態を`deleting`へ更新 |
| DB削除 | `login_sessions` | 対象ユーザの全ログインセッションを削除 |
| バックグラウンド登録 | `AccountDeletionDispatcher` | 物理削除ジョブ登録 |
| Cookie失効 | ブラウザ | ログインセッションCookieを失効 |
| ログ出力 | トレースログ | 削除ジョブ登録失敗の調査情報 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| 未ログイン | 未ログイン応答を返す |
| 対象ユーザなし | 未ログインまたは対象なしとして返す |
| すでに`deleting` | 削除受付済みとして扱い、物理削除ジョブ登録へ進む |
| 状態更新またはセッション削除失敗 | 削除受付失敗としてシステムエラーへ変換し、トレースログへ記録する |
| 削除ジョブ登録失敗 | トレースログへ記録し、削除受付応答は返す |

## 9. ログ出力

状態更新失敗、セッション削除失敗、削除ジョブ登録失敗をtrace_id付きで記録する。未ログイン、すでに`deleting`の扱いは通常処理として記録しない。

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | ユーザ状態更新開始時 |
| 終了点 | ユーザ状態更新、チャット状態更新、ログインセッション削除commit時 |
| rollback条件 | 状態更新またはセッション削除でDB例外が発生した場合 |
| トランザクション外処理 | `AccountDeletionDispatcher` への登録、登録失敗ログ出力、Cookie失効、HTTP応答変換 |
