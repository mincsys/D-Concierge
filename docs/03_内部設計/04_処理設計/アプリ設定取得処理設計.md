# アプリ設定取得処理設計

## 1. 文書の目的

本書は、アプリ設定取得処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: フロントエンドが起動時にアプリ表示設定を取得する。
- 関連文書: `docs/03_内部設計/03_内部IF設計/設定読込IF.md`

## 3. 処理概要

- 起動時に読み込んだ設定から、画面初期表示に必要な歓迎メッセージ、補足案内文、入力候補だけを返す。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `presentation/rest` | API受付と応答変換 |
| `ConfigLoader` | 型付き設定の提供 |
| `TraceLogWriter` | 例外の記録 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | REST境界でtrace_idを生成する。 |
| 2 | DI済み設定オブジェクトからアプリ表示設定を取得する。 |
| 3 | 画面へ公開してよい項目だけを応答スキーマへ詰め替える。 |
| 4 | アプリ設定応答を返す。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- アプリ起動時の設定読込が成功している。

### 6.2. 事後条件

- 歓迎メッセージ、補足案内文、入力候補が設定値に基づいて返る。
- 未設定項目は空または未指定として返る。

### 6.3. 不変条件

- DB、Codex、ファイルパス、ログ出力先など内部設定を画面へ返さない。
- 設定ファイルをリクエストごとに再読込しない。

## 7. 副作用

| 副作用 | 発生箇所 | 補足 |
| --- | --- | --- |
| 設定参照 | ConfigLoader結果 | 起動時読込済み設定を参照 |
| ログ出力 | TraceLogWriter | 失敗 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| 設定がDIされていない | システムエラーとしてHTTP 500へ変換する |
| 応答変換失敗 | システムエラーとしてtraceログへ記録する |

## 9. ログ出力

| タイミング | ログ内容 |
| --- | --- |
| 例外発生 | trace_id、エラー分類、例外種別、スタックトレース |

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | 本処理ではDBトランザクションを開始しない |
| 終了点 | 設定応答作成完了 |
| ロールバック条件 | DB更新を行わないため該当なし |
