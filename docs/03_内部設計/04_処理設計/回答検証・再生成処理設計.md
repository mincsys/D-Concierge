# 回答検証・再生成処理設計

## 1. 文書の目的

本書は、回答検証・再生成処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: 生成用codex execが最終回答候補を返したとき。
- 関連文書: `docs/03_内部設計/03_内部IF設計/Codex実行IF.md`、`docs/03_内部設計/03_内部IF設計/参照元ファイルIF.md`

## 3. 処理概要

- 回答候補の固定検証を行い、回答ブロック、参照元形式、成果物参照の基本制約を確認する。
- Codex出力スキーマの `locator.start_page` / `locator.end_page` を、内部・API・DBで使用する `locator.page_start` / `locator.page_end` へ正規化する。
- Codex出力スキーマの `locator.path` は `readonly/` から始まるPDFパスとして受け取り、固定検証後は共有データソースルートからの相対パスへ正規化する。
- 検証用codex exec起動前に、正規化済みPDFパスが共有データソース内に存在し、指定ページ範囲が実PDFのページ数内に収まることを確認する。
- 回答本文内の成果物リンクは、`artifacts/...` または `./artifacts/...` 形式、許可拡張子、生成用セッション `artifacts/` 配下の実ファイル存在を固定検証する。
- 回答本文に成果物リンクが含まれる場合は、検証用codex exec起動前に生成用セッション `artifacts/` を検証用セッションへシンボリックリンクとして提示する。
- 参照元pathまたはページ範囲の不備が具体的に判定できる場合は、不正な指定を列挙した再生成指示を返す。
- 参照元PDFが共有データソース内に存在するにもかかわらずPDFとして読み取れない場合は、データソース準備側の問題としてシステムエラーにする。
- 検証用codex execへ元のユーザ指示、回答本文、`readonly/` 付き参照元PDFパス、ページ範囲を含む `ValidatorCodexInput` JSONを渡し、参照元整合性を確認する。
- 検証失敗時は再生成可否を判定し、上限未満なら再生成指示を、上限到達時は失敗結果を呼出元へ返す。
- 本処理は生成用codex execを起動しない。再生成指示を受けた `ExecuteChatRunUseCase` が `実行中` へ戻して生成を再実行する。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `ValidateAnswerUseCase` | 検証順序と再生成判断 |
| `AnswerCandidate` | 回答候補の構造制約 |
| `RetryPolicy` | 再生成可否判断 |
| `CodexRunner` | 検証用codex exec実行 |
| `FileReferenceStore` | PDF参照元の安全確認 |
| `ArtifactLinkValidator` | 成果物リンクの形式、拡張子、存在確認 |
| `TraceLogWriter` | 検証失敗理由と再生成回数の記録 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | 回答候補JSONを構文解析し、Codex出力スキーマに適合することを確認する。 |
| 2 | Codex出力の `locator.path` が `readonly/` から始まるPDFパスであることを確認し、`readonly/` を除去して共有データソースルートからの相対パスへ正規化する。 |
| 3 | Codex出力の `locator.start_page` / `locator.end_page` を `locator.page_start` / `locator.page_end` へ正規化し、`AnswerCandidate` として構築する。 |
| 4 | 必須項目、1件以上の回答ブロック、空でないMarkdown本文、`source_type = pdf`、PDF参照元、ページ範囲、危険HTMLを固定検証する。 |
| 5 | 回答本文内のMarkdownリンク、Markdown画像、HTML `href` / `src` を確認し、成果物リンクは `artifacts/...` または `./artifacts/...` 形式、許可拡張子、生成用セッション内の実ファイル存在を固定検証する。 |
| 6 | 参照元ファイルIFでPDF参照元のパス安全性、存在、読込可否、指定ページ範囲が実PDFのページ数内であることを確認する。PDF実体が存在するのに読み取れない場合はシステムエラーを送出する。 |
| 7 | 成果物リンクがある場合は、検証用セッション作業領域に生成用セッション `artifacts/` へのシンボリックリンクを作成する。 |
| 8 | 検証用codex execへ元のユーザ指示、現在の回答本文、参照元情報を `ValidatorCodexInput` として渡す。参照元pathは検証用作業領域上の `readonly/` 付きPDFパスだけを提示し、共有データソース相対パスは提示しない。 |
| 9 | 検証合格なら採用済み回答として返す。 |
| 10 | 検証不合格で再生成可能なら、検証用Codexの `comment`、または固定検証で組み立てた具体的な再生成指示を呼出元へ返す。 |
| 11 | 再生成上限到達時は検証失敗として呼出元へ返す。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- 生成用codex execから最終回答候補が取得できている。
- 検証用Codexホームと検証プロンプトが利用可能である。
- 再生成上限が設定されている。

### 6.2. 事後条件

- 合格時は採用可能な回答ブロック、ブロックごとの参照元、成果物候補が返る。
- 不合格時は再生成理由または終端エラー理由が返る。

### 6.3. 不変条件

- 固定検証を通過しない回答候補は検証用codex execへ渡さない。
- 検証不合格の回答を採用済み回答として保存しない。
- 再生成回数は設定上限を超えない。
- 検証用Codexの `valid` / `comment` は処理中の一時判定結果であり、DBへ永続化しない。
- 検証用Codexへresumeを使う場合でも、元のユーザ指示、現在の回答候補、参照元情報を検証対象の正として渡し、過去の検証文脈だけに依存しない。
- 固定検証通過後の参照元locatorは `page_start` / `page_end` で扱い、`start_page` / `end_page` を内部・API・DBへ持ち込まない。
- 固定検証通過後の参照元pathは共有データソースルートからの相対パスで扱い、Codex作業領域上の `readonly/` 接頭辞を内部・API・DBへ持ち込まない。
- 成果物リンクは `artifacts/...` または `./artifacts/...` だけを許可し、保存前に実ファイル存在を確認する。
- Markdown画像の成果物は `.svg`、`.png`、`.jpg`、`.jpeg` だけを許可する。通常リンクの成果物は `.svg`、`.png`、`.jpg`、`.jpeg`、`.html`、`.csv` だけを許可する。
- 検証用Codexへ成果物を提示する場合はシンボリックリンクだけを使用し、コピーへのフォールバックは行わない。
- 検証用codex execへ渡す前に、参照元PDFが共有データソース内に存在し、読み取り可能で、指定ページ範囲が実PDFのページ数内であることをアプリケーション側で確認する。
- 参照元pathが `readonly/` 配下の実PDFを指さない場合、または参照元PDFが存在しない場合は、不正な `readonly/... .pdf` 指定を全件列挙して再生成指示に含める。
- 参照元ページ範囲が成立しない場合、または実PDFに指定ページが存在しない場合は、不正なpathとページ範囲を全件列挙して再生成指示に含める。
- 参照元PDFが存在するのに読み取れない場合は、生成用Codexへ再生成させず、呼出元がrunをエラー終端できるシステムエラーとして扱う。
- 生成用codex execの再実行は呼出元の責務であり、本処理の副作用に含めない。

## 7. 副作用

| 副作用 | 発生箇所 | 補足 |
| --- | --- | --- |
| codex exec起動 | CodexRunner | 検証用Codexホームを使用 |
| ファイル読込 | FileReferenceStore | PDF参照元の存在、読込可否、ページ数確認 |
| ログ出力 | TraceLogWriter | 固定検証結果、参照元検証結果、再生成回数 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| 回答候補JSON不正 | 検証失敗とし、再生成可能なら再生成指示を返す |
| 出力スキーマ不一致 | 検証失敗とし、再生成可能なら再生成指示を返す |
| 空回答 | 検証失敗とし、再生成可能なら再生成指示を返す |
| `source_type` が `pdf` 以外 | 検証失敗とし、再生成可能なら再生成指示を返す |
| ページ範囲不正 | 検証失敗とし、再生成可能なら不正なpathとページ範囲を列挙した再生成指示を返す |
| 危険HTML検知 | 検証失敗とし、再生成可能なら再生成指示を返す |
| 参照元pathが `readonly/` 配下ではない | 検証失敗とし、再生成可能なら不正なpathを列挙した再生成指示を返す |
| 参照元が存在しない | 検証失敗とし、再生成可能なら不正なpathを列挙した再生成指示を返す |
| 参照元PDFを読み取れない | システムエラーとして呼出元へ送出し、runをエラー終端する |
| 参照元PDFに指定ページが存在しない | 検証失敗とし、再生成可能なら不正なpathとページ範囲を列挙した再生成指示を返す |
| 成果物リンク形式、拡張子、実ファイル存在の不備 | 検証失敗とし、再生成可能なら不正なリンクを列挙した再生成指示を返す |
| 検証用成果物シンボリックリンク作成失敗 | システムエラーとして呼出元へ送出し、runをエラー終端する |
| 検証用codex exec起動失敗 | 検証処理エラーとして呼出元へ返す |
| 再生成上限到達 | runをエラー終端できる失敗結果を返す |

## 9. ログ出力

| タイミング | ログ内容 |
| --- | --- |
| 固定検証開始 | trace_id、run_id、回答候補ID |
| 固定検証失敗 | trace_id、run_id、失敗項目 |
| 参照元検証失敗 | trace_id、run_id、参照元ID、理由 |
| 参照元PDF読み取り失敗 | trace_id、run_id、対象PDF相対パス、例外種別、例外要約 |
| 検証用成果物提示失敗 | trace_id、run_id、例外種別、例外要約 |
| 再生成判断 | trace_id、run_id、再生成回数、上限、検証失敗理由、検証用Codexのcomment要約 |
| 検証合格 | trace_id、run_id |

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | 本処理単独ではDBトランザクションを開始しない |
| 終了点 | 呼出元の回答保存処理へ検証結果を返す時点 |
| ロールバック条件 | 本処理内の副作用はcodex execとファイル読込であり、DB rollback対象を持たない |
