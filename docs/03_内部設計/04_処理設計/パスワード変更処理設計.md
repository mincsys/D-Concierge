# パスワード変更処理設計

## 1. 文書の目的

本書は、パスワード変更処理の内部手順、副作用、例外、トランザクション境界を定義することを目的とする。

## 2. 前提

- 呼出契機: 利用者が設定ダイアログのパスワード変更画面で変更ボタンを押し、フロントエンドが `PATCH /api/account/password` を呼び出す。
- 本処理は認証済みユーザだけが呼び出せる。
- パスワード変更成功時も、現在のログインセッションと他のログインセッションは維持する。

## 3. 処理概要

- REST境界の認証依存関係で現在ユーザを確定する。
- `ChangePasswordUseCase` が現在のパスワード、新しいパスワード、新しいパスワード確認を検証する。
- 現在のパスワードが一致する場合だけ、新しいパスワードハッシュを保存する。
- 成功時は本文なし応答を返す。

## 4. 関連要素

| 要素 | 役割 |
| --- | --- |
| `account_router` | `PATCH /api/account/password` のREST境界 |
| `get_authenticated_user` | ログインセッション検証 |
| `ChangePasswordUseCase` | パスワード変更の調停 |
| `PasswordPolicy` | 新しいパスワード入力規則の判定 |
| `PasswordHasherPort` | 現在のパスワード検証、新しいパスワードハッシュ生成 |
| `AccountRepositoryPort` | パスワードハッシュ取得、更新 |

## 5. 処理シーケンス

| 順序 | 処理 |
| --- | --- |
| 1 | REST境界でtrace_idを発行し、認証依存関係で現在ユーザを取得する。 |
| 2 | リクエストJSONを検証する。 |
| 3 | `ChangePasswordUseCase` を呼び出す。 |
| 4 | 現在のパスワード、新しいパスワード、新しいパスワード確認の入力を検証する。 |
| 5 | Repositoryから認証済みユーザの保存済みパスワードハッシュを取得する。 |
| 6 | 現在のパスワードと保存済みハッシュを検証する。 |
| 7 | 新しいパスワードハッシュを生成する。 |
| 8 | Repositoryで `users.password_hash` を更新する。 |
| 9 | 本文なし応答へ変換する。 |

## 6. 事前条件 / 事後条件 / 不変条件

### 6.1. 事前条件

- 認証済みユーザIDが確定している。
- RepositoryとPasswordHasherが注入済みである。

### 6.2. 事後条件

- `users.password_hash` が新しいハッシュへ更新される。
- ログインセッションは削除されない。

### 6.3. 不変条件

- パスワード生値をDB、ログ、応答へ出さない。
- 現在のパスワードが不一致の場合、DBを更新しない。
- `削除中`ユーザのパスワードは変更しない。

## 7. 副作用

| 種別 | 対象 | 内容 |
| --- | --- | --- |
| DB更新 | `users` | パスワードハッシュと更新日時を更新 |

## 8. 例外処理

| 条件 | 扱い |
| --- | --- |
| 未ログイン | 未ログイン応答を返す |
| 新しいパスワード入力不正 | `new_password` の項目別入力エラーとして返す |
| 新しいパスワード確認不一致 | `new_password_confirmation` の項目別入力エラーとして返す |
| 現在のパスワード不一致 | `current_password` の項目別入力エラーとして返す |
| 対象ユーザなしまたは`削除中` | 未ログインとして返す |
| ハッシュ生成/検証失敗 | システムエラーとして返し、トレースログへ記録する |
| DB更新失敗 | システムエラーとして返し、トレースログへ記録する |

## 9. ログ出力

入力不正、現在のパスワード不一致、未ログインはトレースログへ記録しない。ハッシュ生成/検証失敗、DB更新失敗、予期しない例外をtrace_id付きで記録する。

## 10. トランザクション

| 項目 | 内容 |
| --- | --- |
| 開始点 | パスワードハッシュ更新開始時 |
| 終了点 | パスワードハッシュ更新commit時 |
| rollback条件 | DB例外が発生した場合 |
| トランザクション外処理 | HTTP応答変換 |
