# 共通設計

## 1. 文書の目的

本書は、D-Conciergeの内部設計において横断的に適用する共通ルールを定義することを目的とする。

## 2. 前提

- 関連文書: `docs/03_内部設計/01_アーキテクチャ設計/アーキテクチャ設計.md`、`docs/03_内部設計/01_アーキテクチャ設計/ディレクトリ構成.md`、`docs/03_内部設計/03_内部IF設計/IF一覧.md`、`docs/03_内部設計/04_処理設計/処理一覧.md`、`docs/03_内部設計/05_データ設計/物理データ設計.md`
- 共通化対象は、複数レイヤまたは複数機能で同じ記述になりやすく、個別実装へ重複させると不整合が起きるものに限定する。
- ドメイン固有の業務判断は `domain/` に置き、HTTP、DB、ファイル、Codex実行などの副作用は共通部品へ混ぜない。

## 3. 対象とする共通事項

| 項目 | 扱う内容 |
| --- | --- |
| エラー | 共通基底例外、エラー分類、利用者向けメッセージ、HTTP応答変換 |
| 認証 / セッション | Cookieセッション、パスワードハッシュ、セッショントークン、認証依存関係 |
| trace_id / トレースログ | API、SSE、ユースケース、異常調査ログを関連付けるIDとYAML記録 |
| ID / 相関ID | チャットID、run ID、参照元ID、成果物ID、session ID、trace IDの生成と受け渡し |
| 時刻 / タイムゾーン | 現在時刻取得、DB保存時刻、画面表示時刻の基準 |
| 設定 | `config.yaml` の型付き読込、必須項目検証、公開範囲 |
| パス安全性 | 共有データソース、Codex作業領域、保存済み成果物領域の正規化と許可範囲判定 |
| 固定状態・種別 | 実行状態、チャット状態、検証結果、SSEイベント種別、Codexイベント種別、成果物リンク種別、参照元種別、エラー分類の内部表現 |
| Markdown / HTML / Mermaid / 画像表示 | 回答表示時の安全化、許可URL、外部レンダラ生成DOM向けスタイル |
| フロントエンド共通境界 | `globals.css`、`components/ui`、`backend_mock` の責務境界 |
| テスト支援値 | 固定ID、固定時刻、Fake実装の配置方針 |

## 4. 共通ルール

### 4.1. エラー

| 項目 | ルール |
| --- | --- |
| 共通基底例外 | 複数レイヤで扱う内部構造化エラーは `src/backend/shared/errors/errors.py` の `AppError` を基底にする。 |
| AppErrorの保持情報 | `AppError` は `error_type`、`trace`、`diagnostic_message`、`cause` だけを保持する。`diagnostic_message` はトレースログへ記録する開発者向け診断文だけを入れ、`trace=False` の場合は空文字にする。 |
| エラー分類 | `src/backend/shared/errors/error_type.py` の `ErrorType` で `input`、`not_found`、`conflict`、`configuration`、`forbidden`、`system` を通常Enumとして持つ。 |
| 利用者向けメッセージ | API、SSE、履歴へ表示・保存する文言は `src/backend/shared/user_messages.py` に集約し、HTTP応答やSSE `error` には内部例外、絶対パス、コマンド、生JSONLを含めない。 |
| 削除関連メッセージ | 削除受付失敗は `チャットを削除できませんでした。時間を置いて再度お試しください。`、`削除中`は `このチャットは削除中のため操作できません。`、削除済み扱いは `このチャットは削除されました。` を利用者へ表示する。 |
| 再生成指示メッセージ | 生成用Codexへ渡す再生成指示文は利用者向け文言ではないため `src/backend/application/validation/instruction_messages.py` で生成し、domain層には持たせない。生成用Codex prompt全体は `src/backend/application/execution/generation_prompt.py` で組み立て、元のユーザ指示と検証による修正指示をタグ付きブロックで分離する。 |
| HTTP応答変換 | `presentation/errors/http.py` が `AppError.error_type` をHTTPステータスとAPIエラーコードへ変換し、例外型または分類から `shared/user_messages.py` の利用者向けメッセージを選択する。 |
| 例外捕捉 | 全REST APIとSSE接続で例外を捕捉し、`AppError.trace=True` のエラーと予期しない例外だけをtrace_id付きでトレースログへ出力する。 |
| domainエラー | domain内の不変条件違反はdomain共通エラーとして表し、presentationへ直接依存しない。 |

### 4.2. trace_id / トレースログ

| 項目 | ルール |
| --- | --- |
| trace_id生成 | REST APIとSSE接続の入口でtrace_idを生成する。将来リクエストヘッダーで受け取る場合も、未指定時は生成する。 |
| trace_id伝搬 | presentationからapplication、infrastructure、trace log writerへ明示的に渡す。 |
| ログ形式 | 1異常1YAMLファイルとし、`occurred_at`、`trace_id`、`event_name`、`stage`、`user_id`、`chat_id`、`run_id`、`error_type`、`message` を基本項目にする。 |
| Codex関連項目 | Codex実行時は `codex_exit_status`、`runner_type`、`os_name`、`process_result`、`execution_deadline_at`、`timeout_state`、`cancel_state`、`retry_count`、`validation_failure_reason`、`validation_comment` を必要に応じて記録する。 |
| 例外関連項目 | 例外時は `exception_type` と `stacktrace` を記録する。 |
| 記録対象 | 正常終了、処理開始、処理完了、ユーザキャンセル、検証不合格による通常の再生成は記録しない。タイムアウト、最大試行回数到達、API/SSE例外、起動時回復失敗、アプリ生成失敗は記録する。 |
| AppErrorの記録判断 | 共通ハンドラでは `AppError.trace=True` の場合だけ `diagnostic_message` を `message` に保存する。入力不正、対象なし、表示不可、キャンセル不可、通常の競合は原則として `trace=False` とし、トレースログへ出さない。 |
| 削除関連エラー | `ChatDeletingError` は通常競合として扱い、原則として `trace=False` とする。削除受付失敗、削除ジョブ登録失敗、物理削除失敗、DB削除失敗はトレースログ対象にする。 |
| 文字列上限 | 開発者向け調査情報はマスクしない。`message`、`validation_comment`、`request_validation_errors` などは64KiB、`stacktrace` は1MiBを上限に切り詰める。 |
| ログ失敗 | トレースログ書込失敗は元処理のエラー分類を上書きしない。 |
| 配置 | 実装は `src/backend/shared/tracing/` と `src/backend/infrastructure/trace_log/` に分け、値と出力副作用を混ぜない。 |

### 4.3. ID / 相関ID

| 項目 | ルール |
| --- | --- |
| ID種別 | `ChatId`、`RunId`、`UserInstructionId`、`AnswerBlockId`、`ReferenceId`、`ArtifactId`、`SessionId`、`TraceId` を意味のある型として扱う。 |
| 発番 | 本番のID発番は `src/backend/infrastructure/runtime/UuidGenerator` が行い、applicationはport経由で利用する。 |
| 値の使い回し禁止 | チャットID、run ID、Codex側resume用ID、session ID、trace_idを同じ値として代用しない。 |
| 画面公開 | 画面へ公開するIDはチャットID、run ID、参照元ID、成果物IDに限定し、内部絶対パスやプロセスIDは公開しない。 |
| 型配置 | ドメインID値は `src/backend/domain/shared/ids/`、複数レイヤを接続するtrace_idは `src/backend/shared/tracing/` に置く。 |

### 4.4. 時刻 / タイムゾーン

| 項目 | ルール |
| --- | --- |
| 現在時刻取得 | application層は時刻ライブラリを直接呼ばず、Clock portを経由する。 |
| DB保存 | DBへ保存する時刻、実行deadline、タイムアウト計算はUTC基準の `timestamptz` とする。 |
| 運用者向け日時 | トレースログの日付ディレクトリ、ファイル名、発生日時、保存期間判定は `app.timezone` 基準とする。 |
| 表示 | 履歴一覧や履歴詳細で返す時刻はAPI応答変換時にISO 8601文字列へ変換する。 |
| テスト支援 | 固定時刻は `src/backend/tests/support/` に置き、本番実装と混在させない。 |

### 4.5. 設定

| 項目 | ルール |
| --- | --- |
| 読込 | `src/backend/infrastructure/config/ConfigLoader` がYAMLを読み込み、型付き設定へ変換する。 |
| 検証 | 必須項目、数値範囲、パス、Codexホーム、出力スキーマ、ログ出力先は起動時に検証する。 |
| 公開範囲 | `/api/app-config` は歓迎メッセージと入力候補だけを返し、DB接続、ファイルパス、Codex設定、ログ設定は返さない。 |
| 広いdict禁止 | application層へ設定を渡す場合は、用途別の設定型を使い、生の `dict` を渡さない。 |

### 4.6. パス安全性

| 項目 | ルール |
| --- | --- |
| 正規化 | ファイルアクセス前に `PathSecurityService` で正規化し、許可ルート配下であることを確認する。 |
| 対象ルート | 共有データソース、Codex作業領域、保存済み成果物領域を別の許可ルートとして扱う。 |
| 禁止入力 | 絶対パス、UNC、URL、親ディレクトリ参照、空パス、NUL文字、許可外拡張子を拒否する。 |
| セッション作業領域 | 生成用は `codex/sessions/<user-id>/<session-id>/{readonly,tmp,artifacts}`、検証用は `codex/sessions_validator/<user-id>/<session-id>/{readonly,tmp}` を許可ルートとして扱う。`readonly/` は共有データソースディレクトリへのシンボリックリンクとして提示する。 |
| OS差異 | アプリが実ファイル参照として扱う値はPOSIX相対形式へ標準化し、`readonly\...` や `artifacts\...` の区切り文字差分だけを安全な範囲で `/` へ正規化する。Windows絶対パスやUNCは参照先として解釈しない。 |
| 成果物保存 | セッション内 `artifacts/` は一時領域とし、採用済み回答が参照するファイルだけを保存済み成果物領域へコピーする。 |
| 保存済み成果物領域 | `generator.saved_artifacts_dir/<user-id>/<run_id>/<artifact_id>.<拡張子>` を許可形式とし、DBには保存済み成果物領域からの相対参照を保持する。 |
| 作業領域削除 | チャット物理削除では、生成用・検証用の `<workdir>/<user-id>/<session-id>` だけを削除する。アカウント物理削除では、生成用・検証用の `<workdir>/<user-id>` だけを削除する。`readonly/` シンボリックリンクは削除対象に含めるが、リンク先の共有データソース実体は削除しない。 |
| 成果物削除 | チャット物理削除では、`generator.saved_artifacts_dir` 配下へ正規化できる対象 `storage_path` と空の親runディレクトリだけを削除する。アカウント物理削除では、`generator.saved_artifacts_dir/<user-id>` だけを削除する。 |
| 画面公開 | 実ファイルの絶対パスはAPI応答、SSEへ出さない。トレースログは開発者向け調査情報として絶対パスを保存してよい。 |
| 配置 | パス安全性の実装は `src/backend/infrastructure/filesystem/path_security/` に置き、ファイル入出力実装から共通利用する。 |

### 4.7. 固定状態・種別

| 項目 | ルール |
| --- | --- |
| 内部表現 | ユーザ状態、実行状態、チャット状態、検証結果、dispatcher結果、Codex終了要求結果、SSEイベント種別、Codex JSONLイベント種別、成果物リンク種別、参照元種別、Codex出力種別、エラー分類は通常の `Enum` メンバーとして扱う。 |
| 配置 | 固定状態・結果・種別のEnumは、利用する処理や概念の所有者に近い専用モジュールへ定義し、判定ロジックやDTO定義と同じファイルへ混在させない。 |
| 文字列化境界 | DB保存、API応答、SSE送信、Codex JSONL、Codex出力JSON、Markdown/HTML解析、トレースログなど外部仕様として文字列を扱う箇所だけ、`.value` による文字列化または `Enum(value)` による復元を明示する。 |
| presentation境界 | presentationはAPI応答とSSE送信のpayload変換でdomainのEnumや値オブジェクトを参照し、送信直前に `.value` へ変換する。 |
| 内部分岐 | application、domain、infrastructure内部の比較、分岐、代入では文字列リテラルを直接使わず、`RunState.COMPLETED` などのEnumメンバーを使う。presentationもAPI/SSE契約変換以外の業務判断を持たない。 |
| DB復元 | RepositoryはDBから読み取った状態値・種別値をEnumへ復元し、定義外の値はデータ不整合として扱う。 |
| 外部仕様 | 外部設計、DB制約、API/SSE wire形式、Codex入出力スキーマで定義する値は従来どおり文字列仕様とする。 |
| チャット状態 | `ChatState` は `ACTIVE = "有効"`、`DELETING = "削除中"` を持ち、`src/backend/domain/chat/chat_state.py` に定義する。 |
| ユーザ状態 | `UserState` は `ACTIVE = "通常"`、`DELETING = "削除中"` を持ち、`src/backend/domain/account/user_state.py` に定義する。 |
| 実行状態との分離 | チャット状態はチャット単位のライフサイクル状態として扱う。チャット実行処理の状態列挙には含めない。 |

### 4.8. Markdown / HTML / Mermaid / 画像表示

| 項目 | ルール |
| --- | --- |
| Markdown | 回答ブロック本文のMarkdownは `MarkdownRenderer` でsanitizeして表示する。 |
| HTML許可要素 | Markdown内HTMLは `table`、`thead`、`tbody`、`tr`、`th`、`td`、`p`、`ul`、`ol`、`li`、`strong`、`em`、`code`、`pre`、`a` を許可する。 |
| HTML禁止要素 | `script`、`iframe`、`object`、`embed` は表示しない。 |
| HTML禁止属性 | すべてのイベント属性、`javascript:` URL、任意コード実行や外部通信につながる属性は除去する。 |
| Mermaid | Mermaidは `securityLevel: "strict"` で描画し、失敗時は図を表示できない旨だけを表示する。 |
| 画像 | 画像URLは同一オリジンかつ `/api/artifacts/` 配下だけを許可する。 |
| PDF | 本システムの参照元ビューアはPDFのみを対象にする。 |

### 4.9. フロントエンド共通境界

| 項目 | ルール |
| --- | --- |
| `globals.css` | Tailwind/shadcnテーマ、body初期設定、疑似要素、Markdown/Mermaid/PDF.jsなど外部ライブラリやレンダラ生成DOM向けCSSだけを置く。 |
| コンポーネントCSS | 通常のレイアウト、余白、色、枠線、影、状態表示はReactコンポーネント側のTailwind `className` へ記述する。 |
| `components/ui` | shadcn/ui由来の汎用UIラッパーとして扱い、業務状態やAPI通信を持たせない。 |
| `backend_mock` | フロントエンド単体確認用資産として維持するが、UIコンポーネントと `features/*/api/` から直接参照しない。 |
| モック通信 | モック利用時はVite middlewareが `/api/...` を応答し、画面側の通信パスとAPI境界を本番接続時と同一にする。 |
| API境界 | フロントエンドの通信は `src/frontend/src/features/*/api/` から `/api/...` へ行う。 |

### 4.10. テスト支援値

| 項目 | ルール |
| --- | --- |
| バックエンド | 固定ID、固定時刻、Fake Repository、Fake Codex Runnerは `src/backend/tests/support/` に置く。 |
| フロントエンド | fetch/EventSourceモック、render helper、fixtureは `src/frontend/tests/support/` に置く。 |
| 本番混入防止 | テスト支援実装は `src/backend/infrastructure/runtime/` や `src/frontend/src/` の本番モジュールから参照しない。 |

### 4.11. 認証 / セッション

| 項目 | ルール |
| --- | --- |
| 認証方式 | D-Concierge内のローカル認証とする。ブラウザCookieにログインセッショントークンを保持し、保護対象APIはpresentation層の認証依存関係で検証する。外部IdP連携は行わない。 |
| Cookie値 | Cookieには推測困難なトークン生値だけを入れ、画面モデル、DB、トレースログへ保持しない。 |
| DB照合 | `login_sessions.token_hash` にトークン照合用ハッシュを保存し、Cookie値の生値は保存しない。 |
| 有効期限 | `login_sessions.expires_at` を保存し、認証状態確認時に期限切れセッションを未ログイン扱いにして削除する。アプリケーション起動時にも期限切れセッションを削除する。 |
| ログアウト | 現在Cookieに対応するログインセッション行を削除し、HTTP応答でCookieを失効させる。 |
| アカウント削除 | 対象ユーザの全ログインセッション行を削除し、HTTP応答でCookieを失効させる。 |
| パスワードハッシュ | パスワードハッシュ生成・検証は `PasswordHasherPort` を経由し、具象実装はpasslibとbcryptを使用する。 |
| セッショントークン | セッショントークン発行と照合用ハッシュ生成は `SessionTokenProviderPort` を経由し、具象実装は `secrets` を使用する。 |
| 項目別エラー | domain/application内部の項目キーは実装内の意味で扱い、presentation層でAPI項目名へ変換して `field_errors` へ設定する。 |
| 保護対象ユースケース | チャット系ユースケースには認証済みユーザIDを明示的に渡し、Repositoryでユーザ所有範囲を条件に含める。 |

### 4.12. フロントエンド単体確認用モック

| 項目 | ルール |
| --- | --- |
| 管理対象 | `backend_mock` はアカウント状態、ログインセッション、ユーザ別チャット履歴を `/api/...` 境界の内側で保持する。 |
| デモユーザ | 開発サーバ起動直後の初回認証状態確認でCookieがない場合、`demo-user`、ユーザ名 `デモユーザ`、パスワード `password` のログイン済み状態を作る。 |
| 自動ログイン抑止 | ログアウトまたはアカウント削除後はデモユーザの自動ログインを再発行せず、ログイン画面を表示する。 |
| ユーザ別履歴 | デモユーザには既存モック履歴を紐づける。アカウント登録で作成した新規ユーザは、登録直後はチャット履歴を持たない。 |
| モック削除 | アカウント削除時は、ユーザ、ログインセッション、ユーザ別チャット履歴をモック状態から削除し、同じユーザIDで再登録できる状態にする。 |

## 5. 留意事項

- 共通化は重複を避けるために行うが、業務固有の判断を過度に `shared` へ寄せない。
- `src/backend/shared/` は複数レイヤを接続する横断契約に限定し、ドメイン値は `src/backend/domain/shared/`、副作用実装は `src/backend/infrastructure/` に置く。
- ロールや権限管理を導入する場合は、認可共通設計を本書へ追加する。
