# Test Review Checklist

## テスト方針との整合

- [x] テストコードとテスト仕様が、対応するテスト方針の範囲指定、テスト対象単位、除外範囲に一致しているか。
  - 検証結果: 指摘あり
  - 確認根拠: `docs/04_テスト/03_結合テスト/結合テスト方針.md` は REST API、DB、ファイル、トレースログ連携を結合対象にしているが、`src/backend/tests/integration/test_deletion_recovery_trace_api.py` は削除受付と起動時セッション掃除中心で、物理削除のDB/ファイル境界と起動時再登録を観測していない。
- [x] テスト関連成果物のディレクトリ構成が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 問題なし
  - 確認根拠: 単体テストは `src/backend/tests/unit/application/...`、結合テストは `src/backend/tests/integration/test_deletion_recovery_trace_api.py` に配置されており、単体/結合テスト方針の配置ルールに一致している。
- [x] テスト関連成果物のファイル構成、ファイル名、配置先が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 問題なし
  - 確認根拠: 追加ファイルは `test_chat_deletion_use_cases.py`、`test_account_deletion_recovery_use_cases.py`、`test_deletion_recovery_trace_api.py` で、対象機能単位を表す `test_*.py` 構成になっている。
- [x] 単体、結合、総合テストの役割が混ざっていないか。
  - 検証結果: 問題なし
  - 確認根拠: 単体テストはFakeでUseCase/port契約を確認し、結合テストはASGI、PostgreSQL、TraceLogWriter経由のAPI/起動時境界を確認している。総合テスト相当のブラウザ操作は含まれていない。
- [x] テスト方針で求める観点、カバレッジ、証跡、実行環境、完了条件が満たされているか。
  - 検証結果: 指摘あり
  - 確認根拠: 生成役報告では Red は成立しているが、F007 の物理削除、起動時再登録、削除中通常操作対象外、正式TraceLogRecord契約に対するテスト観点が不足している。
- [x] テスト方針が実装や設計の現状に追従していない場合は、テスト方針側を直す方針を書く。
  - 検証結果: 対象外
  - 確認根拠: 今回確認した不整合はテストコード側の不足・契約ずれであり、単体テスト方針・結合テスト方針の変更が必要な事象ではない。

## 単体テスト

- [x] 公開関数、公開メソッド、コンポーネント、純粋ロジックの責務を単位としているか。
  - 検証結果: 指摘あり
  - 確認根拠: `DeleteChatUseCase`、`ExecuteChatDeletionUseCase`、`ExecuteAccountDeletionUseCase`、`RecoverDeletingAccountsUseCase` は単位化されているが、チャット物理削除の未完了run判定がDTO契約外のFakeメソッドに依存している。
- [x] 外部副作用を Fake、Stub、Mock、fixture で差し替えているか。
  - 検証結果: 指摘あり
  - 確認根拠: DB、dispatcher、cancel、filesystem はFake化されているが、trace logger fake が正式な `TraceLogRecord` / `write(record)` 境界ではなく `write_chat_event` / `write_account_event` を要求している。
- [x] 事前条件、事後条件、不変条件、正常系、異常系、境界値、分岐が確認されているか。
  - 検証結果: 指摘あり
  - 確認根拠: 削除受付成功/dispatcher失敗/対象なし、物理削除延期/正常/ファイル失敗、回復成功/失敗はあるが、DB削除失敗、対象なし正常終了、`not_registered` 準正常、正式TraceLogRecord項目、削除中通常操作対象外の全境界が不足している。
- [x] テストが内部実装詳細に密結合しすぎていないか。
  - 検証結果: 指摘あり
  - 確認根拠: `FakeChatDeletionRepository.list_unfinished_runs_for_chat_deletion()` は設計・port契約にないテスト内メソッドであり、実装をこの内部形状へ誘導する。
- [x] テストが単に例外が出ないことではなく、仕様上意味のある結果、状態変化、副作用、出力を検証しているか。
  - 検証結果: 問題なし
  - 確認根拠: 削除受付応答、DB状態相当のFake記録、dispatcher呼び出し、ファイル削除Fake、traceイベント記録を具体値で検証している。

## 結合テスト

- [x] 外部インターフェース、処理設計、永続化、ファイル、通信境界、表示または操作連携を設計単位で確認しているか。
  - 検証結果: 指摘あり
  - 確認根拠: `DELETE /api/chats/{chat_id}` のHTTP契約と一部DB状態は確認しているが、物理削除のDB/ファイル境界、アカウント起動時再登録、削除中チャットの継続指示/SSE/参照元/成果物拒否が未確認。
- [x] 実外部サービス、実行環境、実行時間や環境差分が大きい対象を不必要に使っていないか。
  - 検証結果: 問題なし
  - 確認根拠: 結合テストはASGITransport、PostgreSQLテストDB、一時ディレクトリを使い、実Codexやブラウザは使っていない。
- [x] Mock、Fake、Stub が過剰で、実際の契約違反や結合不備を隠していないか。
  - 検証結果: 指摘あり
  - 確認根拠: チャット削除dispatcher fake は妥当だが、起動時アカウント回復では dispatcher を観測しておらず、物理削除は結合テストがなくFake単体だけのためRepository/ファイル/trace writerの契約違反を隠す。
- [x] トランザクション、状態更新、競合、ロールバック、エラー変換、ログ依頼が確認されているか。
  - 検証結果: 指摘あり
  - 確認根拠: チャット削除受付の状態更新とdispatcher失敗時ログは確認しているが、物理削除のファイル削除後DB削除、DB削除失敗時rollback/trace、起動時回復登録失敗traceが結合層で未確認。
- [x] テストデータが独立し、実行順序に依存していないか。
  - 検証結果: 問題なし
  - 確認根拠: 各結合テストは `prepare_foundation_database()` と `tmp_path` を使い、固定IDと専用初期データで独立している。
- [x] 時刻、乱数、実行順序、外部通信、ファイルシステム、DB 状態により不安定になるテストになっていないか。
  - 検証結果: 問題なし
  - 確認根拠: 固定ID、`FIXED_CHAT_NOW`、`tmp_path`、テストDB初期化を使っており、実外部通信に依存していない。
- [x] テストデータが実装ロジックのコピーや都合のよい値だけになっておらず、境界値、異常値、業務上意味のある値を含んでいるか。
  - 検証結果: 指摘あり
  - 確認根拠: dispatcher失敗と期限切れセッションは業務上意味があるが、削除中チャットへの継続/SSE/参照元/成果物拒否、物理削除のファイル欠損・削除失敗・DB失敗などの境界値が不足している。
