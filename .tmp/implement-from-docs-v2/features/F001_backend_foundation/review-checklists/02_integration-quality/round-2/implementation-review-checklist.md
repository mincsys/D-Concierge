## 責務分割と依存方向

- [x] 層、機能、モジュール、コンポーネントの責務が設計と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `domain` はEnum、`application/ports` はProtocol/DTO、`infrastructure` は副作用実体、`presentation` はHTTP境界に分離されている。
- [x] 副作用を持つ処理が定義済みの境界へ閉じているか。
  - 検証結果: 指摘なし
  - 確認根拠: DB接続、Alembic、パス正規化、トレースログ書込、設定ファイル読込は `infrastructure` または `app` composition root に閉じている。
- [x] 業務判断やドメイン判断が、表示層、入出力層、インフラ層、設定読込など不適切な場所へ漏れていないか。
  - 検証結果: 指摘なし
  - 確認根拠: F001範囲では状態値定義のみで、状態遷移などの業務判断は未実装またはPort境界に留まる。
- [x] 純粋ロジックが永続化、通信、ファイル、時刻、ID、外部プロセスなどの副作用へ直接依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: `domain` と `application/ports` にFastAPI/SQLAlchemy/infrastructure importがないことを本文とアーキテクチャ契約テストで確認した。
- [x] 上位層が下位層の実装詳細、テスト用実体、内部データ形式へ直接依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: presentationは設定型と認証依存を利用し、application portはSQLAlchemyモデルではなくDTO/Protocolを公開している。
- [x] 依存方向、公開範囲、再利用単位が設計や開発標準と矛盾していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 具象実装の組み立ては `src/backend/app/factory.py` にあり、下位実装をdomain/applicationへ逆流させていない。
- [x] 不要な依存関係、未使用コード、到達不能コード、暫定的な分岐、デバッグ用処理が残っていないか。
  - 検証結果: 指摘なし

## エラー・ログ・セキュリティ

- [x] 利用者向けメッセージと内部調査用情報が分離されているか。
  - 検証結果: 指摘なし
  - 確認根拠: REST応答は `error`/`message` の利用者向けpayloadで、内部診断は `TraceLogRecord.message` とstacktraceへ分離されている。
- [x] 内部パス、秘密情報、スタックトレース、認証情報、個人情報などが利用者向け出力、外部応答、ログ、証跡へ露出していないか。
  - 検証結果: 指摘なし
  - 確認根拠: `/api/app-config` 結合テストはDB接続、Codex設定、trace_log.dirを応答へ返さないことを確認している。
- [x] URL、ポート、ファイルパス、認証情報、環境名、外部サービス名などが不適切にハードコードされていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 本番設定値は `config.yaml` 読込に集約され、テスト固定値は `src/backend/tests/support/` に限定されている。
- [x] エラー分類、追跡情報、状態更新、後続処理の継続または抑止が設計と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `AppError` と `HTTPException` は共通RESTエラーへ変換され、`trace=True` と予期しない例外のみトレースログへ保存される。
- [x] ファイルパス、URL、コンテンツ種別、マークアップ、コマンド、外部入力などの検証が境界で行われているか。
  - 検証結果: 指摘なし
  - 確認根拠: `PathSecurityService` は絶対パス、UNC、URL、親ディレクトリ、NUL、許可外拡張子、symlink逸脱を拒否する。
- [x] 認証、認可、入力制限、出力制御、監査記録が対象システムの設計と整合しているか。
  - 検証結果: 指摘なし
  - 確認根拠: F001範囲の `/api/app-config` は `get_authenticated_user` 依存を通り、未認証時は401共通エラー形式になる。
