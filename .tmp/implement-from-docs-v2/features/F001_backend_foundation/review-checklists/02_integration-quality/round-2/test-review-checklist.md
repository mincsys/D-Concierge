# Test Review Checklist

## テスト方針との整合

- [x] テストコードとテスト仕様が、対応するテスト方針の範囲指定、テスト対象単位、除外範囲に一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: F001 backend 範囲の単体・結合テストとして配置は妥当で、結合テストDB初期化の接続先検証も最新の `src/backend/tests/support/foundation.py` で確認した。
- [x] テスト関連成果物のディレクトリ構成が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: backend単体は `src/backend/tests/unit/`、結合は `src/backend/tests/integration/`、支援コードは `src/backend/tests/support/` に配置されている。
- [x] テスト関連成果物のファイル構成、ファイル名、配置先が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: Pythonテストは `test_*.py` で、F001の設定、共通エラー、DB、Repository、アーキテクチャ、REST、migration、trace境界に対応している。
- [x] 単体、結合、総合テストの役割が混ざっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体はメタデータ・設定・共通部品の契約を直接確認し、結合はFastAPI/DB/Alembic/trace log連携を確認している。ブラウザや総合テスト観点は含めていない。
- [x] テスト方針で求める観点、カバレッジ、証跡、実行環境、完了条件が満たされているか。
  - 検証結果: 指摘あり
  - 確認根拠: 履歴一覧索引降順、TraceLogWriter同日上限、テストDB URL誤設定拒否の観点は追加済み。ただし最新の単体coverage evidenceコマンドは分岐85.71%で方針値95%を下回り、証跡も最新結果と一致しない。
- [x] テスト方針が実装や設計の現状に追従していない場合は、テスト方針側を直す方針を書く。
  - 検証結果: 指摘なし
  - 確認根拠: 不足はテスト方針ではなく実装・テストコード側の不足として判断した。

## 単体テスト

- [x] 公開関数、公開メソッド、コンポーネント、純粋ロジックの責務を単位としているか。
  - 検証結果: 指摘なし
  - 確認根拠: `ConfigLoader`、`AppError`、`new_trace_id`、`PathSecurityService`、DBモデルmetadata、Repository Portを公開責務単位で確認している。
- [x] 外部副作用を Fake、Stub、Mock、fixture で差し替えているか。
  - 検証結果: 指摘なし
  - 確認根拠: 設定ファイルやPathは `tmp_path`、UUID発番は `FixedUuidGenerator`、実DBは単体では使っていない。
- [x] 事前条件、事後条件、不変条件、正常系、異常系、境界値、分岐が確認されているか。
  - 検証結果: 指摘あり
  - 確認根拠: 設定欠落、数値範囲、PathSecurity、DB制約、履歴一覧索引降順、TraceLogWriter起動後カウンタ、テストDB URL誤設定拒否は確認済み。ただし最新coverageでは `TraceLogWriter.prune_expired` と `_unique_log_path` の分岐が未通過。
- [x] テストが内部実装詳細に密結合しすぎていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 契約テストはdocs上の公開契約やSQLAlchemy metadata/Alembic inspect結果を見ており、private実装の逐語的固定には寄っていない。
- [x] テストが単に例外が出ないことではなく、仕様上意味のある結果、状態変化、副作用、出力を検証しているか。
  - 検証結果: 指摘なし
  - 確認根拠: エラー分類、HTTP payload、DB制約違反、trace log YAML項目、公開設定の非漏えいを具体的にassertしている。

## 結合テスト

- [x] 外部インターフェース、処理設計、永続化、ファイル、通信境界、表示または操作連携を設計単位で確認しているか。
  - 検証結果: 指摘なし
  - 確認根拠: F001範囲の `/api/app-config`、REST共通エラー、trace log、Alembic migration、PostgreSQL制約が結合テストで確認されている。
- [x] 実外部サービス、実行環境、実行時間や環境差分が大きい対象を不必要に使っていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 実Codex、実ブラウザ、外部ネットワークは使用していない。DBは方針上のPostgreSQLテストDBを使用する。
- [x] Mock、Fake、Stub が過剰で、実際の契約違反や結合不備を隠していないか。
  - 検証結果: 指摘なし
  - 確認根拠: `/api/app-config` の認証依存だけをテスト用にoverrideし、HTTP middleware、設定読込、Pydantic/dataclass応答、trace_idヘッダーは実経路で確認している。
- [x] トランザクション、状態更新、競合、ロールバック、エラー変換、ログ依頼が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: F001範囲では未完了run部分UNIQUE、DB制約違反、RESTエラー変換、設定失敗時ログ、起動時ログ削除を確認している。後続ユースケースの状態更新はF001外。
- [x] テストデータが独立し、実行順序に依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: `prepare_foundation_database` は `_validate_foundation_test_database_url` でDB種別、DB名、ユーザ、ポートを検証してから `DROP SCHEMA` を実行する。
- [x] 時刻、乱数、実行順序、外部通信、ファイルシステム、DB 状態により不安定になるテストになっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: UUIDは固定値を使い、trace logの日付だけ実行時刻依存だが判定は存在/項目/対応関係で確認している。検証役実行で結合28件は再現して成功した。
- [x] テストデータが実装ロジックのコピーや都合のよい値だけになっておらず、境界値、異常値、業務上意味のある値を含んでいるか。
  - 検証結果: 指摘なし
  - 確認根拠: 未認証、不正Cookie、設定欠落、invalid timezone、Windows絶対パス、UNC、URL、親ディレクトリ、DB重複/不正状態などの異常値を含む。

## 総合テスト

- [x] 利用者が利用者インターフェースまたは実運用に近い経路で業務を完了できることを確認しているか。
  - 検証結果: 対象外
  - 確認根拠: 今回はF001 backendの結合テスト完了検証・実装品質レビューであり、総合テスト工程とfrontend UI確認は対象外。
- [x] 自動確認、手動確認、通信確認、永続化確認、ファイル確認の役割がテスト方針と一致しているか。
  - 検証結果: 対象外
  - 確認根拠: 総合テスト証跡は今回のレビュー対象外。backend結合テストは自動確認として扱った。
- [x] 未実施、一部確認、保留、不合格、再テストの記録がテスト仕様・結果に残っているか。
  - 検証結果: 対象外
  - 確認根拠: 総合テスト仕様・結果ファイルのレビューではない。
- [x] 実行証跡とテストケースが追跡できるか。
  - 検証結果: 対象外
  - 確認根拠: 総合テストケースID単位の証跡は対象外。backend単体・結合evidenceはEvidence checklistで確認した。

## TDD と実装順序

- [x] 実装成果物レビューでは、TDD が要求される範囲で Red、Green、Refactor の証跡または説明があるか。
  - 検証結果: 指摘なし
  - 確認根拠: 生成役報告でF001テストRed、実装後Green、静的テストが提示され、検証役再実行でも単体68件・結合28件が成功した。coverage未達と証跡更新漏れは別項目で指摘した。
- [x] Red を事後的に作れない場合は、レビュー指摘として記録し、今後の修正方針を示す。
  - 検証結果: 指摘なし
  - 確認根拠: F001のRed成立報告は前工程から引き継がれており、事後作成不能とは判断していない。
- [x] テスト追加が不要な場合は、除外理由が方針と整合しているか。
  - 検証結果: 指摘なし
  - 確認根拠: backend範囲外のfrontend総合テストは除外理由がユーザー指定と方針に合う。新規指摘1件は単体テスト補強と証跡更新が必要。

  - 検証結果: 指摘なし
  - 確認根拠: docstringは仕様上の確認観点と期待結果を説明している。
- [x] テスト失敗理由、作成時の都合、環境制約、ツール制約、レビュー指摘への反応など、成果物ではなく作業報告や issue に書くべき内容が混入していないか。
  - 検証結果: 指摘なし
  - 確認根拠: テスト本文と証跡に「指摘されたため」などの作業報告表現は見当たらない。
- [x] `既存実装があるので失敗しない`、`一旦この実装にしている`、`あとで修正する`、`指摘されたため追加`、`カバレッジのため追加`、`環境の都合で確認できない`、`旧仕様ではこうだった`、コードの単なる日本語言い換えなどが残っていないか。

## 修正方針の判断

- [x] テスト不足ならテストコード、テスト仕様、証跡のどれを補うかを書く。

- [x] テストが仕様に従っていて実装が違う場合は、実装側を直す方針を書く。
  - 検証結果: 指摘なし

- [x] テスト方針が過剰または古い場合は、テスト方針と関連成果物を直す方針を書く。
  - 検証結果: 指摘なし
  - 確認根拠: テスト方針を過剰または古いと判断した項目はない。
