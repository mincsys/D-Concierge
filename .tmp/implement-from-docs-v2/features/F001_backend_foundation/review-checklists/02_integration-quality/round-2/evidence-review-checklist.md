- [x] 証跡がどのテストケース、コマンド、範囲指定、実施日時に対応するか追跡できるか。

- [x] 証跡に対象コミット、対象差分、実行環境、主要バージョン、ブラウザ、画面サイズなど、再現に必要な前提が残っているか。

- [x] 証跡に秘密情報、個人情報、絶対パス、不要な詳細ログが含まれていないか。

- [x] 対象、実行日時、実行コマンド、カバレッジ値、総数、通過数、判定結果がそろっているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体証跡に `target/executed_at/command/branch_coverage/covered_branches/total_branches/result`、結合証跡に加えて `design_coverage/tests/failures` が存在する。
- [x] 目標未達を合格扱いしていないか。
  - 検証結果: 指摘あり
  - 確認根拠: 最新の単体coverage JSONは `covered_branches=72`、`total_branches=84`、`branch_coverage=85.71%` で、単体方針の95%以上を下回る。証跡は `branch_coverage=95.71%`、`result=pass` のまま。

  - 検証結果: 対象外
  - 確認根拠: 総合テスト証跡ではなく、単体・結合のcoverage/evidenceで確認した。
- [x] 画面確認、HTTP確認、DB確認、ファイル確認の記録が、ケースの判定に必要な粒度で残っているか。
  - 検証結果: 対象外

## 実行結果の信頼性

- [x] 証跡の実行日時が対象成果物の変更後になっているか。
  - 検証結果: 指摘あり
  - 確認根拠: evidenceの `executed_at=2026-06-21T16:17:09+0900` は、最新の単体68件・結合28件の状態を反映していない。
- [x] 実行コマンドが方針書、設定、package manager、依存管理の現行定義と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡コマンドは `env UV_CACHE_DIR=/tmp/uv-cache uv run pytest ... --cov=src/backend --cov-branch ...` で、uvと一時cache指定、pytest-cov利用方針に合う。
- [x] 証跡と実際のテストコード、テスト仕様、設定ファイルに矛盾がないか。
  - 検証結果: 指摘あり
  - 確認根拠: 検証役が `pytest src/backend/tests/unit -q` で68件、`pytest src/backend/tests/integration -q` で28件を再実行したが、証跡は単体65件相当、結合 `tests=27` のまま。単体coverage値も最新JSONと一致しない。
- [x] 同じ対象に対する古い証跡が残り、最新結果と誤認される状態になっていないか。
  - 検証結果: 指摘あり
  - 確認根拠: backend単体・結合の正式証跡ファイル各1件が古い値を保持しており、最新結果と誤認される状態になっている。

## 修正方針の判断

- [x] 証跡が不足している場合は、証跡の追加、再実行、テスト仕様更新のどれが必要かを書く。
  - 検証結果: 指摘あり
  - 確認根拠: `.issue/implement-from-docs/2026-06-21_16-42-45_単体分岐カバレッジが方針値を下回り証跡も最新結果と一致しない.md` に、単体テスト補強後のcoverage再実行と単体・結合evidence更新が必要と記録した。
- [x] 証跡が方針と矛盾する場合は、証跡を直すべきか、方針を直すべきかを根拠から判断する。
  - 検証結果: 指摘あり
  - 確認根拠: 方針の95%基準は妥当で、証跡が最新実行結果を反映していない。方針ではなく、テスト補強と証跡更新が必要。
- [x] 証跡だけでは判定不能な場合は、必要な再実行条件と対象成果物を具体的に書く。
  - 検証結果: 指摘なし
  - 確認根拠: 指定コマンドを検証役として再実行し、単体coverage未達と証跡不一致を実コマンド結果で判定できた。結合coverage付き再実行のみ検証役環境のPostgreSQL接続差分により5件失敗したため、標準結合28件成功と管理役側の外側実行結果を併記して扱う。
