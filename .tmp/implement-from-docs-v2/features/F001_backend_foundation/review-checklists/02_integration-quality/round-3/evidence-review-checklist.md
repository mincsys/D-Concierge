# Evidence Review Checklist

## 証跡と方針の整合

- [x] 証跡の保存場所、ファイル名、記録項目が対応するテスト方針と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体証跡は `docs/04_テスト/02_単体テスト/evidence/backend-unit-coverage.txt`、結合証跡は `docs/04_テスト/03_結合テスト/evidence/backend-integration-evidence.txt` にあり、方針の保存先と一致する。
- [x] 証跡がどのテストケース、コマンド、範囲指定、実施日時に対応するか追跡できるか。
  - 検証結果: 指摘なし
  - 確認根拠: 両証跡に `target`、`executed_at`、`command`、coverage値、分岐数、`result` が `key=value` 形式で記録されている。
- [x] 証跡に対象コミット、対象差分、実行環境、主要バージョン、ブラウザ、画面サイズなど、再現に必要な前提が残っているか。
  - 検証結果: 指摘なし
  - 確認根拠: backend単体・結合方針で要求される証跡項目は揃っている。ブラウザと画面サイズはbackend F001証跡では不要。
- [x] 再テスト時に古い証跡と新しい証跡の扱いが方針と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体/結合の正式証跡ファイルは同じファイル名で最新値へ上書きされ、単体70件・結合28件の状態を示している。
- [x] 証跡に秘密情報、個人情報、絶対パス、不要な詳細ログが含まれていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡は集約値とコマンドのみで、DBパスワード、個人情報、スタックトレース、詳細ログは含まない。`/tmp` のJSON出力先は再生成可能な一時ファイル指定である。

## カバレッジ証跡

- [x] 方針で指定された指標だけが記録されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡は分岐カバレッジ、通過分岐数、総分岐数、実行件数、失敗件数、判定結果を記録し、行/ステートメント/関数カバレッジは記録していない。
- [x] 対象、実行日時、実行コマンド、カバレッジ値、総数、通過数、判定結果がそろっているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体証跡に `target/executed_at/command/branch_coverage/covered_branches/total_branches/result`、結合証跡に加えて `design_coverage/tests/failures` が存在する。
- [x] 目標未達を合格扱いしていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体 `branch_coverage=96.43%` は95%以上、結合 `branch_coverage=81.25%` は80%以上で、いずれも `result=pass` と整合する。
- [x] 測定から外した範囲がテスト方針と矛盾していないか。
  - 検証結果: 指摘なし
  - 確認根拠: `pyproject.toml` のcoverage設定は `branch = true`、`source = src/backend` で、omitは `tests`、`main.py`、migrations配下など方針上の対象外範囲に留まる。

## 総合テスト証跡

- [x] スクリーンショットやテキスト証跡がテストケースIDと対応しているか。
  - 検証結果: 対象外
  - 確認根拠: 今回はbackend F001の単体・結合証跡レビューであり、総合テスト証跡は対象外。
  - 理由: レビュー対象に総合テスト証跡が含まれないため。
- [x] スクリーンショットやログが、ロード中、エラー画面、古い状態ではなく、判定対象の状態を示しているか。
  - 検証結果: 対象外
  - 確認根拠: スクリーンショットや総合テストログは今回の対象外。
  - 理由: backend単体・結合coverage evidenceのみを確認しているため。
- [x] 必要最小限の証跡になっているか。
  - 検証結果: 対象外
  - 確認根拠: 総合テスト証跡ではなく、単体・結合のcoverage evidenceを対象に確認した。
  - 理由: 総合テスト証跡方針のレビュー工程ではないため。
- [x] 画面確認、HTTP確認、DB確認、ファイル確認の記録が、ケースの判定に必要な粒度で残っているか。
  - 検証結果: 対象外
  - 確認根拠: 画面確認証跡は対象外。HTTP/DB/ファイル確認は結合テスト本文と実行結果で確認した。
  - 理由: 総合テストケース単位の証跡をレビューしていないため。
- [x] 未実施、一部確認、保留、不合格、再テストが曖昧に合格扱いされていないか。
  - 検証結果: 対象外
  - 確認根拠: 総合テストの未実施/保留管理は今回の対象外。
  - 理由: F001 backend結合レビューであり、正式総合テストは後続工程で扱うため。
- [x] 証跡だけを見て、どのケースが合格、不合格、部分確認、未実施、保留なのか誤読なく分かるか。
  - 検証結果: 対象外
  - 確認根拠: 総合テストケース単位の証跡ではないため対象外。
  - 理由: backend単体・結合coverage evidenceを対象にしたレビューのため。

## 実行結果の信頼性

- [x] 証跡の実行日時が対象成果物の変更後になっているか。
  - 検証結果: 指摘なし
  - 確認根拠: evidenceの `executed_at=2026-06-21T16:50:37+09:00` は、state.mdの生成役round-2 coverage/evidence結果と整合し、単体70件・結合28件の最新値を示している。
- [x] 実行コマンドが方針書、設定、package manager、依存管理の現行定義と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡コマンドは `env UV_CACHE_DIR=/tmp/uv-cache uv run pytest ... --cov=src/backend --cov-branch --cov-report=json:... --cov-report= -q` で、uvと一時cache指定、pytest-cov利用方針に合う。
- [x] 証跡と実際のテストコード、テスト仕様、設定ファイルに矛盾がないか。
  - 検証結果: 指摘なし
  - 確認根拠: 検証役再実行で標準単体70件、標準結合28件が通過し、単体coverage JSONも81/84=96.43%で証跡と一致した。結合coverage付き実行は検証役環境でPostgreSQL接続エラーになったが、JSON値117/144=81.25%は証跡と一致し、管理役確認では同コマンド28 passedである。
- [x] 同じ対象に対する古い証跡が残り、最新結果と誤認される状態になっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: backend単体・結合の正式証跡ファイルは各1件で、古い65件/27件の値は残っていない。

## 修正方針の判断

- [x] 証跡が不足している場合は、証跡の追加、再実行、テスト仕様更新のどれが必要かを書く。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡不足は確認していない。
- [x] 証跡が方針と矛盾する場合は、証跡を直すべきか、方針を直すべきかを根拠から判断する。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡と方針の矛盾は確認していない。
- [x] 証跡だけでは判定不能な場合は、必要な再実行条件と対象成果物を具体的に書く。
  - 検証結果: 指摘なし
  - 確認根拠: 標準テスト、単体coverage、evidence、管理役確認結果から判定できる。結合coverage付き実行の検証役環境差分は、標準結合28 passedと管理役のcoverage pass結果を併記して扱う。
