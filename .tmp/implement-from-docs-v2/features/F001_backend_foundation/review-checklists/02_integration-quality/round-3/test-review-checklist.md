# Test Review Checklist

## テスト方針との整合

- [x] テストコードとテスト仕様が、対応するテスト方針の範囲指定、テスト対象単位、除外範囲に一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: F001 backendの単体テストは `src/backend/tests/unit/`、結合テストは `src/backend/tests/integration/` に分かれ、設定、DB、REST、trace、境界契約を方針どおり確認している。
- [x] テスト関連成果物のディレクトリ構成が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: テスト本体は `unit`/`integration`、支援コードは `support` に配置され、backendテストコードの管理対象ディレクトリと一致する。
- [x] テスト関連成果物のファイル構成、ファイル名、配置先が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: Pythonテストは `test_*.py` で、F001対象の設定、共通エラー、DB、Repository、アーキテクチャ、REST、migration、trace logをファイル単位で確認している。
- [x] 単体、結合、総合テストの役割が混ざっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体はクラス/関数/metadata契約を直接確認し、結合はFastAPI、DB、Alembic、trace log連携を確認している。ブラウザや利用者シナリオの総合テスト観点は含めていない。
- [x] テスト方針で求める観点、カバレッジ、証跡、実行環境、完了条件が満たされているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体70件、結合28件が通過し、単体coverageは81/84=96.43%、結合coverage evidenceは117/144=81.25%で方針値を満たす。evidenceも最新値へ更新されている。
- [x] テスト方針が実装や設計の現状に追従していない場合は、テスト方針側を直す方針を書く。
  - 検証結果: 指摘なし
  - 確認根拠: テスト方針側を古いまたは過剰と判断する根拠はない。

## 単体テスト

- [x] 公開関数、公開メソッド、コンポーネント、純粋ロジックの責務を単位としているか。
  - 検証結果: 指摘なし
  - 確認根拠: `ConfigLoader`、`AppError`、`new_trace_id`、`PathSecurityService`、`TraceLogWriter`、DB metadata、Repository Portを公開責務単位で確認している。
- [x] 外部副作用を Fake、Stub、Mock、fixture で差し替えているか。
  - 検証結果: 指摘なし
  - 確認根拠: ファイル系は `tmp_path`、ID発番は固定Generator、DB実体は単体では使用せず、結合テストへ分離している。
- [x] 事前条件、事後条件、不変条件、正常系、異常系、境界値、分岐が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 設定欠落/不正値、PathSecurity拒否条件、UUIDv7、TraceLogWriterの上限/日付変更/失敗書込/保持期間/同名衝突、DBモデル制約が確認されている。
- [x] テストが内部実装詳細に密結合しすぎていないか。
  - 検証結果: 指摘なし
  - 確認根拠: テストはdocs上の契約、公開型、SQLAlchemy metadata、YAML payloadなどの外部的に意味のある結果を確認している。
- [x] テストが単に例外が出ないことではなく、仕様上意味のある結果、状態変化、副作用、出力を検証しているか。
  - 検証結果: 指摘なし
  - 確認根拠: エラー分類、HTTP payload、DB制約、trace log YAML項目、公開設定の非漏えい、ログ削除/保存件数などを具体的にassertしている。

## 結合テスト

- [x] 外部インターフェース、処理設計、永続化、ファイル、通信境界、表示または操作連携を設計単位で確認しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `/api/app-config`、REST共通エラー、trace_id/trace log、設定読込失敗ログ、Alembic migration、PostgreSQL制約を設計単位で確認している。
- [x] 実外部サービス、実行環境、実行時間や環境差分が大きい対象を不必要に使っていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合テストで使う実体は方針上必要なPostgreSQLテストDBとASGI経路であり、実Codex、実ブラウザ、外部ネットワークは使っていない。
- [x] Mock、Fake、Stub が過剰で、実際の契約違反や結合不備を隠していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 認証依存だけをテスト用overrideし、middleware、設定読込、REST応答、trace log、DB migrationは実経路で確認している。
- [x] トランザクション、状態更新、競合、ロールバック、エラー変換、ログ依頼が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: F001範囲の未完了run部分UNIQUE、DB制約違反、RESTエラー変換、設定失敗ログ、起動時ログ削除を確認している。後続ユースケースの状態更新はF001外。
- [x] テストデータが独立し、実行順序に依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: DB結合テストは `prepare_foundation_database` でテストDB URLを検証してからschemaを初期化し、各ケースが独立した状態で実行される。
- [x] 時刻、乱数、実行順序、外部通信、ファイルシステム、DB 状態により不安定になるテストになっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 固定日時/固定UUIDを使う箇所があり、trace logの実時刻依存は存在/項目/対応関係で確認している。標準結合コマンドは検証役環境でも28件成功した。
- [x] テストデータが実装ロジックのコピーや都合のよい値だけになっておらず、境界値、異常値、業務上意味のある値を含んでいるか。
  - 検証結果: 指摘なし
  - 確認根拠: 未認証、不正Cookie、設定欠落、invalid timezone、危険パス、DB重複、不正状態、非テストDB URLなどの異常値を含む。

## 総合テスト

- [x] 利用者が利用者インターフェースまたは実運用に近い経路で業務を完了できることを確認しているか。
  - 検証結果: 対象外
  - 確認根拠: 今回はF001 backendの結合テスト完了検証と実装コード品質レビューであり、総合テスト工程は対象外。
  - 理由: レビュー対象に総合テスト仕様・結果と画面証跡が含まれないため。
- [x] 自動確認、手動確認、通信確認、永続化確認、ファイル確認の役割がテスト方針と一致しているか。
  - 検証結果: 対象外
  - 確認根拠: 総合テストの自動/手動確認役割は今回のレビュー対象外。
  - 理由: backend単体・結合テストとevidenceを対象にしたレビューのため。
- [x] 未実施、一部確認、保留、不合格、再テストの記録がテスト仕様・結果に残っているか。
  - 検証結果: 対象外
  - 確認根拠: 総合テスト仕様・結果ファイルの確認フェーズではない。
  - 理由: F001機能結合レビューであり、正式総合テストは後続のwhole-project phaseで扱うため。
- [x] 実行証跡とテストケースが追跡できるか。
  - 検証結果: 対象外
  - 確認根拠: 総合テストケースID単位の証跡は今回の対象外。backend単体・結合evidenceはEvidence checklistで確認した。
  - 理由: 総合テスト証跡をレビュー対象にしていないため。

## TDD と実装順序

- [x] 実装成果物レビューでは、TDD が要求される範囲で Red、Green、Refactor の証跡または説明があるか。
  - 検証結果: 指摘なし
  - 確認根拠: state.mdにF001テストRed、実装後Green、round-2 coverage未達Red、round-3 coverage Greenの経緯が記録されている。
- [x] Red を事後的に作れない場合は、レビュー指摘として記録し、今後の修正方針を示す。
  - 検証結果: 指摘なし
  - 確認根拠: Red成立記録がstate.mdに残っており、事後作成不能とは判断していない。
- [x] テスト追加が不要な場合は、除外理由が方針と整合しているか。
  - 検証結果: 指摘なし
  - 確認根拠: backend範囲外の総合テストは工程対象外として除外され、F001内の不足分岐は単体テスト追加で補われている。

## 不要なコメントと作業経緯

- [x] テスト名、docstring、コメント、テスト仕様、証跡に、確認対象の理解や再実行に不要な作業経緯、内部事情、言い訳、暫定理由が混入していないか。

  - 確認根拠: テストdocstringは `観点`/`確認` 形式で、作業経緯やレビュー対応理由を含めていない。
- [x] コメントや docstring は、仕様上の意図、契約、前提条件、非自明な制約の説明に限定されているか。
  - 検証結果: 指摘なし
  - 確認根拠: コメント/docstringは仕様上の役割、確認観点、制約の説明に留まっている。
- [x] テスト失敗理由、作成時の都合、環境制約、ツール制約、レビュー指摘への反応など、成果物ではなく作業報告や issue に書くべき内容が混入していないか。
  - 検証結果: 指摘なし
  - 確認根拠: テスト本文と証跡に「指摘されたため」などの作業報告表現は確認していない。
- [x] `既存実装があるので失敗しない`、`一旦この実装にしている`、`あとで修正する`、`指摘されたため追加`、`カバレッジのため追加`、`環境の都合で確認できない`、`旧仕様ではこうだった`、コードの単なる日本語言い換えなどが残っていないか。
  - 検証結果: 指摘なし

  - 検証結果: 指摘なし
  - 確認根拠: PathSecurity、TraceLogWriter、DB migration、REST error、設定読込のテストは観点と確認内容をdocstringで説明している。

- [x] テスト方針が過剰または古い場合は、テスト方針と関連成果物を直す方針を書く。
