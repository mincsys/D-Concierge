## 単体テスト

- [x] 公開関数、公開メソッド、コンポーネント、純粋ロジックの責務を単位としているか。
  - 検証結果: 指摘なし
  - 確認根拠: `RegisterAccountUseCase`、`LoginUseCase`、`AuthenticateSessionUseCase`、`LogoutUseCase`、`ChangeUserNameUseCase`、`ChangePasswordUseCase`、`DeleteAccountUseCase` の `execute(Command)` を公開責務単位で確認している。
- [x] 外部副作用を Fake、Stub、Mock、fixture で差し替えているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストは `FakeAccountRepository`、`FakePasswordHasher`、`FakeSessionTokenProvider`、`FakeAccountDeletionDispatcher`、`FakeTraceLogger`、`FakeTransactionManager`、`FixedClock` を使用している。
- [x] 事前条件、事後条件、不変条件、正常系、異常系、境界値、分岐が確認されているか。
  - 検証結果: 指摘あり
  - 確認根拠: 正常系、重複、短すぎるパスワード、確認不一致、削除中ユーザ、期限切れ、Cookie なし、dispatcher failed は確認しているが、パスワード文字種制約の境界が確認されていない。
  - 指摘: `.issue/implement-from-docs/2026-06-21_22-00-01_パスワード文字種制約が未実装.md`
- [x] テストが内部実装詳細に密結合しすぎていないか。
  - 検証結果: 指摘なし
  - 確認根拠: UseCase テストは Command、戻り値、状態変化、副作用依頼を観測しており、SQLAlchemy や REST 具象には依存していない。
- [x] テストが単に例外が出ないことではなく、仕様上意味のある結果、状態変化、副作用、出力を検証しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `field_errors`、ハッシュ、token hash、セッション削除、ユーザ・チャット状態、dispatcher 呼び出し、trace logger 呼び出しなど、仕様上の観測点を assert している。
