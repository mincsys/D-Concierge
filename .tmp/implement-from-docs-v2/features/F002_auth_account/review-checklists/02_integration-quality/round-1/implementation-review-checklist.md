## 責務分割と依存方向

- [x] 層、機能、モジュール、コンポーネントの責務が設計と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: UseCase が業務調停、Repository が DB 操作、security infra がハッシュ・トークン、REST が HTTP/Cookie 変換を担当しており、層の大枠は設計と一致している。
- [x] 副作用を持つ処理が定義済みの境界へ閉じているか。
  - 検証結果: 指摘あり
  - 確認根拠: DB 更新、Cookie 設定、ハッシュ生成、トークン発行は定義済み境界に閉じている。一方、削除ジョブ登録失敗時のログ副作用は設計上 TraceLogger 境界へ出す必要があるが、REST 配線で `NoopAccountTraceLogger` が注入されるため保存境界へ到達しない。
  - 指摘: `.issue/implement-from-docs/2026-06-21_22-00-02_削除ジョブ登録失敗がトレースログに記録されない.md`
- [x] 業務判断やドメイン判断が、表示層、入出力層、インフラ層、設定読込など不適切な場所へ漏れていないか。
  - 検証結果: 指摘なし
  - 確認根拠: ユーザ状態、入力検証、認証失敗、削除受付の業務判断は application 層にあり、REST 層は request/response、Cookie、dependency 組み立てに寄っている。
- [x] 純粋ロジックが永続化、通信、ファイル、時刻、ID、外部プロセスなどの副作用へ直接依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: validation は文字列検証のみで副作用に依存せず、UseCase は Clock、Repository、Hasher、TokenProvider、Dispatcher を Port 相当で受け取っている。
- [x] 上位層が下位層の実装詳細、テスト用実体、内部データ形式へ直接依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: application 層は `AccountRepositoryLike` や security/runtime Port に依存しており、SQLAlchemy model やテスト Fake を import していない。REST composition は具象依存を組み立てる境界として扱われている。
- [x] 依存方向、公開範囲、再利用単位が設計や開発標準と矛盾していないか。
  - 検証結果: 指摘なし
  - 確認根拠: `src/backend/application/account/` から presentation や infrastructure への逆依存は確認していない。router 登録も `app/router/registration.py` に集約されている。
- [x] 不要な依存関係、未使用コード、到達不能コード、暫定的な分岐、デバッグ用処理が残っていないか。
  - 検証結果: 指摘なし

## エラー・ログ・セキュリティ

- [x] 利用者向けメッセージと内部調査用情報が分離されているか。
  - 検証結果: 指摘なし
  - 確認根拠: REST エラー応答は `error` と利用者向け `message` を返し、`diagnostic_message` や stacktrace は通常応答に含めず TraceErrorMiddleware 側で扱っている。
- [x] 内部パス、秘密情報、スタックトレース、認証情報、個人情報などが利用者向け出力、外部応答、ログ、証跡へ露出していないか。
  - 検証結果: 指摘なし
  - 確認根拠: パスワード生値は DB に保存せず、Cookie 生トークンも DB には `sha256:` 付き hash のみ保存する。テスト証跡にも秘密情報や絶対パスは確認していない。
- [x] URL、ポート、ファイルパス、認証情報、環境名、外部サービス名などが不適切にハードコードされていないか。
  - 検証結果: 指摘なし
  - 確認根拠: DB URL や trace log dir は設定から注入され、F002 実装で本番 URL、認証情報、外部サービス名の直書きは確認していない。Cookie 名と max-age は外部 IF の共通データ項目と一致する定数である。
- [x] エラー分類、追跡情報、状態更新、後続処理の継続または抑止が設計と一致しているか。
  - 検証結果: 指摘あり
  - 確認根拠: 未ログインは 401、入力不正は 400、削除受付後の dispatcher failed は 202 維持という応答分類は実装されている。一方、dispatcher failed の追跡情報保存が REST 配線で実現されていない。
  - 指摘: `.issue/implement-from-docs/2026-06-21_22-00-02_削除ジョブ登録失敗がトレースログに記録されない.md`
- [x] ファイルパス、URL、コンテンツ種別、マークアップ、コマンド、外部入力などの検証が境界で行われているか。
  - 検証結果: 指摘あり
  - 確認根拠: F002 はファイルパスや MIME を扱わないが、外部入力としてのパスワード文字種検証が不足している。
  - 指摘: `.issue/implement-from-docs/2026-06-21_22-00-01_パスワード文字種制約が未実装.md`
- [x] 認証、認可、入力制限、出力制御、監査記録が対象システムの設計と整合しているか。
  - 検証結果: 指摘あり
  - 確認根拠: 保護対象 API への `get_authenticated_user` 適用、削除中ユーザの 401 化、Cookie 生値非保存は整合している。一方、入力制限のパスワード文字種と、削除ジョブ登録失敗の監査・追跡記録に不足がある。
  - 指摘:
    - `.issue/implement-from-docs/2026-06-21_22-00-01_パスワード文字種制約が未実装.md`
    - `.issue/implement-from-docs/2026-06-21_22-00-02_削除ジョブ登録失敗がトレースログに記録されない.md`
