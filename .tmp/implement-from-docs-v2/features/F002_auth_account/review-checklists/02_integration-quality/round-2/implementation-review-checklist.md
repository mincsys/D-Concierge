## 責務分割と依存方向

- [x] 層、機能、モジュール、コンポーネントの責務が設計と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: UseCase が業務調停、Repository が永続化、security infra が秘密情報処理、REST が HTTP/Cookie/trace log adapter を担当しており、層ごとの責務は設計と一致している。
- [x] 副作用を持つ処理が定義済みの境界へ閉じているか。
  - 検証結果: 指摘あり
  - 確認根拠: DB 更新、Cookie、ハッシュ、トークン、削除ジョブ登録は境界へ閉じているが、削除ジョブ登録失敗時のログ副作用で `TraceLogWriter.write()` の例外が UseCase 実行へ波及し得るため、ログ副作用の失敗境界が閉じていない。
  - 指摘: `.issue/implement-from-docs/2026-06-21_22-15-01_削除受付のトレースログ書込失敗が応答を上書きする.md`
- [x] 業務判断やドメイン判断が、表示層、入出力層、インフラ層、設定読込など不適切な場所へ漏れていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 入力検証、認証状態判定、削除受付状態維持は application 層にあり、REST 層は request/response、Cookie、trace log adapter の組み立てに留まっている。
- [x] 純粋ロジックが永続化、通信、ファイル、時刻、ID、外部プロセスなどの副作用へ直接依存していないか。

  - 確認根拠: `validation.py` は副作用を持たず、UseCase は Repository、Clock、Hasher、TokenProvider、Dispatcher、TraceLogger を注入境界経由で使っている。
- [x] 上位層が下位層の実装詳細、テスト用実体、内部データ形式へ直接依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: application 層は Protocol 相当の境界に依存し、SQLAlchemy model やテスト Fake を import していない。具象依存は REST composition と app factory に閉じている。
- [x] 依存方向、公開範囲、再利用単位が設計や開発標準と矛盾していないか。
  - 検証結果: 指摘なし
  - 確認根拠: application から presentation/infrastructure への逆依存はなく、router 登録も `app/router/registration.py` に集約されている。
- [x] 不要な依存関係、未使用コード、到達不能コード、暫定的な分岐、デバッグ用処理が残っていないか。

## エラー・ログ・セキュリティ

- [x] 利用者向けメッセージと内部調査用情報が分離されているか。
  - 検証結果: 指摘なし
  - 確認根拠: REST エラー応答は `error` と利用者向け `message` を返し、`diagnostic_message` や stacktrace は trace log 側へ送る構成である。
- [x] 内部パス、秘密情報、スタックトレース、認証情報、個人情報などが利用者向け出力、外部応答、ログ、証跡へ露出していないか。
  - 検証結果: 指摘なし
  - 確認根拠: パスワード生値と Cookie 生トークンは DB に保存されず、証跡ファイルも key=value の coverage summary のみで秘密情報や絶対パスを含まない。
- [x] URL、ポート、ファイルパス、認証情報、環境名、外部サービス名などが不適切にハードコードされていないか。
  - 検証結果: 指摘なし
  - 確認根拠: DB URL、trace log dir、timezone は設定から取得される。Cookie 名と max-age は外部 IF の共通データ項目に対応する定数であり、不適切な環境依存値ではない。
- [x] エラー分類、追跡情報、状態更新、後続処理の継続または抑止が設計と一致しているか。
  - 検証結果: 指摘あり
  - 確認根拠: 未ログイン 401、入力不正 400、削除受付時 dispatcher failed の 202 維持は実装されている。ただし dispatcher failed の trace log 保存に失敗した場合の継続が保証されず、追跡情報の副作用失敗が後続処理を抑止し得る。
  - 指摘: `.issue/implement-from-docs/2026-06-21_22-15-01_削除受付のトレースログ書込失敗が応答を上書きする.md`
- [x] ファイルパス、URL、コンテンツ種別、マークアップ、コマンド、外部入力などの検証が境界で行われているか。
  - 検証結果: 指摘なし
  - 確認根拠: F002 はファイル配信や MIME を扱わない。外部入力のユーザID、ユーザ名、パスワード、確認値は application validation と REST schema 経由で検証される。
- [x] 認証、認可、入力制限、出力制御、監査記録が対象システムの設計と整合しているか。
  - 検証結果: 指摘あり
  - 確認根拠: 保護対象 API の `get_authenticated_user`、削除中ユーザの 401 化、Cookie 生値非保存、パスワード文字種制約は整合している。一方、削除ジョブ登録失敗の監査・追跡記録は、ログ書き込み失敗時に元応答を維持する契約まで満たしていない。
  - 指摘: `.issue/implement-from-docs/2026-06-21_22-15-01_削除受付のトレースログ書込失敗が応答を上書きする.md`
