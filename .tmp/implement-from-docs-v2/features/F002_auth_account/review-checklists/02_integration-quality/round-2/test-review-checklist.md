# Test Review Checklist

## テスト方針との整合

- [x] テスト関連成果物のディレクトリ構成が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし

  - 検証結果: 指摘なし

  - 指摘: `.issue/implement-from-docs/2026-06-21_22-15-01_削除受付のトレースログ書込失敗が応答を上書きする.md`
- [x] テスト方針が実装や設計の現状に追従していない場合は、テスト方針側を直す方針を書く。
  - 検証結果: 指摘なし
  - 確認根拠: 今回の不足は実装とテスト側で補うべき内容であり、単体テスト方針、結合テスト方針、静的テスト方針が古いと判断する根拠はない。

## 単体テスト

- [x] 公開関数、公開メソッド、コンポーネント、純粋ロジックの責務を単位としているか。
  - 検証結果: 指摘なし
  - 確認根拠: `RegisterAccountUseCase.execute()`、`LoginUseCase.execute()`、`AuthenticateSessionUseCase.execute()`、`LogoutUseCase.execute()`、アカウント管理 UseCase、Port 契約、security provider が公開責務単位で確認されている。
- [x] 外部副作用を Fake、Stub、Mock、fixture で差し替えているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストは `FakeAccountRepository`、`FakeTransactionManager`、`FakePasswordHasher`、`FakeSessionTokenProvider`、`FakeAccountDeletionDispatcher`、`FakeTraceLogger`、`FixedClock` で DB、時刻、ハッシュ、トークン、dispatcher、trace log を差し替えている。
- [x] 事前条件、事後条件、不変条件、正常系、異常系、境界値、分岐が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 登録重複、ユーザID不正、ユーザ名空/長過ぎ、パスワード短過ぎ、許可外文字、確認不一致、ログイン失敗、期限切れ、削除中、dispatcher failed が単体で確認されている。
- [x] テストが内部実装詳細に密結合しすぎていないか。
  - 検証結果: 指摘なし
  - 確認根拠: UseCase の公開 Command/Result、副作用境界、戻り値、Repository 状態を確認しており、private helper や内部呼び出し順だけを固定していない。
- [x] テストが単に例外が出ないことではなく、仕様上意味のある結果、状態変化、副作用、出力を検証しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 正常系ではユーザ/セッション/ハッシュ/状態を、異常系では `field_errors`、副作用未実行、セッション削除、trace logger 受け渡しを確認している。

## 結合テスト

- [x] 外部インターフェース、処理設計、永続化、ファイル、通信境界、表示または操作連携を設計単位で確認しているか。
  - 検証結果: 指摘なし
  - 確認根拠: IF-SB-11 から IF-SB-17 の REST 契約、Cookie、DB 永続化、`field_errors`、削除受付時の trace log 保存成功が確認されている。F002 はファイル配信や表示連携を扱わない。
- [x] 実外部サービス、実行環境、実行時間や環境差分が大きい対象を不必要に使っていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合テストは ASGITransport と PostgreSQL テストDBを使い、実 Codex、外部ネットワーク、実ブラウザ、長時間処理を使っていない。
- [x] Mock、Fake、Stub が過剰で、実際の契約違反や結合不備を隠していないか。
  - 検証結果: 指摘なし
  - 確認根拠: REST 結合テストは実 app factory、REST router、SQLAlchemy repository、security provider、TraceLogWriter を通して確認している。dispatcher failed だけは F002 境界の異常系を作るため monkeypatch で制御している。
- [x] トランザクション、状態更新、競合、ロールバック、エラー変換、ログ依頼が確認されているか。
  - 検証結果: 指摘あり
  - 確認根拠: 状態更新、入力不正の DB 非更新、削除受付時のセッション削除、dispatcher failed 時の trace log 保存成功は確認されている。ただし trace log 書き込み失敗時も元の削除受付応答を維持するログ依頼境界の異常系が未確認である。
  - 指摘: `.issue/implement-from-docs/2026-06-21_22-15-01_削除受付のトレースログ書込失敗が応答を上書きする.md`
- [x] テストデータが独立し、実行順序に依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 各結合テストは `prepare_foundation_database()` と fixture で DB を準備し、`user-001` などの固定値もテストごとの既知状態で使っている。
- [x] 時刻、乱数、実行順序、外部通信、ファイルシステム、DB 状態により不安定になるテストになっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: DB はテストごとに準備し、セッション期限などの時刻は固定データを使っている。trace log は tmp_path 配下で確認し、外部通信は使わない。
- [x] テストデータが実装ロジックのコピーや都合のよい値だけになっておらず、境界値、異常値、業務上意味のある値を含んでいるか。
  - 検証結果: 指摘なし
  - 確認根拠: 短過ぎるパスワード、全角混入、確認不一致、削除中ユーザ、期限切れセッション、Cookie なしログアウト、登録重複、未知ユーザなど、仕様上意味のある正常・異常データが含まれている。
