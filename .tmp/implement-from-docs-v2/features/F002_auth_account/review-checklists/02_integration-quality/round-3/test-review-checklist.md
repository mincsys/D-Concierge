# Test Review Checklist

## テスト方針との整合

- [x] テストコードとテスト仕様が、対応するテスト方針の範囲指定、テスト対象単位、除外範囲に一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: F002 の単体テストは UseCase/Port/security provider 単位、結合テストは REST API/DB/Cookie/trace log 境界単位で、単体・結合テスト方針の範囲に収まっている。
- [x] テスト関連成果物のディレクトリ構成が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストは `src/backend/tests/unit/`、結合テストは `src/backend/tests/integration/`、支援コードは `src/backend/tests/support/` に配置されている。
- [x] テスト関連成果物のファイル構成、ファイル名、配置先が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 認証API、アカウント管理API、DB契約、security provider、UseCase の責務ごとに `test_*.py` が分割され、重複した公開契約テストファイルは残っていない。
- [x] 単体、結合、総合テストの役割が混ざっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体は Fake/Stub、結合は ASGITransport と PostgreSQL テストDB、TraceLogWriter を使う構成であり、総合テスト相当のブラウザ操作や実Codex実行は含まれない。
- [x] テスト方針で求める観点、カバレッジ、証跡、実行環境、完了条件が満たされているか。
  - 検証結果: 指摘なし
  - 確認根拠: state には単体 105 passed、結合 52 passed、静的テスト全種 pass、単体 96.32%、結合 80.26% が記録され、evidence も同じ値へ更新されている。
- [x] テスト方針が実装や設計の現状に追従していない場合は、テスト方針側を直す方針を書く。
  - 検証結果: 指摘なし
  - 確認根拠: 今回の確認ではテスト方針が古い、または過剰である根拠は確認していない。

## 単体テスト

- [x] 公開関数、公開メソッド、コンポーネント、純粋ロジックの責務を単位としているか。
  - 検証結果: 指摘なし
  - 確認根拠: 登録、ログイン、認証状態確認、ログアウト、ユーザ名変更、パスワード変更、削除受付の `execute()` と Port/security provider の公開契約を単位としている。
- [x] 外部副作用を Fake、Stub、Mock、fixture で差し替えているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストは `FakeAccountRepository`、`FakeTransactionManager`、`FakePasswordHasher`、`FakeSessionTokenProvider`、`FakeAccountDeletionDispatcher`、`FakeTraceLogger`、`FixedClock` で副作用を差し替える。
- [x] 事前条件、事後条件、不変条件、正常系、異常系、境界値、分岐が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 重複、不正入力、確認不一致、許可外パスワード文字、期限切れ、削除中、dispatcher failed、セッション削除などの分岐を確認している。
- [x] テストが内部実装詳細に密結合しすぎていないか。
  - 検証結果: 指摘なし
  - 確認根拠: public Command/Result と副作用境界の観測を中心に確認し、private helper や内部呼び出し順の固定に寄っていない。
- [x] テストが単に例外が出ないことではなく、仕様上意味のある結果、状態変化、副作用、出力を検証しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 戻り値、`field_errors`、DB相当状態、セッション削除、hash保存、trace logger 受け渡しなどを検証している。

## 結合テスト

- [x] 外部インターフェース、処理設計、永続化、ファイル、通信境界、表示または操作連携を設計単位で確認しているか。
  - 検証結果: 指摘なし
  - 確認根拠: IF-SB-11 から IF-SB-17 の REST 契約、Cookie、DB状態、`field_errors`、削除受付 trace log 成功/失敗境界が確認されている。F002 はファイル配信や表示連携を扱わない。
- [x] 実外部サービス、実行環境、実行時間や環境差分が大きい対象を不必要に使っていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合テストは ASGITransport、PostgreSQL テストDB、tmp_path の trace log を使い、実外部サービスや実Codex、実ブラウザを使っていない。
- [x] Mock、Fake、Stub が過剰で、実際の契約違反や結合不備を隠していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 通常経路は app factory、REST router、SQLAlchemy repository、security provider、TraceLogWriter の実配線で確認し、dispatcher failed と trace log write failed のみ異常系を作るために monkeypatch している。
- [x] トランザクション、状態更新、競合、ロールバック、エラー変換、ログ依頼が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 入力不正のDB非更新、削除受付時の `deleting` 遷移と全セッション削除、dispatcher failed の trace log 保存、trace log write failed 時の 202 維持が確認されている。
- [x] テストデータが独立し、実行順序に依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 各結合テストは `prepare_foundation_database()` と fixture で DB を既知状態にし、テストごとに必要なユーザ、セッション、チャットを投入している。
- [x] 時刻、乱数、実行順序、外部通信、ファイルシステム、DB 状態により不安定になるテストになっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 固定日時、固定セッショントークン、tmp_path の trace log、テストDB初期化を使っており、外部通信や実行順依存は確認していない。
- [x] テストデータが実装ロジックのコピーや都合のよい値だけになっておらず、境界値、異常値、業務上意味のある値を含んでいるか。
  - 検証結果: 指摘なし
  - 確認根拠: 短過ぎるパスワード、全角混入、確認不一致、削除中ユーザ、期限切れセッション、Cookieなしログアウト、登録重複、ログ書込失敗などの意味ある異常値を含む。

  - 検証結果: 対象外
  - 確認根拠: 今回は F002 結合・品質レビューであり、機能別総合テストまたは正式総合テストのレビューではない。
  - 理由: 総合テストフェーズではないため。
- [x] 自動確認、手動確認、通信確認、永続化確認、ファイル確認の役割がテスト方針と一致しているか。
  - 検証結果: 対象外
  - 確認根拠: 今回の対象は単体・結合テストコードと backend evidence であり、手動確認や画面確認は対象外である。

- [x] 未実施、一部確認、保留、不合格、再テストの記録がテスト仕様・結果に残っているか。
  - 検証結果: 対象外
  - 確認根拠: 機能別総合テスト仕様・結果は未実施フェーズであり、今回のレビュー対象ではない。
  - 理由: 総合テストフェーズではないため。

  - 検証結果: 対象外
  - 確認根拠: 総合テストケース別証跡ではなく、backend 単体・結合の coverage evidence を確認している。
  - 理由: 総合テストフェーズではないため。

  - 確認根拠: state に追加テストの Red 失敗、修正後 Green、単体 105 passed、結合 52 passed、静的テスト pass、coverage 更新が記録されている。
- [x] Red を事後的に作れない場合は、レビュー指摘として記録し、今後の修正方針を示す。

- [x] コメントや docstring は、仕様上の意図、契約、前提条件、非自明な制約の説明に限定されているか。
  - 検証結果: 指摘なし

- [x] テスト失敗理由、作成時の都合、環境制約、ツール制約、レビュー指摘への反応など、成果物ではなく作業報告や issue に書くべき内容が混入していないか。
  - 検証結果: 指摘なし
