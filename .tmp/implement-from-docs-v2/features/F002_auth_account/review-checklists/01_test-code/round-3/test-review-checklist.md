# Test Review Checklist

## テスト方針との整合

- [x] テストコードとテスト仕様が、対応するテスト方針の範囲指定、テスト対象単位、除外範囲に一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `docs/04_テスト/02_単体テスト/単体テスト方針.md` は公開メソッド単位の単体テスト、`docs/04_テスト/03_結合テスト/結合テスト方針.md` は REST API と DB 連携の結合確認を求めている。F002 対象テストは登録、ログイン、認証状態確認、ログアウト、ユーザ名変更、パスワード変更、アカウント削除受付を単体・結合の範囲で確認しており、F007 の物理削除、起動時回復、実行回復を要求していない。
- [x] テスト関連成果物のディレクトリ構成が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: F002 の単体テストは `src/backend/tests/unit/application/account/`、Port 契約は `src/backend/tests/unit/application/ports/`、security 具象は `src/backend/tests/unit/infrastructure/security/`、結合テストは `src/backend/tests/integration/`、補助コードは `src/backend/tests/support/` に配置され、単体・結合テスト方針の配置規則と一致している。
- [x] テスト関連成果物のファイル構成、ファイル名、配置先が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 対象ファイルは `test_*.py` 命名で配置されている。round-2 で問題だった `src/backend/tests/unit/application/account/test_auth_account_use_cases.py` は作業ツリー上で削除され、残るユースケース単体テストは `test_register_and_login_use_cases.py`、`test_authenticate_and_logout_use_cases.py`、`test_account_management_use_cases.py` の Command オブジェクト方式に統一されている。
- [x] 単体、結合、総合テストの役割が混ざっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストは UseCase、Port、security provider を Fake/Stub で確認し、結合テストは FastAPI ASGITransport、SQLAlchemy、テスト DB 経由で HTTP、Cookie、DB 状態を確認している。総合テスト仕様、Playwright、実ブラウザ、実運用フローの証跡確認は含まれていない。
- [x] テスト方針で求める観点、カバレッジ、証跡、実行環境、完了条件が満たされているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体では正常系、入力不正、重複、認証失敗、削除中ユーザ、期限切れ、Cookie なし、削除受付成功/失敗を確認している。結合では IF-SB-11 から IF-SB-17 の REST 契約、Cookie、`field_errors`、DB 状態変化を確認している。Red 結果は `state.md` に単体 `30 failed, 70 passed`、結合 `18 failed, 30 passed` として記録され、失敗理由は F002 本実装未作成に切り分けられている。
- [x] テスト方針が実装や設計の現状に追従していない場合は、テスト方針側を直す方針を書く。
  - 検証結果: 指摘なし
  - 確認根拠: 確認した範囲では、単体テスト方針、結合テスト方針、外部 IF、処理設計、内部 IF に対してテスト方針側の修正を要する乖離は確認していない。

## 単体テスト

- [x] 公開関数、公開メソッド、コンポーネント、純粋ロジックの責務を単位としているか。
  - 検証結果: 指摘なし
  - 確認根拠: `test_register_and_login_use_cases.py`、`test_authenticate_and_logout_use_cases.py`、`test_account_management_use_cases.py` は各 UseCase の `execute(Command)` を単位に戻り値、状態変化、副作用を確認している。キーワード引数 `execute(...)` を別契約として要求していた `test_auth_account_use_cases.py` は削除済みである。
- [x] 外部副作用を Fake、Stub、Mock、fixture で差し替えているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストは `FakeAccountRepository`、`FakePasswordHasher`、`FakeSessionTokenProvider`、`FakeAccountDeletionDispatcher`、`FakeTraceLogger`、`FakeTransactionManager`、`FixedClock` を使い、DB、時刻、ハッシュ、トークン、dispatcher、トレースログの副作用を差し替えている。
- [x] 事前条件、事後条件、不変条件、正常系、異常系、境界値、分岐が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 登録では入力不正とユーザ ID 重複、ログインでは存在しないユーザ、削除中ユーザ、パスワード不一致、認証状態確認では Cookie なし、期限切れ、削除中ユーザ、ログアウトでは Cookie なし、アカウント管理ではユーザ名不正、新パスワード不正、確認不一致、削除受付失敗時の状態維持を確認している。
- [x] テストが内部実装詳細に密結合しすぎていないか。
  - 検証結果: 指摘なし
  - 確認根拠: UseCase 単体テストは公開 Command、戻り値、Port 副作用、状態変化を観測している。SQLAlchemy の内部構造、API route の実装詳細、未定義の互換 API には依存していない。
- [x] テストが単に例外が出ないことではなく、仕様上意味のある結果、状態変化、副作用、出力を検証しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 対象テストは `field_errors`、Cookie 用生トークンと保存用 `token_hash` の分離、セッション削除、ユーザ・チャットの `deleting` 更新、DB 状態、Cookie 属性、レスポンス payload、トレースログ依頼など、仕様上の観測点を assert している。

## 結合テスト

- [x] 外部インターフェース、処理設計、永続化、ファイル、通信境界、表示または操作連携を設計単位で確認しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `test_auth_account_api.py` は IF-SB-11 から IF-SB-14 の登録、ログイン、認証状態確認、ログアウトについて、正常系、入力不正、重複、存在しないユーザ、削除中ユーザ、Cookie なし、Cookie 失効、DB 状態を確認している。`test_account_management_api.py` は IF-SB-15 から IF-SB-17 のユーザ名変更、パスワード変更、削除受付を確認している。
- [x] 実外部サービス、実行環境、実行時間や環境差分が大きい対象を不必要に使っていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合テストは FastAPI `ASGITransport`、PostgreSQL テスト DB、SQLAlchemy、pytest fixture を使う構成で、実 Codex、実ブラウザ、社外サービス、本番データには依存していない。
- [x] Mock、Fake、Stub が過剰で、実際の契約違反や結合不備を隠していないか。
  - 検証結果: 指摘なし
  - 確認根拠: API 結合テストは `create_app`、ASGITransport、実 DB、実 password hasher/token provider を経由しており、F002 の HTTP、Cookie、DB 結合契約を Fake で隠していない。
- [x] トランザクション、状態更新、競合、ロールバック、エラー変換、ログ依頼が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 登録重複、ログイン不存在、ログイン削除中、パスワード不一致ではセッション非作成と `field_errors` を確認している。認証状態確認の期限切れ/削除中ではセッション削除、アカウント管理ではユーザ名・パスワード・削除受付の DB 更新または非更新を確認している。
- [x] テストデータが独立し、実行順序に依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合テストは各テストで `prepare_foundation_database(database_url)` を呼び、必要なユーザ、セッション、チャットを個別に seed しているため、テスト間の実行順序に依存しない構成になっている。
- [x] 時刻、乱数、実行順序、外部通信、ファイルシステム、DB 状態により不安定になるテストになっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストは `FixedClock` と固定 token/hash を使い、結合テストも固定日時と固定 UUID を seed している。DB は各テストで初期化され、外部通信や実ブラウザを使っていない。
- [x] テストデータが実装ロジックのコピーや都合のよい値だけになっておらず、境界値、異常値、業務上意味のある値を含んでいるか。
  - 検証結果: 指摘なし
  - 確認根拠: 登録の不正ユーザ ID、空ユーザ名、短いパスワード、確認不一致、登録済みユーザ ID、ログインの不存在/deleting/不一致、期限切れセッション、Cookie なし、削除受付など、業務上意味のある値を含めている。
