# Evidence Review Checklist

## 証跡と方針の整合

- [x] 証跡の保存場所、ファイル名、記録項目が対応するテスト方針と一致しているか。
  - 検証結果: 問題なし
  - 確認根拠: 単体は `docs/04_テスト/02_単体テスト/evidence/backend-unit-coverage.txt`、結合は `docs/04_テスト/03_結合テスト/evidence/backend-integration-evidence.txt` に保存され、方針のkey=value項目を使用している。
- [x] 証跡がどのテストケース、コマンド、範囲指定、実施日時に対応するか追跡できるか。
  - 検証結果: 問題なし
  - 確認根拠: evidenceには `command`、`executed_at`、`target=backend`、結合側は `tests=75` と `design_coverage` が記録されている。
- [x] 証跡に対象コミット、対象差分、実行環境、主要バージョン、ブラウザ、画面サイズなど、再現に必要な前提が残っているか。
  - 検証結果: 問題なし
  - 確認根拠: 単体/結合方針の正式key=value項目は対象、日時、コマンド、coverage等に限定され、ブラウザや画面サイズは今回のbackend coverage evidenceでは必須項目外。
- [x] 再テスト時に古い証跡と新しい証跡の扱いが方針と一致しているか。
  - 検証結果: 問題なし
  - 確認根拠: 方針どおり同じ evidence ファイルが最新結果で上書きされており、旧結果との併存は確認していない。
- [x] 証跡に秘密情報、個人情報、絶対パス、不要な詳細ログが含まれていないか。
  - 検証結果: 問題なし
  - 確認根拠: evidenceはコマンド、coverage、件数のみで、DB URL、Cookie、絶対パス、個人情報、詳細ログを含まない。

## カバレッジ証跡

- [x] 方針で指定された指標だけが記録されているか。
  - 検証結果: 問題なし
  - 確認根拠: 単体/結合 evidence は分岐カバレッジ、covered/total branches、resultを中心に記録し、行/ステートメント/関数カバレッジは出していない。
- [x] 対象、実行日時、実行コマンド、カバレッジ値、総数、通過数、判定結果がそろっているか。
  - 検証結果: 問題なし
  - 確認根拠: 単体 evidence は `target/executed_at/command/branch_coverage/covered_branches/total_branches/result` を持つ。結合 evidence はこれに加えて `design_coverage/tests/failures` を持つ。
- [x] 目標未達を合格扱いしていないか。
  - 検証結果: 問題なし
  - 確認根拠: 単体 branch 96.20% は95%以上、結合 branch 80.31% は80%以上で、数値上の目標は達成している。
- [x] 測定から外した範囲がテスト方針と矛盾していないか。
  - 検証結果: 問題なし
  - 確認根拠: evidence本文に計測除外の追加や例外扱いは記録されていない。coverageコマンドは `--cov=src/backend --cov-branch` を指定している。

## 総合テスト証跡

- [x] スクリーンショットやテキスト証跡がテストケースIDと対応しているか。
  - 検証結果: 対象外
  - 確認根拠: 本フェーズは結合・品質レビューであり、総合テストのスクリーンショット/テストケースID証跡はレビュー対象外。
- [x] スクリーンショットやログが、ロード中、エラー画面、古い状態ではなく、判定対象の状態を示しているか。
  - 検証結果: 対象外
  - 確認根拠: 今回の証跡はbackend単体/結合coverageテキストであり、画面スクリーンショットは対象外。
- [x] 必要最小限の証跡になっているか。
  - 検証結果: 問題なし
  - 確認根拠: coverage evidenceは方針で指定されたkey=value項目に絞られている。
- [x] 画面確認、HTTP確認、DB確認、ファイル確認の記録が、ケースの判定に必要な粒度で残っているか。
  - 検証結果: 指摘あり
  - 確認根拠: 結合 evidence の `design_coverage` はF004 SSE/current state/startup recoveryを記録しているが、実際のテストはSSEライブ配信と起動時回復の状態別単独ケースを確認できていない。
- [x] 未実施、一部確認、保留、不合格、再テストが曖昧に合格扱いされていないか。
  - 検証結果: 指摘あり
  - 確認根拠: 未確認のSSE購読配信、キャンセル結果分岐、起動時回復ログ/失敗扱いが evidence 上は未確認として分離されず、`result=pass` と `design_coverage` に含まれている。
- [x] 証跡だけを見て、どのケースが合格、不合格、部分確認、未実施、保留なのか誤読なく分かるか。
  - 検証結果: 指摘あり
  - 確認根拠: backend coverage evidenceは個別ケース一覧を持たない方針だが、`design_coverage` に未確認設計単位が含まれ、設計網羅済みと誤読される。

## 実行結果の信頼性

- [x] 証跡の実行日時が対象成果物の変更後になっているか。
  - 検証結果: 問題なし
  - 確認根拠: stateの生成役実装完了報告と evidence の `executed_at=2026-06-22T05:40:07+09:00` / `05:42:49+09:00` はF004実装完了後の証跡として記録されている。
- [x] 実行コマンドが方針書、設定、package manager、依存管理の現行定義と一致しているか。
  - 検証結果: 問題なし
  - 確認根拠: コマンドは `env UV_CACHE_DIR=/tmp/uv-cache uv run pytest ... --cov=src/backend --cov-branch` で、uvとpytest-covを使う方針に一致している。
- [x] 証跡と実際のテストコード、テスト仕様、設定ファイルに矛盾がないか。
  - 検証結果: 指摘あり
  - 確認根拠: 証跡の `design_coverage` はF004 SSE/current state/saved message/terminal answer、cancel boundaries、startup run recoveryを記録しているが、実際のテストコードにはSSE live publish、キャンセル `already_exited` / `not_registered`、回復対象単独ケースがない。
- [x] 同じ対象に対する古い証跡が残り、最新結果と誤認される状態になっていないか。
  - 検証結果: 問題なし
  - 確認根拠: 対象backend evidenceは各1ファイルで上書きされ、複数の古いF004 evidence併存は確認していない。

## 修正方針の判断

- [x] 証跡が不足している場合は、証跡の追加、再実行、テスト仕様更新のどれが必要かを書く。
  - 検証結果: 指摘あり
  - 確認根拠: 4件の実装/テスト修正後に同じ単体・結合・coverageコマンドを再実行し、evidence の `design_coverage` と件数/coverageを更新する必要がある。
- [x] 証跡が方針と矛盾する場合は、証跡を直すべきか、方針を直すべきかを根拠から判断する。
  - 検証結果: 指摘あり
  - 確認根拠: 方針は設計網羅を求めており妥当。矛盾は未確認設計単位を網羅済みに見せている evidence/テスト側の問題として判断した。
- [x] 証跡だけでは判定不能な場合は、必要な再実行条件と対象成果物を具体的に書く。
  - 検証結果: 問題なし
  - 確認根拠: 証跡だけで数値は判定可能で、設計網羅不足はテスト/実装本文の照合により判断できた。判断不能項目はない。
