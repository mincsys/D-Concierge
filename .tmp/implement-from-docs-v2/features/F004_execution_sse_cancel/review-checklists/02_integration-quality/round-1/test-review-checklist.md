# Test Review Checklist

## テスト方針との整合

- [x] テストコードとテスト仕様が、対応するテスト方針の範囲指定、テスト対象単位、除外範囲に一致しているか。
  - 検証結果: 指摘あり
  - 確認根拠: F004用単体/結合テストは範囲内にあるが、結合テスト方針が求めるSSE publish後イベント配信、起動時回復の `accepted`/`running`/`validating` 単独ケース、キャンセルFake終了要求結果の一部を確認していない。
- [x] テスト関連成果物のディレクトリ構成が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 問題なし
  - 確認根拠: 単体は `src/backend/tests/unit/application/execution/`、結合は `src/backend/tests/integration/test_execution_sse_cancel_api.py`、補助は `src/backend/tests/support/execution.py` に配置されている。
- [x] テスト関連成果物のファイル構成、ファイル名、配置先が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 問題なし
  - 確認根拠: `test_cancel_and_recovery_use_cases.py`、`test_sse_event_broker.py`、`test_execution_sse_cancel_api.py` はF004の処理単位と一致している。
- [x] 単体、結合、総合テストの役割が混ざっていないか。
  - 検証結果: 問題なし
  - 確認根拠: 単体ではFakeを使ったUseCase/Broker確認、結合ではASGI経由のAPI/DB確認に分かれている。Playwrightや総合テスト手順は混入していない。
- [x] テスト方針で求める観点、カバレッジ、証跡、実行環境、完了条件が満たされているか。
  - 検証結果: 指摘あり
  - 確認根拠: 生成役報告とevidence上の実行件数/coverageは基準を満たすが、設計単位の網羅に不足があるため完了条件の「未確認項目がない」を満たさない。
- [x] テスト方針が実装や設計の現状に追従していない場合は、テスト方針側を直す方針を書く。
  - 検証結果: 問題なし
  - 確認根拠: 現行テスト方針はF004のSSE API、キャンセル、起動時回復を対象としており、方針側修正は不要。

## 単体テスト

- [x] 公開関数、公開メソッド、コンポーネント、純粋ロジックの責務を単位としているか。
  - 検証結果: 問題なし
  - 確認根拠: `CancelChatRunUseCase.execute()`、`RecoverUnfinishedRunsUseCase.execute()`、`RunEventBroker`、SSE formatter の公開責務を単位にしている。
- [x] 外部副作用を Fake、Stub、Mock、fixture で差し替えているか。
  - 検証結果: 問題なし
  - 確認根拠: Repository、TransactionManager、CodexRunner、EventPublisher、BackgroundExecutor、Clock は `src/backend/tests/support/execution.py` のFakeで差し替えられている。
- [x] 事前条件、事後条件、不変条件、正常系、異常系、境界値、分岐が確認されているか。
  - 検証結果: 指摘あり
  - 確認根拠: accepted/running/終端済み/deleting/状態更新競合/回復基本分岐はあるが、`CodexRunner.cancel()` の `already_exited` / `not_registered` と回復対象取得失敗・ログ記録分岐がない。
- [x] テストが内部実装詳細に密結合しすぎていないか。
  - 検証結果: 問題なし
  - 確認根拠: 単体テストは公開UseCase/Brokerの戻り値、状態更新依頼、イベント発行依頼を確認しており、内部private関数名には依存していない。
- [x] テストが単に例外が出ないことではなく、仕様上意味のある結果、状態変化、副作用、出力を検証しているか。
  - 検証結果: 問題なし
  - 確認根拠: AppError分類、Repository transitions、Codex cancel request、publisher events、broker event ordering、SSE wire payload をassertしている。

## 結合テスト

- [x] 外部インターフェース、処理設計、永続化、ファイル、通信境界、表示または操作連携を設計単位で確認しているか。
  - 検証結果: 指摘あり
  - 確認根拠: IF-SB-06/07のHTTP境界とDB状態は確認しているが、SSE APIのpublish後ライブ配信、RunEventBroker接続、起動時回復の状態別単独ケースを確認していない。
- [x] 実外部サービス、実行環境、実行時間や環境差分が大きい対象を不必要に使っていないか。
  - 検証結果: 問題なし
  - 確認根拠: 実Codex、実ブラウザ、外部サービスは使わず、ASGITransportとPostgreSQLテストDB、Fake/Stub境界で確認している。
- [x] Mock、Fake、Stub が過剰で、実際の契約違反や結合不備を隠していないか。
  - 検証結果: 指摘あり
  - 確認根拠: REST cancel が `NoopCodexRunCancellation` と `_NoopRunEventPublisher` を使うため、キャンセル結果整合とSSEイベント配信の結合不備を結合テストが検出できない。
- [x] トランザクション、状態更新、競合、ロールバック、エラー変換、ログ依頼が確認されているか。
  - 検証結果: 指摘あり
  - 確認根拠: 状態更新・競合・エラー変換は一部確認しているが、起動時回復のトレースログ記録、回復対象取得失敗、`already_exited`/`not_registered` の状態整合が未確認。
- [x] テストデータが独立し、実行順序に依存していないか。
  - 検証結果: 問題なし
  - 確認根拠: 各結合テストで `prepare_foundation_database()` と固定ID seed を使い、テストごとにDBを既知状態へ初期化している。
- [x] 時刻、乱数、実行順序、外部通信、ファイルシステム、DB 状態により不安定になるテストになっていないか。
  - 検証結果: 問題なし
  - 確認根拠: 固定UUID/固定時刻/テストDB初期化を使い、実外部通信や実Codex起動に依存していない。
- [x] テストデータが実装ロジックのコピーや都合のよい値だけになっておらず、境界値、異常値、業務上意味のある値を含んでいるか。
  - 検証結果: 指摘あり
  - 確認根拠: 認証なし/他ユーザ/deleting/終端済みなどはあるが、起動時回復は `cancel_requested` を必ず含む都合のよいデータで、`accepted`のみ等の実運用上意味のある境界がない。

## 総合テスト

- [x] 利用者が利用者インターフェースまたは実運用に近い経路で業務を完了できることを確認しているか。
  - 検証結果: 対象外
  - 確認根拠: 本フェーズは結合テスト完了検証・実装コード品質レビューであり、機能別総合テスト/ブラウザ確認は対象外。
- [x] 自動確認、手動確認、通信確認、永続化確認、ファイル確認の役割がテスト方針と一致しているか。
  - 検証結果: 問題なし
  - 確認根拠: F004 backend結合は自動テストでASGI/API/DB/SSE文字列を確認する方針で、手動確認やPlaywrightを混ぜていない。
- [x] 未実施、一部確認、保留、不合格、再テストの記録がテスト仕様・結果に残っているか。
  - 検証結果: 対象外
  - 確認根拠: 本フェーズの対象は単体/結合テストとcoverage evidenceであり、総合テスト仕様・結果の分類レビューではない。
- [x] 実行証跡とテストケースが追跡できるか。
  - 検証結果: 指摘あり
  - 確認根拠: evidenceはテスト総数とdesign_coverageを持つが、design_coverageが実際には未確認のSSEライブ配信・起動時回復単独ケースまでカバー済みのように読める。
