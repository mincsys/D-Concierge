# Test Review Checklist

## テスト方針との整合

- [x] テストコードとテスト仕様が、対応するテスト方針の範囲指定、テスト対象単位、除外範囲に一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: F004 テストは run 状態遷移、SSE、キャンセル、起動時回復、trace log に限定され、F005 以降の Codex 実行本体や F007 物理削除は要求していない。
- [x] テスト関連成果物のディレクトリ構成が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストは `src/backend/tests/unit/application/execution/`、結合テストは `src/backend/tests/integration/test_execution_sse_cancel_api.py`、補助は `src/backend/tests/support/execution.py` に配置されている。
- [x] テスト関連成果物のファイル構成、ファイル名、配置先が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: cancellation/recovery と broker の単体テスト、SSE/cancel API の結合テスト、trace error boundary の結合テストが責務単位で分かれている。
- [x] 単体、結合、総合テストの役割が混ざっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体は fake/stub で UseCase/broker の契約を確認し、結合は FastAPI route、DB、SSE wire、trace log を確認する。画面操作や総合テスト証跡は含めていない。
- [x] テスト方針で求める観点、カバレッジ、証跡、実行環境、完了条件が満たされているか。
  - 検証結果: 指摘なし
  - 確認根拠: 生成役報告と evidence で unit 144 passed、integration 83 passed、branch coverage は単体 95.88%、結合 80.92% と方針閾値を満たしている。
- [x] テスト方針が実装や設計の現状に追従していない場合は、テスト方針側を直す方針を書く。
  - 検証結果: 対象外
  - 確認根拠: テスト方針側の過不足として扱うべき差分は確認していない。

## 単体テスト

- [x] 公開関数、公開メソッド、コンポーネント、純粋ロジックの責務を単位としているか。
  - 検証結果: 指摘なし
  - 確認根拠: `CancelChatRunUseCase`、`RecoverUnfinishedRunsUseCase`、`RunEventBroker` の公開契約単位でテストされている。
- [x] 外部副作用を Fake、Stub、Mock、fixture で差し替えているか。
  - 検証結果: 指摘なし
  - 確認根拠: repository、dispatcher、publisher、cancellation、clock などはテスト用 fake/stub で差し替え、DB や外部 process へ直接依存していない。
- [x] 事前条件、事後条件、不変条件、正常系、異常系、境界値、分岐が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: accepted/running/validating/cancel_requested、terminal state、`already_exited` / `not_registered`、update conflict、unsubscribe identity の分岐が確認されている。
- [x] テストが内部実装詳細に密結合しすぎていないか。
  - 検証結果: 指摘なし
  - 確認根拠: テストは UseCase の result、repository fake の状態変化、broker の publish/subscribe/unsubscribe 結果を検証し、private 実装へ直接依存していない。
- [x] テストが単に例外が出ないことではなく、仕様上意味のある結果、状態変化、副作用、出力を検証しているか。
  - 検証結果: 指摘なし
  - 確認根拠: state 遷移、publisher event、dispatcher 登録、trace log、SSE payload、DB の run 状態を assert している。

## 結合テスト

- [x] 外部インターフェース、処理設計、永続化、ファイル、通信境界、表示または操作連携を設計単位で確認しているか。
  - 検証結果: 指摘なし
  - 確認根拠: SSE endpoint、cancel endpoint、起動時回復、trace log、DB 状態、認証境界を FastAPI/DB 経由で確認している。
- [x] 実外部サービス、実行環境、実行時間や環境差分が大きい対象を不必要に使っていないか。
  - 検証結果: 指摘なし
  - 確認根拠: Codex 実行本体は `NoopCodexRunCancellation` や test double で境界化し、F004 で不要な外部サービス実行はしていない。
- [x] Mock、Fake、Stub が過剰で、実際の契約違反や結合不備を隠していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合テストは実 DB repository と実 route を通し、runner 完了済み結果や broker live event のみ必要な境界で差し替えている。
- [x] トランザクション、状態更新、競合、ロールバック、エラー変換、ログ依頼が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: cancel の DB 状態、runner completed result の canceled 整合、起動時回復 summary log、取得失敗 log、未ログイン/他ユーザ/deleting/not found の HTTP 変換を確認している。
- [x] テストデータが独立し、実行順序に依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: support helper で account/chat/run を seed し、各テストが独自の user/chat/run を作成している。起動時回復テストは app 作成前後の seed 位置も意図的に分けている。
- [x] 時刻、乱数、実行順序、外部通信、ファイルシステム、DB 状態により不安定になるテストになっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 固定 ID/固定時刻の support helper と DB seed を使い、SSE live event は test broker で制御している。
- [x] テストデータが実装ロジックのコピーや都合のよい値だけになっておらず、境界値、異常値、業務上意味のある値を含んでいるか。
  - 検証結果: 指摘なし
  - 確認根拠: terminal/cancel/requested/running/validating、runner `already_exited`/`not_registered`、削除中チャット、他ユーザ、起動時 read failure など業務境界を含む。
