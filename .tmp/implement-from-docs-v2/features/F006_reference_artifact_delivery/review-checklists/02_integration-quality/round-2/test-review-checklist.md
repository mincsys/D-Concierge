# Test Review Checklist

## テスト方針との整合

- [x] テストコードとテスト仕様が、対応するテスト方針の範囲指定、テスト対象単位、除外範囲に一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストはUseCase/Port単位、結合テストはREST API/DB/ファイル境界単位であり、単体・結合テスト方針の範囲に一致している。
- [x] テスト関連成果物のディレクトリ構成が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストは `src/backend/tests/unit/application/...`、結合テストは `src/backend/tests/integration/` に配置されている。
- [x] テスト関連成果物のファイル構成、ファイル名、配置先が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `test_get_reference_data_use_case.py`、`test_get_artifact_use_case.py`、`test_file_delivery_port_contracts.py`、`test_reference_artifact_delivery_api.py` は対象UseCase/Port/APIに対応する命名である。
- [x] 単体、結合、総合テストの役割が混ざっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体はFake Store/Repository、結合はASGI/DB/一時ファイルを使い、総合テストやPlaywright確認は含めていない。
- [x] テスト方針で求める観点、カバレッジ、証跡、実行環境、完了条件が満たされているか。
  - 検証結果: 指摘なし
  - 確認根拠: evidenceは単体241 passed、branch coverage 95.99%、結合137 passed、branch coverage 81.62%、failures=0を記録している。テストコードはF006の正常系、所有者、削除中、欠損、traversal、許可外MIME、MIME/拡張子不一致、成果物欠損traceを扱う。
- [x] テスト方針が実装や設計の現状に追従していない場合は、テスト方針側を直す方針を書く。
  - 検証結果: 対象外
  - 確認根拠: テスト方針側の陳腐化や過剰指定は確認されなかった。
  - 理由: テスト方針修正対象ではないため。

## 単体テスト

- [x] 公開関数、公開メソッド、コンポーネント、純粋ロジックの責務を単位としているか。
  - 検証結果: 指摘なし
  - 確認根拠: GetReferenceDataUseCase、GetArtifactUseCase、filesystem Port/DTOの公開責務単位でテストされている。
- [x] 外部副作用を Fake、Stub、Mock、fixture で差し替えているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストはRepositoryとStoreをFake化し、DBや実HTTP配信を直接扱っていない。
- [x] 事前条件、事後条件、不変条件、正常系、異常系、境界値、分岐が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 正常、対象なし、別ユーザ、削除中、非PDF参照元、欠損、読込失敗、許可外MIME、MIME/拡張子不一致、HTML/CSV配信、Store未呼出が確認されている。
- [x] テストが内部実装詳細に密結合しすぎていないか。
  - 検証結果: 指摘なし
  - 確認根拠: テストは公開Command、戻り値、AppError分類、Port呼出有無を確認し、private関数の実装詳細を直接固定していない。
- [x] テストが単に例外が出ないことではなく、仕様上意味のある結果、状態変化、副作用、出力を検証しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 正常時のfile_path/mime_type、異常時のErrorType/trace、Store未呼出または対象Store呼出、HTTP本文非返却を確認している。

## 結合テスト

- [x] 外部インターフェース、処理設計、永続化、ファイル、通信境界、表示または操作連携を設計単位で確認しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `test_reference_artifact_delivery_api.py` はIF-SB-08/09のREST、認証Cookie、DB seed、共有データソース、保存済み成果物、一時ファイル、エラー変換、traceログを結合して確認している。
- [x] 実外部サービス、実行環境、実行時間や環境差分が大きい対象を不必要に使っていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合テストはASGITransport、テストDB、一時ファイルを使い、実Codexや実ブラウザを使っていない。
- [x] Mock、Fake、Stub が過剰で、実際の契約違反や結合不備を隠していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合テストは実FastAPIアプリ、認証Cookie、DB seed、保存済みファイルを通しており、主要なF006連携をFakeだけで済ませていない。
- [x] トランザクション、状態更新、競合、ロールバック、エラー変換、ログ依頼が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: F006は読取中心でDB更新ロールバックは対象外だが、削除中409、404/403/401のエラー変換、保存済み成果物欠損時のtraceログ出力が確認されている。
- [x] テストデータが独立し、実行順序に依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 各結合テストはDB初期化、ユーザseed、tmp_path上のファイル配置を個別に行っている。
- [x] 時刻、乱数、実行順序、外部通信、ファイルシステム、DB 状態により不安定になるテストになっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 固定UUID、固定datetime、一時ファイル、ASGITransport、テストDBを使っている。
- [x] テストデータが実装ロジックのコピーや都合のよい値だけになっておらず、境界値、異常値、業務上意味のある値を含んでいるか。
  - 検証結果: 指摘なし
  - 確認根拠: 他ユーザ、削除中、欠損、親ディレクトリ参照、許可外MIME、MIME/拡張子不一致、HTML/CSVなど業務上意味のある値が含まれている。
