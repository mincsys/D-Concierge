# Test Review Checklist

## テスト方針との整合

- [x] テストコードとテスト仕様が、対応するテスト方針の範囲指定、テスト対象単位、除外範囲に一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: F006の対象4テストは単体を `src/backend/tests/unit/`、結合を `src/backend/tests/integration/` に置き、UseCase、Port/DTO、REST API境界を対象にしている。前回指摘の `object` 戻り値注釈は `OpenedReferenceFile` / `OpenedArtifactFile` に修正され、単体テスト方針とコーディング規約の型ルールに沿っている。
- [x] テスト関連成果物のディレクトリ構成が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストは `src/backend/tests/unit/application/references/`、`src/backend/tests/unit/application/artifacts/`、`src/backend/tests/unit/application/ports/` に配置され、結合テストは `src/backend/tests/integration/` に配置されている。単体テスト方針と結合テスト方針が定義する backend テスト配置に一致している。
- [x] テスト関連成果物のファイル構成、ファイル名、配置先が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `test_get_reference_data_use_case.py` は `GetReferenceDataUseCase`、`test_get_artifact_use_case.py` は `GetArtifactUseCase`、`test_file_delivery_port_contracts.py` は filesystem Port/DTO、`test_reference_artifact_delivery_api.py` は IF-SB-08/09 のREST結合境界に対応しており、ファイル名と配置先が対象責務を表している。
- [x] 単体、結合、総合テストの役割が混ざっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストはRepository/StoreをFake化してUseCaseとPort契約を確認し、結合テストはASGI経由でREST API、認証、DB、テスト用ファイルを確認している。Playwrightや総合テスト仕様・証跡は追加されていない。
- [x] テスト方針で求める観点、カバレッジ、証跡、実行環境、完了条件が満たされているか。
  - 検証結果: 指摘なし
  - 確認根拠: 生成役報告で、Red確認、ruff pass、`git diff --check` pass が記録されている。coverageはRed先行フェーズのため未実行であり、Green化後の実行対象として扱われている。所有者/削除中チャット、配信元ファイル欠損・読込失敗、`text/html` / `text/csv` 配信、DTO戻り値注釈の観点は対象テストへ反映済みである。
- [x] テスト方針が実装や設計の現状に追従していない場合は、テスト方針側を直す方針を書く。
  - 検証結果: 対象外
  - 確認根拠: 確認した範囲では、単体テスト方針、結合テスト方針、処理設計、内部IF設計が古いまたは過剰であるとは判断していない。
  - 理由: テスト方針側の修正対象は確認されなかったため。

## 単体テスト

- [x] 公開関数、公開メソッド、コンポーネント、純粋ロジックの責務を単位としているか。
  - 検証結果: 指摘なし
  - 確認根拠: `test_get_reference_data_use_case.py` と `test_get_artifact_use_case.py` は各UseCaseの `execute`、`test_file_delivery_port_contracts.py` は filesystem Port/DTO の公開契約を対象にしており、公開責務単位で構成されている。
- [x] 外部副作用を Fake、Stub、Mock、fixture で差し替えているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストでは `FakeReferenceDeliveryRepository`、`FakeReferenceStore`、`FakeArtifactDeliveryRepository`、`FakeArtifactStore` を使い、DBや実ファイル読込を直接行わずに依頼内容と戻り値を確認している。
- [x] 事前条件、事後条件、不変条件、正常系、異常系、境界値、分岐が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 参照元UseCaseでは正常PDF、対象なし、別ユーザ、削除中、PDF以外、PDF実体欠損、PDF読込失敗が確認されている。成果物UseCaseでは正常SVG、対象なし、別ユーザ、削除中、許可外MIME、`text/html` / `text/csv`、保存済みファイル欠損、読込失敗が確認されている。
- [x] テストが内部実装詳細に密結合しすぎていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストはUseCaseの公開Command、Repository/Store Port相当のFake、戻り値または `AppError` を確認しており、内部privateメソッドや実装内の処理順序には依存していない。
- [x] テストが単に例外が出ないことではなく、仕様上意味のある結果、状態変化、副作用、出力を検証しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 正常系ではRepositoryへのuser_id/reference_idまたはartifact_idの依頼、Storeへの相対pathまたはArtifactDataの依頼、返却file_pathとmime_typeを確認している。異常系ではStore未呼出または対象Store呼出、`ErrorType`、trace有無を確認している。

## 結合テスト

- [x] 外部インターフェース、処理設計、永続化、ファイル、通信境界、表示または操作連携を設計単位で確認しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `test_reference_artifact_delivery_api.py` は IF-SB-08/09 の正常配信、未ログイン、パストラバーサル、他ユーザ所有、削除中チャット、実ファイル欠損、許可外MIME、`text/html` / `text/csv` の保存済みMIME配信をASGI/DB/ファイル境界で確認している。
- [x] 実外部サービス、実行環境、実行時間や環境差分が大きい対象を不必要に使っていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合テストは `ASGITransport`、テストDB、`tmp_path` 上のテスト用PDF/成果物を使い、実Codex、実ブラウザ、外部サービスに依存していない。
- [x] Mock、Fake、Stub が過剰で、実際の契約違反や結合不備を隠していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合テストは `create_app`、認証Cookie、DB seed、実ファイル配置を通してREST境界を呼び出しており、配信APIの主要層間連携をFakeだけで済ませていない。404/409系も共通エラーpayloadを検証しており、未登録ルートの404だけで偶然通る形ではない。
- [x] トランザクション、状態更新、競合、ロールバック、エラー変換、ログ依頼が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: F006は読取中心でDB更新やロールバックは対象外だが、削除中チャットのHTTP 409、対象なし/別ユーザ/実ファイル欠損のHTTP 404、パストラバーサル/許可外MIMEのHTTP 403、未ログインのHTTP 401 が共通エラー形式で確認されている。
- [x] テストデータが独立し、実行順序に依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 各結合テストは `prepare_foundation_database(database_url)`、`seed_chat_user(database_url)`、`tmp_path` を使ってDBとファイルを個別に用意しており、テスト間共有状態に依存する記述は確認されなかった。
- [x] 時刻、乱数、実行順序、外部通信、ファイルシステム、DB 状態により不安定になるテストになっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: IDは固定UUID、時刻は固定datetime、ファイルは `tmp_path` 配下、HTTPはASGITransport、DBはテスト用初期化を使っており、外部通信や実行順序に依存していない。
- [x] テストデータが実装ロジックのコピーや都合のよい値だけになっておらず、境界値、異常値、業務上意味のある値を含んでいるか。
  - 検証結果: 指摘なし
  - 確認根拠: `../secret.pdf`、`application/octet-stream`、Cookieなし、別ユーザ所有、`deleting` チャット、実ファイル欠損、非画像の許可MIMEという業務上重要な値が含まれている。
