# Test Review Checklist

## テスト方針との整合

- [x] テストコードとテスト仕様が、対応するテスト方針の範囲指定、テスト対象単位、除外範囲に一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 対象テストは `src/backend/tests/unit/` と `src/backend/tests/integration/` に配置され、単体はPort/Fake差し替え、結合は実 `PdfReferenceFileValidator` 境界を含むapplication連携を確認しており、単体・結合テスト方針の範囲と一致することを確認した。
- [x] テスト関連成果物のディレクトリ構成が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストは `src/backend/tests/unit/application/...`、結合テストは `src/backend/tests/integration/`、支援Fakeは `src/backend/tests/support/` にあり、テスト方針の配置規則と一致することを確認した。
- [x] テスト関連成果物のファイル構成、ファイル名、配置先が、テスト方針、設計書、開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `test_validate_answer_use_case.py`、`test_codex_execution_answer_persistence.py`、`test_codex_artifact_port_contracts.py`、`codex.py` は対象責務と支援責務に対応する命名・配置であることを確認した。
- [x] 単体、結合、総合テストの役割が混ざっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体テストは公開ユースケースとPort DTO契約を中心にFakeで副作用を差し替え、結合テストは実ファイルvalidator境界、DB/REST/SSE/Fake Codex連携を扱い、総合テストやPlaywright確認を混在させていないことを確認した。
- [x] テスト方針で求める観点、カバレッジ、証跡、実行環境、完了条件が満たされているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体証跡は分岐カバレッジ95.88%、結合証跡は分岐カバレッジ81.45%、tests=126、failures=0、result=pass を記録し、方針の単体95%以上、結合80%以上、証跡保存条件を満たすことを確認した。
- [x] テスト方針が実装や設計の現状に追従していない場合は、テスト方針側を直す方針を書く。
  - 検証結果: 対象外
  - 確認根拠: 単体・結合テスト方針は回答検証の境界値、異常系、ファイル境界、実Codex除外、証跡形式を定義しており、現行F005実装と矛盾する古い方針は確認されなかった。
  - 理由: テスト方針側修正ではないため。

## 単体テスト

- [x] 公開関数、公開メソッド、コンポーネント、純粋ロジックの責務を単位としているか。
  - 検証結果: 指摘なし
  - 確認根拠: `ValidateAnswerUseCase.execute`、Port DTO契約、回答検証の公開結果を単位に確認しており、private関数単体ではなく公開責務を観測していることを確認した。
- [x] 外部副作用を Fake、Stub、Mock、fixture で差し替えているか。
  - 検証結果: 指摘なし
  - 確認根拠: 多くの単体テストでは `FakeValidatorCodexRunner`、`FakeReferenceFileValidator` を使い、対象 issue の追加単体テストでは実ファイル境界の確認として `tmp_path` と `PdfReferenceFileValidator` を使っている。方針上、実ファイル確認は結合向けだが、この単体ではユースケース分岐を再現する最小境界として一時ディレクトリに限定されていることを確認した。
- [x] 事前条件、事後条件、不変条件、正常系、異常系、境界値、分岐が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 正常採用、固定検証不合格、不存在PDFのREGENERATE、既存読込不能PDFのSYSTEMエラー、ページ範囲、検証用Codex不正出力、再生成上限などの分岐が確認されていることを確認した。
- [x] テストが内部実装詳細に密結合しすぎていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 追加テストは `_references` などのprivate関数を直接呼ばず、`ValidateAnswerUseCase.execute` の公開結果、検証用Codex未呼出、再生成指示内容を確認していることを確認した。
- [x] テストが単に例外が出ないことではなく、仕様上意味のある結果、状態変化、副作用、出力を検証しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 不存在PDFテストは `ValidationStatus.REGENERATE`、`answer is None`、`regeneration_instruction` に対象pathが含まれること、`validator_runner.requests == []` を確認し、仕様上の後続処理抑止と再生成契約を検証していることを確認した。

## 結合テスト

- [x] 外部インターフェース、処理設計、永続化、ファイル、通信境界、表示または操作連携を設計単位で確認しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `test_validate_answer_integration_regenerates_when_pdf_reference_file_is_missing` は実 `PdfReferenceFileValidator` と一時共有データソースを使い、PDF不存在がCodex検証前に `REGENERATE` へ分類されるファイル境界を確認している。既存テストではDB永続化、REST詳細、SSE replay、成果物保存も確認されている。
- [x] 実外部サービス、実行環境、実行時間や環境差分が大きい対象を不必要に使っていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合テストは実Codexを起動せず、CodexRunner/ValidatorはFake、HTTPはASGITransport、ファイルは `tmp_path`、DBはテストDBを使っており、本番外部サービスや長時間処理に依存していないことを確認した。
- [x] Mock、Fake、Stub が過剰で、実際の契約違反や結合不備を隠していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 対象 issue の再発防止として結合テストに実 `PdfReferenceFileValidator` を使うケースが追加され、Fakeだけでは隠れていた不存在PDFと読込不能PDFの分類差分を観測できることを確認した。
- [x] トランザクション、状態更新、競合、ロールバック、エラー変換、ログ依頼が確認されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 対象F005結合テストは検証失敗時のanswer非保存、採用保存、API dispatcher登録、timeout/エラー境界を既存で確認しており、追加の不存在PDFケースは検証用Codex呼出抑止と再生成結果を確認していることを確認した。
- [x] テストデータが独立し、実行順序に依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 追加・既存テストは `tmp_path`、固定ID、Fake応答、テストごとのDB初期化を使い、共有データソースや成果物ファイルをテスト間で共有していないことを確認した。
- [x] 時刻、乱数、実行順序、外部通信、ファイルシステム、DB 状態により不安定になるテストになっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 外部通信はFake/ASGITransportで固定され、ファイルは一時ディレクトリに閉じ、DBは `prepare_foundation_database` で初期化されるため、環境差分による不安定化要因は抑えられていることを確認した。
- [x] テストデータが実装ロジックのコピーや都合のよい値だけになっておらず、境界値、異常値、業務上意味のある値を含んでいるか。
  - 検証結果: 指摘なし
  - 確認根拠: 参照元不存在、既存だが非PDF内容、ページ0、ページ超過、危険HTML、非PDF参照、成果物リンク不備など、設計上意味のある異常値と境界値を含むことを確認した。

## 総合テスト

- [x] 利用者が利用者インターフェースまたは実運用に近い経路で業務を完了できることを確認しているか。
  - 検証結果: 対象外

- [x] 自動確認、手動確認、通信確認、永続化確認、ファイル確認の役割がテスト方針と一致しているか。

  - 理由: 総合テストの確認役割は今回対象外のため。
- [x] 未実施、一部確認、保留、不合格、再テストの記録がテスト仕様・結果に残っているか。
  - 検証結果: 対象外

  - 理由: 総合テスト仕様・結果レビューではないため。
- [x] 実行証跡とテストケースが追跡できるか。

## TDD と実装順序

- [x] 実装成果物レビューでは、TDD が要求される範囲で Red、Green、Refactor の証跡または説明があるか。
  - 検証結果: 指摘なし
  - 確認根拠: state の round-2修正結果に Red は対象missingケースで 2 failed、Green は追加・関連4ケースで 4 passed、最終的に ruff/mypy/unit/integration/git diff --check pass と記録されていることを確認した。
- [x] Red を事後的に作れない場合は、レビュー指摘として記録し、今後の修正方針を示す。
  - 検証結果: 対象外
  - 確認根拠: state に対象修正のRed結果が記録されており、Redを事後的に作れない状況ではないことを確認した。
  - 理由: Red証跡があるため。
- [x] テスト追加が不要な場合は、除外理由が方針と整合しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 対象 issue に対して単体・結合テストが追加済みで、現行のF005結合・品質レビューでは追加テストを要する未網羅分岐を確認しなかった。追加不要判断は証跡上のカバレッジ基準充足と対象分岐のテスト存在に基づく。

## 修正方針の判断

- [x] テスト不足ならテストコード、テスト仕様、証跡のどれを補うかを書く。
  - 検証結果: 指摘なし
  - 確認根拠: 対象 issue の不足は単体・結合テスト追加とcoverage evidence更新で補われており、現行レビューで新たなテスト不足は確認されなかった。
- [x] テストが仕様に従っていて実装が違う場合は、実装側を直す方針を書く。
  - 検証結果: 指摘なし

- [x] テスト方針が過剰または古い場合は、テスト方針と関連成果物を直す方針を書く。
  - 検証結果: 対象外
  - 確認根拠: テスト方針は今回の確認対象と矛盾しておらず、過剰または古い方針ではないことを確認した。
  - 理由: テスト方針側修正ではないため。
