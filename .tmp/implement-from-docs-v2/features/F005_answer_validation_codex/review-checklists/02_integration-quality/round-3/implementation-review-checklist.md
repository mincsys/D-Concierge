# Implementation Review Checklist

## 仕様・設計との照合

- [x] 実装が要件、外部設計、内部設計、テスト方針、開発標準と整合しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `回答検証・再生成処理設計.md` と `ValidateAnswerUseCaseクラス設計.md` は参照元PDF不存在を再生成指示、存在するが読めないPDFをシステムエラーに分ける契約である。`ReferenceValidationResult.exists`、`PdfReferenceFileValidator`、`ValidateAnswerUseCase._references` を照合し、不存在は `REGENERATE` 診断へ、既存読込不能は `AppError(ErrorType.SYSTEM)` へ分岐していることを確認した。
- [x] ディレクトリ構成が設計書や開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 修正対象はPort DTO、application validation、infrastructure filesystem、unit/integration/support tests、内部IF設計、coverage evidence に限定され、既存の層別ディレクトリ構成に沿って配置されていることを確認した。
- [x] ファイル構成、ファイル名、配置先が設計書や開発標準と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `dto.py` はPort DTO、`validate_answer.py` は回答検証ユースケース、`reference_file_validator.py` は共有データソース上のPDF参照元検証、対象テストは unit/integration/support に置かれており、責務に対応するファイル構成であることを確認した。
- [x] 外部インターフェース、永続化、ファイル、設定、ログ、エラー、状態名などが設計された契約と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `Codex実行IF.md` の `ReferenceValidationResult` 説明がPDF存在有無、読込可否、ページ数を含む形へ更新され、実装でも存在しないPDFは固定検証不合格、読込不能PDFはSYSTEMエラーとして扱われることを確認した。
- [x] 設計書にない状態、入出力項目、設定値、永続化項目、操作導線、利用者向け文言が追加されていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 追加された `exists` は内部Port DTOの固定検証結果であり、DB永続化項目、REST/SSE外部応答、設定値、画面導線、利用者向け文言には新規追加がないことを確認した。
- [x] 実装が正しく、仕様書側が古い可能性がある場合は、仕様書側の修正方針を書く。
  - 検証結果: 対象外
  - 確認根拠: 今回の対象 issue は設計が正で、実装が設計に追従していなかった問題である。現行実装と `Codex実行IF.md` は設計契約へ整合しているため、仕様書側を古いものとして直す指摘は不要と判断した。
  - 理由: 仕様書側修正が主対象ではないため。
- [x] 対象成果物に存在しない技術要素や実行形態を前提にして指摘していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 確認対象は現存する Python backend、Port DTO、PDF参照元検証、pytest単体/結合テスト、coverage evidence、内部設計書に限定し、実Codex起動や総合テスト実施を前提にした指摘をしていない。

## 責務分割と依存方向

- [x] 層、機能、モジュール、コンポーネントの責務が設計と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 存在有無と読込可否の検出は `PdfReferenceFileValidator`、再生成またはSYSTEMエラーへの業務分類は `ValidateAnswerUseCase` に残っており、infrastructure層とapplication層の責務分割が保たれていることを確認した。
- [x] 副作用を持つ処理が定義済みの境界へ閉じているか。
  - 検証結果: 指摘なし
  - 確認根拠: ファイル存在確認とPDF読込は `PdfReferenceFileValidator` に閉じ、`ValidateAnswerUseCase` はPortを通じて結果を受ける構成で、ファイル副作用がapplication層へ直接漏れていないことを確認した。
- [x] 業務判断やドメイン判断が、表示層、入出力層、インフラ層、設定読込など不適切な場所へ漏れていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 不存在PDFを生成回答不備として再生成へ回す判断は application 層の `_references` にあり、infrastructure層は存在、読込可否、ページ数を返すだけであることを確認した。
- [x] 純粋ロジックが永続化、通信、ファイル、時刻、ID、外部プロセスなどの副作用へ直接依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 回答候補JSON検証、パス正規化、診断組み立ては `ValidateAnswerUseCase` 内でPort結果を使って行われ、ファイル実体への直接アクセスは `ReferenceFileValidatorPort` の外へ出ていないことを確認した。
- [x] 上位層が下位層の実装詳細、テスト用実体、内部データ形式へ直接依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: application層は `ReferenceFileValidatorPort` と `ReferenceValidationResult` に依存し、`PdfReferenceFileValidator` の `pypdf` 実装詳細やテストFakeへ依存していないことを確認した。
- [x] 依存方向、公開範囲、再利用単位が設計や開発標準と矛盾していないか。
  - 検証結果: 指摘なし
  - 確認根拠: `application/ports/codex` のDTO/Protocolを境界として、`infrastructure/filesystem` が具象実装を提供する依存方向であり、applicationからinfrastructureへの逆依存は確認されなかった。
- [x] 不要な依存関係、未使用コード、到達不能コード、暫定的な分岐、デバッグ用処理が残っていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 追加された `exists` 分岐は対象 issue の解消に直接対応しており、対象ファイル内にレビュー対応理由、デバッグ出力、未使用の暫定分岐は確認されなかった。

## 型と構造化データ

- [x] 使用言語、フレームワーク、開発標準に照らして、構造化データが意味のある型、データ構造、スキーマ、列挙値で表現されているか。
  - 検証結果: 指摘なし
  - 確認根拠: `ReferenceValidationResult` はdataclassで `path`、`page_start`、`page_end`、`exists`、`readable`、`page_count` を明示し、不存在、読込不能、ページ範囲外を判定できる構造になっていることを確認した。
- [x] 広すぎる型、未検証の動的データ、説明できない型変換、暗黙のデータ形状に依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 対象差分では `dict[str, object]`、広すぎる `Any`、`cast(...)` は使われず、テスト支援データもTypedDict、Protocol、dataclassで表現されていることを確認した。
- [x] 外部境界、永続化境界、設定境界、表示境界、処理内部のデータ構造が必要に応じて分かれているか。
  - 検証結果: 指摘なし
  - 確認根拠: Codex境界DTO、回答検証内部dataclass、DB保存DTO、テストFakeの記録型が分かれており、PDF参照元検証結果を外部応答やDB構造へ混在させていないことを確認した。
- [x] ID、状態、種別、payload、メタデータが文字列や汎用コンテナだけに埋もれず、意味のある表現になっているか。
  - 検証結果: 指摘なし
  - 確認根拠: 検証結果は `ValidationStatus`、エラー分類は `ErrorType`、PDF参照元検証は `ReferenceValidationResult` で扱われ、文字列だけの暗黙分類に戻っていないことを確認した。
- [x] 入力値の検証、正規化、変換が境界で行われ、内部処理が未検証データを前提にしていないか。
  - 検証結果: 指摘なし
  - 確認根拠: `data_source/` 接頭辞、PDF拡張子、ページ範囲、ファイル存在、PDF読込可否、ページ数超過の確認がCodex検証前に行われ、未検証の参照元を検証用Codexへ渡さない構成であることを確認した。

## エラー・ログ・セキュリティ

- [x] 利用者向けメッセージと内部調査用情報が分離されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 不存在PDFは再生成指示の診断に含め、読込不能PDFだけ `diagnostic_message="PDF読み取り失敗: ..."` のSYSTEMエラーにしており、利用者向け状態と内部調査用診断の境界が維持されていることを確認した。
- [x] 内部パス、秘密情報、スタックトレース、認証情報、個人情報などが利用者向け出力、外部応答、ログ、証跡へ露出していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 再生成指示に含めるパスは `data_source/{normalized_path}` の形式で、絶対パスや `tmp_path`、スタックトレース、認証情報は証跡やテスト期待値に含まれていないことを確認した。
- [x] URL、ポート、ファイルパス、認証情報、環境名、外部サービス名などが不適切にハードコードされていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 実装は `data_source_dir` と候補相対pathを使い、追加修正でポート、URL、認証情報、環境名を新たにハードコードしていないことを確認した。
- [x] エラー分類、追跡情報、状態更新、後続処理の継続または抑止が設計と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: `not validation.exists` は診断に積んで `continue` し、固定検証失敗として `ValidationStatus.REGENERATE` を返す。`exists=True/readable=False` は `AppError(ErrorType.SYSTEM, trace=True)` で後続の検証用Codex呼び出しを抑止することを確認した。
- [x] ファイルパス、URL、コンテンツ種別、マークアップ、コマンド、外部入力などの検証が境界で行われているか。
  - 検証結果: 指摘なし
  - 確認根拠: `PdfReferenceFileValidator` は `resolve()` と `is_relative_to()` で共有データソース外を除外し、存在しないファイルを `exists=False`、読み取れない既存ファイルを `exists=True/readable=False` として区別することを確認した。
- [x] 認証、認可、入力制限、出力制御、監査記録が対象システムの設計と整合しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 今回の修正は回答検証内部境界に限定され、既存のチャット所有者確認、REST/SSE出力制御、成果物配信認可を変更していないことを確認した。

## 修正方針の判断

- [x] 実装が設計から外れている場合は、実装と関連テストを直す方針を書く。
  - 検証結果: 指摘なし
  - 確認根拠: 対象 issue の修正方針どおり、結果型、実ファイルvalidator、ユースケース分岐、単体・結合テスト、内部IF設計が更新済みであり、追加の実装修正方針を要する乖離は確認されなかった。
- [x] 設計が実装済みの正しい振る舞いを表せていない場合は、設計とテスト仕様を直す方針を書く。
  - 検証結果: 対象外
  - 確認根拠: 現行設計は「不存在PDFは再生成、既存読込不能PDFはSYSTEMエラー」として実装済みの正しい振る舞いを表せているため、設計やテスト仕様を直す追加方針は不要と判断した。
  - 理由: 設計側の不足ではないため。
- [x] テストや証跡だけが古い場合は、実装を直す前提にせず、テスト仕様、テストコード、証跡の修正方針を書く。
  - 検証結果: 対象外
  - 確認根拠: テストと証跡は更新済みで、テストや証跡だけが古い状態ではないことを確認した。
  - 理由: 古いテストまたは証跡のみを直す状況ではないため。
- [x] 判断にユーザ合意が必要な場合は、どの成果物を確定根拠にするかを修正方針に書く。

  - 確認根拠: 設計書、対象issue、生成役修正報告、実装、テスト、証跡から解消判断が可能であり、ユーザ合意待ちの仕様判断は確認されなかった。

- [x] 特定のアプリ、言語、通信方式、画面構成、保存方式に依存した判断ではなく、レビュー対象の仕様と設計に基づいて指摘する。
  - 検証結果: 指摘なし
  - 確認根拠: 判定は F005 の回答検証・再生成設計、Codex実行IF、実装、単体/結合テスト、coverage evidence の照合に限定し、設計外の一般論や別機能の前提を根拠にしていないことを確認した。
