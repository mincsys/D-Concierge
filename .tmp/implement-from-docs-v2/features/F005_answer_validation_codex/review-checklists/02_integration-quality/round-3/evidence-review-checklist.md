# Evidence Review Checklist

## 証跡と方針の整合

- [x] 証跡の保存場所、ファイル名、記録項目が対応するテスト方針と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体証跡は `docs/04_テスト/02_単体テスト/evidence/backend-unit-coverage.txt`、結合証跡は `docs/04_テスト/03_結合テスト/evidence/backend-integration-evidence.txt` に保存され、各方針が定めるkey=value形式の項目で記録されていることを確認した。
- [x] 証跡がどのテストケース、コマンド、範囲指定、実施日時に対応するか追跡できるか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体証跡は `src/backend/tests/unit`、結合証跡は `src/backend/tests/integration` のcoverage付き実行コマンド、実施日時、対象、結果を記録しており、今回の生成役報告と対応付けられることを確認した。
- [x] 証跡に対象コミット、対象差分、実行環境、主要バージョン、ブラウザ、画面サイズなど、再現に必要な前提が残っているか。
  - 検証結果: 対象外
  - 確認根拠: 単体・結合テスト方針の証跡項目は target、executed_at、command、coverage値、件数、result 等に限定され、コミット、ブラウザ、画面サイズを必須項目としていないことを確認した。
  - 理由: 今回の証跡形式では方針上必須ではないため。
- [x] 再テスト時に古い証跡と新しい証跡の扱いが方針と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 方針は同一ファイル名で最新結果へ上書きすると定義している。対象2ファイルはいずれも 2026-06-22T22:03:48+09:00 の修正後結果を記録し、round-2以前の古いcoverage値ではないことを確認した。
- [x] 証跡に秘密情報、個人情報、絶対パス、不要な詳細ログが含まれていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡には target、executed_at、command、coverage値、設計網羅説明、tests/failures、result だけがあり、DB URL、絶対パス、個人情報、スタックトレース、詳細ログは確認されなかった。

## カバレッジ証跡

- [x] 方針で指定された指標だけが記録されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 対象証跡は分岐カバレッジ、covered_branches、total_branchesを記録し、行カバレッジ、ステートメントカバレッジ、関数カバレッジを記録していないことを確認した。
- [x] 対象、実行日時、実行コマンド、カバレッジ値、総数、通過数、判定結果がそろっているか。
  - 検証結果: 指摘なし
  - 確認根拠: `backend-unit-coverage.txt` は target、executed_at、command、branch_coverage、covered_branches、total_branches、result を備える。`backend-integration-evidence.txt` はさらに design_coverage、tests、failures を備えることを確認した。
- [x] 目標未達を合格扱いしていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体は95%基準に対して95.88%、結合は80%基準に対して81.45%で、どちらも `result=pass` と整合することを確認した。
- [x] 測定から外した範囲がテスト方針と矛盾していないか。
  - 検証結果: 指摘なし
  - 確認根拠: `pyproject.toml` の coverage 除外は `src/backend/tests/*`、薄いentrypointの `src/backend/main.py`、database migrationsであり、方針のテスト支援コード、薄いエントリポイント、管理対象外を分岐カバレッジ対象外にする扱いと矛盾しないことを確認した。

## 総合テスト証跡

- [x] スクリーンショットやテキスト証跡がテストケースIDと対応しているか。
  - 検証結果: 対象外
  - 確認根拠: 現在フェーズは結合・品質レビューであり、今回の対象証跡は単体/結合coverage evidenceである。state上も機能別総合テストは未実施であることを確認した。
  - 理由: 総合テスト証跡レビューではないため。
- [x] スクリーンショットやログが、ロード中、エラー画面、古い状態ではなく、判定対象の状態を示しているか。
  - 検証結果: 対象外
  - 確認根拠: 今回のレビュー対象に画面スクリーンショットや画面ログ証跡は含まれず、対象は単体/結合のテキスト証跡であることを確認した。
  - 理由: 画面証跡が対象外のため。
- [x] 必要最小限の証跡になっているか。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡は方針で指定されたkey=valueの結果要約に限定され、詳細pytestログ、JSON coverage、JUnit XML、行カバレッジレポートを含まないことを確認した。
- [x] 画面確認、HTTP確認、DB確認、ファイル確認の記録が、ケースの判定に必要な粒度で残っているか。
  - 検証結果: 対象外
  - 確認根拠: 結合証跡は設計単位の網羅結果を記録する形式であり、個別画面確認や総合テストケース別のHTTP/DB/ファイル確認記録は今回フェーズの成果物ではないことを確認した。
  - 理由: 総合テスト証跡の粒度確認は対象外のため。
- [x] 未実施、一部確認、保留、不合格、再テストが曖昧に合格扱いされていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体証跡は `result=pass`、結合証跡は `tests=126`、`failures=0`、`result=pass` を明記しており、保留や未実施を合格扱いする記述は確認されなかった。
- [x] 証跡だけを見て、どのケースが合格、不合格、部分確認、未実施、保留なのか誤読なく分かるか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体証跡は対象範囲とpass判定、結合証跡は設計網羅説明、tests、failures、resultを明記しており、今回対象範囲が合格であることを誤読なく確認できる。

## 実行結果の信頼性

- [x] 証跡の実行日時が対象成果物の変更後になっているか。
  - 検証結果: 指摘なし
  - 確認根拠: state の round-2修正結果と前回修正報告は単体222 passed、結合126 passed、分岐95.88%/81.45%を報告しており、対象証跡の 2026-06-22T22:03:48+09:00 の値と一致することを確認した。
- [x] 実行コマンドが方針書、設定、package manager、依存管理の現行定義と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡のコマンドは `env UV_CACHE_DIR=/tmp/uv-cache uv run pytest ...` であり、AGENTS指示および単体・結合テスト方針の `uv` 実行、キャッシュ指定、pytest-covによる分岐coverage取得に沿うことを確認した。
- [x] 証跡と実際のテストコード、テスト仕様、設定ファイルに矛盾がないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合証跡の `design_coverage` には `real PDF reference missing-file regeneration boundary` が含まれ、実際に `test_validate_answer_integration_regenerates_when_pdf_reference_file_is_missing` が存在することを確認した。単体証跡の対象も `src/backend/tests/unit` と一致する。
- [x] 同じ対象に対する古い証跡が残り、最新結果と誤認される状態になっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 対象 evidence ファイルは各1ファイルで最新結果へ上書きされており、同一ディレクトリ内の今回対象バックエンド証跡として別名の古い結果は確認されなかった。

## 修正方針の判断

- [x] 証跡が不足している場合は、証跡の追加、再実行、テスト仕様更新のどれが必要かを書く。
  - 検証結果: 指摘なし
  - 確認根拠: 対象証跡は方針の必須項目と生成役報告の結果を満たしており、追加証跡、再実行、テスト仕様更新を要する不足は確認されなかった。
- [x] 証跡が方針と矛盾する場合は、証跡を直すべきか、方針を直すべきかを根拠から判断する。
  - 検証結果: 指摘なし
  - 確認根拠: 保存先、key=value形式、分岐カバレッジ、tests/failures、result、最新上書きの扱いは方針と矛盾しないため、証跡または方針を直すべき問題は確認されなかった。
- [x] 証跡だけでは判定不能な場合は、必要な再実行条件と対象成果物を具体的に書く。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡、テスト方針、`pyproject.toml`、対象テストコードを照合し、現時点の証跡だけで合否、coverage基準、対象 issue の再発防止テスト存在を判断できることを確認した。
