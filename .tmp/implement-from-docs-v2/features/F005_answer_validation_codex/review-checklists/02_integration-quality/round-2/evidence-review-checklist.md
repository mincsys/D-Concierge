# Evidence Review Checklist

## 証跡と方針の整合

- [x] 証跡の保存場所、ファイル名、記録項目が対応するテスト方針と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体証跡は `docs/04_テスト/02_単体テスト/evidence/backend-unit-coverage.txt`、結合証跡は `docs/04_テスト/03_結合テスト/evidence/backend-integration-evidence.txt` にあり、各方針の保存先とkey=value項目に一致することを確認した。
- [x] 証跡がどのテストケース、コマンド、範囲指定、実施日時に対応するか追跡できるか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体証跡は `src/backend/tests/unit` のcoverage付きコマンドと実施日時、結合証跡は `src/backend/tests/integration` のcoverage付きコマンド、設計単位網羅、tests/failuresを記録していることを確認した。
- [x] 証跡に対象コミット、対象差分、実行環境、主要バージョン、ブラウザ、画面サイズなど、再現に必要な前提が残っているか。
  - 検証結果: 対象外
  - 確認根拠: 単体・結合テスト方針の証跡項目は target、executed_at、command、coverage値、件数、result 等に限定され、コミット、ブラウザ、画面サイズを必須項目としていないことを確認した。
  - 理由: 今回の証跡形式では方針上必須ではないため。
- [x] 再テスト時に古い証跡と新しい証跡の扱いが方針と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 方針は同一ファイル名で最新結果へ上書きすると定義しており、対象2ファイルはいずれも 2026-06-22 19時台の最新round修正後結果のみを記録していることを確認した。
- [x] 証跡に秘密情報、個人情報、絶対パス、不要な詳細ログが含まれていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡には target、executed_at、command、coverage値、設計網羅説明、tests/failures、result のみがあり、DB URL、絶対パス、個人情報、詳細ログは確認されなかった。

## カバレッジ証跡

- [x] 方針で指定された指標だけが記録されているか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体・結合証跡はいずれも分岐カバレッジ、covered_branches、total_branchesを記録し、行・ステートメント・関数カバレッジを出力していないことを確認した。
- [x] 対象、実行日時、実行コマンド、カバレッジ値、総数、通過数、判定結果がそろっているか。
  - 検証結果: 指摘なし
  - 確認根拠: `backend-unit-coverage.txt` は target、executed_at、command、branch_coverage、covered_branches、total_branches、result を備え、`backend-integration-evidence.txt` は加えて design_coverage、tests、failures を備えていることを確認した。
- [x] 目標未達を合格扱いしていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体は95%基準に対して95.83%、結合は80%基準に対して81.20%で、どちらも `result=pass` と整合することを確認した。
- [x] 測定から外した範囲がテスト方針と矛盾していないか。
  - 検証結果: 指摘なし
  - 確認根拠: `pyproject.toml` の coverage 除外は `src/backend/tests/*`、薄い entrypoint の `src/backend/main.py`、migration 配下であり、テスト方針のテスト支援コード・薄いエントリポイント・本体コード外の管理対象を分岐カバレッジ対象から外す扱いと矛盾しないことを確認した。

## 総合テスト証跡

- [x] スクリーンショットやテキスト証跡がテストケースIDと対応しているか。
  - 検証結果: 対象外
  - 確認根拠: 現在フェーズは結合・品質レビューであり、機能別総合テスト証跡は state 上も未実施で、今回の対象証跡は単体/結合 coverage evidence であることを確認した。
  - 理由: 総合テスト証跡レビューではないため。
- [x] スクリーンショットやログが、ロード中、エラー画面、古い状態ではなく、判定対象の状態を示しているか。
  - 検証結果: 対象外
  - 確認根拠: 今回のレビュー対象にスクリーンショット証跡は含まれず、対象は単体/結合のテキスト証跡であることを確認した。
  - 理由: 画面証跡が対象外のため。
- [x] 必要最小限の証跡になっているか。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡は方針で指定されたkey=valueの結果要約に限定され、詳細ログやJUnit XML、行カバレッジレポートを含んでいないことを確認した。
- [x] 画面確認、HTTP確認、DB確認、ファイル確認の記録が、ケースの判定に必要な粒度で残っているか。
  - 検証結果: 対象外
  - 確認根拠: 結合証跡は設計単位の網羅説明を記録しているが、個別画面確認や総合テストケース別証跡は今回フェーズの成果物ではないことを確認した。
  - 理由: 総合テスト証跡の粒度確認は対象外のため。
- [x] 未実施、一部確認、保留、不合格、再テストが曖昧に合格扱いされていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 単体証跡は `result=pass`、結合証跡は `tests=125`、`failures=0`、`result=pass` を明記しており、保留や未実施を合格扱いする記述は確認されなかった。
- [x] 証跡だけを見て、どのケースが合格、不合格、部分確認、未実施、保留なのか誤読なく分かるか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合証跡は個別ケース一覧ではないが、方針上の証跡項目である設計網羅説明、tests、failures、result を備え、対象範囲の合否は誤読なく確認できることを確認した。

## 実行結果の信頼性

- [x] 証跡の実行日時が対象成果物の変更後になっているか。
  - 検証結果: 指摘なし
  - 確認根拠: state の round-1修正結果と証跡はいずれも 2026-06-22 19時台の単体/結合 pass として整合し、前回round-1の 95.74%/80.34% ではなく修正後の 95.83%/81.20% に更新されていることを確認した。
- [x] 実行コマンドが方針書、設定、package manager、依存管理の現行定義と一致しているか。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡のコマンドは `UV_CACHE_DIR=/tmp/uv-cache uv run pytest ...` を使用し、AGENTS指示とテスト方針の `uv` 実行・キャッシュ指定に沿うことを確認した。
- [x] 証跡と実際のテストコード、テスト仕様、設定ファイルに矛盾がないか。
  - 検証結果: 指摘なし
  - 確認根拠: 結合証跡の `design_coverage` にはF005 fake Codex、固定回答検証、成果物保存、REST/SSE replay、API dispatcher登録、CodexRunner JSONL契約が含まれ、実際の対象テストファイルと対応することを確認した。
- [x] 同じ対象に対する古い証跡が残り、最新結果と誤認される状態になっていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 対象 evidence ファイルは各1ファイルで最新結果へ上書きされており、同一ディレクトリ内に今回対象の別名古い証跡は確認していない。

## 修正方針の判断

- [x] 証跡が不足している場合は、証跡の追加、再実行、テスト仕様更新のどれが必要かを書く。
  - 検証結果: 指摘なし
  - 確認根拠: 既存証跡は方針の必須項目を満たしているため証跡単体の不足はない。新規 issue 修正後は生成役が同じ単体/結合coverage条件で再実行し、対象 evidence を最新結果へ上書きする必要があることを確認した。
- [x] 証跡が方針と矛盾する場合は、証跡を直すべきか、方針を直すべきかを根拠から判断する。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡ファイルの保存先、key=value形式、分岐カバレッジ、tests/failures、result は方針と矛盾していないため、証跡または方針を直すべき矛盾は確認されなかった。
- [x] 証跡だけでは判定不能な場合は、必要な再実行条件と対象成果物を具体的に書く。
  - 検証結果: 指摘なし
  - 確認根拠: 証跡、テスト方針、`pyproject.toml` の coverage 設定を照合し、現時点の証跡だけで合否と基準充足を判断できることを確認した。新規 issue 修正後は同じ coverage コマンドで再実行し、2つの evidence を更新する必要がある。
