## 責務分割と依存方向

- [x] 層、機能、モジュール、コンポーネントの責務が設計と一致しているか。
  - 検証結果: 指摘あり
  - 確認根拠: `PdfReferenceFileValidator` が存在有無と読込可否の区別を返せず、`ValidateAnswerUseCase` が設計上の再生成可能不備をシステムエラーに分類しているため、参照元検証境界の責務契約が不足していることを確認した。
  - 指摘: `.issue/implement-from-docs/2026-06-22_21-49-52_F005参照元PDF不存在が再生成ではなくシステムエラーになる.md`
- [x] 副作用を持つ処理が定義済みの境界へ閉じているか。
  - 検証結果: 指摘なし
  - 確認根拠: DB更新はRepository、Codex実行はCodexRunner、成果物保存はArtifactStore、PDF読込は `PdfReferenceFileValidator`、SSEはRunEventBroker publisherへ閉じていることを確認した。
- [x] 業務判断やドメイン判断が、表示層、入出力層、インフラ層、設定読込など不適切な場所へ漏れていないか。
  - 検証結果: 指摘なし
  - 確認根拠: REST層は認証、入力、ユースケース呼び出し、HTTP/SSE payload変換に留まり、回答採用、再生成、状態遷移判断は application 層で扱っていることを確認した。
- [x] 純粋ロジックが永続化、通信、ファイル、時刻、ID、外部プロセスなどの副作用へ直接依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: 検証・実行ユースケースはProtocol/Port経由でRepository、CodexRunner、ArtifactStore、Clock、TraceLoggerを利用しており、副作用実体への直接依存は確認されなかった。
- [x] 上位層が下位層の実装詳細、テスト用実体、内部データ形式へ直接依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: `presentation/rest/chat.py` は `RunExecutionDispatcherLike` を通じてdispatcherを参照し、application層はPort DTO/Protocolを通じて下位実装を扱っていることを確認した。
- [x] 依存方向、公開範囲、再利用単位が設計や開発標準と矛盾していないか。
  - 検証結果: 指摘なし
  - 確認根拠: `application` から `infrastructure` への直接importはなく、`factory.py` と `run_execution_dispatcher.py` が具象実装の組み立てを担う構成で、開発標準の依存方向に沿っていることを確認した。
- [x] 不要な依存関係、未使用コード、到達不能コード、暫定的な分岐、デバッグ用処理が残っていないか。
  - 検証結果: 指摘なし
  - 確認根拠: 対象ファイルの追加処理はdispatcher配線、回答検証、成果物リンク、PDF参照元検証、テスト追加に対応しており、明示的なデバッグ出力や暫定分岐は確認されなかった。

## 型と構造化データ

- [x] 使用言語、フレームワーク、開発標準に照らして、構造化データが意味のある型、データ構造、スキーマ、列挙値で表現されているか。
  - 検証結果: 指摘あり
  - 確認根拠: 多くの要求・結果はdataclass、Protocol、StrEnumで表現されているが、`ReferenceValidationResult` は参照元不存在とPDF読込不能を区別する意味のある結果種別を持たず、設計の分類を表現できないことを確認した。
  - 指摘: `.issue/implement-from-docs/2026-06-22_21-49-52_F005参照元PDF不存在が再生成ではなくシステムエラーになる.md`
- [x] 広すぎる型、未検証の動的データ、説明できない型変換、暗黙のデータ形状に依存していないか。
  - 検証結果: 指摘なし
  - 確認根拠: Codex候補JSONは `JsonValue` と内部dataclassへ変換され、テスト支援もTypedDictやProtocolを使っているため、広すぎる `Any` や安易な `cast` への依存は確認されなかった。
- [x] 外部境界、永続化境界、設定境界、表示境界、処理内部のデータ構造が必要に応じて分かれているか。
  - 検証結果: 指摘なし
  - 確認根拠: Codex DTO、DB DTO、REST schema、SSE payload、検証内部dataclassが分かれており、境界ごとの構造分離は確認できた。
- [x] ID、状態、種別、payload、メタデータが文字列や汎用コンテナだけに埋もれず、意味のある表現になっているか。
  - 検証結果: 指摘なし

- [x] 入力値の検証、正規化、変換が境界で行われ、内部処理が未検証データを前提にしていないか。
  - 検証結果: 指摘なし

  - 指摘: `.issue/implement-from-docs/2026-06-22_21-49-52_F005参照元PDF不存在が再生成ではなくシステムエラーになる.md`
